# Audit Logs

**Audit logs** are chronological records that document events, actions, and changes within a system, application, or network, providing an immutable trail of **who did what, when, and from where**. They are a foundational component of [[security monitoring]], [[compliance frameworks]], and [[digital forensics]], enabling organizations to detect anomalies, investigate incidents, and demonstrate regulatory adherence. Audit logs are distinct from general system logs in that they are specifically designed to capture security-relevant events for accountability and non-repudiation purposes.

---

## Overview

Audit logs exist because modern computing environments involve thousands of interactions per second across users, services, and systems — and organizations need a reliable mechanism to reconstruct past events. The concept originates from traditional financial auditing, where every transaction is recorded for accountability. In IT security, this translates to recording authentication attempts, privilege escalations, file access, configuration changes, and administrative actions. Without audit logs, a breach investigation is essentially guesswork; with them, analysts can reconstruct an entire attack chain from initial access to exfiltration.

Regulatory frameworks have made audit logging a legal and contractual obligation across many industries. [[PCI DSS]] requires organizations handling cardholder data to maintain audit trails for at least one year, with three months immediately available. [[HIPAA]] mandates activity logs for all access to electronic protected health information (ePHI). [[SOX]] requires audit trails for financial system integrity. [[NIST SP 800-92]] provides the foundational guidance for log management in federal environments. Failure to maintain adequate audit logs is itself a compliance violation that can result in fines independent of any breach.

From an operational standpoint, audit logs serve several distinct functions simultaneously. As a **detective control**, they reveal anomalies — a service account logging in at 3 AM, a user accessing 10,000 records in a single session, or a configuration change rolled back five minutes after being applied. As a **deterrent control**, users who know their actions are logged tend to behave more carefully. As a **corrective control**, audit logs provide the information needed to reverse unauthorized changes and restore systems to their known-good state. This multi-role utility makes them one of the highest-return security investments an organization can make.

The challenge with audit logs is not generating them — virtually every modern OS, application, and network device produces logs by default — but managing them effectively. Log volume at enterprise scale is staggering; a single Active Directory domain controller in a medium-sized organization can generate millions of events per day. This has given rise to [[SIEM]] (Security Information and Event Management) platforms, log aggregation infrastructure, and the discipline of log management, which involves collection, normalization, storage, analysis, and ultimately deletion or archival of log data in accordance with retention policies.

Audit logs must themselves be protected from tampering. An attacker who can delete or modify logs can erase evidence of their activity, a technique documented extensively in post-breach investigations. High-value audit logs should be written to append-only, remotely stored locations immediately upon generation, so that even if a system is compromised, the historical record remains intact. This is why forwarding logs to a centralized, hardened [[SIEM]] or dedicated log management platform is considered a security best practice rather than a convenience.

## How It Works

### Event Generation

Every security-relevant event generates a log entry at the source system. On Windows, the **Security Event Log** captures authentication, privilege use, object access, and policy changes via the Windows Event Log service (`eventlog`). On Linux, audit events are generated by the **Linux Audit Framework** (`auditd`), which intercepts system calls at the kernel level.

**Windows Security Event Log — Key Event IDs:**
```
4624  - Successful logon
4625  - Failed logon attempt
4648  - Logon using explicit credentials (pass-the-hash indicator)
4656  - Object (file/registry) access attempt
4672  - Special privileges assigned to new logon (admin)
4698  - Scheduled task created
4720  - User account created
4728  - Member added to security-enabled global group
4768  - Kerberos TAS ticket requested (AS-REQ)
4771  - Kerberos pre-authentication failed
4776  - NTLM authentication attempt
5140  - Network share accessed
7045  - New service installed
```

**Querying Windows Security Logs with PowerShell:**
```powershell
# Retrieve all failed logon events in the last 24 hours
Get-WinEvent -FilterHashtable @{
    LogName   = 'Security'
    Id        = 4625
    StartTime = (Get-Date).AddHours(-24)
} | Select-Object TimeCreated, Message | Format-List

# Find account creation events
Get-WinEvent -FilterHashtable @{LogName='Security'; Id=4720} |
    ForEach-Object {
        $xml = [xml]$_.ToXml()
        [PSCustomObject]@{
            Time        = $_.TimeCreated
            NewAccount  = ($xml.Event.EventData.Data | Where-Object {$_.Name -eq 'TargetUserName'}).'#text'
            CreatedBy   = ($xml.Event.EventData.Data | Where-Object {$_.Name -eq 'SubjectUserName'}).'#text'
        }
    }
```

### Linux Audit Framework (auditd)

The `auditd` daemon runs as a privileged process and intercepts system calls before they are executed. Rules are loaded into the kernel via `auditctl` and define which events to record.

**Installing and configuring auditd (Debian/Ubuntu):**
```bash
sudo apt install auditd audispd-plugins -y
sudo systemctl enable --now auditd
```

**Defining audit rules in `/etc/audit/rules.d/audit.rules`:**
```bash
# Monitor all authentication events
-w /etc/passwd -p wa -k identity_changes
-w /etc/shadow -p wa -k identity_changes
-w /etc/sudoers -p wa -k sudoers_changes

# Monitor privileged command execution
-a always,exit -F arch=b64 -S execve -F euid=0 -k root_commands

# Monitor SSH login attempts
-w /var/log/auth.log -p rwa -k ssh_access

# Monitor file deletions
-a always,exit -F arch=b64 -S unlink -S unlinkat -S rename -S renameat -k file_deletion

# Lock audit configuration (must be last)
-e 2
```

**Searching audit logs with `ausearch`:**
```bash
# Find all sudo usage in the last hour
sudo ausearch -k root_commands --start recent

# Find failed login attempts
sudo ausearch -m USER_AUTH --success no

# Find events by user
sudo ausearch -ua 1001 -i

# Generate a human-readable report
sudo aureport --summary
sudo aureport --auth --failed --summary
```

### Log Forwarding and Centralization

Logs should be forwarded in near-real-time to a centralized collector. The standard protocol for network log forwarding is **Syslog (UDP/514 or TCP/514)** with the more secure **Syslog over TLS (TCP/6514)** preferred for sensitive environments.

**Configuring rsyslog to forward logs to a SIEM (Linux):**
```bash
# /etc/rsyslog.conf — forward all auth logs over TLS
$DefaultNetstreamDriver gtls
$ActionSendStreamDriverMode 1
$ActionSendStreamDriverAuthMode x509/name

auth,authpriv.* @@siem.cocytus.internal:6514
*.* @@siem.cocytus.internal:6514
```

**Windows Event Forwarding (WEF) — Collector-initiated subscription:**
```powershell
# On the collector, configure WinRM and subscription
winrm quickconfig
wecutil cs subscription.xml

# Subscription XML defines source computers and event filter
# Events are forwarded to the ForwardedEvents log
Get-WinEvent -LogName "ForwardedEvents" -MaxEvents 50
```

### Log Structure

A well-formed audit log entry contains these fields (often mapped to [[Common Event Format]] or JSON for SIEM ingestion):

| Field | Description | Example |
|---|---|---|
| Timestamp | UTC time of event | `2024-11-14T03:22:41Z` |
| Event ID | Numeric identifier | `4625` |
| Source | Originating host | `WS-ACCOUNTING-04` |
| Subject | Who performed the action | `DOMAIN\jsmith` |
| Object | What was acted upon | `\\SERVER\Finance\Q4.xlsx` |
| Action | What happened | `ReadData` |
| Outcome | Success or failure | `Failure` |
| Source IP | Network origin | `192.168.10.44` |

## Key Concepts

- **Non-repudiation**: The property that ensures a subject cannot deny having performed an action. Audit logs, especially when cryptographically signed or stored on tamper-evident systems, provide non-repudiation because the log entry serves as evidence that cannot be unilaterally deleted by the actor.

- **Log Integrity**: The guarantee that a log entry has not been altered after creation. Techniques for ensuring log integrity include write-once storage (WORM), cryptographic hash chaining (each entry includes a hash of the previous entry), digital signatures, and forwarding to an independent system the original host cannot access after transmission.

- **Retention Policy**: A documented rule specifying how long logs must be kept before deletion or archival. Retention requirements vary by regulation: PCI DSS requires 12 months, HIPAA requires 6 years, and some state privacy laws specify their own periods. Retention must be long enough to cover the average breach detection time (IBM's 2023 report placed this at 204 days).

- **Log Aggregation**: The process of collecting logs from multiple disparate sources — endpoints, servers, network devices, cloud services — into a centralized repository for unified analysis. Aggregation enables correlation: connecting a failed VPN login from Event ID 4625 on a domain controller to a subsequent successful login from a foreign IP 30 minutes later.

- **Audit Trail**: A complete sequence of audit log entries that document a specific transaction or event from beginning to end. In an investigation, reconstructing the audit trail means assembling all log entries related to an incident into a coherent timeline. A complete audit trail answers the six investigative questions: who, what, when, where, how, and why.

- **Syslog Severity Levels**: The RFC 5424 syslog standard defines 8 severity levels (0=Emergency, 1=Alert, 2=Critical, 3=Error, 4=Warning, 5=Notice, 6=Informational, 7=Debug). Security audit events typically fall in levels 3–6; properly tuned SIEM alerts trigger on unexpected patterns within those levels.

- **WORM Storage**: Write Once Read Many storage ensures logs cannot be overwritten or deleted once written. Enterprise solutions include NetApp SnapLock, AWS S3 Object Lock, and Azure Immutable Blob Storage. WORM storage is required by some regulatory frameworks as evidence preservation.

## Exam Relevance

**SY0-701 Domain Mapping:** Audit logs appear primarily in Domain 4 (Security Operations) under log monitoring and analysis, but also appear in Domain 5 (Security Program Management) in the context of compliance and data retention.

**High-Frequency Question Patterns:**

- **"Which control type is audit logging?"** — Audit logging is primarily a **detective control**. It can also be **deterrent** (people avoid misbehavior when they know they're logged). It is *not* preventive. If a question asks what type of control audit logs represent, detective is almost always correct.

- **"What ensures non-repudiation?"** — Audit logs, combined with strong authentication (so the log entry is tied to a verified identity), provide non-repudiation. This is a frequent pairing question.

- **"What should be done with logs to protect their integrity?"** — Forward them to a centralized, remote system immediately. The answer will often be "send to a SIEM" or "use a write-once log server," not "encrypt them locally."

- **"What is the purpose of a SIEM?"** — Aggregate, correlate, and alert on log data from multiple sources. SIEMs do not prevent attacks; they detect them. Don't confuse SIEM with IPS.

**Common Gotchas:**

- Audit logs are not the same as **event logs** (event logs are general-purpose; audit logs are security-focused and have specific integrity requirements).
- **Log clipping** (also called log masking) is an attack technique where an attacker modifies the logging threshold so that their activity falls below the minimum severity to be recorded — this is why tamper-evident logging matters.
- The exam distinguishes between **audit logs** (records of events) and **audit trails** (reconstructed sequences of those records). Both terms appear and are not interchangeable in exam questions.
- **Timestamps must be synchronized via [[NTP]]** — without time synchronization, correlating events across systems is unreliable, and this is explicitly called out in frameworks like PCI DSS Requirement 10.6.

## Security Implications

### Attacks Against Audit Logs

**Log Tampering and Deletion** is the most common attacker behavior post-compromise. After gaining elevated privileges, attackers routinely clear Windows Event Logs using built-in commands:
```cmd
wevtutil cl Security
wevtutil cl System
wevtutil cl Application
```
On Linux, attackers may truncate or delete log files:
```bash
echo "" > /var/log/auth.log
shred -u /var/log/auth.log
```
Event ID **1102** (Security log cleared) and **104** (System log cleared) on Windows are high-fidelity indicators of post-compromise activity and should trigger immediate alerts.

**Log Flooding / Log Injection**: An attacker can deliberately generate enormous volumes of log noise to obscure their malicious activity within the flood, or inject crafted log entries containing special characters or format strings to exploit log parsers. [[Log4Shell]] (CVE-2021-44228) was exploited precisely through log injection — user-supplied data written to logs contained JNDI lookup strings that the Log4j library evaluated, triggering remote code execution on the logging system itself.

**Log Forging**: An attacker with write access to log files can insert fabricated entries to create false alibis or implicate innocent users. This is countered by cryptographic log signing and WORM storage.

**Lateral Movement Evasion**: Sophisticated attackers (APT groups) are aware of what actions generate log entries and deliberately choose techniques that avoid detection. For example, using **living-off-the-land binaries (LOLBins)** generates log entries that resemble legitimate activity, making detection dependent on behavioral baselines rather than signature matching.

### Real Incidents

- **SolarWinds (2020)**: Attackers specifically targeted and disabled Windows Defender and manipulated audit logging on compromised systems as part of their tradecraft, demonstrating that audit log availability cannot be taken for granted.
- **Target Breach (2013)**: Post-incident analysis revealed that SIEM alerts generated by the intrusion were not acted upon, highlighting that log generation alone is insufficient — logs must be reviewed and acted upon.
- **Uber (2016)**: Forensic investigators used AWS CloudTrail audit logs (S3 access logs) to reconstruct how attackers accessed and exfiltrated 57 million user records from an S3 bucket, demonstrating the critical role of cloud audit logging.

## Defensive Measures

**1. Enable Comprehensive Logging at the Source**

Configure Windows audit policy via Group Policy to capture all security-relevant event categories:
```powershell
# Enable advanced audit policy via auditpol.exe
auditpol /set /subcategory:"Logon" /success:enable /failure:enable
auditpol /set /subcategory:"Account Logon" /success:enable /failure:enable
auditpol /set /subcategory:"Account Management" /success:enable /failure:enable
auditpol /set /subcategory:"Privilege Use" /success:enable /failure:enable
auditpol /set /subcategory:"Object Access" /success:enable /failure:enable
auditpol /set /subcategory:"Policy Change" /success:enable /failure:enable
auditpol /set /subcategory:"System" /success:enable /failure:enable

# Verify current settings
auditpol /get /category:*
```
