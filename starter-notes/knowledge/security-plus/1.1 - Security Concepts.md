# 1.1 - Security Concepts

The foundation of all cybersecurity practice rests on a set of core principles that define **what we protect**, **why we protect it**, and **how protection is achieved**. This section introduces the **CIA Triad**, **non-repudiation**, **authentication frameworks**, **cryptographic concepts**, and fundamental **risk terminology** that underpin every other domain in the Security+ exam and real-world security operations. Mastery of these concepts provides the conceptual vocabulary necessary to understand [[Threat Intelligence]], [[Risk Management]], and [[Cryptography]] as they appear throughout the field.

---

## Overview

Security as a discipline emerged from the need to protect information assets — first in military and government contexts, then expanding into commercial and personal computing. The earliest formalized frameworks, such as the U.S. Department of Defense's Trusted Computer System Evaluation Criteria (TCSEC, "Orange Book") published in 1983, laid the groundwork for modern security models by defining levels of trust, access control, and assurance. Today, frameworks like NIST SP 800-53, ISO/IEC 27001, and the MITRE ATT&CK Matrix carry this tradition forward into enterprise and cloud environments.

The **CIA Triad** — Confidentiality, Integrity, and Availability — is the canonical three-part model that describes the goals of any security program. Confidentiality ensures that information is accessible only to authorized parties. Integrity ensures that data has not been altered in unauthorized ways. Availability ensures that systems and data are accessible when needed by authorized users. Every security control, policy, or technology can be evaluated by asking which pillar(s) of the CIA Triad it addresses.

Beyond the CIA Triad, modern security adds two critical properties: **non-repudiation** and **authentication**. Non-repudiation is the assurance that a party cannot later deny having performed an action — achieved through digital signatures, audit logs, and cryptographic proofs. Authentication verifies that an entity is who it claims to be, and sits at the heart of access control. These properties collectively form the basis of **AAA** — Authentication, Authorization, and Accounting — a framework implemented by protocols like RADIUS, TACACS+, and modern identity providers.

Security concepts also encompass the formal language of **risk** — the possibility that a threat will exploit a vulnerability to cause harm to an asset. Organizations must understand the difference between threats (potential causes of harm), vulnerabilities (weaknesses that can be exploited), and risk (the intersection of likelihood and impact). The discipline of risk management involves identifying, assessing, and treating these risks through controls classified as preventive, detective, corrective, deterrent, compensating, or directive in nature.

Cryptography serves as the technical enabler for nearly all security properties. Symmetric encryption (e.g., AES), asymmetric encryption (e.g., RSA, ECC), hashing (e.g., SHA-256), and digital signatures form the toolkit used to enforce confidentiality, integrity, and non-repudiation at the technical layer. Understanding *when* and *why* each cryptographic primitive is used — not just *how* — is essential for both the exam and for designing real security architectures.

## How It Works

### The CIA Triad in Practice

**Confidentiality** is enforced through:
- **Encryption**: AES-256 for data at rest; TLS 1.3 for data in transit
- **Access Controls**: RBAC (Role-Based Access Control), DAC, MAC
- **Steganography and data masking** for sensitive fields

**Integrity** is enforced through:
- **Cryptographic hashing**: SHA-256, SHA-3 produce a fixed-length digest; any change to the input produces a completely different hash
- **Digital signatures**: A sender signs a hash of data with their private key; the recipient verifies using the sender's public key
- **File integrity monitoring (FIM)**: Tools like Tripwire or AIDE track changes to critical files

```bash
# Generate a SHA-256 hash of a file
sha256sum /etc/passwd

# Verify a downloaded file against a published checksum
echo "expectedhash  filename.iso" | sha256sum --check
```

**Availability** is enforced through:
- **Redundancy**: RAID arrays, clustered services, load balancers
- **Disaster Recovery / Business Continuity Planning**
- **DDoS mitigation**: Rate limiting, CDN scrubbing centers, BGP blackholing

### Authentication Factors

Authentication is categorized into **five factor types**:

| Factor | Type | Example |
|--------|------|---------|
| Something you know | Knowledge | Password, PIN |
| Something you have | Possession | Smart card, OTP token |
| Something you are | Inherence | Fingerprint, retina scan |
| Somewhere you are | Location | GPS geofencing, IP range |
| Something you do | Behavior | Typing cadence, gait |

**Multi-Factor Authentication (MFA)** requires two or more *different* factor types. Two passwords is NOT MFA — they are both "something you know."

### Non-Repudiation with Digital Signatures

```
1. Alice creates a message: "Transfer $10,000 to account 12345"
2. Alice's software computes SHA-256 hash of the message
3. Alice signs the hash with her RSA PRIVATE key → Digital Signature
4. Alice sends: [Message] + [Digital Signature] + [Alice's Public Certificate]
5. Bob retrieves Alice's PUBLIC key from her certificate
6. Bob decrypts the Digital Signature using Alice's PUBLIC key → hash_A
7. Bob independently computes SHA-256 of the received message → hash_B
8. If hash_A == hash_B: Message is authentic AND unaltered
9. Alice CANNOT deny sending it — only her private key could produce that signature
```

This process is implemented in protocols like S/MIME, PGP, and SSH certificate authentication.

### AAA Framework

**Authentication, Authorization, and Accounting** is implemented at both network and application layers:

```
Network AAA (RADIUS - UDP 1812/1813):
  Client (NAS) → RADIUS Server → Directory (LDAP/AD)
  
  1. User attempts VPN connection to Network Access Server
  2. NAS forwards credentials to RADIUS server (UDP 1812)
  3. RADIUS queries Active Directory via LDAP (TCP 389)
  4. AD returns group memberships
  5. RADIUS sends Access-Accept + authorization attributes
  6. NAS grants access based on returned policy
  7. Accounting records sent to RADIUS (UDP 1813)
```

**TACACS+** (TCP 49) differs from RADIUS by **separating Authentication, Authorization, and Accounting** into distinct phases and encrypting the *entire* payload (vs. RADIUS which only encrypts the password field).

### Cryptographic Primitives Summary

```
Symmetric Encryption (same key for encrypt/decrypt):
  AES-128/192/256 (block cipher, current standard)
  ChaCha20 (stream cipher, used in TLS 1.3)
  3DES (legacy, being phased out)

Asymmetric Encryption (public/private key pair):
  RSA-2048/4096 (key exchange, digital signatures)
  ECC / ECDSA (smaller keys, equal strength — used in modern TLS)
  Diffie-Hellman / ECDH (key exchange only, not encryption)

Hashing (one-way, no key):
  SHA-256 / SHA-3 (current standards)
  MD5 / SHA-1 (BROKEN — collision attacks demonstrated)
  PBKDF2, bcrypt, scrypt (password hashing — adds salt + work factor)

Message Authentication Code (integrity + authenticity):
  HMAC-SHA256 (hash + shared secret key)
```

### Security Control Categories and Types

Controls are classified along two axes:

**By Category (mechanism):**
- **Technical**: Firewalls, encryption, IDS/IPS
- **Administrative (Managerial)**: Policies, procedures, risk assessments
- **Physical**: Locks, bollards, badge readers, security guards

**By Type (function):**
- **Preventive**: Stops the attack (firewall blocks port)
- **Detective**: Identifies an attack (SIEM alerts on failed logins)
- **Corrective**: Restores after attack (patch applied, backup restored)
- **Deterrent**: Discourages attack (warning banners, visible cameras)
- **Compensating**: Alternative control when primary is infeasible (manual review instead of automated scanning)
- **Directive**: Guides behavior (policy stating "passwords must be 12+ characters")

## Key Concepts

- **CIA Triad**: The three foundational security goals — **Confidentiality** (restrict access to authorized parties), **Integrity** (ensure data is unaltered and trustworthy), and **Availability** (ensure systems are accessible when needed). Every security control maps to one or more of these pillars.

- **Non-Repudiation**: The property that prevents a party from **denying** a past action. Achieved technically through digital signatures using asymmetric cryptography, and procedurally through tamper-evident audit logs. Critical in legal and financial contexts.

- **Authentication vs. Authorization vs. Accounting**: **Authentication** verifies identity ("Are you who you claim to be?"). **Authorization** determines permissions ("What are you allowed to do?"). **Accounting** records activity ("What did you actually do, and when?"). These are distinct processes that operate sequentially.

- **Threat vs. Vulnerability vs. Risk**: A **threat** is any potential danger (e.g., ransomware group). A **vulnerability** is a weakness that can be exploited (e.g., unpatched CVE). **Risk** = Threat × Vulnerability × Impact — it only exists when all three elements are present. A vulnerability with no threat actor capable of exploitation represents lower risk.

- **Symmetric vs. Asymmetric Cryptography**: **Symmetric** uses one shared secret key for both encryption and decryption — fast, but requires secure key distribution (e.g., AES). **Asymmetric** uses mathematically linked public/private key pairs — slower, but solves key distribution; the public key encrypts or verifies, the private key decrypts or signs (e.g., RSA, ECC).

- **Hashing**: A **one-way mathematical function** that produces a fixed-length digest (fingerprint) of arbitrary input. Identical inputs always produce identical outputs; any change produces a radically different output (**avalanche effect**). Cannot be reversed. Used for integrity verification, password storage, and digital signatures.

- **Zero Trust Architecture**: A security model that assumes **no implicit trust** based on network location — every access request must be authenticated, authorized, and continuously validated regardless of whether the request originates inside or outside the perimeter. Replaces the legacy "castle-and-moat" approach.

- **Least Privilege Principle**: Users, processes, and systems should be granted **only the minimum permissions** required to perform their function — nothing more. Limits the blast radius of compromised accounts or processes.

## Exam Relevance

**SY0-701 Domain 1.0 — General Security Concepts** carries approximately **12% of exam weight**. Questions in this domain are primarily conceptual and scenario-based.

**High-frequency question patterns:**

1. **"Which CIA pillar is affected?"** — Given a scenario (data leaked, files corrupted, website down), identify which CIA property is violated. Key: A DDoS attack violates *Availability*, not Confidentiality. A man-in-the-middle that modifies data violates *Integrity*.

2. **"What type of control is this?"** — Given a control description, identify its type (preventive/detective/corrective) AND category (technical/administrative/physical). A **security camera** is a *physical, detective* control. A **firewall rule** is a *technical, preventive* control.

3. **"Which authentication factor type?"** — A smart card is *something you have*. A PIN is *something you know*. A retina scan is *something you are*. Two PINs ≠ MFA.

4. **"Symmetric or Asymmetric?"** — AES = symmetric. RSA, ECC, Diffie-Hellman = asymmetric. HMAC uses symmetric (shared secret). Digital signatures use asymmetric.

5. **"Non-repudiation mechanism?"** — The answer is almost always **digital signatures**. Hashing alone does NOT provide non-repudiation (anyone can hash). Encryption alone does NOT provide non-repudiation.

**Common gotchas:**
- **MD5 and SHA-1 are considered broken** for collision resistance — do not use for digital signatures or certificates, though MD5 may still appear in non-cryptographic contexts
- **Diffie-Hellman is a key *exchange* protocol**, not an encryption algorithm — it produces a shared secret used by symmetric algorithms
- **RADIUS encrypts only the password; TACACS+ encrypts the entire body** — this distinction appears in network access control questions
- **Compensating controls** are used when a required control cannot be implemented, not just as an addition to existing controls
- **Deterrent controls** don't stop attacks — they *discourage* them. A warning banner is deterrent, not preventive

## Security Implications

### Attacks Against CIA Properties

**Confidentiality attacks:**
- **Eavesdropping/Sniffing**: Passive capture of unencrypted traffic (Wireshark, tcpdump). Protocols still seen in cleartext include Telnet (TCP 23), FTP (TCP 21), HTTP (TCP 80), SNMPv1/v2 (UDP 161)
- **Cryptanalysis**: Attacking weak encryption. RC4 (WEP) was broken through statistical bias in keystream generation. MD5 collision attacks demonstrated in 2004 (Wang et al.)
- **Shoulder surfing / Social engineering**: Bypasses technical controls entirely

**Integrity attacks:**
- **CVE-2014-0160 (Heartbleed)**: OpenSSL vulnerability that allowed attackers to read server memory, compromising private keys and session data — a massive integrity AND confidentiality failure
- **Supply chain attacks**: SolarWinds SUNBURST (2020) — malicious code injected into software build pipeline, integrity of updates compromised at source
- **Hash collision exploitation**: The Flame malware (2012) used an MD5 collision to forge a fraudulent Microsoft code-signing certificate

**Availability attacks:**
- **DDoS (Distributed Denial of Service)**: Volumetric (bandwidth exhaustion), Protocol (SYN flood — TCP state exhaustion), Application Layer (HTTP flood targeting specific endpoints)
- **Ransomware**: Encrypts data making it unavailable until ransom paid. NotPetya (2017) caused $10B+ in damages, primarily targeting availability

**Against Authentication:**
- **Credential stuffing**: Reusing breached username/password combinations at scale (automated using tools like Hydra, Burp Suite)
- **Pass-the-Hash (PtH)**: Capturing NTLM hash from memory (Mimikatz) and using it directly without cracking — exploits Windows authentication protocol weakness
- **Kerberoasting**: Requesting Kerberos service tickets for service accounts and cracking them offline — targets weak service account passwords

## Defensive Measures

### Protecting Confidentiality
- **Enforce TLS 1.2+ everywhere**; disable SSL 3.0, TLS 1.0/1.1 in web server configs
  ```nginx
  ssl_protocols TLSv1.2 TLSv1.3;
  ssl_ciphers ECDHE-ECDSA-AES256-GCM-SHA384:ECDHE-RSA-AES256-GCM-SHA384;
  ```
- **Encrypt data at rest**: BitLocker (Windows), LUKS (Linux), AWS KMS for cloud storage
- **Implement RBAC** with quarterly access reviews; disable accounts within 24 hours of offboarding
- **Data classification policy**: Define Confidential/Internal/Public tiers with corresponding handling requirements

  ```bash
  # Run daily check (add to cron)
  sudo aide --check
  ```
