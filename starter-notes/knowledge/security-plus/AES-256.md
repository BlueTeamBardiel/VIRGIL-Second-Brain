# AES-256

**AES-256** (**Advanced Encryption Standard with 256-bit key**) is a [[Symmetric Encryption]] block cipher standardized by [[NIST]] in 2001 (FIPS 197) that operates on 128-bit blocks using a 256-bit key derived through a process called the **Rijndael key schedule**. It is the most widely deployed encryption algorithm in the world, used in everything from [[TLS]] handshakes to full-disk encryption. AES-256 is considered **quantum-resistant** for the foreseeable future, requiring approximately 2¹²⁸ operations even under a theoretical [[Grover's Algorithm]] attack.

---

## Overview

AES was selected by NIST in 2001 following a public competition to replace the aging [[DES]] (Data Encryption Standard) and its stopgap successor [[3DES]]. The winning algorithm, Rijndael, was designed by Belgian cryptographers Joan Daemen and Vincent Rijmen. NIST standardized three key sizes: 128-bit (AES-128), 192-bit (AES-192), and 256-bit (AES-256). Of these, AES-256 offers the highest security margin and is mandated for protecting classified U.S. government information at the TOP SECRET level by NSA Suite B cryptography guidelines.

AES is a **substitution-permutation network (SPN)**, not a Feistel cipher like DES. This architectural choice means that every bit of the output depends on every bit of the input after just a few rounds of processing — a property known as the **avalanche effect**. AES-256 performs 14 rounds of transformation (compared to 10 for AES-128 and 12 for AES-192), making brute-force attacks computationally infeasible with current and projected classical computing hardware.

The algorithm has been subjected to over two decades of intensive cryptanalysis and remains unbroken in its standard form. The best known theoretical attacks (such as biclique cryptanalysis) reduce the effective key strength by only a trivial margin — from 256 bits to approximately 254.4 bits — which has no practical significance. This robustness makes AES-256 the algorithm of choice for governments, financial institutions, cloud providers, and security-conscious homelab practitioners alike.

In practical deployment, AES-256 is almost always paired with a **mode of operation** that defines how the cipher handles data larger than a single 128-bit block. Common modes include CBC (Cipher Block Chaining), GCM (Galois/Counter Mode), CTR (Counter Mode), and ECB (Electronic Codebook — generally considered insecure for most use cases). The choice of mode dramatically affects both the security properties and performance characteristics of the resulting encryption scheme. AES-256-GCM is the current gold standard, providing both encryption and authentication ([[AEAD]]).

## How It Works

### Block and Key Structure

AES-256 encrypts data in fixed **128-bit (16-byte) blocks**. The plaintext block is arranged into a 4×4 matrix of bytes called the **state array**. The 256-bit key undergoes **key expansion** (Rijndael key schedule) to generate 15 round keys (one initial + 14 round keys), each 128 bits wide, totalling 240 bytes of key material.

### Round Structure (14 Rounds)

Each of the 14 rounds (except the final) applies four transformations to the state array:

#### 1. SubBytes
Each byte in the 4×4 state is replaced using the **AES S-box**, a fixed 256-entry lookup table derived from the multiplicative inverse in GF(2⁸). This provides **non-linearity** — the critical property that prevents linear cryptanalysis.

```
Original byte: 0x53
S-box output:  0xED
```

#### 2. ShiftRows
Each row of the state array is cyclically shifted left by a different offset:
- Row 0: no shift
- Row 1: shift 1 byte left
- Row 2: shift 2 bytes left
- Row 3: shift 3 bytes left

This provides **diffusion** — spreading byte influence across columns.

#### 3. MixColumns
Each column of the state is treated as a polynomial over GF(2⁸) and multiplied by a fixed polynomial. This is the most computationally expensive step and ensures that each output byte depends on all four input bytes of the column. The final round **omits MixColumns** to make encryption and decryption structurally symmetric.

#### 4. AddRoundKey
The current round key (derived from key schedule) is XORed with the state array. Without knowledge of the key, this step is the cryptographic lock that an attacker cannot bypass.

```
State byte:      0xAB
Round key byte:  0x37
Result:          0x9C  (XOR operation)
```

### Key Expansion (Rijndael Key Schedule)

The 256-bit (32-byte) original key is expanded into 60 32-bit words (W[0]–W[59]):

```
W[i] = W[i-Nk] XOR temp
```
Where `Nk = 8` for AES-256. Special operations (SubWord, RotWord, Rcon) are applied at regular intervals to ensure key material thoroughly mixes.

### Full Encryption Flow

```
Input: 128-bit plaintext block + 256-bit key
│
├─ Initial AddRoundKey (Round Key 0)
│
├─ Rounds 1–13:
│   ├─ SubBytes
│   ├─ ShiftRows
│   ├─ MixColumns
│   └─ AddRoundKey
│
└─ Round 14 (Final):
    ├─ SubBytes
    ├─ ShiftRows
    └─ AddRoundKey  (no MixColumns)
│
Output: 128-bit ciphertext block
```

### Practical Usage with OpenSSL

```bash
# Encrypt a file using AES-256-CBC
openssl enc -aes-256-cbc -salt -pbkdf2 -iter 100000 \
  -in plaintext.txt -out encrypted.bin -pass pass:YourStrongPassphrase

# Decrypt
openssl enc -aes-256-cbc -d -pbkdf2 -iter 100000 \
  -in encrypted.bin -out decrypted.txt -pass pass:YourStrongPassphrase

# Encrypt using AES-256-GCM (preferred)
openssl enc -aes-256-gcm -salt -pbkdf2 -iter 100000 \
  -in plaintext.txt -out encrypted_gcm.bin -pass pass:YourStrongPassphrase

# Generate a random 256-bit key (32 bytes) as hex
openssl rand -hex 32

# Benchmark AES performance on your system
openssl speed aes-256-cbc aes-256-gcm
```

### Python Implementation (PyCryptodome)

```python
from Crypto.Cipher import AES
from Crypto.Random import get_random_bytes
import base64

# Generate key and nonce
key = get_random_bytes(32)       # 256-bit key
nonce = get_random_bytes(16)     # 128-bit nonce for GCM

# Encrypt using AES-256-GCM (authenticated encryption)
cipher = AES.new(key, AES.MODE_GCM, nonce=nonce)
ciphertext, tag = cipher.encrypt_and_digest(b"Hello, [YOUR-LAB]!")

print(f"Key (hex):        {key.hex()}")
print(f"Nonce (hex):      {nonce.hex()}")
print(f"Ciphertext (b64): {base64.b64encode(ciphertext).decode()}")
print(f"Auth Tag (hex):   {tag.hex()}")

# Decrypt and verify
cipher2 = AES.new(key, AES.MODE_GCM, nonce=nonce)
plaintext = cipher2.decrypt_and_verify(ciphertext, tag)
print(f"Decrypted: {plaintext.decode()}")
```

## Key Concepts

- **Block Cipher**: AES-256 encrypts exactly 128 bits (16 bytes) per operation. Data larger than one block requires a **mode of operation** (CBC, CTR, GCM) to handle the full message securely.
- **Key Size vs. Block Size**: AES always uses a **128-bit block size** regardless of key length. The 256-bit figure refers exclusively to the key — a critical distinction frequently tested on Security+.
- **Modes of Operation**: Define how sequential blocks are processed. **GCM** (Galois/Counter Mode) provides **AEAD** (Authenticated Encryption with Associated Data), combining confidentiality and integrity in a single pass. **ECB mode** is cryptographically weak because identical plaintext blocks produce identical ciphertext blocks.
- **Avalanche Effect**: A single-bit change in plaintext or key causes approximately 50% of ciphertext bits to change after all rounds, making statistical analysis of the relationship between input and output infeasible.
- **Key Derivation**: Raw passphrases must never be used directly as AES keys. A **KDF** (Key Derivation Function) such as [[PBKDF2]], [[bcrypt]], or [[Argon2]] must be used to derive a proper 256-bit key from a human-readable password, with appropriate salt and iteration count.
- **Hardware Acceleration (AES-NI)**: Modern Intel and AMD processors include dedicated **AES-NI** instructions that accelerate AES operations 3–10× compared to software implementation, virtually eliminating performance overhead for encrypted storage and networking.
- **Initialization Vector (IV) / Nonce**: Modes like CBC and GCM require a unique, unpredictable value per encryption operation to prevent ciphertext patterns from leaking. **IV reuse is catastrophic** in GCM mode, potentially allowing full plaintext recovery and authentication forgery.

## Exam Relevance

**Security+ SY0-701** tests AES-256 knowledge across multiple domains. Key areas to focus on:

**Common Question Patterns:**
- "Which algorithm provides the STRONGEST encryption?" → AES-256 (when compared to AES-128, 3DES, DES, RC4)
- "Which algorithm is approved for TOP SECRET data?" → AES-256 (NSA Suite B / CNSA)
- "What is the block size of AES?" → **128 bits**, NOT 256 bits. This is the #1 gotcha.
- "What mode provides both encryption and integrity?" → **GCM** (AEAD)
- "Which mode should NEVER be used?" → **ECB** mode

**Critical Gotchas:**
- AES-256 is **symmetric** — the same key encrypts and decrypts. Do not confuse with [[RSA]] or [[ECC]] (asymmetric).
- The number **256** refers to the **key length**, not block size, not rounds. Block size is always 128 bits.
- AES-256 performs **14 rounds** (not 10, not 16). AES-128 = 10 rounds, AES-192 = 12 rounds, AES-256 = 14 rounds.
- In the context of VPNs and TLS, AES-256 is used for **bulk data encryption** while [[RSA]] or [[ECDH]] handles **key exchange** — they complement each other.
- **Perfect Forward Secrecy** is a property of the key exchange mechanism ([[DHE]], [[ECDHE]]), not of AES itself.
- Questions about "quantum resistance" of symmetric keys: Grover's algorithm halves the effective key strength, so AES-256 → ~128-bit post-quantum security. AES-128 → ~64-bit (potentially vulnerable). AES-256 remains recommended.

## Security Implications

### Known Attack Vectors

**Theoretical Attacks (Non-Practical):**
- **Biclique Cryptanalysis (2011)**: The best known attack against AES-256, requiring 2^254.4 operations — effectively no improvement over brute force for practical purposes. AES-256 has never been broken in its full-round form.
- **Related-Key Attacks**: Theoretical attacks exist against AES-256 in related-key scenarios (where an attacker can obtain encryptions under related keys). These require conditions that never occur in properly implemented systems.

### Real-World Vulnerabilities (Implementation, Not Algorithm)

**CVE-2016-2183 (SWEET32)**: Not AES-specific, but illustrates the importance of block size. 64-bit block ciphers (3DES, Blowfish) are vulnerable to birthday attacks after ~32GB of data. AES's 128-bit block size makes equivalent attacks require 2^64 blocks — currently infeasible.

**IV/Nonce Reuse in GCM**:
- If the same (key, nonce) pair is reused in AES-GCM, an attacker who captures two ciphertexts can XOR them to eliminate the keystream, potentially recovering plaintext. More critically, nonce reuse in GCM allows full authentication key recovery, breaking integrity guarantees entirely.
- Real incident: Multiple TLS library implementations were found to have nonce-generation weaknesses.

**Padding Oracle Attacks (CBC Mode)**:
- **CVE-2014-3566 (POODLE)**: Exploited SSLv3 CBC padding to decrypt sessions byte-by-byte. Demonstrates that even strong ciphers become vulnerable when modes are implemented incorrectly or when unauthenticated padding is exposed.
- Mitigation: Use **GCM** instead of **CBC**, or always apply **MAC-then-encrypt** with authenticated encryption.

**Key Management Failures**:
- The overwhelming majority of AES-256 compromises in practice involve key theft, weak key derivation, or hardcoded keys — not cryptanalytic attacks.
- Example: Hardcoded AES keys found in IoT firmware, cloud secrets stored in cleartext configuration files, improper key storage in mobile apps.

**Side-Channel Attacks**:
- Cache-timing attacks against software AES implementations (not using AES-NI) can leak key material by measuring execution time variations. AES-NI hardware instructions eliminate most timing side-channels.

## Defensive Measures

### Algorithm and Mode Selection
- Always prefer **AES-256-GCM** over AES-256-CBC for new implementations. GCM provides authentication without requiring a separate HMAC.
- Never use **AES-ECB** mode. If identical blocks produce identical ciphertext, patterns in data (especially structured formats) can reveal information about the plaintext.
- For disk encryption, **XTS mode** (AES-256-XTS) is standard (used by [[BitLocker]], [[LUKS]], FileVault). XTS is designed specifically to handle fixed-size sectors without expanding the ciphertext.

### Key Generation and Storage
```bash
# Generate a cryptographically secure 256-bit key
openssl rand -hex 32 > /etc/app/aes.key
chmod 600 /etc/app/aes.key

# Use HashiCorp Vault for production key management
vault secrets enable transit
vault write transit/keys/myapp-key type=aes256-gcm96

# Encrypt data via Vault Transit (no key ever leaves Vault)
vault write transit/encrypt/myapp-key \
  plaintext=$(base64 <<< "sensitive data")
```

### Key Derivation (Never Use Raw Passwords)
```python
import hashlib, os

# PBKDF2-HMAC-SHA256 key derivation
password = b"user_passphrase"
salt = os.urandom(32)                          # 256-bit random salt
key = hashlib.pbkdf2_hmac(
    'sha256',
    password,
    salt,
    iterations=600_000,                        # NIST 2023 recommendation
    dklen=32                                   # 256-bit output
)
```
