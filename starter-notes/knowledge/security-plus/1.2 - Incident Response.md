# 1.2 - Incident Response

**Incident Response (IR)** is the structured methodology an organization uses to detect, contain, eradicate, and recover from **security incidents**—any event that threatens the confidentiality, integrity, or availability of information systems. The [[NIST Special Publication 800-61]] framework and the [[SANS Incident Response Process]] are the two most referenced models in the industry, and both inform the Security+ SY0-701 exam. Effective IR reduces [[dwell time]]—the window between initial compromise and detection—and minimizes operational damage from breaches, ransomware, and insider threats.

---

## Overview

Incident response exists because no security posture is impenetrable. Organizations face a continuous stream of threats ranging from commodity malware to sophisticated nation-state actors, and the question is never *if* an incident will occur but *when*. A well-defined IR process ensures that when a breach happens, trained personnel follow a repeatable, legally defensible procedure rather than improvising under pressure. This reduces both technical damage and reputational harm.

The NIST SP 800-61 lifecycle defines four phases: **Preparation**, **Detection and Analysis**, **Containment, Eradication, and Recovery**, and **Post-Incident Activity**. The SANS model uses six steps—Preparation, Identification, Containment, Eradication, Recovery, and Lessons Learned—often abbreviated as **PICERL**. Though the naming differs, both frameworks converge on the same operational logic: build readiness before incidents occur, accurately identify and scope incidents when they do, limit damage, remove the threat, restore operations, and institutionalize lessons.

Real-world IR involves multiple stakeholders simultaneously. The **Security Operations Center (SOC)** provides continuous monitoring through a [[SIEM]] (Security Information and Event Management) platform. Tier-1 analysts triage alerts, escalating confirmed incidents to Tier-2 and Tier-3 analysts or dedicated **Incident Response Teams (IRTs)**. Legal counsel, HR, executive leadership, and external forensic vendors may all become involved depending on the severity and nature of the incident—particularly if personally identifiable information (PII) is involved and breach notification laws apply.

IR is deeply intertwined with [[Digital Forensics]]. Evidence preservation, chain of custody, and forensic imaging must be conducted correctly from the moment an incident is confirmed; mishandled evidence can compromise criminal prosecution or civil litigation. The concept of **evidence volatility** dictates the order in which data is collected: CPU registers and RAM contents are lost the moment a system is powered off, so live memory acquisition takes priority over disk imaging.

Organizations formalize their IR capability through an **Incident Response Plan (IRP)**, an **Incident Response Policy**, and individual **Incident Response Playbooks**. Playbooks are prescriptive, step-by-step guides for specific incident types—ransomware, phishing, data exfiltration, insider threat—ensuring consistency and reducing decision fatigue during high-stress events. These documents must be tested through **tabletop exercises** and **full simulations** to remain operationally current.

## How It Works

### Phase 1: Preparation

Preparation is the foundation of effective IR and is the only phase that occurs *before* an incident. Key activities include:

- Establishing and training an **Incident Response Team (IRT)** with defined roles: Incident Commander, Lead Analyst, Communications Lead, Legal/Compliance Liaison
- Documenting the **Incident Response Plan** and individual playbooks
- Deploying monitoring infrastructure: [[SIEM]], [[IDS/IPS]], EDR agents, netflow collectors, centralized logging (syslog, Windows Event Forwarding)
- Creating and maintaining a **contact list** for internal and external stakeholders (FBI, CISA, legal counsel, forensic vendors)
- Preparing a **jump bag** or IR toolkit—pre-loaded USB drives or virtual machines with forensic tools

**Critical tools deployed during preparation:**

```bash
# Deploy Velociraptor agent for endpoint visibility (Linux example)
sudo dpkg -i velociraptor_linux_amd64.deb
velociraptor config generate -i   # Interactive config generation

# Configure Windows Event Forwarding (PowerShell)
winrm quickconfig -q
wecutil cs subscription.xml       # Create WEF subscription

# Deploy Wazuh HIDS agent
sudo systemctl enable wazuh-agent
sudo systemctl start wazuh-agent
```

### Phase 2: Detection and Analysis

An incident begins with an **indicator**—either an **Indicator of Compromise (IoC)** such as a known malicious IP or hash, or an **Indicator of Attack (IoA)** such as anomalous lateral movement behavior.

**Common detection sources:**
- SIEM correlation rules (e.g., multiple failed logins followed by success = brute force)
- AV/EDR alerts on [[malware]] execution
- User reports ("I can't access my files" → potential ransomware)
- External notification from law enforcement or threat intelligence partners
- Honeypot triggers

**Triage process:**

```
Alert Received
     │
     ▼
Is it a True Positive? ──No──▶ Document as False Positive, tune rule
     │
    Yes
     ▼
Classify Severity (P1/P2/P3/P4)
     │
     ▼
Notify Incident Commander
     │
     ▼
Begin Evidence Collection (volatile first)
```

**Severity classification example:**

| Priority | Definition | Example |
|----------|-----------|---------|
| P1 - Critical | Active breach, data exfiltration in progress | Ransomware spreading across domain |
| P2 - High | Confirmed compromise, contained | Single workstation malware infection |
| P3 - Medium | Suspicious activity, unconfirmed | Anomalous outbound traffic |
| P4 - Low | Policy violation, no system impact | User visiting blocked website |

**Evidence collection order (Order of Volatility per RFC 3227):**
1. CPU registers, cache
2. RAM / running processes
3. Network connections, routing tables, ARP cache
4. Running processes and open files
5. Disk contents
6. Remote logging and monitoring data
7. Physical configuration, network topology
8. Archival media (backups, tapes)

```bash
# Live memory acquisition with LiME (Linux)
sudo insmod lime.ko "path=/media/usb/memory.lime format=lime"

# Capture RAM on Windows with WinPmem
winpmem_mini_x64.exe memory.aff4

# Capture volatile network state
netstat -anob > connections.txt         # Active connections + process names
arp -a > arp_cache.txt
ipconfig /all > network_config.txt
```

### Phase 3: Containment

Containment prevents further spread without destroying forensic evidence. It is split into **short-term** and **long-term** containment.

**Short-term containment:**
- Isolate the affected system (network quarantine via VLAN reassignment or EDR kill-switch)
- Block malicious IPs/domains at the firewall or DNS sinkholes
- Disable compromised user accounts in Active Directory

```powershell
# Disable AD account immediately
Disable-ADAccount -Identity "jsmith"

# Block IP at Windows Firewall
New-NetFirewallRule -DisplayName "Block C2" -Direction Outbound `
  -RemoteAddress 198.51.100.47 -Action Block

# Isolate host via Cisco switch port shutdown
# (executed via SSH to switch)
interface GigabitEthernet1/0/12
  shutdown
```

**Long-term containment:**
- Rebuild or reimage systems with known-good baseline images
- Apply emergency patches for the exploited vulnerability
- Rotate all credentials that may have been exposed

### Phase 4: Eradication

Remove the root cause and all artifacts of the compromise:
- Delete malware binaries and persistence mechanisms (registry run keys, scheduled tasks, cron jobs, startup scripts)
- Remove threat actor accounts and SSH authorized keys
- Patch the exploited vulnerability
- Scan all systems in the same network segment with updated signatures

```bash
# Check for persistence mechanisms (Linux)
crontab -l -u root
ls -la /etc/cron.d/ /etc/cron.daily/
systemctl list-units --type=service | grep -v "loaded active"
cat /root/.ssh/authorized_keys

# Windows persistence checks
reg query HKLM\SOFTWARE\Microsoft\Windows\CurrentVersion\Run
schtasks /query /fo LIST /v | findstr "Task Name\|Run As User\|Task To Run"
```

### Phase 5: Recovery

Restore systems to normal operation with increased monitoring:
- Restore from clean backups verified against hash values
- Return systems to production in phases, monitoring closely
- Implement additional controls to detect recurrence

### Phase 6: Lessons Learned (Post-Incident Activity)

Conducted within 2 weeks of incident closure:
- **After-Action Report (AAR)** documents timeline, root cause, actions taken, gaps identified
- Update playbooks, detection rules, and security controls
- Calculate metrics: MTTD (Mean Time to Detect), MTTR (Mean Time to Respond/Recover)
- Brief leadership and update risk register

## Key Concepts

- **Incident vs. Event**: An **event** is any observable occurrence in a system or network (a log entry, a login attempt). An **incident** is an event that violates security policy or threatens the organization—not all events are incidents, but all incidents begin as events.

- **Chain of Custody**: A **chronological, documented record** of who collected, handled, transferred, and analyzed evidence. Breaks in chain of custody can render evidence inadmissible in legal proceedings and must be maintained with signed documentation throughout the IR process.

- **CIRT/CSIRT**: A **Computer Incident Response Team / Computer Security Incident Response Team** is the dedicated group responsible for handling security incidents. They may be internal to the organization or provided as a retainer service by an external vendor (e.g., CrowdStrike Incident Response Services, Mandiant).

- **Dwell Time**: The period between initial **attacker compromise** and **detection** by defenders. The 2023 Mandiant M-Trends report placed global median dwell time at 16 days. Reducing dwell time is one of the primary measurable goals of an IR program.

- **Playbook vs. Runbook**: A **playbook** is a high-level strategic guide for responding to a category of incident (ransomware playbook, phishing playbook). A **runbook** contains the specific, technical step-by-step procedures that analysts execute—runbooks operationalize playbooks.

- **Evidence Volatility**: The principle that certain evidence types are **more transient than others** and must be collected before less volatile evidence. Defined in RFC 3227, it dictates collection order from RAM and running processes down to archival media.

- **Tabletop Exercise**: A **discussion-based simulation** where stakeholders walk through an incident scenario to test the IRP without live system impact. Distinguished from a **full-scale exercise** which involves actual technical teams executing response actions in a test environment.

- **Threat Intelligence Integration**: Structured threat data in formats like **STIX/TAXII** or from feeds like **MISP** is consumed by the SIEM and IR platform to automatically correlate IoCs against known threat actor TTPs (Tactics, Techniques, and Procedures) from the [[MITRE ATT&CK]] framework.

## Exam Relevance

**SY0-701 Exam Tips:**

The Security+ exam tests IR heavily under Domain 4 (Security Operations) but foundational concepts appear in objective 1.2 (Threat Intelligence) due to the relationship between threat identification and response triggering. Expect **scenario-based questions** that describe an incident and ask you to identify the correct *next step* in the IR process.

**Common question patterns:**

1. **"A security analyst discovers encrypted files with ransom notes on a file server. What should be done FIRST?"**
   - Answer is almost always **containment** (isolate the system) — not eradication, not notifying users. The exam loves testing whether you jump to remediation before containing the spread.

2. **"Which phase of incident response involves documenting lessons learned?"**
   - NIST calls it **Post-Incident Activity**; SANS calls it **Lessons Learned**. Know both names.

3. **"An analyst needs to preserve evidence from a compromised Linux server. What should be collected first?"**
   - **RAM/volatile memory** — always collect most volatile data first (RFC 3227 order of volatility).

4. **"What document provides step-by-step procedures for responding to a specific type of attack?"**
   - **Playbook** (some questions say runbook — context matters; if it's specific tactical steps it's a runbook, if it's the overall strategy it's a playbook).

**Key gotchas:**
- **PICERL** (Preparation, Identification, Containment, Eradication, Recovery, Lessons Learned) = SANS model
- **NIST 800-61** has 4 phases, not 6 — containment/eradication/recovery are one combined phase in NIST
- **Containment comes before eradication** — never skip to eradication; you'll cause more damage
- **Lessons Learned is NOT optional** — the exam treats it as a required phase, not an afterthought
- Know the difference between **incident response** (process) and **disaster recovery** (restoring operations after catastrophic events) — they overlap but are distinct

## Security Implications

**Failure modes in IR that attackers exploit:**

Threat actors specifically study and evade IR processes. Common adversarial techniques include:

- **Log tampering**: Attackers with admin access delete or modify Windows Event Logs (`wevtutil cl Security`) and syslog entries to blind IR teams. Centralized, write-once log storage is the countermeasure.

- **Living off the Land (LotL)**: Using built-in OS tools (PowerShell, WMI, certutil, mshta) instead of custom malware means no malicious binary to detect via hash. This was central to the **SolarWinds SUNBURST** incident (2020), where attackers used legitimate Orion software update mechanisms to avoid detection for months.

- **Delayed payload execution**: Malware dormant for 30+ days before activating, ensuring backups used for recovery are also compromised. The **NotPetya** (2017) attack spread through the MeDoc accounting software update channel and caused an estimated $10 billion in damages partly because organizations couldn't trust their own backup infrastructure.

- **Anti-forensic techniques**: Timestomping (modifying file metadata timestamps using tools like Metasploit's `timestomp`), steganography to exfiltrate data, encrypted C2 channels over legitimate services (Cobalt Strike using HTTPS to legitimate cloud providers).

- **CVE-2021-44228 (Log4Shell)**: The Log4j vulnerability allowed remote code execution via a JNDI lookup string in log messages. IR teams were challenged because exploitation attempts appeared *in their own SIEM log data*, potentially triggering the vulnerability in logging systems parsing the attack traffic.

**Detection gaps commonly found during IR:**
- No visibility into encrypted east-west traffic (lateral movement)
- Incomplete EDR coverage (unmanaged assets, legacy systems)
- Alert fatigue leading to true positives being dismissed
- No baselining, making anomaly detection impossible

## Defensive Measures

**Building a defensible IR capability:**

**1. SIEM Deployment and Tuning**
Deploy a SIEM (Splunk, IBM QRadar, Microsoft Sentinel, or open-source Wazuh + ELK Stack) with tuned correlation rules. Critical log sources:
- Windows Security Event Log (Event IDs: 4624 logon, 4625 failed logon, 4688 process creation, 4698 scheduled task, 4720 account creation)
- Sysmon (enhanced Windows process telemetry)
- Firewall/proxy logs
- DNS query logs (for detecting C2 via DNS tunneling)

**2. Endpoint Detection and Response (EDR)**
Deploy an EDR solution (CrowdStrike Falcon, SentinelOne, Microsoft Defender for Endpoint, open-source Velociraptor) on all endpoints. Configure for automatic isolation of systems matching high-confidence malicious behavior rules.

**3. Network Segmentation**
Implement [[VLANs]] and firewall rules to limit lateral movement. The blast radius of an incident is directly proportional to how flat the network is.
