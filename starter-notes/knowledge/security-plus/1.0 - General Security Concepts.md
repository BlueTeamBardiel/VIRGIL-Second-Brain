# 1.0 - General Security Concepts

**General Security Concepts** form the foundational vocabulary and theoretical framework upon which all practical cybersecurity work is built. This domain covers essential topics including **security controls**, **cryptographic principles**, **authentication mechanisms**, and **non-repudiation** — all of which appear extensively in the [[CompTIA Security+ SY0-701]] exam. Understanding these concepts is prerequisite knowledge for every other domain, from threat intelligence to cloud security.

---

## Overview

General Security Concepts (Domain 1.0) represents approximately **12% of the Security+ SY0-701 exam** and is intentionally broad, serving as a conceptual anchor for the entire certification. It establishes shared language and mental models that security professionals use daily when designing systems, evaluating risk, or responding to incidents. Rather than focusing on a single technology, this domain builds the cognitive scaffolding needed to reason about *why* security decisions are made.

At its core, this domain emphasizes the **CIA Triad** — Confidentiality, Integrity, and Availability — which is the fundamental model for evaluating any security control or incident. Confidentiality ensures that sensitive data is accessible only to authorized parties; integrity ensures that data has not been altered improperly; and availability ensures that authorized users can access resources when needed. Nearly every security technology, from firewalls to encryption, can be traced back to protecting one or more of these three properties.

The domain also addresses the **AAA framework** — Authentication, Authorization, and Accounting — which governs how systems verify identity, enforce access decisions, and maintain records of activity. This triad is implemented in technologies ranging from [[RADIUS]] and [[TACACS+]] servers to operating system login processes and web application session management. Without a robust AAA model, even well-encrypted systems are vulnerable to unauthorized access.

Cryptographic concepts are a major sub-component of this domain, covering symmetric and asymmetric encryption, hashing, digital signatures, and certificates. These mechanisms underpin virtually every secure communication protocol in use today, including [[TLS]], [[SSH]], [[IPsec]], and [[PGP]]. Understanding *when* to apply each cryptographic tool, and *why* it works, is critical for both exam success and practical security engineering.

Finally, this domain introduces security control categories and types, non-repudiation, the concept of zero trust architecture, and physical security fundamentals. These topics establish the governance and architectural vocabulary that security professionals use when writing policies, designing networks, or auditing existing systems.

## How It Works

### The CIA Triad in Practice

The CIA Triad is not just theoretical — it is operationalized in every security control deployed in an organization:

- **Confidentiality** is enforced through encryption (AES-256 at rest, TLS 1.3 in transit), access control lists, data classification policies, and need-to-know principles.
- **Integrity** is enforced through cryptographic hashes (SHA-256, SHA-3), message authentication codes (HMAC), digital signatures, and file integrity monitoring tools like [[Tripwire]] or [[AIDE]].
- **Availability** is enforced through redundancy, failover clustering, DDoS mitigation, backups, and UPS systems.

```bash
# Verify file integrity using SHA-256 (Linux)
sha256sum /etc/passwd
# Output: e3b0c44298fc1c149afb... /etc/passwd

# Compare against a known-good baseline
sha256sum -c checksums.txt
```

### Authentication Mechanisms

Authentication is the process of proving identity. The three classical factors are:

| Factor | Type | Example |
|--------|------|---------|
| Something you know | Knowledge | Password, PIN |
| Something you have | Possession | Smart card, TOTP token |
| Something you are | Inherence | Fingerprint, retina scan |

**Multi-Factor Authentication (MFA)** requires two or more distinct factors. Combining a password (knowledge) with a TOTP code from an authenticator app (possession) is the most common deployment pattern. FIDO2/WebAuthn has emerged as a phishing-resistant standard that replaces passwords with cryptographic key pairs stored in hardware authenticators (e.g., YubiKey).

```bash
# Generate a TOTP secret using oathtool (Linux)
oathtool --totp --base32 "JBSWY3DPEHPK3PXP"
# Output: 123456 (changes every 30 seconds)
```

### Cryptographic Mechanisms

**Symmetric Encryption** uses the same key for encryption and decryption. It is fast and efficient for bulk data encryption but requires a secure key exchange mechanism.

- Common algorithms: AES (128/192/256-bit), 3DES (legacy, deprecated)
- Common modes: GCM (authenticated), CBC, CTR

**Asymmetric Encryption** uses a mathematically linked key pair — a public key for encryption and a private key for decryption. It solves the key distribution problem but is computationally expensive.

- Common algorithms: RSA (2048/4096-bit), ECC (256-bit), Diffie-Hellman (key exchange)
- Used in: TLS handshakes, digital signatures, certificate authorities

```
Symmetric Key Exchange Problem:
Alice and Bob need to share a secret key over an untrusted channel.
Solution: Diffie-Hellman Key Exchange

1. Alice and Bob agree on public parameters (g=2, p=23)
2. Alice selects private a=6, sends A = g^a mod p = 64 mod 23 = 18
3. Bob selects private b=15, sends B = g^b mod p = 215 mod 23 = 2
4. Alice computes s = B^a mod p = 2^6 mod 23 = 64 mod 23 = 18
5. Bob computes s = A^b mod p = 18^15 mod 23 = 18
6. Shared secret = 18 (never transmitted)
```

**Hashing** produces a fixed-length digest from arbitrary input. It is one-way and deterministic — the same input always produces the same hash. Used for password storage, file integrity, and digital signatures.

| Algorithm | Output Size | Status |
|-----------|-------------|--------|
| MD5 | 128-bit | Broken (collision attacks) |
| SHA-1 | 160-bit | Deprecated (SHAttered attack) |
| SHA-256 | 256-bit | Current standard |
| SHA-3/Keccak | 256/512-bit | NIST approved |
| bcrypt | 184-bit | Password hashing (includes salt) |

**Digital Signatures** combine asymmetric cryptography with hashing to provide **non-repudiation** and **integrity**:

```
Signing:   Hash(document) → Encrypted with sender's PRIVATE key = Signature
Verifying: Decrypt(signature) with sender's PUBLIC key → Compare to Hash(document)
```

### Security Control Categories and Types

Security controls are organized by **category** (how they are implemented) and **type** (what they do):

**Categories:**
- **Technical/Logical** — Firewalls, encryption, access control systems
- **Administrative/Managerial** — Policies, procedures, security awareness training
- **Physical/Operational** — Locks, guards, badge readers, security cameras

**Types:**
- **Preventive** — Stop an attack from succeeding (firewall, MFA)
- **Detective** — Identify that an attack has occurred (IDS, SIEM, audit logs)
- **Corrective** — Restore systems after an incident (backups, patch management)
- **Deterrent** — Discourage attackers (warning banners, visible cameras)
- **Compensating** — Alternative control when primary is not feasible
- **Directive** — Require behavior through policy (acceptable use policy)

### Zero Trust Architecture

Zero Trust operates on the principle of **"never trust, always verify"** — no user, device, or network segment is inherently trusted, even if inside the corporate perimeter. Key tenets include:

1. Verify explicitly — Always authenticate and authorize based on all available data points
2. Use least privilege access — Limit access to minimum required, use Just-In-Time (JIT) access
3. Assume breach — Minimize blast radius, segment networks, encrypt everything, monitor continuously

## Key Concepts

- **CIA Triad**: The three foundational security properties — **Confidentiality** (only authorized parties access data), **Integrity** (data is accurate and unaltered), and **Availability** (systems are accessible when needed). All security controls ultimately protect one or more of these properties.

- **Non-Repudiation**: The assurance that a party cannot deny having performed an action. Achieved through **digital signatures** and **audit logs** — a digitally signed email proves the sender's private key was used, making denial impossible without claiming key compromise.

- **AAA Framework**: **Authentication** (who are you?), **Authorization** (what are you allowed to do?), and **Accounting** (what did you do?). Implemented by protocols like [[RADIUS]], [[TACACS+]], and [[LDAP]], and is foundational to both network access control and cloud IAM.

- **Symmetric vs. Asymmetric Cryptography**: **Symmetric** encryption (AES) uses a single shared key and is fast — ideal for bulk data. **Asymmetric** encryption (RSA, ECC) uses a key pair and solves key distribution — ideal for secure key exchange, digital signatures, and certificates. In practice, TLS uses asymmetric methods to exchange a symmetric session key.

- **Hashing vs. Encryption**: **Hashing** is one-way — data cannot be recovered from a hash. **Encryption** is two-way — data can be decrypted with the correct key. Passwords should be *hashed* (with salt), not encrypted. Confusing these is a common security mistake and a common exam trap.

- **Steganography**: The practice of **hiding data within other data** — for example, concealing a text message within the pixel values of an image file. Unlike cryptography, steganography aims to conceal the *existence* of a message rather than its content.

- **Obfuscation**: Making information or code **difficult to understand without fundamentally encrypting it** — used legitimately in software licensing and maliciously by attackers to evade detection (e.g., obfuscated PowerShell payloads).

- **Gap Analysis**: A formal assessment that compares an organization's **current security posture** against a desired target state (such as NIST CSF or ISO 27001), identifying deficiencies that need remediation.

## Exam Relevance

**SY0-701 Weight:** ~12% of exam (Domain 1 of 5)

**Common Question Patterns:**

1. **Scenario-based control identification**: "A company wants to prevent tailgating into a secure facility. Which type of control is a mantrap?" Answer: **Preventive + Physical**. Watch for questions that ask for *both* category and type.

2. **CIA Triad mapping**: Given an incident description, identify which property was violated. A ransomware attack that encrypts files violates **availability** (and potentially integrity). A data exfiltration attack violates **confidentiality**.

3. **Cryptographic algorithm selection**: Know when to use each algorithm. RSA for key exchange and signatures; AES for bulk encryption; SHA-256 for integrity; bcrypt/Argon2 for password storage. MD5 and SHA-1 are **broken** — their appearance in questions usually signals incorrect usage.

4. **Authentication factor types**: A PIN is **something you know** even if used with a smart card. Questions often try to trick you into counting a PIN+card as two factors when the PIN is *unlocking the card* (single factor, something you have). True MFA requires factors from *different categories*.

**Gotchas:**

- **Hashing is not encryption** — this distinction appears frequently. If a question says passwords are "encrypted," verify context carefully.
- **Non-repudiation requires asymmetric cryptography** — HMAC provides integrity but NOT non-repudiation because both parties share the key.
- **Deterrent vs. Preventive** — a fence can deter but not prevent a determined attacker; a locked door prevents. Context matters.
- **Compensating controls** are used *when the primary control cannot be implemented*, not just as an additional layer.
- **Zero Trust ≠ no perimeter** — it means the perimeter is not sufficient alone; you still use firewalls.

## Security Implications

### Cryptographic Weaknesses

- **MD5 collision attacks**: Demonstrated in 2004 by Xiaoyun Wang et al., and exploited in practice in the **Flame malware (2012)**, which used a forged Microsoft code-signing certificate generated via an MD5 collision. This allowed Flame to pass Windows Update as legitimate.
- **SHA-1 SHAttered attack (2017)**: Google researchers demonstrated the first practical SHA-1 collision, producing two different PDF files with the same SHA-1 hash. This invalidated millions of TLS certificates and git object integrity. CVE tracking was not applicable but CAs immediately began revoking SHA-1 certificates.
- **RSA key generation flaws (ROCA, CVE-2017-15361)**: A vulnerability in Infineon TPM chips caused weak RSA key generation, affecting millions of government smart cards, laptops (including Lenovo, HP), and YubiKey 4 tokens. Keys as large as 2048-bit were factorable within hours.

### Authentication Attacks

- **Pass-the-Hash (PtH)**: Attackers capture NTLM password hashes from memory (using tools like [[Mimikatz]]) and use them to authenticate without knowing the plaintext password. This bypasses the knowledge factor entirely.
- **Kerberoasting**: Attackers request Kerberos service tickets for accounts with SPNs, then crack them offline. Targets service accounts with weak passwords.
- **MFA Bypass via Prompt Bombing**: Attacker floods victim with MFA push notifications until they accidentally approve one (also called MFA fatigue). Used in the **Uber breach (2022)** to compromise internal systems.

### Control Failures

- **Over-reliance on detective controls**: Organizations that deploy IDS/SIEM without corresponding preventive controls create detection-only environments where breaches occur and are *noticed* but not *stopped*.
- **Misconfigured compensating controls**: Using IP allowlisting as a compensating control for lack of MFA fails when attackers compromise an allowlisted IP.

## Defensive Measures

### Cryptographic Hygiene
- **Deprecate MD5 and SHA-1** in all systems. Audit using `openssl s_client` or SSL Labs scanner.
- Use **AES-256-GCM** for symmetric encryption (authenticated encryption — provides both confidentiality and integrity).
- Use **RSA-2048 minimum** or prefer **ECC P-256/P-384** for new systems (smaller keys, equivalent security, faster operations).
- Store passwords using **bcrypt, scrypt, or Argon2** — never MD5, SHA-1, or unsalted SHA-256.

```bash
# Audit TLS configuration on a target server
openssl s_client -connect example.com:443 -tls1
# Should fail if server correctly rejects TLS 1.0

# Check certificate signature algorithm
openssl s_client -connect example.com:443 2>/dev/null | openssl x509 -noout -text | grep "Signature Algorithm"
```

### Authentication Hardening
- Enforce **MFA on all privileged accounts** and all remote access (VPN, SSH, RDP) using FIDO2/WebAuthn where possible.
- Implement **account lockout policies**: 5 failed attempts → 30-minute lockout.
- Use **privileged access workstations (PAWs)** for administrative tasks.
- Disable NTLM where possible; enforce **NTLMv2 minimum** if required.
- Implement **Just-In-Time (JIT) access** for administrative privileges — time-limited, approval-based elevation.

### Access Control Policies
- Apply **principle of least privilege**: users receive minimum permissions needed for their role.
- Conduct **quarterly access reviews** to revoke stale permissions.
- Implement **separation of duties**: no single person should control an entire sensitive process (e.g., the person who creates a payment should not also approve it).
- Deploy **privileged identity management (PIM)** solutions (e.g., CyberArk, Azure PIM) for service accounts.
