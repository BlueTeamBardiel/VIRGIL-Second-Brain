# Anti-Forgery Token

An **anti-forgery token** (also called a **CSRF token**, **synchronizer token**, or **request verification token**) is a unique, unpredictable value generated by a web application and embedded in forms or request headers to prevent [[Cross-Site Request Forgery]] (CSRF) attacks. It ensures that state-changing requests originate from the legitimate application's own pages rather than from a malicious third-party site exploiting an authenticated user's [[Session Management|session]]. Although classified here under "attack" because it directly relates to CSRF exploitation, the token itself is a defensive primitive — its absence or mishandling is what enables the attack.

---

## Overview
Anti-forgery tokens exist because [[HTTP]] is a stateless protocol in which browsers automatically attach ambient authentication credentials — [[Cookies]], [[HTTP Basic Authentication]] headers, client certificates, or [[Kerberos]]/NTLM tickets — to every request to a given origin, regardless of which page initiated the request. This means that if a user is authenticated to `bank.example.com` and then visits `evil.example`, the attacker's page can issue a cross-origin request (via an HTML form POST, an `<img>` tag, or `fetch()` with credentials) that the browser will happily send with the victim's session cookie attached. Without an additional per-request secret that the attacker cannot read, the bank has no way to distinguish a forged request from a legitimate one.

The **Synchronizer Token Pattern**, formalized by the [[OWASP]] CSRF Prevention Cheat Sheet, is the canonical countermeasure. The server issues a cryptographically random token tied to the user's session, embeds it in every form it serves, and requires that the token be echoed back on every state-changing request. Because the attacker's page cannot read the token (it lives on a different origin and is protected by the [[Same-Origin Policy]]), forged requests arrive without a valid token and are rejected.

Anti-forgery tokens are now deeply integrated into mainstream web frameworks: ASP.NET Core's `AntiForgeryToken`, Django's `{% csrf_token %}`, Ruby on Rails' `protect_from_forgery`, Laravel's `@csrf` directive, and Spring Security's `CsrfFilter` all implement variants of the same pattern. They remain relevant even in the era of [[SameSite Cookie]] attributes, which provide defense-in-depth but do not fully replace tokens due to edge cases involving `SameSite=None` cookies, subdomain trust, and older browsers.

Historically, CSRF has been a top-ten web vulnerability since the OWASP Top Ten's inception and has been implicated in real-world incidents including the 2008 attacks on home router configuration interfaces (ING Direct, YouTube, and Netflix all had CSRF flaws disclosed between 2007–2009) and persistent issues in IoT and embedded device management consoles.

## How It Works
The **Synchronizer Token Pattern** operates as follows:

1. **Token generation.** When the user authenticates or first requests a form, the server generates a cryptographically random value (typically 128–256 bits from a CSPRNG such as `/dev/urandom` or `RandomNumberGenerator`).
2. **Token storage.** The token is stored server-side, bound to the user's session (session ID → token mapping), or encoded in a signed/encrypted cookie (the **Double-Submit Cookie** variant).
3. **Token delivery.** The token is embedded in rendered HTML, usually as a hidden form field or a `meta` tag readable by JavaScript:
   ```html
   <form method="POST" action="/transfer">
     <input type="hidden" name="__RequestVerificationToken"
            value="CfDJ8Nf...a9Q" />
     <input name="amount" />
     <input name="to_account" />
   </form>
   ```
4. **Token submission.** On submit, the browser sends the token back as a form field or custom header (`X-CSRF-Token`, `X-XSRF-TOKEN`).
5. **Token validation.** The server compares the submitted token to the stored/signed value using a **constant-time comparison** to prevent timing attacks. Mismatched or missing tokens cause the request to be rejected (usually `403 Forbidden`).

Three major implementation variants exist:

- **Synchronizer Token Pattern** — token stored server-side in session state. Most robust but requires server state.
- **Double-Submit Cookie** — token sent in both a cookie and a request parameter/header; server verifies they match. Stateless, but vulnerable if subdomains can set cookies (see [[Cookie Tossing]]).
- **Signed Double-Submit** — the cookie value is an HMAC of the session ID, preventing cookie injection forgeries.

Example in Django:
```python
# settings.py enables CsrfViewMiddleware by default
# Template:
<form method="post">{% csrf_token %}...</form>
# AJAX:
fetch('/api/', {
  headers: {'X-CSRFToken': getCookie('csrftoken')},
  credentials: 'same-origin'
})
```

Example in ASP.NET Core:
```csharp
services.AddAntiforgery(opts => opts.HeaderName = "X-CSRF-TOKEN");
// In Razor: @Html.AntiForgeryToken()
// On controller: [ValidateAntiForgeryToken]
```

Modern implementations also bind tokens to the authenticated user and rotate them on privilege change (login, logout, password reset) to prevent **session fixation + CSRF** chains.

## Key Concepts
- **Synchronizer Token Pattern**: Server-stored random token matched against a token submitted with each state-changing request.
- **Double-Submit Cookie**: Stateless variant where the same token appears in a cookie and in the request body/header; server verifies equality.
- **Constant-Time Comparison**: Use of `hmac.compare_digest` or equivalent to prevent [[Timing Attack|timing side channels]] during token validation.
- **Origin/Referer Header Check**: A complementary defense verifying that the `Origin` or `Referer` header matches the expected host; often used alongside tokens.
- **Token Entropy**: Tokens must be drawn from a [[CSPRNG]] with ≥128 bits of entropy; predictable tokens (counters, timestamps) defeat the defense.
- **Token Scope**: Tokens should be bound to the user's session and, ideally, to the specific action or form to prevent replay across endpoints.
- **Token Rotation**: Issuing a new token on login, logout, and privilege elevation to limit the window of a stolen token.

## Exam Relevance
For **SY0-701**, anti-forgery tokens appear under the **Application Security** and **Secure Coding Practices** objectives (primarily 2.3 and 4.1). Expect:

- Recognizing CSRF as the attack that hidden form tokens defend against — distinct from [[Cross-Site Scripting|XSS]] (which tokens do **not** stop; XSS defeats CSRF tokens because attacker JS can read them).
- Knowing that **input validation**, **output encoding**, and **anti-forgery tokens** are distinct controls addressing distinct classes of flaw.
- Pairing defenses: tokens + [[SameSite Cookie]] + `Secure` + `HttpOnly` + Origin checks = layered defense.
- Gotcha: a CSRF token inside a page vulnerable to XSS is **useless** — XSS always trumps CSRF defenses.
- Gotcha: GET requests should never perform state changes; tokens are typically only validated on POST/PUT/DELETE.

## Security Implications
When anti-forgery tokens are missing, misconfigured, or validated incorrectly, CSRF becomes trivially exploitable. Real-world failure modes include:

- **Token not validated on sensitive endpoints** — e.g., CVE-2020-13777 was a separate issue, but countless CVEs (e.g., **CVE-2018-0114** in Cisco's node-jose, **CVE-2021-22117** in RabbitMQ management UI) stem from routes that skip token checks.
- **Token leaked via Referer header** — tokens embedded in URLs leak to third-party sites via `Referer`.
- **Predictable tokens** — early Rails versions (<2.x) used weak token generation.
- **XSS bypass** — any [[Cross-Site Scripting]] defeats the token because attacker-controlled JS on the same origin can read it. The 2010 Twitter "onMouseOver" worm demonstrated how XSS bypasses all CSRF protections.
- **Subdomain takeover enabling cookie tossing** — attackers with control of `evil.victim.com` can set cookies for `victim.com`, defeating Double-Submit Cookie without HMAC binding.
- **Login CSRF** — even without tokens being leaked, an attacker can log a victim into the attacker's account to capture subsequent actions; defended by token checks on the login endpoint itself.

Tools to identify missing or weak tokens include **Burp Suite** (CSRF PoC generator), **OWASP ZAP** (CSRF scanner), and **Nikto**. Automated scanners flag `POST` endpoints that accept requests without tokens or with predictable token patterns.

## Defensive Measures
- **Enable framework CSRF middleware globally** and whitelist explicitly rather than opt-in per route.
- **Use `SameSite=Lax`** (or `Strict` where UX permits) on session cookies as a second layer — modern browsers default to `Lax`.
- **Enforce `Secure` and `HttpOnly`** on session cookies; use `__Host-` prefix for domain-locked cookies.
- **Validate `Origin`/`Referer` headers** on state-changing requests and reject mismatches.
- **Reject ambiguous content types** on sensitive endpoints; require `application/json` with a preflighted custom header to leverage [[CORS]] preflight as an implicit CSRF defense.
- **Rotate tokens** on login, logout, password change, and MFA elevation.
- **Pen-test regularly** — run [[Burp Suite]]'s scanner and manual CSRF PoC generation against every state-changing form.
- **Pair with strong [[Content Security Policy]]** to reduce XSS risk that would otherwise defeat tokens.
