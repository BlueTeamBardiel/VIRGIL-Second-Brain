# III. Malicious Activity
**Understanding how to detect, analyze, and respond to suspicious behavior across your security infrastructure**

---

## Overview

Malicious activity rarely announces itself with a single smoking gun—instead, it leaves breadcrumbs scattered across logs, network traffic, and user behavior patterns. Security analysts hunt for these breadcrumbs by correlating events, analyzing deviations from normal operations, and understanding how attackers typically behave. You need to master detection techniques because your organization's response speed depends entirely on how quickly you spot the threat.

---

## Key Concepts

### Log Analysis and Event Correlation

**Analogy**: Imagine you're a detective investigating a robbery. A single eyewitness report is interesting, but when you cross-reference it with security camera timestamps, credit card transactions, and phone records, the full picture emerges. That's log correlation.

**Definition**: [[Log Analysis]] involves collecting events from systems, applications, and infrastructure, then identifying patterns—especially when multiple events tie together—to determine scope (is this isolated or enterprise-wide?) and impact (what data or systems are affected?).

**Core Analysis Tasks**:
- Identify which [[systems]] and [[users]] were involved
- Classify the [[data]] potentially exposed
- Detect beaconing or [[command-and-control]] patterns
- Map event sequences to attack timelines

**Tools and Methods**:

| Method | Use Case | Analyst Value |
|--------|----------|---|
| [[SIEM]] | Real-time correlation across thousands of events | Automated alerting and pattern detection |
| Log Aggregation (ELK, Splunk) | Centralized storage and search | Faster investigation, better retention |
| Manual Review | Focused investigation of specific incidents | Deep understanding, catches edge cases |
| Trend Analysis | Baseline vs. anomaly comparison | Spots slow, creeping attacks |

---

### Application and Service Monitoring

**Analogy**: Your server is like a worker on a factory floor. You don't just care if they show up (uptime)—you care if they're working fast, handling transactions correctly, and not acting strangely.

**Definition**: [[Application Monitoring]] tracks multiple dimensions: whether services stay online, how quickly they respond, whether transactions complete successfully, and what actions applications perform.

**What Analysts Actually Monitor**:
- **Uptime/downtime events** → Detect DoS or crashes
- **Performance metrics** (CPU, memory, latency) → Spot resource exhaustion attacks
- **Transactional logs** → Identify failed authentication, permission denials, or data access anomalies
- **Application-level logging** → Catch malicious behavior at the code level

Deviations from baseline behavior are your first alert that something is wrong.

---

### Application and Service Anomaly Detection

**Analogy**: Most car check-engine lights are false alarms (bad sensor, loose gas cap), but sometimes they mean your engine is failing. You need to distinguish the noise from the real threat.

**Definition**: [[Anomaly Detection]] identifies unusual application behavior, which can stem from innocent causes (authentication failures, permission issues, failed dependencies) or malicious causes (exploitation, backdoors, malware).

**Common Non-Malicious Causes**:
- Failed authentication attempts (typos, expired credentials)
- Missing or updated dependencies
- Permission configuration errors
- Startup failures or process crashes

**Malicious Behavior Requires**:
- [[Antivirus]] and [[EDR]] (Endpoint Detection and Response)
- [[File Integrity Monitoring]] to detect unauthorized changes
- Allow-listing to prevent unauthorized executables
- Behavioral heuristics to catch unknown threats

**Critical Truth**: Logs are your forensic record. Without them, you're investigating blind.

---

### Application Logs

**Analogy**: Your application logs are like a flight recorder on an airplane—useless until something goes wrong, but invaluable when it does.

**Definition**: [[Application Logs]] are text records generated by software that document operational events, errors, and user actions.

**Location by Platform**:
- **[[Linux]]**: Typically `/var/log/` (application-specific subdirectories like `/var/log/apache2/`)
- **Windows**: Event Viewer (built-in) or forwarded to centralized log aggregators
- **Cloud/SaaS**: Often API-based or cloud-native logging services

**Critical Best Practices**:
- Enable logging *before* an incident occurs (you can't retroactively collect deleted logs)
- Protect logs with write-once storage or [[SIEM]] archival to prevent attacker tampering
- Centralize critical logs so an attacker can't delete local copies
- Set retention policies based on compliance and investigation depth needs

### Application Behavior Analysis

**Analogy**: You know your friend's daily routine—they always leave for work at 8 AM, grab coffee at the same place, and call their mom on Sundays. One day they're buying plane tickets to Argentina at 3 AM with borrowed credit cards. Something's wrong.

**Definition**: [[Behavior Analysis]] compares current application activity against a baseline of normal operation to identify compromise indicators.

**Compromise Indicators to Hunt For**:
- **Unexpected outbound traffic** → C2 beaconing, data exfiltration
- **New files or accounts** → Persistence mechanisms, backdoors
- **Abnormal resource consumption** → Cryptominers, DDoS bots
- **Process execution from unusual paths** → Malware execution
- **Unusual API calls** → Exploitation attempts

**Detection Requires**:
- Pre-established baseline data on normal behavior
- Comprehensive logging at the process/system level
- Heuristic or behavior-based detection tools (not just signature-based [[AV]])
- Analyst skill at recognizing subtle deviations

**Key Insight**: Anomalous behavior is often the *first* indicator of compromise, appearing before malware signatures or network IOCs become visible.

### New Account Creation

**Analogy**: If an attacker breaks into a house, the first thing they might do is create a duplicate key so they can return later. New accounts are attackers' duplicate keys.

**Definition**: [[Account Creation]] monitoring detects when new user or service accounts are provisioned, a common [[persistence]] technique used by attackers.

**Red Flags**:
- New privileged or admin accounts
- Bulk account creation (suggests automation or scripting)
- Accounts created at unusual times (3 AM on a Sunday)
- Accounts created from unusual geographic locations (VPNs, compromised systems)
- Service accounts with console logon permissions

**Scope**: Applies to both on-premises systems and cloud applications (AWS IAM, Azure, Google Workspace).

**Exam Context**: The CySA+ treats this as [[application-related]] detection, not just directory services.

### Email Security

**Analogy**: Email is like your front door—it's where most attackers knock, often with malware or spoofed packages. You need multiple locks and verification systems.

**Definition**: [[Email Security]] prevents unauthorized senders from spoofing your domain and delivers malicious attachments to quarantine instead of inboxes.

**Core Authentication Technologies**:

| Protocol | Function | How It Works |
|----------|----------|---|
| **[[SPF]]** (Sender Policy Framework) | Restricts who can send email from your domain | DNS TXT record lists authorized mail servers |
| **[[DKIM]]** (DomainKeys Identified Mail) | Cryptographically verifies email hasn't been altered | Sender signs email with private key; recipient verifies with public key |
| **[[DMARC]]** (Domain-based Message Authentication, Reporting, and Conformance) | Enforces SPF/DKIM results and reports violations | Tells receiving server: "If SPF/DKIM fails, quarantine or reject" |

**DMARC Best Practice Deployment**:
1. Start in `monitoring` mode (`p=none`) — log violations but don't block
2. Analyze DMARC reports for 2-4 weeks
3. Move to `quarantine` mode (`p=quarantine`) — suspicious mail goes to spam
4. Finally, enforce strict rejection (`p=reject`) — only approved senders deliver

**Technical Constraints**: SPF records have a 255-character limit, so complex organizations must use `SPF includes` to reference multiple policy files.

### File Analysis

**Analogy**: A suspicious USB drive might look innocent until a forensic examiner plugs it in and sees what data it's trying to copy. File analysis is that examination process.

**Definition**: [[File Analysis]] examines suspicious files for malicious code, obfuscation, or behavior indicators without fully executing them in production.

**Common Attacker Techniques**:
- **Packing**: Compressing executable to evade detection
- **Encryption**: Hiding malicious code inside encrypted containers
- **Polymorphism**: Malware that changes its binary signature on each execution

**Analysis Methods**:

| Technique | What It Does | Speed | Depth |
|-----------|---|---|---|
| **Hash Comparison** (MD5, SHA-256) | Compares file hash against known malware databases | Fast | Shallow (only catches known malware) |
| **Strings Analysis** | Extracts readable text from binary files to spot suspicious URLs, registry keys, or commands | Fast | Medium (reveals intent without execution) |
| **Sandboxing** | Detonates file in isolated environment and observes behavior | Slow | Deep (reveals actual malicious behavior) |

**Hash Properties** (Why they matter):
- Fixed length (SHA-256 is always 64 hex characters)
- One-way (can't reverse a hash to get original file)
- Deterministic (same input always produces same hash)
- Collision-resistant (two different files won't have same hash)

**Exam Scope**: CySA+ expects only basic file analysis concepts; real-world incident response goes much deeper.

### Sandboxing

**Analogy**: Giving a suspicious toy to a robot in a locked room, watching what it does, then destroying the room if it turns out to be dangerous.

**Definition**: [[Sandboxing]] executes potentially malicious files in an isolated, monitored environment that's completely disconnected from production systems.

**What Analysts Observe in a Sandbox**:
- **Behavioral indicators** → What system calls does the malware make?
- **Network traffic** → Does it beacon to C2 servers or attempt data exfiltration?
- **File system changes** → What files does it create, modify, or delete?
- **Registry modifications** (Windows) → How does it establish persistence?
- **API calls** → Which Windows/Linux APIs does it invoke?

**Common Sandbox Tools**:
- **Joe Sandbox** → Commercial, comprehensive reporting
- **Cuckoo Sandbox** → Open-source, self-hosted option
- **VirusTotal** → Free online service (may use multiple AV engines + sandboxing)

**Analyst Value**: Sandboxing provides behavioral proof without risking your network, but requires time to set up and analyze results.

### User Behavior Analysis (UEBA)

**Analogy**: Your bank notices you usually spend $50 a day locally but suddenly $5,000 is being transferred to North Korea. That deviation triggers fraud alerts.

**Definition**: [[UEBA]] (User and Entity Behavior Analytics) establishes baselines for normal user actions, then flags deviations that suggest account compromise or insider threats.

**Behavioral Red Flags**:
- **Off-hours logins** → Access from 3 AM when user typically works 9-5
- **Unusual geographic locations** → Login from Tokyo when user is known to work from New York
- **Impossible travel** → Logon in New York, then logon in London 30 minutes later (impossible without supersonic flight)
- **Unexpected privilege escalation** → Junior analyst suddenly running administrative queries
- **Data access anomalies** → User accessing folders they've never touched before
- **Mass file downloads or emails** → Exfiltration indicators

**How UEBA Tools Work**:
1. Establish statistical baseline of normal behavior (first 30-90 days)
2. Apply machine learning to detect deviations
3. Flag anomalies with risk scores
4. Alert analysts to investigate

**Critical Limitation**: UEBA requires data, so new users or newly privileged accounts won't be protected until baselines build.
