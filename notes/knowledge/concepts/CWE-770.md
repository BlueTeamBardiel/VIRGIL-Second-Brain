# CWE-770

## What it is
Like a hotel that hands out room keys to every guest who asks — without checking reservations or limiting how many rooms one person can claim — this weakness occurs when software allocates resources (memory, file handles, threads, connections) without any limit or throttling. Precisely: CWE-770 is the failure to restrict how much of a resource an actor can consume, enabling resource exhaustion attacks.

## Why it matters
In 2016, the Mirai botnet exploited IoT devices running services with no connection-rate limits, flooding targets with TCP SYN requests until their connection tables filled completely — legitimate traffic was denied without any vulnerability in the protocol itself, just absent allocation controls. A simple fix — capping concurrent connections per source IP — would have dramatically reduced impact.

## Key facts
- **Not the same as a memory leak (CWE-401):** CWE-770 is about missing *limits*, not unreleased memory; an attacker deliberately triggers the exhaustion.
- **Affects multiple resource types:** CPU cycles, sockets, threads, file descriptors, database connections, and disk space all qualify.
- **Common attack vector:** Unauthenticated endpoints with no rate limiting are the classic trigger — think login pages, file upload APIs, or WebSocket endpoints.
- **Defense pattern:** Implement quotas, caps, and timeouts at the application layer (e.g., `MAX_CONNECTIONS`, request throttling middleware) and infrastructure layer (firewall rate limits).
- **CVSS impact:** Typically scores as Availability impact only, but can chain into Confidentiality/Integrity loss if resource starvation causes fallback to insecure modes.
