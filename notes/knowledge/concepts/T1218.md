# T1218

## What it is
Like a criminal using a police uniform to walk past security checkpoints unchallenged, attackers use Windows' own trusted, signed system binaries to execute malicious payloads — bypassing application whitelisting entirely. T1218 (System Binary Proxy Execution) describes the technique of leveraging legitimate, Microsoft-signed utilities to run attacker-controlled code, making malicious activity appear as normal system operations.

## Why it matters
In the 2020 SolarWinds-era lateral movement playbook, attackers used `mshta.exe` to execute remote HTA files containing VBScript payloads — a classic T1218 sub-technique. Because `mshta.exe` is a signed Microsoft binary, traditional AV and application whitelisting tools allowed execution without alerting, giving attackers persistent access while blending into normal Windows telemetry.

## Key facts
- **Sub-techniques include**: `mshta.exe`, `rundll32.exe`, `regsvr32.exe` (Squiblydoo), `certutil.exe`, `msiexec.exe`, and `wscript.exe` — each abuses a legitimate binary
- **Regsvr32 (Squiblydoo)** is particularly notorious: it can fetch and execute a COM scriptlet from a remote URL with zero registry footprint
- **Defense evasion** is the primary MITRE tactic — these binaries bypass application whitelisting (AppLocker, WDAC) because they are Microsoft-signed
- **Detection focus**: Look for anomalous parent-child process relationships (e.g., `Word.exe` spawning `mshta.exe`) and unexpected network connections from these binaries
- **Mitigation**: Use Windows Defender Application Control (WDAC) policies to block unnecessary LOLBins; enable Script Block Logging (PowerShell) and Sysmon Event ID 1 for process creation monitoring
