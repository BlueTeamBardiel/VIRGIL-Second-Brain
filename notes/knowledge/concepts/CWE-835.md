# CWE-835

## What it is
Like a washing machine that keeps spinning because the sensor checking "is the cycle done?" never actually triggers the stop condition, a loop without exit is code that iterates indefinitely because the termination condition is unreachable, always false, or never updated. CWE-835 describes infinite loops that consume CPU and memory until the process hangs or crashes.

## Why it matters
In 2016, a crafted malicious packet sent to certain Cisco IOS implementations triggered an infinite parsing loop, causing routers to exhaust CPU resources and drop all traffic — a textbook Denial of Service without any memory corruption. Defenders look for this in code review by ensuring every loop has a reachable, guaranteed-to-be-met exit condition, and that external input cannot manipulate loop counters or conditions.

## Key facts
- **DoS impact category**: CWE-835 is classified under Availability impact — it doesn't leak data or grant access, it *kills uptime*
- **Common trigger**: Attacker-controlled input that sets a loop bound to a value the termination condition can never satisfy (e.g., counter increments but condition checks a flag that's never flipped)
- **Parser/protocol code is highest risk**: Network parsers, file format readers, and state machines frequently iterate over attacker-supplied data, making them prime targets
- **Distinct from CWE-400 (Uncontrolled Resource Consumption)**: CWE-835 specifically involves a loop structure; CWE-400 covers resource exhaustion more broadly
- **Detection**: Static analysis tools (SAST) flag loops where loop variables are not modified within the loop body or where exit conditions depend on external state that's never written
