# CWE-639

## What it is
Imagine a hotel where room numbers are sequential — if you have Room 101, you can guess Room 102 belongs to someone else and simply try the door. CWE-639 (Authorization Bypass Through User-Controlled Key) occurs when an application uses a predictable or user-manipulable value (like a numeric ID in a URL) as the sole access control mechanism, allowing attackers to access other users' resources by simply changing that value.

## Why it matters
In 2021, an IDOR vulnerability in Parler's API allowed researchers to enumerate and download millions of posts and user data in sequential order before the platform went offline — the API accepted user-supplied numeric IDs with no ownership verification. This is a textbook CWE-639 exploit: change the number, get someone else's data. Proper server-side authorization checks (verify the requesting user *owns* the resource) would have prevented the breach entirely.

## Key facts
- CWE-639 is a subtype of **Insecure Direct Object Reference (IDOR)**, which appears on the OWASP Top 10 under **Broken Access Control (A01:2021)**
- The core flaw is **missing server-side authorization** — the application authenticates the user but never verifies they're *authorized* for the specific object requested
- Common attack vectors include manipulating URL parameters (`?invoice_id=1042`), POST body fields, cookies, or API path segments
- Defense requires **indirect reference maps** (random UUIDs instead of sequential integers) combined with server-side ownership validation — one alone is insufficient
- Testing method: authenticate as User A, capture a resource URL, then access it while authenticated as User B — successful access confirms the vulnerability
