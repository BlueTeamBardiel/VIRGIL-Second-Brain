# CWE-799

## What it is
Like a bouncer who lets anyone into a club as long as they show *any* ID — even an expired one from a different country — this weakness occurs when software fails to properly control the *frequency or rate* of interactions it allows. Precisely: CWE-799 is Improper Control of Interaction Frequency, where an application does not enforce limits on how often an actor can perform an action, enabling abuse through repetition.

## Why it matters
A login portal vulnerable to CWE-799 has no lockout or throttling mechanism, allowing an attacker to run a credential-stuffing or brute-force attack — automatically cycling through thousands of username/password pairs per second without triggering any defense. This was a contributing factor in the 2016 GitHub brute-force incident, where attackers exploited rate-limitless endpoints to test stolen credential lists at scale.

## Key facts
- CWE-799 is the parent of CWE-307 (Improper Restriction of Excessive Authentication Attempts) — knowing the hierarchy matters for exam questions
- Mitigations include rate limiting, CAPTCHA, exponential backoff, and account lockout policies
- Distinguished from CWE-400 (Uncontrolled Resource Consumption) — 799 focuses on *interaction frequency by an actor*, not resource exhaustion per se
- APIs are a primary attack surface: without rate limiting, attackers can enumerate users, scrape data, or flood endpoints at machine speed
- NIST SP 800-63B recommends limiting failed authentication attempts as a baseline control, directly addressing this weakness
