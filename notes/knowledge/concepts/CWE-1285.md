# CWE-1285

## What it is
Like a bouncer who checks your ID but never verifies the ID is actually *valid* — just that one exists — improper validation of specified index, position, or offset means code accepts a value without confirming it falls within a legitimate range. Precisely: CWE-1285 occurs when software receives an index or offset but fails to verify it is within expected bounds before using it to access memory or data structures.

## Why it matters
An attacker supplying a negative index or an oversized offset to an array access can read or overwrite memory outside the intended buffer — a classic path to both information disclosure and arbitrary code execution. This pattern appeared in numerous CVEs affecting media parsers (like libjpeg and FFmpeg), where malformed image files supplied crafted offsets that caused out-of-bounds reads, leaking heap memory contents including pointers that defeat ASLR.

## Key facts
- Distinct from CWE-129 (improper validation of array index) — CWE-1285 is the *parent* class covering any index, position, or offset, not just array indices
- Root cause is almost always missing boundary checks: no `if (offset < 0 || offset >= max)` guard before use
- Consequences include out-of-bounds read (CWE-125), out-of-bounds write (CWE-787), and buffer overflow — all high-severity findings
- Commonly exploited in file parsers, network packet handlers, and codec libraries where external input directly controls offset calculations
- Defense: always validate that `0 ≤ index < collection_size` *before* dereferencing; use memory-safe languages (Rust, Go) or compiler sanitizers (ASan, UBSan) to catch violations at runtime
