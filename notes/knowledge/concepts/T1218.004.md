# T1218.004

## What it is
Like hiring a licensed contractor to smuggle contraband past a checkpoint, attackers use `InstallUtil.exe` — Microsoft's trusted .NET installer utility — to execute malicious code that bypasses application whitelisting controls. It is a signed-binary proxy execution technique where the CLR (Common Language Runtime) is abused through `InstallUtil`'s `[RunInstaller(true)]` attribute to invoke attacker-controlled code embedded in a specially crafted .NET assembly.

## Why it matters
In documented APT campaigns, threat actors have dropped a malicious `.exe` compiled as a .NET installer class, then executed it via `InstallUtil.exe /logfile= /LogToConsole=false /U malicious.exe` — the `/U` flag triggers the `Uninstall()` method, running the payload while leaving minimal traces. Defenders monitoring only for unsigned binaries would miss this entirely since `InstallUtil.exe` lives in `C:\Windows\Microsoft.NET\Framework\` and carries Microsoft's signature.

## Key facts
- **Location:** `C:\Windows\Microsoft.NET\Framework[64]\<version>\InstallUtil.exe` — legitimately signed by Microsoft
- **Bypass mechanism:** Executes code through the `Installer` class's `Install()` or `Uninstall()` methods, not direct PE execution — evades many application whitelisting tools like older AppLocker configurations
- **Key flag:** `/U` (uninstall) is commonly used in attacks because it calls `Uninstall()` without requiring an actual installation context
- **Detection pivot:** Monitor process creation events for `InstallUtil.exe` spawning unexpected child processes, or network connections originating from it (unusual for a legitimate installer tool)
- **MITRE category:** Falls under Defense Evasion (TA0005) and is part of the broader "Signed Binary Proxy Execution" (T1218) technique family
