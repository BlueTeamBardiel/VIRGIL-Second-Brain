# CWE-120

## What it is
Imagine pouring a gallon of milk into a pint glass — the overflow spills everywhere, potentially ruining everything nearby. CWE-120, "Buffer Copy without Checking Size of Input," is exactly that: a program copies data into a fixed-size buffer without first verifying the input won't exceed the buffer's capacity. The overflow corrupts adjacent memory, which attackers can weaponize to hijack program execution.

## Why it matters
The 1988 Morris Worm exploited a buffer overflow in `fingerd` by sending an oversized string to a `gets()` call — no bounds checking, instant shell. Modern exploits still target this pattern; a classic attack overwrites the saved return address on the stack with a pointer to attacker-controlled shellcode, redirecting execution flow entirely. Defenders mitigate this with compiler protections like stack canaries (`-fstack-protector`) and ASLR, but vulnerable C code without these controls remains a direct path to remote code execution.

## Key facts
- **Root cause**: Using unsafe C functions like `gets()`, `strcpy()`, `sprintf()` that perform no length validation before copying.
- **Consequences**: Can lead to arbitrary code execution, privilege escalation, or denial of service — ranked critically in CVSS scoring.
- **Classic attack primitive**: Overwriting the saved return address (RIP/EIP) on the stack to redirect execution to attacker-supplied code.
- **Mitigations**: Use safe alternatives (`strncpy`, `snprintf`, `strlcpy`); enable compiler defenses (stack canaries, ASLR, NX/DEP bits).
- **Prevalence**: Consistently appears in OWASP and CWE Top 25 Most Dangerous Software Weaknesses; foundational to understanding memory corruption vulnerabilities.
