# CWE-457

## What it is
Like a chef who grabs a pan without checking if it still has yesterday's oil — using it as if it's clean — a program reads a variable before ever writing a value to it. CWE-457 is the use of uninitialized variables: memory locations accessed before being explicitly assigned a value, potentially containing whatever garbage bytes happened to occupy that address.

## Why it matters
In 2017, researchers demonstrated that uninitialized stack memory in Linux kernel drivers could leak sensitive kernel memory contents to unprivileged user-space processes, enabling KASLR bypass. An attacker who can trigger a code path that reads uninitialized heap or stack memory may recover cryptographic keys, session tokens, or pointer addresses that break exploit mitigations like ASLR.

## Key facts
- Uninitialized stack variables inherit whatever data was previously written to that memory by earlier function calls — this is not random, it's **attacker-influenceable** in some contexts
- This weakness is distinct from a NULL pointer dereference (CWE-476); the variable has *some* value, just not a *known safe* one
- Languages like C and C++ are primary targets; Java and Python initialize variables by default, making this largely a memory-unsafe language problem
- Exploitation can enable **information disclosure** (reading stale sensitive data) or **logic corruption** (branching on a garbage boolean value)
- Static analysis tools (Coverity, Clang's `-Wuninitialized`) can detect many instances at compile time, making this a preventable vulnerability class
