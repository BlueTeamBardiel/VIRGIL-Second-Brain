# T1098.004

## What it is
Like a thief who doesn't pick the lock but instead gets a copy of their key added to the landlord's approved list — SSH Authorized Keys manipulation is when an attacker adds their own public key to a user's `~/.ssh/authorized_keys` file, granting themselves persistent, password-free remote access. This technique falls under Account Manipulation and enables durable backdoor access that survives password resets.

## Why it matters
In the 2021 TeamTNT cryptomining campaign, attackers compromised cloud instances and immediately appended their SSH public keys to root's `authorized_keys` file, ensuring re-entry even after victims patched the initial vulnerability. Defenders who only reset passwords completely missed the persistence mechanism because the key remained untouched.

## Key facts
- The `authorized_keys` file lives at `~/.ssh/authorized_keys` (or `/root/.ssh/authorized_keys` for root) and lists public keys permitted to authenticate without a password
- An attacker only needs **write access to the file once** — the inserted key then provides unlimited future logins
- This technique is nearly invisible to password-based monitoring since no credentials are ever transmitted
- Detection relies on file integrity monitoring (FIM) tools like AIDE or auditd watching for modifications to `authorized_keys`
- Defenders should enforce `AuthorizedKeysFile` to a centrally managed, root-owned path in `sshd_config`, preventing user-writable key files
- Cloud environments (AWS, GCP) are high-value targets because instances often run with root SSH enabled by default
