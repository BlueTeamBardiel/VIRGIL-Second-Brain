# RFC 5568

## What it is
Like a relay race where a runner hands off the baton mid-stride without breaking pace, RFC 5568 defines **Mobile IPv6 Fast Handovers (FMIPv6)** — a protocol extension that allows a mobile node to pre-establish a new care-of address and tunnel packets *before* completing a network handoff, minimizing disruption to active connections.

## Why it matters
In a scenario where a VoIP call is active on a mobile device roaming between access points, standard Mobile IPv6 handovers introduce latency and packet loss that break the session. FMIPv6 mitigates this by pre-signaling the target access router, but the early binding update messages it sends can be exploited by an attacker performing a **redirect attack** — intercepting the handover process to reroute a victim's traffic through a malicious node before authentication completes.

## Key facts
- FMIPv6 operates by exchanging **RtSolPr (Router Solicitation for Proxy)** and **PrRtAdv (Proxy Router Advertisement)** messages to discover the target router in advance
- Two modes exist: **predictive mode** (handover completed before moving) and **reactive mode** (handover completed after moving, used as fallback)
- The **HI (Handover Initiate)** and **HAck (Handover Acknowledge)** messages coordinate tunnel setup between the Previous Access Router (PAR) and New Access Router (NAR)
- FMIPv6 does **not** replace Mobile IPv6 (RFC 6275) — it is an optimization layer on top of it
- A key vulnerability is that early care-of address binding can occur before full **IPSec/IKEv2 authentication**, creating a window for traffic hijacking
