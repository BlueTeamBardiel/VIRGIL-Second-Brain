# CWE-424

## What it is
Like a bouncer who checks IDs at the front door but leaves the fire exit wide open, this weakness occurs when a system enforces access control on one channel while leaving an alternate path completely unprotected. CWE-424 (Improper Protection of Alternate Path) describes a vulnerability where a developer secures the obvious entry point but fails to account for secondary routes to the same resource or functionality.

## Why it matters
A classic exploitation scenario: a web application properly authenticates users through `/login`, but a developer left a legacy API endpoint at `/api/v1/user/data` that returns the same sensitive records without requiring a token. An attacker who discovers this alternate path bypasses all front-end security controls entirely, accessing data as if authentication never existed. This is why security testing must enumerate *all* endpoints, not just the documented ones.

## Key facts
- Alternate paths include hidden endpoints, direct file access, debug interfaces, backup URLs, and unused API versions
- Attackers discover alternate paths through directory brute-forcing (Gobuster/Dirbuster), JavaScript source analysis, and API documentation leaks
- The fix requires centralized authorization enforcement — every path to a resource must pass through the same access control logic, not just the primary one
- This weakness is distinct from CWE-287 (Improper Authentication); CWE-424 specifically involves *bypassing* the authentication/authorization mechanism via a different route
- Common in legacy systems where new security controls were layered onto existing infrastructure without auditing all previous entry points
