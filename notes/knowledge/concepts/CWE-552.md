# CWE-552

## What it is
Like leaving your filing cabinet unlocked in the company lobby, CWE-552 occurs when a web application exposes files or directories through its web-accessible root that should never be reachable by external users. Specifically, it involves insufficient protection of files that are stored in locations accessible via a web server but lack proper access controls or should have been excluded entirely.

## Why it matters
In 2017, a misconfigured AWS S3 bucket (a classic CWE-552 manifestation) exposed Verizon customer data — roughly 14 million records — because the bucket was publicly readable with no authentication required. An attacker simply had to know the bucket URL to download sensitive call records, demonstrating that improper file storage controls can cause massive breaches without any exploitation of complex vulnerabilities.

## Key facts
- CWE-552 is distinct from path traversal (CWE-22); the files are *intentionally served* by the web server — they're just not supposed to be publicly accessible
- Common examples include: exposed `.git` directories, database backup files (`.sql`, `.bak`), configuration files (`config.php`, `.env`), and log files placed under the web root
- Exposed `.git` directories allow attackers to reconstruct full source code using tools like `git-dumper`, often revealing hardcoded credentials
- Remediation involves moving sensitive files *outside* the web root, configuring server deny rules (e.g., `.htaccess`, Nginx `deny all`), and disabling directory listing
- OWASP rates misconfigured file access as part of Security Misconfiguration (A05:2021), making CWE-552 a top-tier finding in web application assessments
