# T1073

## What it is
Like a counterfeiter who prints fake IDs that look real enough to pass a bouncer's glance, DLL Side-Loading tricks a legitimate, trusted application into loading a malicious DLL by placing it where the app expects to find a real one. Specifically, an attacker drops a crafted DLL alongside a signed executable so that Windows' DLL search order causes the malicious file to load instead of the legitimate library.

## Why it matters
In the 2017 CCleaner supply chain attack, adversaries embedded malicious code into a legitimate signed binary that loaded a rogue DLL, allowing them to compromise millions of systems while evading antivirus tools that trusted the signed parent process. Defenders learned that process integrity monitoring — not just file signature checks — is essential, because the *parent* process is legitimate even when its loaded module is not.

## Key facts
- Exploits the Windows DLL search order: current directory is checked before system directories, so a local malicious DLL wins over the legitimate system one
- The attack gains the *trust context* of the signed host application, bypassing application whitelisting and UAC in many configurations
- Commonly used by APT groups (e.g., APT41, Lazarus) to maintain persistence and evade EDR solutions
- Detection focuses on monitoring unexpected DLL loads via Sysmon Event ID 7 (Image Loaded) and looking for unsigned DLLs loaded by signed processes
- Mitigation includes using absolute paths in application manifests, enabling Safe DLL Search Mode (`HKLM\System\CurrentControlSet\Control\Session Manager\SafeDllSearchMode`), and enforcing code-signing policies
