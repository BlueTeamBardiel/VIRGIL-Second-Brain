# RFC 8259

## What it is
Think of JSON like a universal packing slip for shipping data between warehouses — everyone agrees on the box format so any warehouse can read it without confusion. RFC 8259 is the IETF standard that defines the JSON (JavaScript Object Notation) data interchange format, specifying precise syntax rules for encoding structured data as human-readable text. It obsoletes RFC 7159 and is the authoritative specification for JSON parsing and serialization.

## Why it matters
Inconsistent JSON parsing between libraries has been exploited in security-critical contexts — notably in JWT (JSON Web Token) authentication bypass attacks where an attacker sends a JSON payload with a duplicate key (e.g., two `"alg"` fields), and different parsers in a pipeline read different values, allowing algorithm confusion attacks like substituting `"none"` for `RS256`. RFC 8259 explicitly states that duplicate key behavior is undefined, which is exactly the gap attackers exploit when a gateway and backend parser disagree on which duplicate key wins.

## Key facts
- RFC 8259 defines JSON as text encoded in **UTF-8** (mandatory for network transmission), deprecating UTF-16 and UTF-32 for interchange
- The spec defines **six structural characters**: `{ } [ ] : ,` — these delimiters are common injection targets in JSON injection attacks
- RFC 8259 explicitly **does not define behavior for duplicate keys** — this ambiguity is a documented attack surface in JWT and API security
- JSON supports six value types: **string, number, object, array, true/false, null** — type confusion between these is exploited in deserialization attacks
- Maximum interoperability requires parsers to accept numbers in the range **±(2⁵³-1)** without loss, since JSON has no integer type — floating-point precision bugs can affect security logic
