# CWE-359

## What it is
Like a hospital posting a patient's medical records on a public bulletin board "just for staff convenience," this weakness occurs when software exposes private information to actors who shouldn't have access to it. CWE-359 specifically covers the exposure of private personal information — PII, health records, financial data — through insufficient access controls or unintended disclosure channels.

## Why it matters
In 2021, a misconfigured Twitch API endpoint exposed subscriber financial data and creator payout records to unauthenticated requests — a textbook CWE-359 scenario. An attacker simply crafted direct object reference requests to harvest sensitive data without any authentication bypass required, because the application failed to scope what private information each user role could retrieve.

## Key facts
- CWE-359 is a child of CWE-200 (Exposure of Sensitive Information) but specifically targets **personal private information** governed by laws like HIPAA, GDPR, and CCPA
- Common triggers include: over-permissive API responses returning full user objects, debug logs containing PII, and error messages echoing sensitive input
- Distinct from a data breach caused by SQL injection — CWE-359 often requires **no exploitation**; the data is simply accessible to unauthorized parties by design flaw
- CVSS scores for CWE-359 instances frequently land in the **Medium (4.0–6.9)** range, though regulatory fines can dwarf technical severity ratings
- OWASP maps this to **API3:2023 (Broken Object Property Level Authorization)** and **A02:2021 (Cryptographic Failures)** depending on context
