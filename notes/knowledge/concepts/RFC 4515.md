# RFC 4515

## What it is
Think of it as a specialized search syntax for a filing cabinet full of user accounts — you write a precise query ticket, hand it to the clerk, and get back exactly the records matching your criteria. RFC 4515 formally defines the **LDAP Search Filter string representation**, specifying the syntax used to query Lightweight Directory Access Protocol (LDAP) directories for entries matching specific attribute conditions. It standardizes how filters like `(uid=jsmith)` or `(&(objectClass=user)(memberOf=Admins))` are constructed and parsed.

## Why it matters
Attackers exploit improperly sanitized LDAP search filters to perform **LDAP injection attacks** — the directory equivalent of SQL injection. For example, a login form that constructs the filter `(uid=<input>)` can be subverted by entering `*)(uid=*))(|(uid=*` to collapse authentication logic and return all users, bypassing credential checks entirely. Defenders must understand RFC 4515 filter syntax to recognize and sanitize malicious inputs before they reach the directory service.

## Key facts
- RFC 4515 defines **six filter types**: equality (`=`), substring (`*`), present (`=*`), greater-or-equal (`>=`), less-or-equal (`<=`), and approximate match (`~=`)
- **Boolean operators** `&` (AND), `|` (OR), and `!` (NOT) allow complex compound filters using **prefix (Polish) notation** — e.g., `(&(a=1)(b=2))`
- Special characters `( ) * \ NUL` must be **escaped using backslash-hex encoding** (e.g., `\2a` for `*`); failure to enforce this enables injection
- LDAP injection can expose **all directory objects** including password hashes, group memberships, and organizational structure
- RFC 4515 **obsoletes RFC 2254** and is the current authoritative standard for LDAP filter strings
