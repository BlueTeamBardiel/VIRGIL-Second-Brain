# ssh.log

## What it is
Think of it as the doorman's ledger at an exclusive club — every attempt to enter, whether successful or turned away, gets written down with a timestamp and description. In Zeek (formerly Bro) network analysis, `ssh.log` is a structured log file that records metadata about every SSH connection observed on the network, including authentication outcomes, client/server versions, and geographic data when enriched.

## Why it matters
During a credential-stuffing attack, an adversary may attempt thousands of SSH logins against exposed servers using breached username/password pairs. Analysts hunting this activity pivot to `ssh.log` to identify hosts with abnormally high `auth_attempts` counts, repeated `auth_success: false` entries followed by a sudden `auth_success: true` — the classic brute-force-then-success pattern that signals a compromised account.

## Key facts
- `ssh.log` is generated by **Zeek**, not by the Linux OS itself — it captures network-layer SSH metadata, whereas `/var/log/auth.log` or `/var/log/secure` captures host-level authentication events
- Key fields include: `uid`, `id.orig_h`, `id.resp_h`, `auth_success`, `auth_attempts`, `client` (SSH client version string), `server` (SSH server version string), and `remote_location.country_code`
- The `client` field can fingerprint adversary tooling — Metasploit, Hydra, and Medusa each emit distinctive SSH version strings
- `auth_success` is a **boolean** field; correlating `false` → `true` transitions from the same source IP is a primary brute-force detection technique
- SSH version strings in `client`/`server` fields support **passive OS/tool fingerprinting** without sending a single active probe packet
