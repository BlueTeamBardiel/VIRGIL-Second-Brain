# CWE-122

## What it is
Imagine pouring 10 gallons of water into a 5-gallon bucket — the overflow spills onto the floor and damages whatever is nearby. A heap-based buffer overflow works exactly like this: an attacker writes data beyond the allocated boundary of a heap memory buffer, corrupting adjacent memory structures. Unlike stack overflows, this targets dynamically allocated memory (malloc/new), making exploitation more complex but equally dangerous.

## Why it matters
In 2021, a heap overflow in the Windows TCP/IP stack (CVE-2021-24086) allowed attackers to trigger remote code execution by sending malformed IPv6 packets — no user interaction required. Defenders patched by validating packet length before processing, illustrating the core fix: **always check input size against allocated buffer size before writing**.

## Key facts
- Heap overflows corrupt heap metadata (free lists, chunk headers), allowing attackers to redirect execution by overwriting function pointers or vtables
- Unlike stack overflows, heap overflows do **not** directly overwrite the return address — exploitation typically targets adjacent heap objects or management structures
- Common root cause: using `strcpy()`, `memcpy()`, or `sprintf()` without bounds checking on heap-allocated buffers
- Mitigations include **heap canaries**, **safe unlinking** (glibc), **Address Space Layout Randomization (ASLR)**, and **heap integrity checks** (Windows Low Fragmentation Heap)
- CWE-122 is a child of CWE-119 (Improper Restriction of Operations within Bounds of a Memory Buffer) — knowing the hierarchy matters for exam classification questions
