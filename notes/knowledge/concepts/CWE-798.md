# CWE-798

## What it is
Like a locksmith who engraves the master key combination directly onto the front door — the secret is baked into the object it's supposed to protect. CWE-798 describes hardcoded credentials: usernames, passwords, cryptographic keys, or tokens embedded directly in source code, firmware, or compiled binaries. Unlike stolen credentials, these cannot be revoked without redeploying the entire application.

## Why it matters
In 2016, researchers discovered that over 80 Cisco switches and routers shipped with a hardcoded SSH key, meaning any attacker who extracted the key from one device could authenticate to every other affected device worldwide. The only remediation was a firmware update — passive monitoring alone couldn't detect exploitation because the login appeared completely legitimate to the system.

## Key facts
- Hardcoded credentials are classified under **CWE-798** and are a child of CWE-259 (hardcoded password) and CWE-321 (hardcoded cryptographic key)
- Tools like **truffleHog**, **GitLeaks**, and **SAST scanners** detect hardcoded secrets by scanning git history and source trees using regex and entropy analysis
- Credentials hardcoded in compiled binaries can be extracted using **strings**, **Ghidra**, or **Binwalk** — obfuscation provides only marginal protection
- OWASP lists this under **A07:2021 – Identification and Authentication Failures**; it also maps to **NIST 800-53 IA-5** (authenticator management)
- The correct remediation is **externalized secret management** using tools like HashiCorp Vault, AWS Secrets Manager, or environment variables injected at runtime — never commit secrets to version control
