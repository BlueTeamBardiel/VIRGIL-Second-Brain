# False Positives

## What it is
Like a smoke detector that screams every time you make toast — triggering the alarm without an actual fire — a false positive is when a security system flags a benign activity as malicious. Precisely: it is an alert generated by an IDS, SIEM, or antivirus where the identified threat does not actually exist, causing security teams to investigate a non-event.

## Why it matters
In a SOC environment, an overly aggressive IDS signature might flag legitimate port scanning from an internal vulnerability scanner as a reconnaissance attack. Analysts who spend hours chasing these ghost alerts suffer **alert fatigue**, which is exactly when real intrusions slip through undetected — the 2013 Target breach involved overlooked alerts buried under noise.

## Key facts
- **False Positive Rate (FPR)** is the key tuning metric: FPR = FP / (FP + TN). A high FPR wastes analyst time and erodes trust in tooling.
- False positives are the primary cause of **alert fatigue**, a condition where analysts begin ignoring or auto-dismissing alerts, degrading detection capability.
- On the **Security+/CySA+ exam**, false positives are contrasted with false negatives (missed real threats) — false negatives are generally considered *more dangerous* to security posture.
- Reducing false positives requires **tuning detection rules**, whitelisting known-good behavior, and using **threat intelligence context** to validate alerts before escalation.
- SIEM platforms reduce false positives through **correlation rules** — requiring multiple conditions to fire before generating an alert, rather than triggering on any single event.
