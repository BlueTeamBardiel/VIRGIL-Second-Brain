# CWE-320

## What it is
Like leaving your house key under the doormat and trusting nobody will look there, weak key management means your encryption is only as strong as how you handle the keys — not the algorithm itself. CWE-320 is a category covering improper key management practices including weak key generation, insecure key storage, missing key rotation, and failure to protect keys from unauthorized access. It's a meta-weakness: the cipher can be perfect while the implementation collapses around it.

## Why it matters
The 2011 RSA SecurID breach is a textbook example — attackers didn't break the algorithm; they stole the seed values (essentially the keys) via a spear-phishing attack, then used them to clone SecurID tokens and compromise defense contractor networks including Lockheed Martin. The lesson: cryptographic strength is irrelevant when key material is exfiltrated in plaintext.

## Key facts
- CWE-320 is a **category**, not a base weakness — it encompasses child CWEs like CWE-321 (hard-coded keys), CWE-322 (key exchange without authentication), and CWE-324 (use of expired keys)
- Hard-coded cryptographic keys (CWE-321) are consistently flagged in OWASP Top 10 under **A02: Cryptographic Failures**
- Proper key lifecycle management requires: generation → distribution → storage → rotation → revocation → destruction
- Keys stored in source code, config files, or environment variables without encryption are a top finding in cloud penetration tests (AWS, GCP)
- NIST SP 800-57 provides the authoritative guidance on key management; Security+ candidates should know it prescribes **cryptoperiods** — maximum lifespans for key usage
