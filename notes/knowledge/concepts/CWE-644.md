# CWE-644

## What it is
Like a hospital that posts patient records on the public bulletin board instead of using sealed envelopes, this weakness occurs when an application improperly neutralizes HTTP headers that contain sensitive data before transmission. Specifically, CWE-644 describes the failure to properly handle HTTP response headers, allowing sensitive information — like session tokens or internal server details — to be exposed in cleartext or improperly secured contexts.

## Why it matters
An attacker performing a man-in-the-middle attack against an unencrypted HTTP connection can trivially harvest session cookies if the application transmits them without the `Secure` or `HttpOnly` flags set in the `Set-Cookie` header. A real-world example: Firesheep (2010) exploited exactly this pattern against Facebook and Twitter, capturing live session cookies over shared Wi-Fi to hijack authenticated sessions instantly.

## Key facts
- The `Secure` flag on cookies ensures the browser **only** transmits the cookie over HTTPS, directly mitigating this weakness
- The `HttpOnly` flag prevents JavaScript from reading cookie values, blocking XSS-based cookie theft as a complementary control
- Missing or misconfigured `Strict-Transport-Security (HSTS)` headers compound this weakness by allowing protocol downgrade attacks
- Tools like **Burp Suite** and **OWASP ZAP** can detect improperly set security headers during a web application assessment
- CWE-644 is distinct from CWE-614 (Sensitive Cookie Without Secure Flag) — 644 is the broader category covering any sensitive HTTP header mishandling
