# CWE-131

## What it is
Like a chef who eyeballs "a handful" of flour instead of measuring precisely, then overflows the mixing bowl — calculating a buffer size incorrectly leads to allocating too little memory for the actual data. CWE-131 occurs when software computes the size of a memory buffer using faulty logic, producing a buffer smaller than required and setting the stage for a heap or stack overflow when data is written into it.

## Why it matters
The classic exploitation scenario: an attacker sends an unexpectedly long string — say, a crafted username during authentication — and the application, having under-allocated the buffer due to miscalculated `sizeof()` usage, overwrites adjacent memory. This enabled real-world exploits in network daemons where developers mistakenly used `sizeof(pointer)` (typically 4 or 8 bytes) instead of `sizeof(struct)` (potentially hundreds of bytes), handing attackers control of the instruction pointer.

## Key facts
- The most common mistake is using `sizeof(ptr)` on a pointer variable instead of the structure or array it points to — on 64-bit systems this silently returns 8 regardless of actual data size
- Off-by-one errors in size math (e.g., forgetting the null terminator `\0` in C strings) are a subclass of this CWE
- Incorrect size calculation is a *precondition* vulnerability — it enables buffer overflows (CWE-120, CWE-122) but is distinct from them
- Languages like C and C++ are primary targets; memory-safe languages (Rust, Java) eliminate this class by design
- Static analysis tools (Coverity, CodeQL) specifically flag `sizeof` misuse patterns to catch CWE-131 at compile time
