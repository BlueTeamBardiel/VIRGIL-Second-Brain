# T1098.001

## What it is
Like a thief who doesn't steal the master key but instead quietly adds their own copy to the lockbox, Account Manipulation: Additional Cloud Credentials involves attackers adding new authentication credentials to an existing cloud account without removing the legitimate ones. Specifically, adversaries create new access keys, service principal credentials, or certificates on already-compromised cloud identities (AWS IAM users, Azure service principals, GCP service accounts) to maintain persistent access even if the original compromise vector is closed.

## Why it matters
In the 2019 Capital One breach, the attacker exploited a misconfigured WAF to obtain credentials, then leveraged cloud metadata to access IAM roles — a pattern where additional credential creation would have ensured persistence even after initial credential rotation. Defenders catching this technique in AWS look for `CreateAccessKey` API calls in CloudTrail logs, especially when the calling user differs from the target user.

## Key facts
- AWS IAM users can have **up to two active access keys simultaneously** — attackers exploit this to add a second key while the first remains functional and unsuspected
- The technique appears in CloudTrail as `CreateAccessKey`, `CreateLoginProfile`, or `UpdateLoginProfile` events
- In Azure, attackers may add new client secrets or certificates to existing **App Registrations** or **Managed Identities**
- This is a **persistence** technique (TA0003), not initial access — the account must already be compromised first
- Detection signal: access key created by a **different principal** than the key owner, or creation outside business hours/expected IP ranges
