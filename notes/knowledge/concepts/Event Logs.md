# event logs

## What it is
Like a ship's captain's log recording every course change, weather event, and crew action, event logs are timestamped records automatically generated by operating systems, applications, and devices that document what happened, when, and who did it. On Windows systems, these are organized into channels (Security, System, Application) and viewed via Event Viewer or forwarded to a SIEM for correlation.

## Why it matters
During the 2020 SolarWinds attack, investigators relied heavily on Windows Security event logs — specifically Event ID 4624 (successful logon) and 4672 (special privileges assigned) — to trace lateral movement by the threat actor through victim networks. Without intact logs, attribution and scope determination would have been nearly impossible, which is exactly why attackers routinely attempt to clear logs (Event ID 1102) to cover their tracks.

## Key facts
- **Windows critical Event IDs to memorize:** 4624 (logon success), 4625 (logon failure), 4648 (explicit credential use), 4672 (privileged logon), 4688 (process creation), 1102 (audit log cleared)
- **Linux equivalent** lives in `/var/log/` — `auth.log` for authentication, `syslog` for system events, and `audit/audit.log` for SELinux/auditd activity
- **Log retention** is a compliance requirement — PCI-DSS mandates 12 months of logs with 3 months immediately available for analysis
- **Log forwarding** to a centralized SIEM (Splunk, QRadar, Microsoft Sentinel) prevents tampering; an attacker who owns a local machine can delete local logs but cannot easily erase forwarded copies
- **Syslog** (UDP/514 or TCP/6514 for TLS) is the standard protocol for shipping logs across a network; RFC 5424 defines the message format including severity levels 0–7
