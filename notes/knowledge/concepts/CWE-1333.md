# CWE-1333

## What it is
Like a bouncer who gets so tangled checking one suspicious ID that the entire line backs up indefinitely, an inefficient regex pattern can be fed crafted input that causes exponential backtracking, freezing the application. CWE-1333 is Improper Regular Expression Complexity — a vulnerability where regex engines consume catastrophic amounts of CPU time processing specially crafted strings, leading to a Denial of Service condition known as ReDoS (Regular Expression Denial of Service).

## Why it matters
In 2016, a single malicious post containing a carefully crafted string exploited a vulnerable regex in Stack Overflow's comment parser, taking down the site for 34 minutes. Attackers can identify ReDoS vulnerabilities by fuzzing input fields that perform regex validation (email, URL, username fields) and submitting strings designed to trigger worst-case backtracking in poorly written patterns like `(a+)+` or `(.*a){x}`.

## Key facts
- **Catastrophic backtracking** occurs when a regex with nested quantifiers (e.g., `(a+)+b`) is given input like `"aaaaaac"` — the engine tries every possible combination before failing
- **Polynomial vs. exponential**: Some patterns degrade polynomially (bad), others exponentially (catastrophic); exponential patterns can freeze a server with input of just ~30 characters
- **Mitigation**: Use atomic groups, possessive quantifiers, or regex engines with linear-time guarantees (RE2, Rust's regex crate) instead of backtracking engines
- **Detection tools**: Static analysis tools like `safe-regex`, `rxxr2`, and `vuln-regex-detector` can identify vulnerable patterns before deployment
- **CWE classification**: Falls under the broader category of resource management weaknesses; maps to OWASP's "Denial of Service" category and can satisfy CAPEC-492
