# CWE-824

## What it is
Like a librarian handing you a sticky note with a shelf number written on it before the library has even been built — your pointer references a memory location that was never properly initialized. CWE-824 is the "Access of Uninitialized Pointer" weakness, where code dereferences a pointer variable before assigning it a valid memory address, causing unpredictable behavior or exploitable memory corruption.

## Why it matters
In 2014, a vulnerability in the Linux kernel's `n_tty` driver involved an uninitialized pointer that allowed local users to escalate privileges by triggering a kernel crash or arbitrary code execution. An attacker who can influence program flow before a pointer is set can redirect execution to shellcode or a ROP chain by controlling what garbage value happens to reside in that memory location at runtime.

## Key facts
- Uninitialized pointers contain whatever value was previously in that memory region ("garbage values"), making exploitation environment-dependent but not impossible
- Distinct from use-after-free (CWE-416): the pointer was *never* valid, not previously valid then freed
- Compilers like GCC with `-Wall` and static analysis tools (Coverity, Clang Analyzer) can flag many instances at build time
- Exploitation often requires pairing with a heap spray or memory grooming technique to reliably control the garbage value
- MITRE scores this as a Class-level weakness, meaning it abstracts across multiple languages (C, C++, assembly); Java/Python are largely immune due to runtime null initialization
