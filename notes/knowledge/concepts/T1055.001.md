# T1055.001

## What it is
Like a puppeteer slipping their hand inside a marionette, an attacker reaches into a legitimate running process and injects malicious code directly into its memory space. Dynamic Link Library (DLL) Injection is a technique where an adversary forces a target process to load and execute a malicious DLL without the process ever "asking" for it, using Windows APIs like `OpenProcess`, `VirtualAllocEx`, `WriteProcessMemory`, and `CreateRemoteThread`.

## Why it matters
During the 2017 NotPetya outbreak, attackers used process injection techniques to execute malicious payloads inside trusted system processes like `lsass.exe`, making detection dramatically harder because security tools saw clean, signed processes doing the dirty work. Defenders watching for suspicious *processes* would miss it entirely — the alert only appears when monitoring *memory regions* and inter-process API calls.

## Key facts
- The four-API injection chain (`OpenProcess` → `VirtualAllocEx` → `WriteProcessMemory` → `CreateRemoteThread`) is a textbook signature that EDR tools specifically hunt for
- Injected code inherits the target process's privileges and security context — injecting into a SYSTEM-level process grants SYSTEM access
- Detection focus: look for processes loading DLLs from unusual paths (e.g., `%TEMP%` or `AppData`) and mismatched parent-child process relationships
- Unlike legitimate DLL loading, malicious injection often results in memory-backed (not disk-backed) executable regions — flagged by tools checking `MEM_PRIVATE` vs. `MEM_IMAGE` pages
- Reflective DLL injection is a stealthier variant that loads a DLL entirely from memory without touching disk, bypassing many file-based AV signatures
