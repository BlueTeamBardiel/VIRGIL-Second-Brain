# CWE-789

## What it is
Like a hotel that lets guests request any number of rooms based purely on their word — then tries to actually reserve them all — this flaw occurs when a program allocates memory based on an untrusted, unvalidated size value. Specifically, CWE-789 describes uncontrolled memory allocation where user-supplied input directly controls how much memory is requested without bounds checking.

## Why it matters
An attacker submitting a crafted packet claiming a payload length of 2 billion bytes can force a server to attempt a 2GB heap allocation — crashing the process or triggering a denial-of-service condition. This is essentially what made early Heartbleed-adjacent bugs so dangerous: trusting a length field from the network without verifying it against actual available or reasonable bounds.

## Key facts
- Classified under the broader **CWE-400 (Uncontrolled Resource Consumption)** family — exhausting memory is a resource exhaustion attack
- The vulnerability lives in **input validation failure**: the program never checks if the requested size is within a safe, expected range
- Exploitation commonly causes **DoS via memory exhaustion or crash**, but can also enable **heap-based buffer overflows** if combined with subsequent writes
- Mitigations include **enforcing maximum allocation limits**, **validating size parameters against protocol specifications**, and using **safe memory allocation wrappers** that reject unreasonable values
- Languages with manual memory management (C, C++) are most vulnerable, but high-level languages can still suffer if they blindly pass attacker-controlled sizes to `array.resize()` or similar calls
