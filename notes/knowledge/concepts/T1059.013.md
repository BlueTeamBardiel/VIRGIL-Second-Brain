# T1059.013

## What it is
Like a stage magician using a theater's own spotlight and microphone against the audience, this technique turns Apple's built-in scripting engine into an attacker's weapon. T1059.013 refers to **AppleScript abuse**, where adversaries leverage macOS's native automation language to execute malicious commands, control applications via GUI scripting, and move laterally — all using a trusted, pre-installed interpreter.

## Why it matters
In the XCSSET malware campaign targeting macOS developers, attackers used AppleScript to silently interact with browsers and steal session cookies by scripting GUI elements — actions that appeared legitimate because the OS itself was executing them. Defenders monitoring for unusual `osascript` process invocations can catch this, as most legitimate users rarely invoke AppleScript programmatically at runtime.

## Key facts
- AppleScript runs via the `osascript` binary, which is the primary forensic artifact to hunt for in process logs
- It can control any "scriptable" application, including browsers, Finder, and Terminal — enabling data theft, credential harvesting, and persistence without dropping traditional malware binaries
- GUI scripting capability (enabled via Accessibility permissions) allows AppleScript to click buttons, read screen content, and fill forms autonomously
- Commonly delivered via malicious Office macros or `.scpt` files that invoke `osascript` as a subprocess
- Because it's a signed Apple binary, many endpoint tools whitelist it by default, making detection dependent on behavioral analytics rather than signature matching
