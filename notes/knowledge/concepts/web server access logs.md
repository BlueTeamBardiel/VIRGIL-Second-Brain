# web server access logs

## What it is
Think of a web server access log like a hotel doorman's notebook — every guest who walks in gets an entry: who they are, when they arrived, what room they asked for, and whether they got in. Precisely, web server access logs are records automatically generated by HTTP servers (Apache, Nginx, IIS) that capture each incoming request, including source IP, timestamp, HTTP method, requested URI, status code, user-agent string, and bytes transferred.

## Why it matters
During a SQL injection attack, the malicious payloads appear directly in the access log's URI field — strings like `' OR 1=1--` embedded in GET parameters are a dead giveaway. A SOC analyst reviewing logs after an incident can reconstruct the attacker's entire reconnaissance path: which pages they probed, which returned 200 vs. 403, and how quickly requests fired (indicating automated scanning rather than human browsing).

## Key facts
- The **Common Log Format (CLF)** is the standard structure: `host ident authuser date request status bytes` — recognizing this format is foundational for log parsing.
- HTTP **status codes** in logs tell the story: 200 (success), 403 (forbidden — possible auth bypass attempt), 404 (scanning for hidden paths), 500 (possible injection triggering server errors).
- **Unusually high 404 rates** from a single IP are a classic indicator of directory brute-forcing (e.g., DirBuster activity).
- Access logs do **not** capture request bodies — POST payloads are invisible, which is why they alone are insufficient for detecting form-based attacks.
- Log files are a **prime attacker target** for deletion or manipulation to cover tracks; immutable, centralized log storage (SIEM ingestion) is a critical countermeasure.
