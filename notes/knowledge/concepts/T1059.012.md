# T1059.012

## What it is
Like a burglar who learns to mimic the homeowner's voice to fool the security system, an attacker uses the **hypervisor's own management language against it**. T1059.012 covers abuse of **Hypervisor CLI** — specifically using administrative command-line interfaces (like VMware's `esxcli`, `vim-cmd`, or PowerCLI) that are built into hypervisor platforms to execute malicious commands and control virtual infrastructure.

## Why it matters
During the 2023 ESXiArgs ransomware campaign, attackers exploited exposed VMware ESXi hosts and used native `esxcli` and `vim-cmd` commands to enumerate running VMs, power them down, and encrypt their VMDK disk files — all through the hypervisor's legitimate tooling, which evaded endpoint detection tools running inside guest VMs that never saw the attacker at all.

## Key facts
- Targets hypervisor-native CLIs: VMware `esxcli`, `vim-cmd`, `esxcfg-*`, and similar tools native to ESXi/vSphere environments
- Attackers who compromise the **hypervisor layer** sit *above* all guest VMs — no endpoint agent installed inside a VM can detect these commands
- Common malicious actions include: powering off VMs, detaching snapshots, modifying virtual disk paths, and disabling security tools running as VMs
- Detection requires **hypervisor-level logging** (ESXi shell audit logs, vCenter event logs) — SIEM integration at the hypervisor layer is critical
- This sub-technique reflects a broader MITRE trend of cataloging **platform-native scripting languages** as distinct attack surfaces, similar to T1059.001 (PowerShell) but for virtualization infrastructure
