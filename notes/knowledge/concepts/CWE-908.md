# CWE-908

## What it is
Like serving a customer food on a plate that was never washed from the previous diner, this vulnerability hands an attacker data left over from a prior process. CWE-908 (Use of Uninitialized Resource) occurs when a program uses a resource — memory, variable, file descriptor — before it has been explicitly set to a known safe value, meaning whatever garbage was in that memory location before becomes attacker-observable or influenceable.

## Why it matters
In 2014, the OpenSSL Heartbleed vulnerability (CVE-2014-0160) demonstrated a related class of uninitialized/over-read memory flaw where a server would return up to 64KB of heap memory contents in response to a crafted heartbeat request — leaking private keys, passwords, and session tokens. Defenders must enforce compiler flags like `-ftrivial-auto-var-init=zero` and conduct static analysis to catch uses of variables before assignment.

## Key facts
- Uninitialized memory often contains residual data from prior allocations, making it a reliable side-channel for information disclosure attacks
- Languages like C and C++ do **not** zero-initialize local variables by default; developers must do it explicitly
- CWE-908 is a child of CWE-665 (Improper Initialization) in the MITRE CWE hierarchy
- Exploitation can yield privilege escalation if kernel or privileged-process memory remnants are exposed to a lower-privileged process
- Mitigations include safe language use (Rust, Java auto-initialize), compiler sanitizers (`-fsanitize=memory`), and mandatory code review checkpoints for resource allocation paths
