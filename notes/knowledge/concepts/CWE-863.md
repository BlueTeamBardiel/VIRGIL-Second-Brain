# CWE-863

## What it is
Imagine a nightclub bouncer who checks your ID at the door but never verifies whether your VIP wristband actually grants access to the rooftop bar — you're authenticated, but your *permissions* go unchecked. CWE-863 (Incorrect Authorization) occurs when software correctly identifies *who* a user is but then makes flawed decisions about *what* they're allowed to do. The authorization logic exists but produces wrong outcomes due to missing checks, logic errors, or improper trust assumptions.

## Why it matters
In 2019, a Shopify vulnerability allowed any authenticated merchant to access other merchants' customer data by manipulating API parameters — the server verified login credentials but failed to confirm the requesting account actually owned the resource. An attacker simply swapped an ID in the request to harvest sensitive customer records from arbitrary stores. Proper object-level authorization checks (IDOR prevention) would have stopped this cold.

## Key facts
- Distinct from CWE-862 (Missing Authorization) — here the check *exists* but produces an **incorrect result** due to logic flaws
- Commonly manifests as **Insecure Direct Object Reference (IDOR)**, where resource IDs are guessable and ownership isn't verified server-side
- Often appears in **role-based access control (RBAC)** implementations where privilege escalation is possible by manipulating roles or parameters
- Ranked in OWASP Top 10 under **A01:2021 – Broken Access Control**, the #1 web application risk category
- Defense requires **server-side enforcement** of authorization on every request — client-side checks alone are never sufficient
