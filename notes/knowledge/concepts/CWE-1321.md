# CWE-1321

## What it is
Like a locksmith who blindly follows a master key template even when that template has been tampered with by a customer, JavaScript prototype pollution lets an attacker inject properties into `Object.prototype` — the ancestral blueprint all objects inherit from. This means malicious properties propagate to *every* object in the application, silently corrupting logic without touching any specific variable directly.

## Why it matters
In 2019, the `lodash` library (used by millions of npm packages) was found vulnerable to prototype pollution via its `merge()` and `defaultsDeep()` functions. An attacker could send a crafted JSON payload like `{"__proto__": {"isAdmin": true}}`, causing all subsequent objects to inherit `isAdmin: true` — bypassing authorization checks across the entire application without ever touching the auth code itself.

## Key facts
- **Attack vector**: Malicious input targets `__proto__`, `constructor`, or `prototype` keys during recursive object merging or cloning operations
- **Impact range**: Denial of Service, privilege escalation, or Remote Code Execution depending on how the polluted property is consumed downstream
- **Root cause**: Insufficient input key sanitization before merging user-controlled JSON into application objects
- **Fix**: Use `Object.create(null)` to create prototype-free objects, validate/block reserved keys (`__proto__`, `constructor.prototype`), or use `Object.freeze(Object.prototype)`
- **Common sink locations**: `lodash.merge()`, `jQuery.extend()`, `hoek`, and custom recursive merge utilities in Node.js applications
