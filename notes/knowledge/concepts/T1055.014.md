# T1055.014

## What it is
Like slipping a note into someone's pocket while shaking their hand, VDSO hijacking injects malicious code by corrupting the Virtual Dynamic Shared Object — a small memory region the Linux kernel automatically maps into every process's address space to accelerate syscalls without privilege switches. An attacker overwrites or abuses this shared memory region to execute arbitrary code within a target process's context.

## Why it matters
In a post-exploitation scenario on a Linux server, an attacker who has gained limited foothold can use VDSO hijacking to escalate privileges or evade detection by injecting shellcode into a legitimate process like `sshd` or `nginx` — making malicious activity appear as normal system behavior in process monitoring tools. Security teams relying solely on process-level telemetry may miss the injection entirely since no new process is spawned.

## Key facts
- VDSO (Virtual Dynamic Shared Object) is mapped by the Linux kernel into every process automatically, containing frequently-used syscall wrappers like `gettimeofday()` and `clock_gettime()`
- Exploitation typically requires write access to `/proc/[pid]/mem` or root-equivalent privileges, making it more common as a privilege escalation or persistence technique post-compromise
- Unlike DLL injection on Windows, VDSO hijacking is Linux/Unix-specific and targets a kernel-controlled memory region rather than user-land libraries
- Detection signatures include anomalous writes to VDSO memory regions, unexpected executable mappings, and use of `ptrace()` syscalls against unrelated processes
- This technique leaves no file on disk, classifying it as a fileless/in-memory attack — making forensic recovery significantly harder
