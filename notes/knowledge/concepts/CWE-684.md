# CWE-684

## What it is
Like a contractor who builds a house using blueprints from a different building entirely, this weakness occurs when a function doesn't correctly implement the behavior its caller expects based on its documented specification. CWE-684 is "Incorrect Provision of Specified Functionality" — the code exists and runs, but silently violates its own API contract or documented behavior, producing wrong results without throwing errors.

## Why it matters
In 2014, Apple's infamous "goto fail" SSL/TLS bug demonstrated this precisely: the `SSLVerifySignedServerKeyExchange()` function was documented to verify certificate signatures, but due to a duplicated `goto fail` line, it returned success *without actually verifying anything*. Attackers performing MitM attacks could present invalid certificates that the function would approve, silently compromising TLS for Safari and other Apple software on iOS and macOS.

## Key facts
- This is a **silent failure** weakness — no exception is thrown, no error logged; the function returns as if correct behavior occurred
- Distinct from CWE-683 (wrong argument order) — CWE-684 is about the *implementation* diverging from the *specification*, not just misuse by callers
- Especially dangerous in **security-critical functions** like authentication, cryptographic validation, and input sanitization where incorrect behavior isn't immediately observable
- Testing must include **contract testing** (verifying outputs match documented specs), not just functional testing (verifying it doesn't crash)
- CVSS scores for this weakness can reach **Critical (9.8)** when the diverging function bypasses authentication or cryptographic controls
