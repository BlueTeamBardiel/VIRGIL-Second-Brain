# CWE-427

## What it is
Like a locksmith who grabs the first "hammer" tool from an unlabeled bin without checking if it's actually theirs, an application using an uncontrolled search path will load the first library or executable it finds in a directory listing — not necessarily the legitimate one. CWE-427 (Uncontrolled Search Path Element) occurs when a program uses an attacker-influenced directory (via PATH, LD_LIBRARY_PATH, or similar environment variables) to locate resources, allowing malicious code substitution.

## Why it matters
A classic exploitation scenario is **DLL hijacking on Windows**: an attacker drops a malicious `version.dll` into a writable directory that appears earlier in the application's search order than the legitimate Windows system folder. When a privileged installer or service launches, it loads the attacker's DLL instead, achieving code execution — a technique heavily used in privilege escalation and persistence by APT groups.

## Key facts
- **DLL Search Order Hijacking** is the most common Windows manifestation; Windows searches the application directory before `System32` by default
- **Linux equivalent**: manipulating `LD_LIBRARY_PATH` or placing a malicious shared object in `/tmp` when it's listed before `/lib` in the path
- **PATH variable hijacking**: placing a malicious binary named `python` or `git` earlier in PATH than `/usr/bin` tricks scripts into executing attacker code
- **CVE-2010-3190** (Visual C++ redistributable) is a well-known real-world instance of DLL planting via uncontrolled search path
- Remediation: use **absolute paths** for all resource loading, validate path integrity, and apply **principle of least privilege** to writable directories in search paths
