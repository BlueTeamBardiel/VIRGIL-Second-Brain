# pe.log

## What it is
Like a flight recorder capturing every takeoff and landing at an airport, `pe.log` is Zeek's (formerly Bro) dedicated log file that records every executable file (PE — Portable Executable) observed traversing the network. It specifically captures metadata about Windows PE binaries (.exe, .dll, .sys) extracted from network traffic without executing them.

## Why it matters
During a malware delivery investigation, an analyst can query `pe.log` to identify when a suspicious `.exe` was downloaded, revealing its compile timestamp, whether it was compiled for 32 or 64-bit architecture, and whether it imports suspicious libraries like `WinInet.dll` (network communication) before ever running the file in a sandbox. This passive network-level artifact often catches malware that endpoint agents miss because it logs the transfer, not the execution.

## Key facts
- `pe.log` is generated by Zeek's PE analyzer and captures fields including `ts`, `id`, `machine` (target CPU architecture), `compile_ts` (compile timestamp), `is_exe`, `is_64bit`, and `uses_aslr`
- The `compile_ts` field is a prime threat-hunting indicator — malware often has suspiciously recent or epoch-zero (1970-01-01) compile timestamps indicating timestamp stomping
- `uses_aslr`, `uses_dep`, and `uses_seh` fields reveal whether the binary has modern exploit mitigations enabled — legitimate software almost always does
- `pe.log` correlates to other Zeek logs via the `id` connection identifier, linking it to `http.log` or `files.log` for full context
- A missing or disabled `is_64bit` flag combined with modern OS targets can indicate a dropper or shellcode loader
