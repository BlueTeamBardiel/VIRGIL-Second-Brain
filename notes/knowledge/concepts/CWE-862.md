# CWE-862

## What it is
Like a nightclub bouncer who only checks IDs at the front door but waves everyone through the side entrance — the system authenticates users but forgets to verify *authorization* before performing sensitive actions. CWE-862 (Missing Authorization) occurs when software does not perform an access control check to verify that the authenticated user actually has permission to perform a requested operation or access a resource.

## Why it matters
In 2019, a vulnerability in the Fortnite API allowed authenticated players to access *other users'* account data simply by changing a user ID parameter in the request — the server confirmed they were logged in but never verified the data belonged to them. Defenders address this by implementing server-side authorization checks on every sensitive endpoint, never relying solely on what the client sends or what the UI hides.

## Key facts
- Distinct from CWE-306 (Missing Authentication): CWE-862 assumes the user *is* authenticated but lacks a permission check, while 306 skips authentication entirely.
- Commonly manifests as **Insecure Direct Object Reference (IDOR)** — changing `user_id=123` to `user_id=124` in a request exposes another user's data.
- Maps to **OWASP A01:2021 – Broken Access Control**, the top web application vulnerability category.
- The fix requires *enforced*, server-side authorization logic — client-side hiding of buttons or UI elements is **not** a valid control.
- Frequently appears in REST APIs where developers implement AuthN (authentication) middleware globally but forget per-resource AuthZ (authorization) checks on individual endpoints.
