# T1055.009

## What it is
Like a parasite that hijacks a host cell's machinery to replicate itself, Process Hollowing carves out the legitimate code from a running process and replaces it with malicious payload — all while the process retains its trusted identity. Specifically, an attacker spawns a legitimate process in a suspended state, unmaps its memory, writes malicious code into that space, and resumes execution under the original process's credentials and name.

## Why it matters
The Cobalt Strike framework frequently uses process hollowing to inject beacon payloads into suspended `svchost.exe` or `explorer.exe` instances, making the malicious activity appear as routine Windows system processes in Task Manager and basic monitoring tools. Defenders caught flat-footed by this technique learned that process name alone is an unreliable indicator — memory integrity checks and behavioral analysis became essential countermeasures.

## Key facts
- The attack sequence is: `CreateProcess` (suspended) → `ZwUnmapViewOfSection` → `VirtualAllocEx` → `WriteProcessMemory` → `SetThreadContext` → `ResumeThread`
- Because the hollowed process inherits the legitimate process's PID, token, and name, signature-based AV tools frequently miss it entirely
- Detection relies on comparing the on-disk image hash of a process against its in-memory mapped image — a mismatch is a red flag
- Tools like Process Hacker and Volatility can expose hollowed processes by flagging discrepancies between the PEB (Process Environment Block) image path and actual loaded code
- This is a sub-technique of T1055 (Process Injection), classified separately because it replaces rather than supplements existing code
