# CWE-297

## What it is
Imagine checking that a hotel key card *exists* but never verifying it matches *your room number* — that's exactly this flaw. CWE-297 is improper validation of a certificate with a host mismatch: a system accepts a valid, legitimately signed certificate but fails to verify that the certificate's Common Name (CN) or Subject Alternative Name (SAN) actually matches the host it's communicating with.

## Why it matters
In a man-in-the-middle attack, an adversary can present a valid certificate issued for `evil.com` during a TLS handshake intended for `bank.com`. If the client code skips hostname verification — a mistake common in embedded systems and poorly written API clients — the connection proceeds, and all "encrypted" traffic flows through the attacker. This vulnerability enabled real-world exploits in Android apps and IoT devices that used OpenSSL incorrectly by setting `SSL_VERIFY_NONE` or ignoring hostname callback errors.

## Key facts
- **Root cause**: Certificate chain validation and hostname validation are *separate steps* — passing one does not imply passing the other.
- **Common trigger**: Developers disable hostname checks to "fix" SSL errors during testing, then ship that code to production.
- **CVE history**: CVE-2012-5784 (Apache Axis) and CVE-2014-1266 ("goto fail" in Apple iOS) both involved improper certificate/host verification failures.
- **Detection**: Static analysis tools (Veracode, Checkmarx) flag calls to `setHostnameVerifier(ALLOW_ALL_HOSTNAME_VERIFIER)` in Java as direct indicators.
- **Defense**: Always validate both the certificate chain *and* that the CN/SAN field matches the intended hostname using well-tested TLS libraries rather than custom verification logic.
