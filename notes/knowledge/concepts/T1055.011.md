# T1055.011

## What it is
Like a parasite that hollows out a living host and wears its skin, Extra Window Memory (EWM) injection hides malicious code inside the reserved memory slots attached to Windows GUI objects. Specifically, Windows allocates a small buffer of extra bytes (EWM) per window or control class, accessible via `SetWindowLong`/`GetWindowLong` — attackers abuse these slots to store and execute shellcode pointers inside a legitimate process's address space.

## Why it matters
FinFisher/FinSpy spyware used EWM injection to embed shellcode references into window objects of trusted processes, allowing it to execute malicious payloads while appearing as legitimate GUI activity — making memory scanning and behavioral detection significantly harder. Defenders monitoring for classic `WriteProcessMemory` + `CreateRemoteThread` patterns would miss this technique entirely.

## Key facts
- EWM slots are limited (~40 bytes on 32-bit Windows), so attackers typically store a **pointer** to shellcode rather than the shellcode itself
- Uses Windows API calls: `SetWindowLong` (32-bit) or `SetWindowLongPtr` (64-bit) to write into EWM, and `SendNotifyMessage` or `PostMessage` to trigger execution via a callback
- The injection runs **within the target process's legitimate thread**, bypassing many thread-based detection heuristics
- Because it abuses documented Windows GUI functionality (not an exploit), it can bypass application whitelisting controls that only monitor executable sections
- Detection focus: monitor for `SetWindowLong`/`SetWindowLongPtr` calls made **cross-process** (one PID writing into another PID's window memory)
