# RFC 6749

## What it is
Think of it like a hotel valet system: you hand the valet a limited-use ticket instead of your master key, so they can park your car without accessing your room. RFC 6749 is the OAuth 2.0 authorization framework specification, defining how a third-party application can obtain delegated, scoped access to a resource on behalf of a user without ever handling that user's credentials directly.

## Why it matters
In 2022, attackers exploited misconfigured OAuth implementations to perform token hijacking — intercepting authorization codes via redirect URI manipulation to gain persistent access to victims' cloud accounts. RFC 6749's redirect URI validation requirements exist precisely to prevent this, but developers who implement it carelessly leave this door wide open, making OAuth misconfigurations a recurring finding in cloud penetration tests and breach investigations.

## Key facts
- Defines **four grant types**: Authorization Code, Implicit (deprecated), Resource Owner Password Credentials, and Client Credentials — Authorization Code with PKCE is the current best-practice
- Introduces the concept of **scopes** to limit what access a token grants (e.g., `read:email` vs. full account access)
- Separates roles into four distinct actors: **Resource Owner, Client, Authorization Server, and Resource Server**
- Access tokens are **short-lived by design**; refresh tokens allow new access tokens without re-prompting the user
- RFC 6749 is an **authorization** framework, not authentication — OAuth 2.0 alone does not verify identity; OpenID Connect (OIDC) layers identity on top of it
