# CWE-918

## What it is
Imagine a hotel concierge who will fetch *anything* from anywhere in the building simply because a guest writes the destination on a slip of paper — including the manager's private safe. Server-Side Request Forgery (SSRF) works the same way: an attacker supplies a crafted URL that tricks the server into making HTTP requests on their behalf, reaching internal resources the attacker could never access directly. The server becomes an unwitting proxy into protected network space.

## Why it matters
In the 2019 Capital One breach, an attacker exploited an SSRF vulnerability in a misconfigured WAF to query AWS's internal metadata service (`http://169.254.169.254/`), retrieving IAM credentials that granted access to over 100 million customer records. Defending against this requires blocking requests to link-local and private IP ranges (`10.x.x.x`, `172.16-31.x.x`, `192.168.x.x`, `169.254.x.x`) and enforcing allowlists on any server-side URL fetching functionality.

## Key facts
- SSRF targets the **server's network context**, not the client's — the attack bypasses firewalls that block external access to internal services
- Cloud metadata endpoints (`169.254.169.254` on AWS/GCP/Azure) are the most exploited SSRF targets, leaking IAM tokens and instance credentials
- **Blind SSRF** returns no response to the attacker but can still exfiltrate data via DNS lookups or timing side-channels
- SSRF is listed in the **OWASP Top 10 (2021) as its own category (#10)**, reflecting its explosive growth in cloud environments
- Mitigation requires **allowlisting** outbound destinations, not just blocklisting — blocklists are bypassed via DNS rebinding or IPv6 representations of IPv4 addresses
