# T1098.007

## What it is
Like a thief who doesn't steal your house key but instead adds their own copy to your keychain, this technique involves an adversary adding additional authentication methods to an existing account rather than changing credentials. T1098.007 specifically covers **Additional Cloud Credentials** — where attackers add new API keys, OAuth tokens, service principal secrets, or SSH keys to cloud accounts they've compromised, ensuring persistent access even if the original credentials are rotated.

## Why it matters
In the 2023 Microsoft/Storm-0558 breach, attackers forged authentication tokens using a stolen MSA signing key — a prime example of credential abuse in cloud environments. Defenders who only monitor password changes would miss an attacker quietly adding a secondary service principal secret to an Azure AD application, granting long-term backdoor access that survives password resets and MFA enforcement on the primary account.

## Key facts
- Affects major cloud providers: AWS (access keys), Azure (service principal certificates/secrets), GCP (service account keys)
- Detection focus: monitor `CreateAccessKey`, `AddClientSecret`, and `iam.serviceAccounts.keys.create` API calls in CloudTrail/Azure Activity Log/GCP Audit Log
- Attackers prefer this over credential theft because added keys often bypass MFA requirements applied to human login flows
- Legitimate admin activity (DevOps automation) generates significant noise, making behavioral baselining critical for reducing false positives
- Sub-technique of T1098 (Account Manipulation) — part of the Persistence tactic in the MITRE ATT&CK framework
