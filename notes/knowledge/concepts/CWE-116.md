# CWE-116

## What it is
Like a postal worker who stamps "FRAGILE" on a package without actually wrapping the contents in bubble wrap, improper encoding or escaping means a program labels data as safe without actually neutralizing dangerous characters. CWE-116 occurs when an application fails to correctly encode or escape output before it's interpreted by a downstream component — such as a browser, shell, or database engine. The data arrives at its destination still carrying executable teeth.

## Why it matters
The most notorious consequence is Cross-Site Scripting (XSS): a web app accepts user input containing `<script>alert('pwned')</script>` and renders it directly in HTML without HTML-encoding the angle brackets. A victim's browser interprets the injected markup as executable code, allowing session cookie theft or credential harvesting. Defense requires context-aware encoding — HTML encoding for HTML contexts, JavaScript encoding for JS contexts — not a single catch-all sanitization pass.

## Key facts
- CWE-116 is a *class weakness* — it is the parent of more specific issues like XSS (CWE-79) and OS Command Injection (CWE-78)
- Encoding and escaping are distinct: encoding transforms characters into a safe representation (e.g., `<` → `&lt;`); escaping adds a prefix character to neutralize special meaning (e.g., `\'`)
- Context matters critically — correct HTML encoding is wrong in a SQL context; mixing them creates false security
- Output encoding at the point of use (not input sanitization at ingestion) is the OWASP-recommended primary defense
- Failures here are consistently listed in the OWASP Top 10 under "Injection" and "XSS" categories, making this high-frequency exam territory
