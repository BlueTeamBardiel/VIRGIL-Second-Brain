# CWE-444

## What it is
Imagine two postal workers reading the same envelope but disagreeing on where it's addressed — one delivers it to the front door, the other to the back. HTTP Request Smuggling exploits exactly this: when a front-end proxy and back-end server interpret ambiguous HTTP request boundaries differently, an attacker can "smuggle" a hidden request inside a legitimate one. CWE-444 formally describes this as "Inconsistent Interpretation of HTTP Requests."

## Why it matters
In 2019, researcher James Kettle demonstrated HTTP Request Smuggling against major platforms including Netflix and Atlassian, achieving cache poisoning and session hijacking by exploiting disagreements between CDN proxies (like Varnish) and origin servers over `Content-Length` vs. `Transfer-Encoding: chunked` headers. An attacker could prefix a malicious request to a victim's legitimate traffic, stealing their credentials mid-flight.

## Key facts
- The core ambiguity arises when both `Content-Length` and `Transfer-Encoding: chunked` headers are present — RFC 7230 says TE takes priority, but not all implementations comply
- Two main variants: **CL.TE** (front-end uses Content-Length, back-end uses Transfer-Encoding) and **TE.CL** (reversed), each enabling different smuggling payloads
- Attackers can use smuggling to bypass security controls, poison shared connection queues, hijack other users' requests, or perform reflected XSS
- Defenses include normalizing ambiguous requests at the edge (reject or rewrite conflicting headers), disabling HTTP/1.1 keep-alive between proxy layers, or using HTTP/2 end-to-end (which eliminates the ambiguity)
- CVSS scores for real-world instances have reached 9.8 — classified Critical — because exploitation requires no authentication
