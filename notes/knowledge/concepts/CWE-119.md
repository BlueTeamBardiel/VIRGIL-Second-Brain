# CWE-119

## What it is
Like stuffing 10 pounds of groceries into a 5-pound bag until the seams burst and food spills onto the neighbor's shelf — this is what happens when software writes data beyond the boundaries of an allocated memory buffer. CWE-119 is the parent category for improper restriction of operations within the bounds of a memory buffer, encompassing both overflows and underflows in stack, heap, or global memory regions.

## Why it matters
The 2003 SQL Slammer worm exploited a buffer overflow in Microsoft SQL Server, spreading to 75,000 hosts in just 10 minutes — no user interaction required. Attackers craft malicious input that overwrites adjacent memory, redirecting execution flow to attacker-controlled shellcode. Defenders mitigate this through ASLR, DEP/NX bits, stack canaries, and memory-safe language adoption.

## Key facts
- CWE-119 is the **parent** of more specific weaknesses including CWE-120 (classic buffer overflow), CWE-125 (out-of-bounds read), and CWE-787 (out-of-bounds write)
- Consistently ranked in the **CWE Top 25 Most Dangerous Software Weaknesses** alongside SQL injection and XSS
- Exploitation can lead to **arbitrary code execution, privilege escalation, or denial of service** — making it a critical-severity class vulnerability
- Primary defenses: **compiler protections** (stack canaries, SafeStack), **OS-level mitigations** (ASLR, DEP), and **input validation/bounds checking** in code
- Languages like C and C++ are most vulnerable due to **manual memory management**; Rust and Go provide memory safety guarantees by design
