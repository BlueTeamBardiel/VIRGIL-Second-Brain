# CWE-404

## What it is
Like a restaurant that never closes its kitchen after service — leaving gas burners lit and food out overnight — CWE-404 describes software that fails to properly release resources (file handles, network sockets, memory) when they're no longer needed. Specifically, it covers improper resource shutdown or release, where a program opens or allocates a resource but lacks the correct cleanup logic when that resource's lifecycle ends.

## Why it matters
In 2021, a vulnerability in Apache HTTP Server exploited improper socket cleanup under specific mod_proxy configurations, allowing attackers to trigger resource exhaustion and cause denial-of-service conditions. An attacker who can repeatedly force a process to open resources without releasing them can starve a system of file descriptors or memory until it crashes or becomes unresponsive — no exotic exploit required, just volume.

## Key facts
- CWE-404 is a parent class weakness; child weaknesses include CWE-772 (missing release of resource after effective lifetime) and CWE-775 (missing release of file descriptor)
- The primary attack impact is **Denial of Service via resource exhaustion**, affecting availability — directly maps to the CIA triad's "A"
- Common in languages without automatic garbage collection (C, C++) but also occurs in Java/Python when streams or connections aren't explicitly closed
- Secure coding patterns that mitigate CWE-404 include `try-finally` blocks, RAII (Resource Acquisition Is Initialization) in C++, and Python's `with` statement
- CWE-404 contributes to the broader **CWE-400 (Uncontrolled Resource Consumption)** weakness cluster, relevant to both application security and network hardening
