# T1055.3

## What it is
Like a puppeteer slipping their hand into someone else's glove puppet mid-performance, Thread Execution Hijacking seizes control of an already-running thread in a legitimate process and redirects it to execute malicious code. Specifically, an attacker suspends a thread in a target process, overwrites its instruction pointer (via `SetThreadContext`), injects shellcode, then resumes the thread — all within a trusted process's memory space.

## Why it matters
The Cobalt Strike framework uses thread hijacking as a post-exploitation technique to migrate shellcode into legitimate processes like `explorer.exe`, making the malicious activity blend into normal process telemetry. Defenders relying solely on process-level monitoring will miss this entirely because no new process is spawned — the malicious code runs under the host process's identity and inherited trust.

## Key facts
- Uses Windows API sequence: `SuspendThread` → `GetThreadContext` → `WriteProcessMemory` → `SetThreadContext` → `ResumeThread`
- Executes entirely within the virtual address space of a legitimate process, bypassing many application allowlisting controls
- No new process creation event (Event ID 4688) is generated, defeating process-birth-based detection
- Detected through behavioral monitoring: unexpected calls to `SetThreadContext` on threads belonging to unrelated processes, flagged by EDR tools watching cross-process API access
- Classified under Process Injection (T1055) parent technique; shares the defense-evasion and privilege-escalation dual-purpose with sibling techniques
