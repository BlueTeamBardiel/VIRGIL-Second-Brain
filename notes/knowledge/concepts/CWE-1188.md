# CWE-1188

## What it is
Like a bank vault installed with the combination set to "0000" by the factory and never changed, this weakness occurs when a system initializes a resource with an insecure default value that grants unintended access or privilege. Precisely: CWE-1188 (Initialization of a Resource with an Insecure Default) describes situations where software initializes a resource in a state that is either overly permissive, unauthenticated, or otherwise unsafe before proper configuration occurs.

## Why it matters
In 2016, the Mirai botnet compromised hundreds of thousands of IoT devices — routers, cameras, DVRs — almost entirely because manufacturers shipped them with hardcoded default credentials like `admin/admin` or `root/12345`. Attackers simply scanned for these devices and logged in without any exploitation required, demonstrating that an insecure default is functionally equivalent to no security at all.

## Key facts
- Distinct from hardcoded credentials (CWE-798): CWE-1188 covers any insecure initial state, not just passwords — including open firewall rules, world-readable file permissions, or disabled authentication flags
- "Secure by default" is the direct mitigation principle: systems should require explicit action to *reduce* security, not *enable* it
- Common in embedded systems, IoT firmware, and enterprise software installers that prioritize ease of setup over security posture
- NIST SP 800-160 and CIS Controls both mandate default-hardening as a baseline requirement before deployment
- A resource initialized insecurely may be exploitable in a brief window even if reconfigured later — timing attacks target this gap (TOCTOU-adjacent risk)
