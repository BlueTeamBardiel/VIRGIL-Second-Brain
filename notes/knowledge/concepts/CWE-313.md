# CWE-313

## What it is
Like writing your house alarm PIN on a sticky note taped to the front door, cleartext storage in a file means sensitive data is saved in human-readable form with no encryption protecting it. Precisely: CWE-313 occurs when an application stores sensitive information (passwords, keys, PII) in a file without encrypting or otherwise protecting it from direct reading.

## Why it matters
In 2021, researchers discovered that the Passwordstate password manager stored credentials in a cleartext configuration file; attackers who compromised the update mechanism could harvest every stored credential simply by reading that file — no cracking required. A proper defense requires encrypting stored secrets at rest using AES-256 and restricting file permissions to least privilege so even OS-level access doesn't immediately expose data.

## Key facts
- **No cracking needed**: Cleartext secrets are immediately usable by any attacker with file read access — no dictionary attack, no brute force.
- **Log files are a common culprit**: Applications that log debug output frequently write passwords and tokens to cleartext log files, often in world-readable directories.
- **Differs from CWE-312**: CWE-312 covers cleartext storage broadly (memory, DB, etc.); CWE-313 is specifically scoped to **files** on disk.
- **File permissions are not a fix**: Relying solely on OS ACLs is insufficient — if the system is compromised or permissions misconfigured, everything is exposed.
- **Compliance impact**: PCI-DSS Requirement 3 and HIPAA explicitly prohibit storing sensitive data (PANs, PHI) in cleartext files, making this a direct audit finding.
