# CWE-367

## What it is
Like checking whether a door is locked, walking away to grab your keys, and assuming it's still locked when you return — a TOCTOU (Time-of-Check Time-of-Use) race condition occurs when a program verifies a security condition at one moment but acts on it later, after the state may have changed. Precisely: CWE-367 is a race condition where the gap between checking a resource's state (e.g., file permissions, process identity) and using that resource allows an attacker to swap the resource in between.

## Why it matters
A classic exploit involves the `/tmp` directory on Unix systems: a privileged program checks that `/tmp/datafile` is safe to write, then an attacker uses a symlink swap to replace it with `/etc/passwd` before the write occurs — the program then overwrites a critical system file. This is exactly how several historical privilege escalation exploits in Linux utilities (like `sudo` and `screen`) were achieved.

## Key facts
- TOCTOU is a subset of **race conditions (CWE-362)**; CWE-367 specifically addresses file system checks followed by file operations
- The attack window is called the **race window** — smaller windows are harder but not impossible to exploit
- Mitigation involves **atomic operations**: open the file with `O_NOFOLLOW` and `O_CREAT` flags, or use file descriptors obtained at check-time for all subsequent use
- On Linux, `access()` followed by `open()` is a textbook vulnerable pattern — use `faccessat()` with `AT_EACCESS` or just attempt the `open()` and handle errors
- CVSS scores for TOCTOU vulnerabilities often reflect **local privilege escalation**, making them high-severity despite requiring local access
