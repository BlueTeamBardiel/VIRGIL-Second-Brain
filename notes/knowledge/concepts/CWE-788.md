# CWE-788

## What it is
Like a librarian who grabs books from the shelf *next door* to the one they were told to access, this flaw occurs when code reads memory *after* the end of a buffer — but still within the intended memory region's boundary. CWE-788 (Access of Memory Location After End of Buffer) is a specific out-of-bounds read where the accessed offset exceeds the allocated buffer but hasn't yet escaped the broader memory segment.

## Why it matters
The Heartbleed vulnerability (CVE-2014-0160) in OpenSSL is a textbook cousin of this class: a crafted TLS heartbeat request caused the server to read up to 64KB beyond the intended buffer, leaking private keys, session tokens, and credentials from adjacent memory — all without crashing, making it silently catastrophic. Defenders patch this by enforcing strict bounds checking and using memory-safe languages or compiler flags like `-D_FORTIFY_SOURCE=2`.

## Key facts
- **Read-only but still dangerous**: Unlike buffer overflows that write data, CWE-788 leaks data — private keys, passwords, or cryptographic material sitting in adjacent memory.
- **Silent attacker**: The process rarely crashes, meaning the vulnerability can be exploited repeatedly without triggering alerts from crash-based detection.
- **Common triggers**: Off-by-one errors in loop conditions (`i <= length` instead of `i < length`) and improper use of `memcpy` or `strcpy` with unchecked length parameters.
- **Language exposure**: C and C++ are primary victims; Rust and Java enforce bounds checks at compile/runtime, preventing this class entirely.
- **Detection tools**: Static analysis tools (Coverity, Clang Static Analyzer) and dynamic tools (Valgrind, AddressSanitizer) can identify this before deployment.
