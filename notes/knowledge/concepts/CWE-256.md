# CWE-256

## What it is
Like writing your house alarm code on a sticky note attached to the keypad, plaintext storage of passwords keeps credentials visible to anyone who looks. CWE-256 describes the vulnerability where a system stores passwords (or other credentials) in cleartext — unencrypted and unobfuscated — in files, databases, registries, or memory. No cryptographic protection stands between an attacker and the secret.

## Why it matters
In the 2013 Adobe breach, attackers exfiltrated a database containing ~153 million user records where passwords were stored with reversible encryption (effectively plaintext-equivalent). Because the storage provided no real one-way hashing, attackers could cross-reference encrypted values and crack passwords en masse — compromising accounts across multiple platforms where users reused credentials. A proper bcrypt or Argon2 implementation would have made bulk cracking computationally infeasible.

## Key facts
- CWE-256 is distinct from weak hashing (CWE-916) — here, no cryptographic transformation occurs at all; the credential is stored as-is
- Common locations: config files, `.env` files, Windows Registry, database columns, application logs, and embedded firmware
- The correct mitigation is **one-way adaptive hashing** (bcrypt, scrypt, Argon2) with per-user salts — not encryption, which is reversible
- Violates PCI-DSS Requirement 8.3.1 and NIST SP 800-63B guidelines, creating direct compliance liability
- Tools like **grep**, **strings**, and secrets scanners (TruffleHog, GitLeaks) are used in both red-team and blue-team operations to detect this flaw
