# CWE-414

## What it is
Imagine a bank vault with a combination lock, but the lock is never actually spun after closing — it just *looks* locked. CWE-414 (Missing Lock Check) occurs when code acquires a lock to protect a shared resource but fails to verify whether the lock was successfully obtained before proceeding to access that resource. This means the "protected" section runs without guaranteed mutual exclusion, exposing data to race conditions.

## Why it matters
In a multi-threaded server application, if a thread skips verifying a mutex lock was acquired before reading a session token, a concurrent attacker-controlled thread can modify that token mid-read — producing a time-of-check/time-of-use (TOCTOU) style corruption. This has appeared in database connection pool implementations where failed lock acquisition silently proceeds, leading to corrupted transactions or privilege escalation across user sessions.

## Key facts
- **Root cause**: The lock *call* is made but the *return value* is never checked; many locking APIs (e.g., `pthread_mutex_trylock`) return non-zero on failure rather than blocking.
- **Consequence**: Violates mutual exclusion — two threads can simultaneously enter a critical section, causing data races, corruption, or security boundary bypass.
- **Common languages**: Appears frequently in C/C++ and Java where locking is manual and errors are silent by default.
- **Detection**: Static analysis tools (Coverity, Fortify) flag unchecked return values from lock primitives; TSAN (ThreadSanitizer) catches it dynamically.
- **CWE relationship**: Child of CWE-667 (Improper Locking); sibling to CWE-362 (Race Condition).
