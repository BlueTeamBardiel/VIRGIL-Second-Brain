# T1059.010

## What it is
Like a puppeteer pulling strings through a stage curtain, AutoHotkey and AutoIT let attackers script complex actions while hiding behind the appearance of legitimate automation tools. These are scripting languages designed for Windows GUI automation and task scripting that adversaries abuse to execute malicious payloads, bypass defenses, and automate attacker workflows — all while blending in with normal sysadmin activity.

## Why it matters
In documented RAT delivery campaigns, attackers have compiled AutoIT scripts into standalone executables (.exe) that download and execute remote access trojans. Because AutoIT compilers produce legitimate-looking binaries and are widely used in enterprise IT, endpoint defenses that rely on reputation or binary signatures often fail to flag them, giving attackers a reliable initial access and persistence mechanism.

## Key facts
- AutoHotkey (AHK) uses `.ahk` script files; AutoIT uses `.au3` files — both can be compiled into self-contained `.exe` files that embed the interpreter, making analysis harder
- Both tools can call Windows APIs, manipulate processes, write to the registry, and send keystrokes — essentially full system access through a "legitimate" scripting engine
- Defenders should monitor for `AutoHotkey.exe` or `AutoIt3.exe` spawning child processes, especially network-connecting or file-writing activity
- Script compilation artifacts and the presence of interpreter executables in unusual directories (e.g., `%TEMP%` or `%APPDATA%`) are strong indicators of abuse
- This technique is categorized under Command and Scripting Interpreter (T1059), sharing the same parent as PowerShell, VBScript, and Python abuse techniques
