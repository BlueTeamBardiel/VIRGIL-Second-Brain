# CWE-125

## What it is
Like a librarian reaching past the last shelf to grab a book that doesn't exist — and pulling back random junk from the storage room — an out-of-bounds read occurs when a program reads memory *beyond* the allocated buffer boundary. Precisely: CWE-125 is a memory safety weakness where software reads data from a memory location before the start or after the end of an intended buffer, returning unintended data to the caller.

## Why it matters
The 2014 Heartbleed vulnerability (CVE-2014-0160) is the canonical CWE-125 example: OpenSSL's heartbeat handler trusted a user-supplied length value without validation, reading up to 64KB of server memory beyond the intended buffer — leaking private keys, session tokens, and passwords with no authentication required. Defenders patch this by enforcing strict bounds checking before every read operation.

## Key facts
- **CWE-125 is read-only**; it reads past boundaries rather than writing (contrast with CWE-787, out-of-bounds write). This means it typically causes information disclosure rather than direct code execution.
- The CVSS impact is primarily **Confidentiality**, not Integrity or Availability — though crashes (DoS) can also result.
- Root cause is almost always **trusting attacker-controlled length or index values** without validating them against actual buffer size.
- Common in **C/C++** where manual memory management provides no automatic bounds enforcement; memory-safe languages (Rust, Go) largely eliminate this class.
- Appears in **Top 25 Most Dangerous Software Weaknesses** (MITRE CWE list), making it high-priority for secure code review and SAST tooling.
