# CWE-338 Use of Cryptographically Weak PRNG

## What it is
Imagine a casino shuffling cards with a machine that always starts from the same position if you know the time it was turned on — a determined player could predict every hand dealt. A cryptographically weak Pseudo-Random Number Generator (PRNG) works exactly like this: it produces sequences that *look* random but are statistically predictable to an attacker who can determine or guess the seed value or algorithm state.

## Why it matters
In 2008, researchers cracked the Debian OpenSSL vulnerability (CVE-2008-0166), where a flawed patch accidentally reduced the PRNG seed space to only ~32,768 possible values — attackers could brute-force all possible SSH and SSL keys in hours. Any session token, cryptographic key, or nonce generated by a weak PRNG becomes guessable, enabling session hijacking, key reconstruction, or authentication bypass.

## Key facts
- **Standard PRNGs** (e.g., `rand()` in C, `Math.random()` in JavaScript) use linear congruential generators — fast but *never* suitable for security purposes
- **CSPRNGs** (Cryptographically Secure PRNGs) like `/dev/urandom`, `CryptGenRandom()`, or `java.security.SecureRandom` are the required alternatives
- Weak PRNGs are frequently seeded with predictable values like system timestamps, making seed guessing trivial
- This weakness commonly leads to predictable session tokens, weak encryption keys, and bypassable CAPTCHA/nonce values
- NIST SP 800-90A defines approved deterministic random bit generators (DRBGs) for cryptographic use — knowing this standard is exam-relevant
