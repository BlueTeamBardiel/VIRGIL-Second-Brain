# T1078.004

## What it is
Like using a master key that opens every door in a hotel because it was left hanging at the front desk, cloud accounts abuse valid credentials to access cloud environments without triggering "break-in" alerts. Precisely: adversaries use legitimate cloud provider accounts (AWS IAM, Azure AD, GCP service accounts) that were compromised, misconfigured, or exposed to authenticate and operate within cloud infrastructure as authorized users.

## Why it matters
In the 2019 Capital One breach, a misconfigured AWS WAF allowed an attacker to exploit an SSRF vulnerability and retrieve IAM role credentials from the EC2 metadata service — effectively stealing a valid cloud identity and exfiltrating over 100 million customer records while appearing as a legitimate internal service. Defenders must monitor for unusual API call patterns from valid accounts, not just failed authentication attempts.

## Key facts
- Cloud account credentials exposed in **EC2 instance metadata (169.254.169.254)** are a primary attack vector for credential theft in AWS environments
- Service accounts and IAM roles with **overly permissive policies** (e.g., `AdministratorAccess`) dramatically amplify the blast radius of a compromised cloud identity
- Adversaries commonly use stolen cloud credentials to perform **discovery actions** (`ListBuckets`, `DescribeInstances`) before escalating to data exfiltration or persistence
- Detection relies heavily on **CloudTrail, Azure Monitor, and GCP Audit Logs** — anomalies like unusual geographic logins or API calls from new user agents are key indicators
- Unlike on-prem attacks, cloud account abuse often **leaves no malware footprint**, making behavioral analytics (UEBA) the primary detection mechanism
