# T1055.005

## What it is
Like a pickpocket slipping their hand into someone else's jacket to do their dirty work, Thread Local Storage (TLS) injection plants malicious code into a process by abusing the TLS callback mechanism — functions Windows automatically executes before a thread's main entry point runs. Precisely, it is a process injection technique where an attacker writes shellcode into a target process and registers it as a TLS callback, causing it to execute with the privileges and identity of that host process.

## Why it matters
TLS injection was observed in sophisticated malware like Ramnit and early APT loaders specifically because TLS callbacks execute *before* the main executable entry point — meaning security tools watching for suspicious thread creation via `CreateRemoteThread` can be completely bypassed. A defender tuning EDR rules on standard injection APIs (e.g., `CreateRemoteThread`, `NtCreateThreadEx`) would miss this technique unless they specifically monitor for TLS callback table modifications in PE headers.

## Key facts
- TLS callbacks are defined in the PE header's TLS directory (`IMAGE_TLS_DIRECTORY`) and are called automatically by the Windows loader for every thread creation.
- Attackers typically use `VirtualAllocEx` + `WriteProcessMemory` to stage shellcode, then overwrite the TLS callback address in the target process's TLS directory.
- Execution occurs in the context of the **target process**, inheriting its token, privileges, and network identity — ideal for defense evasion.
- This technique can execute code **before debuggers fully attach**, complicating dynamic analysis and sandbox detection.
- Detection focus: monitor for writes to `.tls` section regions and anomalous PE header modifications at runtime using tools like Sysmon (Event ID 8) or memory scanning.
