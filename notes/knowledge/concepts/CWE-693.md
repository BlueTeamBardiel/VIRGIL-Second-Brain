# CWE-693

## What it is
Like a fortress that relies solely on a locked front gate while leaving the side door, roof hatch, and basement window unguarded — this weakness describes software that depends on a **single protection mechanism** as its entire security strategy. CWE-693 (Protection Mechanism Failure) occurs when a system fails to implement or properly use a security control, allowing attackers to bypass it entirely rather than defeat it technically.

## Why it matters
In 2017, Equifax's breach stemmed partly from failing to apply Apache Struts patches — their perimeter controls were the *only* relied-upon layer, and once the vulnerability existed in the application itself, there was no compensating control to prevent data exfiltration. Had defense-in-depth been implemented (network segmentation, egress filtering, anomaly detection), the blast radius would have been drastically reduced.

## Key facts
- CWE-693 is a **class-level weakness** — it's a category that encompasses child weaknesses like CWE-184 (incomplete allowlist) and CWE-807 (reliance on untrusted inputs in security decisions)
- It directly maps to the **defense-in-depth** principle violation; Security+ explicitly tests that no single control should be a single point of failure
- Common manifestations include: relying only on client-side validation, trusting only a single authentication factor, or depending solely on obscurity
- MITRE maps CWE-693 to **CAPEC-2** (Inducing Account Lockout) and other bypass attack patterns where one failed gate = full access
- The fix is **layered controls**: preventive + detective + corrective mechanisms so no single failure is catastrophic
