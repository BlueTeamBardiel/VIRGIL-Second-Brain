# T1059.007

## What it is
Like a Swiss Army knife hidden inside a web browser, JavaScript can be weaponized far beyond displaying web pages. T1059.007 describes adversaries using JavaScript (and related technologies like JScript) as a scripting interpreter to execute malicious commands, download payloads, or establish persistence — all through an engine trusted by the operating system.

## Why it matters
The Lazarus Group leveraged malicious JavaScript embedded in watering hole attacks targeting financial institutions, where visiting a compromised site silently executed JS that profiled victims and dropped secondary loaders. Defenders watching for `wscript.exe` or `cscript.exe` spawning unusual child processes can catch JScript-based attacks that bypass traditional macro-focused detections.

## Key facts
- JavaScript/JScript can execute outside the browser via Windows Script Host (`wscript.exe`, `cscript.exe`), making it a native living-off-the-land binary (LOLBin) scenario
- Malicious `.js` or `.jse` (encoded JScript) email attachments are a common delivery vector, often disguised as invoices or shipping notices
- Node.js can also be abused under this technique — attackers who plant Node.js in an environment can run fully capable server-side JS payloads
- Defense evasion is built-in: JS is interpreted, obfuscation is trivial, and script content is rarely inspected by endpoint solutions focused on PE files
- MITRE maps this under the parent technique T1059 (Command and Scripting Interpreter), alongside PowerShell (T1059.001) and Python (T1059.006)
