# CWE-277

## What it is
Like a bank vault that keeps its combination written on a sticky note during a "temporary" renovation that never ends, CWE-277 occurs when insecure permissions assigned during an installation or setup phase are never tightened afterward. Specifically, it describes inadequate protection of privileges during a system's lifecycle — permissions granted for convenience during initialization persist into production, creating a permanently exposed attack surface.

## Why it matters
During a web application deployment, a developer grants world-readable permissions to a configuration directory containing database credentials to "get things working," intending to fix it later. An attacker who compromises any low-privilege process on that server can read the credentials directly, pivot to the database, and exfiltrate sensitive data — all because a temporary permission was never revoked. Defense requires automated post-install hardening scripts that enforce least privilege as part of the deployment pipeline.

## Key facts
- CWE-277 falls under the broader **CWE-732** (Incorrect Permission Assignment) family, but specifically targets the *lifecycle phase* where permissions degrade over time or are never corrected after initialization.
- Common in software installers that require elevated rights during setup but fail to drop those privileges once installation completes.
- Frequently exploited in **privilege escalation** attacks — a low-privilege user inherits or discovers lingering elevated permissions left from the install phase.
- Automated configuration management tools (Ansible, Chef, Puppet) help mitigate this by enforcing known-good permission states continuously, not just at install time.
- CVSS scoring for vulnerabilities in this class often reflects **high confidentiality/integrity impact** when the over-privileged resource contains credentials or executable code.
