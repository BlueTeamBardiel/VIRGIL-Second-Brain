# CWE-415

## What it is
Like a parking valet who hands your car keys to two different people — the second handoff is catastrophic because the resource is already gone. Double Free is a memory vulnerability where a program calls `free()` on the same memory address twice, corrupting the heap allocator's internal structures and creating exploitable conditions.

## Why it matters
In 2019, a double free vulnerability in the Linux kernel's DCCP networking code (CVE-2017-6074) allowed local privilege escalation — an attacker could manipulate the corrupted heap to overwrite function pointers and execute arbitrary code as root. Defenders mitigated this class of bug by enabling heap hardening features like glibc's `MALLOC_CHECK_` environment variable and Address Space Layout Randomization (ASLR).

## Key facts
- A double free corrupts heap metadata (forward/back pointers in free-list chunks), which attackers leverage to write arbitrary values to arbitrary memory addresses — the classic "write-what-where" primitive
- Modern allocators like `tcmalloc` and hardened glibc detect double frees via "safe-unlinking" checks and pointer encryption, but not all environments enable these
- The vulnerability is categorized under **CWE-415**, a child of **CWE-825** (Expired Pointer Dereference) and sibling of **CWE-416** (Use-After-Free)
- Common root cause: complex ownership semantics — two code paths each believe they own the pointer and are responsible for freeing it
- Mitigations include setting pointers to `NULL` immediately after `free()` (nullifying dangling references), using smart pointers in C++ (`std::unique_ptr`), and fuzzing with AddressSanitizer (ASan) to catch double frees at runtime
