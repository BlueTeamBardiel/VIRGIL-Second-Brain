# CWE-130

## What it is
Imagine a shipping dock worker who reads the label on a box to know how many items to unload — but a malicious sender lies on the label, saying "1000 items" when only 10 are inside. The worker allocates a massive pallet for nothing. CWE-130 (Improper Handling of Length Parameter Inconsistency) occurs when software uses an attacker-controlled length field that doesn't match the actual data size, causing incorrect memory allocation, reads, or writes.

## Why it matters
The classic exploitation path appears in protocols like TLS — most infamously, the Heartbleed bug (CVE-2014-0160). An attacker sent a HeartbeatRequest with a payload of 1 byte but declared a length of 65,535 bytes, causing OpenSSL to read and return up to 64KB of adjacent server memory, leaking private keys, passwords, and session tokens without any authentication.

## Key facts
- The length value and actual data size are attacker-controlled and inconsistent, causing the program to trust the declared length over reality
- Can lead to buffer over-read (reading beyond intended bounds), heap disclosure, or memory corruption depending on how the length is consumed
- Distinct from a classic buffer overflow: no overflow of a target buffer is required — the vulnerability is in *trusting* a mismatched length value
- Commonly found in binary protocols, custom serialization formats, and TLS/SSL implementations
- Mitigations include validating that declared length ≤ actual received data length *before* any allocation or copy operation
