# CWE-791

## What it is
Like a bouncer who only checks IDs at the front door but lets anyone slip in through the side entrance, this flaw occurs when an application performs input filtering in one location but not consistently everywhere the data flows. CWE-791 is "Incomplete Filtering of One or More Instances of Special Elements" — a vulnerability where sanitization logic misses at least one code path, allowing malicious input to reach sensitive sinks unfiltered.

## Why it matters
A classic manifestation appears in SQL injection defenses: a developer sanitizes user input in the login form handler but forgets to apply the same filter to a password-reset endpoint. An attacker probes alternate entry points, finds the unguarded route, and achieves full database compromise despite the primary filter being correctly implemented. Defense requires mapping *every* data entry point, not just the obvious ones.

## Key facts
- CWE-791 is a child of CWE-790 (Incomplete Filtering of Special Elements) and a descendant of CWE-20 (Improper Input Validation)
- The flaw is about *coverage gaps* — the filtering logic itself may be correct, but it's simply not applied universally
- Common in large codebases where multiple developers handle different endpoints without a shared input-validation library
- Discovered frequently during code audits by tracing data flow from all sources to sinks and checking whether sanitization exists on every path
- Mitigation best practice: centralize input validation in a single trusted library/layer rather than implementing ad-hoc checks per endpoint
