# CWE-1220

## What it is
Like a nightclub bouncer who checks ID at the front door but forgets there's also a side entrance — insufficient compartmentalization means security controls only protect *some* pathways to a sensitive resource, leaving others wide open. Precisely: CWE-1220 occurs when a product fails to adequately separate or isolate different components, processes, or data domains, allowing an attacker to bypass intended security boundaries by accessing the system through an unprotected channel.

## Why it matters
In embedded systems and SoC (System-on-Chip) designs, debug interfaces like JTAG are sometimes left electrically connected to sensitive hardware blocks even when "disabled" in software — a classic compartmentalization failure. An attacker with physical access can probe the debug port and directly read cryptographic keys or firmware from memory regions that the software stack believed were protected, bypassing all authentication logic entirely.

## Key facts
- CWE-1220 is classified under the **Hardware Design** category and frequently appears in IoT and embedded security assessments
- Compartmentalization failures often manifest as **shared memory regions** between trusted and untrusted processes with no enforced access controls
- This weakness differs from CWE-653 (Insufficient Compartmentalization) in scope — CWE-1220 specifically targets **inadequate separation** rather than absence of separation
- Defense-in-depth architectures mitigate this by enforcing isolation at **multiple layers**: hardware, hypervisor, OS kernel, and application
- Common in **multi-tenant cloud environments** where VM-to-VM isolation depends entirely on hypervisor integrity; a compartmentalization gap can enable cross-tenant data leakage
