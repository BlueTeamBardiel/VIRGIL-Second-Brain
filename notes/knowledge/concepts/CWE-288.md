# CWE-288

## What it is
Like a nightclub with a VIP entrance around back that bypasses the bouncer entirely, this weakness occurs when a system implements authentication on its primary interface but leaves an alternative path completely unguarded. CWE-288 (Authentication Bypass Using an Alternate Path or Channel) describes a flaw where an attacker circumvents authentication controls by using a different communication channel, API endpoint, or access method that the developer forgot to protect.

## Why it matters
In 2020, attackers targeted misconfigured Kubernetes dashboards by accessing the management API directly on port 10250 instead of the authenticated web UI — harvesting credentials and executing commands without ever touching the login page. Defenders must enumerate *all* access paths to a resource, not just the obvious front door, and ensure authentication is enforced uniformly across every channel.

## Key facts
- The bypass doesn't break the authentication mechanism itself — it simply routes around it entirely, making it harder to detect in code review
- Common real-world instances include: unauthenticated debug interfaces, direct database ports exposed alongside an authenticated app layer, and internal API endpoints reachable from external networks
- This differs from CWE-287 (Improper Authentication) — here authentication *exists* but is *not applied* to alternate paths, rather than being implemented incorrectly
- Mitigation requires a "deny by default" architecture where all paths require explicit authentication grant, not just the primary interface
- Common in IoT devices where a web UI is secured but Telnet/UART debug ports remain open without credentials
