# T1055.008

## What it is
Like a parasite that hijacks a healthy organism's body to avoid detection, this technique injects malicious code into a running process by manipulating its execution environment. Ptrace Injection (T1055.008) is a Linux/Unix process injection method where an attacker uses the `ptrace()` system call — normally reserved for debuggers — to attach to a running process, write shellcode into its memory, and redirect execution to that payload.

## Why it matters
In a documented APT campaign targeting Linux servers, attackers gained initial access via a web shell, then used ptrace injection to implant a backdoor inside a legitimate `apache2` worker process — making the malicious code invisible to basic process listings while inheriting the web server's network permissions. Defenders caught it only by monitoring for unexpected `ptrace()` calls from non-debugger processes using auditd rules.

## Key facts
- `ptrace()` is a POSIX system call used legitimately by tools like `gdb` and `strace`; abuse requires the attacker to have equivalent or higher privileges than the target process
- The injection workflow: `ptrace(ATTACH)` → write shellcode via `PTRACE_POKEDATA` → redirect instruction pointer with `PTRACE_SETREGS` → `ptrace(DETACH)`
- Linux kernel hardening via `ptrace_scope` (Yama LSM) restricts ptrace to parent processes by default — setting `/proc/sys/kernel/yama/ptrace_scope` to `1` or higher mitigates casual abuse
- This technique evades many AV solutions because the malicious code lives only in the memory of a legitimate process — no new process is spawned
- Detection signatures include: unprivileged processes issuing `ptrace(PTRACE_ATTACH)`, seccomp policy violations, and unusual `/proc/[pid]/mem` access patterns
