# CWE-669

## What it is
Like a bank teller who hands a customer both their cash *and* another customer's account statement in the same envelope — CWE-669 is **Incorrect Resource Transfer Between Spheres**: when software moves data or resources from one trust zone to another without applying appropriate controls or sanitization during the handoff.

## Why it matters
In 2021, a common manifestation appeared in server-side request forgery (SSRF) attacks where web applications forwarded user-supplied URLs to internal cloud metadata APIs (like AWS IMDSv1 at `169.254.169.254`). The application blindly transferred an external user request into the internal network sphere, exposing IAM credentials. Blocking this requires explicit validation that the resource being transferred belongs in the destination trust context.

## Key facts
- **Root cause**: the software fails to enforce trust-boundary rules when a resource crosses from one security domain (e.g., public internet) to another (e.g., internal network or kernel space).
- **Common child weaknesses**: SSRF (CWE-918), path traversal (CWE-22), and symlink following (CWE-61) are all specializations of this broader pattern.
- **Not just data**: the "resource" can be files, network connections, process handles, or memory references — anything that carries privilege or sensitivity.
- **Defense pattern**: apply the principle of least privilege *at the boundary* — validate, strip, or re-label the resource before it crosses trust zones, never after.
- **CVSS relevance**: vulnerabilities rooted in CWE-669 frequently achieve high Confidentiality impact scores because they expose resources the attacker was never authorized to reach.
