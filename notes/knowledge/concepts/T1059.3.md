# T1059.3

## What it is
Like a maintenance worker using a building's master key to move through every room undetected, attackers use Windows Command Shell (cmd.exe) — a tool already trusted by the OS — to execute malicious commands without raising suspicion. T1059.3 describes adversary abuse of the Windows Command Shell (`cmd.exe`) to execute commands, scripts, and batch files as part of an attack chain.

## Why it matters
During the 2017 NotPetya campaign, attackers used `cmd.exe` to chain together commands that spread laterally, wiped MBRs, and disabled recovery options — all through a tool Windows treats as legitimate. Defenders monitoring for `cmd.exe` spawned from unusual parent processes (like `winword.exe` or `outlook.exe`) can catch phishing-to-execution chains early, making process lineage analysis a critical detection control.

## Key facts
- `cmd.exe` can execute inline commands (`/c`), run batch scripts (`.bat`, `.cmd`), and chain multiple commands using `&&`, `||`, and `|` — making it highly flexible for attackers
- A strong detection signal: `cmd.exe` spawned as a child of Office applications (`winword.exe`, `excel.exe`) almost always indicates malicious macro execution
- Attackers use `cmd.exe` to call other utilities (e.g., `net.exe`, `sc.exe`, `reg.exe`) rather than doing malicious work directly — creating multi-hop command chains that complicate attribution
- Logging via Windows Event ID **4688** (Process Creation) with command-line auditing enabled captures `cmd.exe` invocations — a critical SIEM data source
- Defense: Application control policies (AppLocker, WDAC) can restrict which users or processes are permitted to launch `cmd.exe`
