# T1003.008

## What it is
Like a thief who knows a building's master key is stored in a specific drawer in every office, attackers know exactly where Linux systems keep password data. T1003.008 refers to credential dumping via `/etc/passwd` and `/etc/shadow` — the two files that together store Linux user account names, metadata, and hashed passwords. Reading `/etc/shadow` (which holds the actual hashes) requires root privileges, making privilege escalation a prerequisite.

## Why it matters
During the 2021 Kaseya supply chain breach, post-exploitation tools on compromised Linux endpoints targeted shadow files to harvest credentials for lateral movement across managed service provider networks. Defenders monitoring for unauthorized reads of `/etc/shadow` via auditd rules can catch this activity before hashes are cracked offline and reused.

## Key facts
- `/etc/passwd` is world-readable and contains usernames, UIDs, GIDs, and shell info — but passwords are replaced with `x`, pointing to `/etc/shadow`
- `/etc/shadow` stores password hashes in format: `$algorithm$salt$hash` (e.g., `$6$` = SHA-512)
- Root access (or equivalent) is required to read `/etc/shadow` — making this a post-privilege-escalation technique
- Attackers use tools like `unshadow` (combines both files) then feed output to `hashcat` or `john` for offline cracking
- Detection: auditd rule `-w /etc/shadow -p r -k shadow_read` flags any read attempt on the shadow file
