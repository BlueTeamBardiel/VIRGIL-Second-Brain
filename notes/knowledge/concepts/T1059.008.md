# T1059.008

## What it is
Like a master key hidden inside a building's own intercom system, Network Device CLI abuse lets attackers use the router or switch's own command-line interface as their weapon. Precisely, this technique involves adversaries issuing commands directly through the native CLI of network infrastructure devices (Cisco IOS, Juniper Junos, etc.) to execute malicious actions, persist, or exfiltrate data — without ever touching an endpoint OS.

## Why it matters
In the 2022 Cisco IOS XE campaign, threat actors exploited CVE-2023-20198 to gain root access and then used the IOS CLI to create backdoor accounts and install implants directly on the device — completely bypassing EDR tools that only monitor Windows/Linux endpoints. Defenders who only watch host-based logs would be entirely blind to this activity.

## Key facts
- Targets include routers, switches, firewalls, and load balancers running proprietary operating systems (IOS, JunOS, EOS, NX-OS)
- Attackers commonly access the CLI via SSH, Telnet, or through web-based management interfaces after credential theft or exploitation
- Commands like `ip route`, `copy run tftp`, and `debug` can be weaponized for reconnaissance, config exfiltration, or traffic manipulation
- Traditional EDR/AV is **ineffective** — network devices rarely support endpoint agents, making CLI-based attacks nearly invisible to standard SIEM pipelines
- Detection relies on NetFlow analysis, AAA logging (TACACS+/RADIUS), and configuration change monitoring via SNMP traps or RANCID/Oxidized diffs

## Why it matters
Detection requires enabling `archive` logging on Cisco devices and shipping logs to a centralized syslog server — native device storage is too limited and too easily wiped by an attacker already holding privileged CLI access.
