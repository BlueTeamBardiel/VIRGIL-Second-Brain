# T1055.004

## What it is
Like a thief who doesn't break into a store but instead hides inside a mannequin already on display, Asynchronous Procedure Call (APC) Injection works by sneaking malicious code into the legitimate execution queue of an already-running thread. Specifically, attackers inject shellcode into a target process and then queue an APC function call to a thread in alertable state, forcing that thread to execute the malicious payload the next time it "wakes up" to check its queue.

## Why it matters
The AtomBombing attack technique and tools like Cobalt Strike leverage APC injection because the malicious code executes under the memory space of a trusted, legitimate process — making it appear benign to endpoint detection tools checking process origin. Defenders must look beyond *what* is running to *where* it was queued and *how* threads transition into alertable states (e.g., via `SleepEx`, `WaitForSingleObjectEx`).

## Key facts
- APC queues exist per-thread in Windows; a thread must be in **alertable state** to execute queued APCs, triggered by specific Win32 wait functions
- Attackers commonly use `QueueUserAPC()` to inject their shellcode pointer into a target thread's APC queue
- **Early Bird APC injection** is a variant that injects before the process fully initializes, bypassing many AV hooks that activate post-load
- Detection focus: monitor for `WriteProcessMemory` + `QueueUserAPC` API call combinations from unexpected parent processes
- Unlike classic DLL injection, APC injection does **not** require `CreateRemoteThread`, making it stealthier against detections tuned for that call
