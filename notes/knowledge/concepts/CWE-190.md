# CWE-190

## What it is
Like a car odometer that rolls back to 00000 after hitting 99999, an integer overflow occurs when arithmetic pushes a value past the maximum boundary a data type can hold, causing it to wrap around to a small (or negative) number. Precisely: CWE-190 is an integer overflow or wraparound vulnerability where a computation produces a result too large for its storage type, yielding an unexpected truncated value. This corrupted value is then used by the program as if it were valid.

## Why it matters
In 2002, an integer overflow in the Sun RPC XDR library allowed attackers to trigger a heap buffer overflow by sending a crafted packet that caused a length variable to wrap to a tiny value, leading to remote code execution across thousands of systems. The root cause was a multiplication overflow — `nelem * elsize` wrapped to near-zero, causing `malloc()` to allocate far too little memory before data was written into it. This pattern — overflow leading to undersized allocation leading to heap corruption — is one of the most dangerous exploitation chains in memory safety.

## Key facts
- Integer overflow is **undefined behavior in C/C++** for signed integers, but wraps predictably for unsigned integers (modulo 2ⁿ)
- A classic trigger: `malloc(user_input * sizeof(int))` — if `user_input = 1073741825` and `sizeof(int) = 4`, the multiplication overflows a 32-bit int to 4 bytes allocated
- CWE-190 is distinct from CWE-191 (integer underflow) — overflow goes above MAX, underflow goes below MIN
- Safe mitigations include: compiler flags (`-ftrapv`), SafeInt libraries, and explicit pre-multiplication bounds checks
- Frequently appears in **image parsers, network protocol handlers, and memory allocators** — high-value targets for exploitation
