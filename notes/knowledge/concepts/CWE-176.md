# CWE-176

## What it is
Like a translator who mishears a foreign accent and writes down the wrong word entirely, a system that improperly handles Unicode encoding can misinterpret byte sequences and process characters in completely unintended ways. CWE-176 refers to software that fails to correctly handle Unicode encoding, leading to incorrect character interpretation, bypassed input validation, or unexpected behavior when multibyte character sequences are processed.

## Why it matters
In 2008, researchers demonstrated that IIS web servers could be tricked using overlong UTF-8 encodings (e.g., representing `/` as `%c0%af`) to bypass path traversal filters — the validator saw a "safe" byte sequence while the decoder reconstructed a dangerous path. This class of bug allows attackers to sneak forbidden characters past security filters by encoding them in ways the validator doesn't recognize but the backend happily decodes.

## Key facts
- UTF-8 allows multiple byte representations of the same character (overlong encodings); secure parsers must reject non-canonical forms
- Unicode normalization forms (NFC, NFD, NFKC, NFKD) can transform visually identical strings differently — validation must occur **after** normalization
- Homograph attacks exploit CWE-176 adjacent behavior: characters from different scripts look identical to humans but differ in encoding (e.g., Cyrillic 'а' vs. Latin 'a')
- The "Best Fit" mapping in Windows can silently convert Unicode characters to ASCII equivalents, causing sanitized input to mutate after validation
- OWASP classifies improper Unicode handling as an input validation failure — relevant to both injection and access control bypass vulnerabilities
