# CWE-1286

## What it is
Like a bouncer who checks IDs but never verifies they're not expired or counterfeit — just that *something* was presented — this weakness occurs when software fails to properly validate the **type** of input, not just its presence. CWE-1286 is "Improper Validation of Syntactic Correctness of Input," where an application accepts data that is structurally malformed because it only checks whether input exists, not whether it conforms to the expected format or grammar.

## Why it matters
A real-world example: an XML parser that accepts a document with mismatched tags or injected control characters because it never validates against the schema. An attacker can craft a malformed XML payload that confuses downstream parsers into disclosing data or executing unintended logic — a technique seen in XML External Entity (XXE) attack chains. Proper schema validation at the intake point would have rejected the input before it ever reached vulnerable parsing code.

## Key facts
- Distinct from CWE-20 (Improper Input Validation) — CWE-1286 specifically targets **syntactic** correctness (structure/grammar), not just presence or range
- Malformed inputs exploiting CWE-1286 often cause **parser confusion**, where two components interpret the same data differently (a technique called "parsing differentials")
- Common in protocols and file formats with complex grammars: JSON, XML, HTTP headers, MIME, and SQL
- Attackers exploit this to bypass WAFs and security filters that expect well-formed input, smuggling malicious payloads inside structurally deviant data
- Defense: use strict, allowlist-based parsers and validate input against a formal grammar or schema **before** processing
