# T1059

## What it is
Like a building manager handing a master key to a contractor — trusting them completely because they work "inside the system" — attackers abuse command and script interpreters to run malicious code using tools the OS already trusts. T1059 covers adversary use of built-in interpreters (PowerShell, Bash, cmd.exe, Python, etc.) to execute commands directly on a compromised host.

## Why it matters
In the 2020 SolarWinds supply chain attack, adversaries used PowerShell (T1059.001) extensively post-compromise to run encoded commands that evaded signature-based detection — because defenders were reluctant to block PowerShell entirely given its legitimate administrative use. Detecting abnormal interpreter invocations (e.g., PowerShell spawned by a Word macro) is now a core CySA+ skill.

## Key facts
- **T1059 has sub-techniques** by interpreter: .001 PowerShell, .002 AppleScript, .003 Windows Command Shell (cmd.exe), .004 Unix Shell, .006 Python, .007 JavaScript, among others
- **Living-off-the-land (LotL)** describes this tactic — attackers use legitimate, pre-installed tools to avoid dropping suspicious executables
- **PowerShell logging** (Script Block Logging via Group Policy) and **AMSI (Antimalware Scan Interface)** are primary defenses against scripted attacks
- **Constrained Language Mode** in PowerShell limits interpreter capability and is an effective hardening control
- **Detection signal**: watch for base64-encoded command strings (`-EncodedCommand`), interpreters spawned by Office applications, and unusual parent-child process relationships
