# T1098.006

## What it is
Like a thief who doesn't pick the lock but instead gets a copy of the master key added to the building's authorized key list, this technique involves adding adversary-controlled SSH keys to cloud or on-premises accounts. Specifically, T1098.006 (Additional Cloud Credentials – SSH Authorized Keys) covers attackers modifying the `~/.ssh/authorized_keys` file on cloud-hosted VMs to establish persistent, password-free access under a legitimate user's identity.

## Why it matters
During the TeamTNT cryptomining campaign, attackers compromised Docker APIs, then injected their public SSH keys into root's `authorized_keys` on victim cloud instances — surviving reboots, credential rotations, and even user password changes. Defenders who only monitored login failures missed the intrusion entirely because the attacker's SSH logins appeared completely legitimate.

## Key facts
- The `authorized_keys` file lives at `~/.ssh/authorized_keys` and lists every public key permitted to authenticate as that user — adding an entry requires only write access to that file, not knowledge of any password.
- This technique grants **persistent, credential-independent access**: even if the user's password is reset or MFA is enforced on the console, SSH key auth bypasses both.
- Cloud providers (AWS, Azure, GCP) often inject initial SSH keys via instance metadata at boot — attackers can abuse instance metadata APIs or user-data scripts to plant keys at provisioning time.
- Detection relies on **file integrity monitoring (FIM)** on `authorized_keys` and auditing `AuthorizedKeysFile` changes via auditd or cloud-native tools like AWS CloudTrail (for EC2 metadata API calls).
- Mitigation includes enforcing `AuthorizedKeysCommand` to pull keys from a centralized, monitored directory rather than flat files, and disabling direct SSH in favor of managed access tools (AWS SSM Session Manager, Azure Bastion).
