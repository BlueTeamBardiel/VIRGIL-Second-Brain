# T1055.003

## What it is
Like a puppeteer slipping their hand into someone else's glove puppet mid-performance, Thread Execution Hijacking redirects a legitimate thread inside a running process to execute attacker-controlled code instead. Specifically, an attacker suspends a thread in a target process, overwrites its execution context (the CONTEXT structure, particularly the instruction pointer), injects shellcode into the process memory, then resumes the thread — which now runs the malicious code. This is a sub-technique of Process Injection (T1055).

## Why it matters
The Cobalt Strike framework uses thread execution hijacking as a post-exploitation technique to migrate beacon payloads into trusted system processes like `svchost.exe`, making detection extraordinarily difficult because the malicious code runs under a legitimate process's identity and inherits its security token. Defenders relying solely on process creation events will miss this entirely — it requires monitoring API calls like `SuspendThread`, `SetThreadContext`, and `ResumeThread` in sequence.

## Key facts
- Uses Windows APIs in a characteristic sequence: `OpenThread` → `SuspendThread` → `VirtualAllocEx` → `WriteProcessMemory` → `SetThreadContext` → `ResumeThread`
- Bypasses security tools that monitor for new process creation since no child process is spawned
- The malicious code inherits the target thread's privileges, security token, and process context — enabling privilege escalation if targeting a high-privilege process
- Detectable via ETW (Event Tracing for Windows) and tools like Sysmon monitoring for cross-process thread modification
- Operates entirely in memory, leaving minimal disk artifacts, which classifies it as a fileless technique
