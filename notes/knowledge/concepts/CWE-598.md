# CWE-598

## What it is
Like shouting your ATM PIN while asking a bank teller for help, this weakness exposes sensitive data by embedding it in a URL where anyone nearby can hear. CWE-598 occurs when a web application uses HTTP GET requests to transmit sensitive information (passwords, session tokens, API keys) as query string parameters, making them visible in browser history, server logs, and HTTP Referer headers. The data travels in plaintext within the URL itself rather than the protected request body.

## Why it matters
In 2012, a major vulnerability in several OAuth implementations transmitted `access_token` values as GET parameters — those tokens then leaked into third-party analytics services via the Referer header when users clicked external links. An attacker monitoring server logs or a shared analytics dashboard could harvest valid session tokens without ever touching the authentication system. Using POST requests with tokens in the request body would have prevented this lateral exposure entirely.

## Key facts
- GET parameters appear in **browser history**, **server access logs**, **proxy logs**, and **Referer headers** — up to four separate leakage vectors
- The HTTP **Referer header** automatically forwards the full URL (including query strings) to the next site a user navigates to, bypassing TLS protections
- OWASP classifies query-string credential exposure under **A02:2021 – Cryptographic Failures**
- Mitigation requires using **HTTP POST** with credentials in the request body, combined with proper `Cache-Control: no-store` headers
- This is distinct from CWE-312 (cleartext storage) — the vulnerability here is **transmission context**, not encryption failure
