# CWE-674

## What it is
Like a hall of mirrors that never ends, where each reflection spawns two more — uncontrolled recursion is when a function keeps calling itself without a proper base case or depth limit, consuming stack memory until the program crashes. Precisely, CWE-674 occurs when software allows recursive calls to proceed without adequate termination conditions, enabling stack exhaustion or denial-of-service conditions.

## Why it matters
Attackers can craft malicious input — such as a deeply nested XML document, JSON payload, or recursive archive (like a "zip bomb" variant) — that triggers unbounded recursion in a parser. This crashes the service or exhausts memory, achieving a Denial of Service without needing to exploit memory corruption. The 2003 Linden Lab Second Life XML parser crash is a classic example of deeply nested structures triggering recursive parser failure.

## Key facts
- **Stack overflow mechanism:** Each recursive call consumes stack frame memory; without a depth limit, the call stack overflows, typically raising a segmentation fault or stack overflow exception.
- **Common attack vectors:** Deeply nested XML/JSON/HTML, recursive archive formats (zip bombs), and malformed ASN.1 structures are frequent triggers.
- **Distinct from infinite loops:** Recursion exhausts *stack* space quickly; infinite loops exhaust *CPU time or heap memory* — both are DoS but through different mechanisms.
- **Mitigations include:** enforcing explicit recursion depth counters, using iterative algorithms instead of recursive ones for untrusted input processing, and setting stack size limits at the OS/runtime level.
- **Languages most at risk:** C and C++ (no automatic tail-call optimization or stack guards by default); Python enforces a default recursion limit of 1000 to partially mitigate this.
