# CWE-476

## What it is
Like a surgeon who grabs a scalpel that isn't there and cuts anyway, a NULL Pointer Dereference occurs when a program attempts to use a memory address that points to nothing. Precisely: the application dereferences a pointer that has been assigned NULL (or never initialized), causing a crash or, in some cases, exploitable behavior. It falls under the broader category of improper memory handling vulnerabilities.

## Why it matters
In 2021, a NULL pointer dereference in the Linux kernel's `nf_tables` subsystem (CVE-2021-20226) allowed local attackers to trigger a kernel panic or escalate privileges. Defenders watching for unexpected application crashes or kernel oops messages should treat repeated NULL dereferences as potential exploitation attempts rather than simple bugs — attackers can sometimes map memory at address 0x0 and redirect execution there.

## Key facts
- Classified under **CWE-119** (Improper Restriction of Memory Operations) family; consistently appears in MITRE's Top 25 Most Dangerous Software Weaknesses
- On Linux, the historical "mmap_min_addr" mitigation prevents mapping memory at NULL (address 0), closing the classic NULL dereference privilege-escalation path
- Typically results in a **segmentation fault** (SIGSEGV) on Unix/Linux or an **access violation** on Windows — making it a common crash indicator in fuzzing
- Languages like C and C++ are primary victims; memory-safe languages (Rust, Java) eliminate this class by design
- Secure coding fix: always validate pointer values before dereference (`if (ptr != NULL)`) and initialize pointers explicitly at declaration
