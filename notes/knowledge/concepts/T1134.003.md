# T1134.003

## What it is
Like a con artist borrowing a judge's robes to sign their own pardon, Make and Impersonate Token allows an attacker to forge Windows access tokens from scratch — without needing to steal one first. Specifically, an adversary with `SeImpersonatePrivilege` calls `LogonUser()` to create a new token for a known user (using valid credentials), then uses `ImpersonateLoggedOnUser()` to assume that identity within their process.

## Why it matters
During the 2020 SolarWinds campaign, attackers with domain credentials used token manipulation techniques to pivot laterally by impersonating high-privilege accounts without triggering standard pass-the-hash detections. Defenders who only monitor for stolen token usage miss the "create-and-impersonate" variant entirely, making detection rules incomplete if they ignore `LogonUser` API calls originating from unexpected parent processes.

## Key facts
- Requires `SeImpersonatePrivilege` or `SeAssignPrimaryTokenPrivilege` — both are granted by default to service accounts and IIS worker processes, making them common jump-off points
- Differs from T1134.001 (Token Impersonation/Theft) because the token is *created* using known credentials, not stolen from an existing process
- Key Windows APIs involved: `LogonUser()`, `ImpersonateLoggedOnUser()`, `CreateProcessWithTokenW()`
- Detection focus: monitor for `LogonUser` calls with logon type 9 (NewCredentials) from non-LSASS processes using Sysmon Event ID 10 or ETW providers
- Unlike Pass-the-Hash, this technique requires *plaintext* credentials, but avoids writing them to disk if credentials are passed programmatically in memory
