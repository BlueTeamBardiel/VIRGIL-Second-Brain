# T1021.004

## What it is
Like using a master key that every Unix room recognizes without requiring a face check, SSH lets attackers move laterally by slipping authenticated shell sessions between machines. Precisely: SSH (Secure Shell) Remote Services abuse occurs when adversaries leverage valid or stolen SSH credentials, keys, or agent forwarding to pivot through a network and access remote systems.

## Why it matters
In the 2020 SolarWinds-related intrusions, threat actors harvested SSH keys from compromised Linux systems and used them to hop between internal servers without triggering password-based authentication alerts. Defenders who monitor for unusual SSH login times, source IPs, or connections from unexpected user accounts can catch this lateral movement before attackers reach crown-jewel systems.

## Key facts
- **SSH agent hijacking** allows attackers to steal forwarded authentication sockets (`/tmp/ssh-*/`) and impersonate legitimate users to downstream hosts without knowing the private key passphrase
- Default SSH port is **TCP 22**; attackers often tunnel over custom ports to evade detection
- **Authorized_keys file** manipulation is a persistence + lateral movement combo — adding a public key grants future access even after credential rotation
- Unlike password auth, **key-based SSH abuse leaves minimal credential-based alerts**, making it stealthier than brute force or pass-the-hash techniques
- Defenders should audit `~/.ssh/authorized_keys`, enable `AllowUsers` restrictions in `sshd_config`, and log via **auditd** or SIEM for `ssh` process anomalies
