# T1021.007

## What it is
Like a master key that opens a digital window instead of a door, cloud shell services let attackers hop into a fully-authenticated terminal session through a browser — no SSH keys, no VPN required. T1021.007 describes adversary use of cloud provider-native interactive shell environments (AWS CloudShell, Azure Cloud Shell, GCP Cloud Shell) to move laterally and execute commands within a victim's cloud environment using already-compromised credentials.

## Why it matters
In the 2023 Scattered Spider intrusions, attackers used legitimate cloud shell access after stealing SSO credentials to enumerate IAM roles, exfiltrate S3 data, and pivot across cloud services — all while blending into normal administrative activity. Defenders struggle here because cloud shell sessions are often excluded from standard endpoint monitoring and appear as trusted, native management traffic.

## Key facts
- Cloud shells are browser-based, persistent compute environments pre-authenticated via the cloud provider's IAM/identity layer — no separate credential prompt after login
- AWS CloudShell sessions inherit the IAM permissions of the logged-in console user, making privilege level entirely dependent on the compromised account's role
- Artifacts are scarce: commands run in cloud shells may not appear in CloudTrail as granular CLI events the same way direct API calls do, creating detection gaps
- Defenders should monitor `StartSession`, `GetConsoleOutput`, and cloud shell-specific API calls in provider logs as indicators of abuse
- MFA enforcement on cloud console access is the primary control — without it, a stolen password grants immediate cloud shell access
