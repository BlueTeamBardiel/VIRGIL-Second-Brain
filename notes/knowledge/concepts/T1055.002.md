# T1055.002

## What it is
Like a puppeteer cutting the strings of a marionette and replacing them with their own while the audience keeps watching the same show, Portable Executable Injection writes malicious code directly into the memory space of a running process using a legitimate PE (executable) file as the payload. The attacker allocates memory in a remote process, writes a full PE file into it, and manually maps its sections—bypassing the normal Windows loader entirely.

## Why it matters
During the 2017 NotPetya campaign, attackers injected PE payloads into legitimate Windows processes like `rundll32.exe` to execute destructive code while appearing as trusted system activity, evading basic process-name monitoring. Defenders who only watch *what* process is running—not *what code is actually inside it*—will completely miss this technique.

## Key facts
- Unlike basic DLL injection, PE injection does **not** require dropping a file to disk—the entire executable lives only in memory, making forensic recovery significantly harder.
- The attack chain typically uses `VirtualAllocEx` → `WriteProcessMemory` → manual section mapping → `CreateRemoteThread` or `SetThreadContext` to redirect execution.
- PE injection requires fixing the **base relocations** and **import address table** of the injected image manually, since the Windows loader isn't involved.
- Detection focus: look for `VirtualAllocEx` + `WriteProcessMemory` API calls originating from non-standard processes, especially combined with `PAGE_EXECUTE_READWRITE` memory permissions.
- Commonly used to inject into **trusted processes** (`svchost.exe`, `explorer.exe`) to inherit their network trust and evade application-layer firewalls.
