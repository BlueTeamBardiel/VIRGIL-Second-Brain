# CWE-347

## What it is
Like a bouncer who waves people in because they *claim* to be on the VIP list without ever checking it, this weakness occurs when software accepts a cryptographic signature without actually verifying its authenticity or integrity. Precisely: CWE-347 is the failure to properly verify a cryptographic signature before trusting the signed data, allowing attackers to forge or tamper with messages, tokens, or code.

## Why it matters
The 2015 JWT "alg:none" attack exploited this exact weakness — many JSON Web Token libraries accepted tokens where the algorithm was set to "none," meaning no signature was required at all, yet the token was trusted as fully authenticated. Attackers could craft arbitrary tokens with escalated privileges simply by omitting the signature entirely. Patching required libraries to explicitly reject unsigned tokens regardless of what the header claimed.

## Key facts
- **JWT alg:none** is the canonical exploit: setting the algorithm field to `none` bypasses signature verification in vulnerable implementations
- Affects **code signing pipelines** — malware can be distributed as "trusted" software updates if signature checks are skipped or improperly implemented
- Common in **OAuth/OIDC flows** where ID tokens are decoded but the `iss` (issuer) and signature are never cryptographically confirmed
- Root cause is often **trusting attacker-controlled metadata** (e.g., the algorithm field) to determine how — or whether — to verify the signature
- CVSS scores for real-world instances frequently reach **7.5–9.8 (High/Critical)** due to direct authentication bypass impact
