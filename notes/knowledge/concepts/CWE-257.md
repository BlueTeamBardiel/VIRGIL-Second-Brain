# CWE-257

## What it is
Imagine a hotel that stores every guest's room key code in a binder at the front desk — readable by any curious employee. CWE-257 is the weakness where a system stores passwords in a recoverable format, meaning the original plaintext can be retrieved through decryption or direct reading rather than being irreversibly hashed.

## Why it matters
In the 2013 Adobe breach, approximately 153 million user records were exposed — and because Adobe had encrypted (not hashed) passwords, attackers could cross-reference identical ciphertext blocks to crack millions of credentials. Had passwords been stored as salted hashes, recovering the originals would have been computationally infeasible, dramatically limiting the damage.

## Key facts
- **Encryption ≠ hashing**: Encryption is reversible by design; hashing is one-way. Storing passwords encrypted still violates this CWE because a key compromise exposes everything.
- **Acceptable alternatives**: bcrypt, scrypt, Argon2, and PBKDF2 are approved password hashing algorithms — they are slow by design to resist brute-force attacks.
- **Symmetric key risk**: If the decryption key is stored alongside the password database, an attacker who compromises one likely compromises both.
- **Compliance trigger**: Violating CWE-257 can constitute non-compliance with PCI-DSS Requirement 8.2.1 and NIST SP 800-63B guidelines, which explicitly forbid reversible password storage.
- **Detection signal**: During a CySA+ assessment, finding Base64-encoded or AES-encrypted password fields in a database schema is a red flag for CWE-257, not a security control.
