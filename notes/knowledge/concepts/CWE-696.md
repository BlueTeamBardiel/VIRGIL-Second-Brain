# CWE-696

## What it is
Like a surgeon who sterilizes the incision *after* making the cut, CWE-696 describes calling functions in the wrong order — performing operations in an incorrect sequence that violates the assumed preconditions of a system. Formally, it is a weakness where a product calls multiple functions, methods, or operations in an incorrect order, creating a state where security controls or data integrity assumptions no longer hold.

## Why it matters
A classic exploitation of this pattern occurs in authentication flows: if an application checks a user's permissions *after* executing the privileged action rather than before, an attacker can trigger the action window before the gate closes. This was observed in certain OpenSSL implementations where cleanup and initialization functions were called out of sequence, leading to use-after-free conditions that attackers leveraged for arbitrary code execution.

## Key facts
- CWE-696 is a parent of several concrete weaknesses, including improper use of initialization functions and race conditions introduced by sequencing errors
- It differs from CWE-362 (race condition) in that the ordering error is *logical and deterministic*, not timing-dependent
- Common in cryptographic libraries where key generation, seeding, and initialization must follow strict sequences — skipping or reordering steps weakens entropy
- TOCTOU (Time-of-Check to Time-of-Use) vulnerabilities are a closely related class where the *state* changes between the check and the use, often caused by incorrect sequencing
- Secure coding standards (CERT C, MISRA) explicitly mandate initialization-before-use rules to prevent this class of defect
