# CWE-697

## What it is
Like a bouncer who checks if your age "looks about right" instead of comparing it to exactly 18, this weakness occurs when code uses an incorrect or overly loose comparison operator to make a security decision. CWE-697 (Incorrect Comparison) is when a program compares two values using a method that doesn't properly reflect the intended logic, leading to incorrect authorization, authentication, or control-flow outcomes.

## Why it matters
In PHP's early `==` (loose comparison) versus `===` (strict comparison), an attacker submitting the string `"0e123456"` as a password hash could match another hash beginning with `"0e"` because PHP interprets both as scientific notation equaling zero. This "Magic Hash" attack bypassed authentication in real-world login systems, including vulnerable WordPress plugins, simply by exploiting how the comparison was performed rather than breaking the hash itself.

## Key facts
- CWE-697 is a **parent weakness** under the improper comparison category; specific child weaknesses include CWE-595 (Comparison of Object References) and CWE-696 (Incorrect Behavior Order).
- Type coercion in dynamic languages (PHP, JavaScript) is a primary root cause — `"1" == 1` evaluates `true` in both languages.
- In authentication contexts, incorrect comparison can lead to **authentication bypass** without ever cracking a credential.
- Using `strcmp()` in C returns 0 on match, but checking `if(strcmp(a,b))` catches *non-matches*, not matches — a logic inversion that has appeared in CTF and real CVEs.
- Secure coding fixes include using **strict equality operators**, type-safe comparison functions, and language linters that flag loose comparisons in security-sensitive code paths.
