# CWE-284

## What it is
Like a nightclub bouncer who waves everyone through without checking IDs, improper access control means software fails to correctly restrict what authenticated (or unauthenticated) users can do. CWE-284 is the parent category for weaknesses where a system does not sufficiently enforce who can access resources, functions, or data. It encompasses missing authorization checks, incorrect permission assignments, and bypassed access control mechanisms.

## Why it matters
In 2019, a Facebook bug allowed any user to post to any Facebook Group as if they were an admin — a textbook CWE-284 failure where the API endpoint performed authentication but skipped the authorization check for whether the user held admin rights. Attackers regularly probe API endpoints with low-privileged tokens to find endpoints that authenticate the request but forget to verify the caller's role or ownership of the resource being accessed.

## Key facts
- CWE-284 is a **parent weakness** that contains more specific children like CWE-285 (Improper Authorization), CWE-732 (Incorrect Permission Assignment), and CWE-862 (Missing Authorization)
- The distinction from authentication (CWE-287): authentication asks *"Who are you?"*; access control asks *"What are you allowed to do?"*
- Consistently appears in OWASP Top 10 as **Broken Access Control** (ranked #1 in 2021)
- Horizontal privilege escalation (accessing another user's data at the same privilege level) and vertical privilege escalation (gaining higher-privilege functions) are both manifestations of CWE-284
- Common in REST APIs where developers enforce access control on UI routes but forget to enforce it on the underlying API endpoints — a mismatch attackers exploit directly
