# Rainbow Table Attack

## What it is
Imagine a phone book where instead of names mapping to numbers, cracked passwords map to their hashes — a precomputed lookup table that trades storage space for cracking speed. A rainbow table attack uses these massive precomputed tables of hash-to-plaintext mappings to reverse password hashes almost instantly, bypassing the need to hash each guess in real time. It's a time-memory trade-off attack that collapses what could take days of brute forcing into a simple table lookup.

## Why it matters
In 2012, the LinkedIn breach exposed 6.5 million unsalted SHA-1 password hashes. Attackers used rainbow tables to crack the majority of them within hours, because without unique salts, identical passwords produced identical hashes — making bulk lookups trivially fast. This breach directly accelerated industry adoption of salted hashing algorithms like bcrypt and Argon2.

## Key facts
- Rainbow tables are defeated entirely by **salting** — appending a unique random value to each password before hashing, making precomputed tables useless
- They represent a **time-memory trade-off**: attackers invest storage (tables can reach hundreds of GB) to gain cracking speed
- Rainbow tables differ from simple lookup tables by using **hash chains** with reduction functions to compress storage while maintaining coverage
- Common targets are weak, fast hashing algorithms: **MD5, SHA-1, NTLM** — all considered unsuitable for password storage
- On Security+, rainbow tables are the canonical reason why **salting is a required control**, not an optional enhancement, for password storage
