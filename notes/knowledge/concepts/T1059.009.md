# T1059.009

## What it is
Like a contractor who uses the building's own intercom system to issue unauthorized commands, attackers exploit **Cloud API** interfaces — the legitimate management channels built into platforms like AWS, Azure, and GCP — to execute malicious actions through normal-looking API calls. Precisely, T1059.009 describes adversary use of cloud provider APIs to interact with cloud services and execute commands, enumerate resources, or manipulate configurations without deploying traditional malware.

## Why it matters
In the 2019 Capital One breach, the attacker used AWS metadata service API calls to extract credentials from a misconfigured WAF role, then issued S3 API commands to exfiltrate over 100 million customer records. Defenders who monitor only endpoint telemetry would have missed this entirely — the attack lived entirely within legitimate API traffic, making CloudTrail log analysis the critical detection surface.

## Key facts
- Attackers commonly use AWS CLI, Azure PowerShell, or GCP `gcloud` SDK to issue API calls that appear indistinguishable from authorized administrative activity
- Credential theft (via SSRF against metadata endpoints like `169.254.169.254`) typically precedes API abuse — valid tokens are the entry ticket
- CloudTrail (AWS), Azure Monitor Activity Logs, and GCP Cloud Audit Logs are the primary sources for detecting anomalous API call patterns
- Indicators of compromise include unusual `AssumeRole`, `ListBuckets`, or `DescribeInstances` calls from unexpected IP ranges or at odd hours
- Defense-in-depth includes least-privilege IAM policies, MFA for API access, and alerting on high-volume or cross-region API calls
