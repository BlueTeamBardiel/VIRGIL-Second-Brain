# T1003.002

## What it is
Like a locksmith making a wax impression of every key in a building by pressing them into a single block, SAM Database Dumping extracts all local account password hashes from one file. Specifically, T1003.002 refers to adversaries accessing the Security Account Manager (SAM) database — a file at `C:\Windows\System32\config\SAM` — which stores NTLM hashes of local Windows user credentials.

## Why it matters
During the 2017 NotPetya campaign, attackers used credential dumping techniques including SAM extraction to harvest local admin hashes, then weaponized those hashes in Pass-the-Hash attacks to move laterally across networks without ever cracking a single password. Defenders who enforce unique local administrator passwords via LAPS (Local Administrator Password Solution) directly neutralize this lateral movement chain.

## Key facts
- The SAM file is locked by the OS while Windows runs; attackers bypass this using Volume Shadow Copies (`vssadmin`), registry exports (`reg save HKLM\SAM`), or tools like **Mimikatz** and **secretsdump.py**
- Hashes in SAM are stored as **NTLM (NT hashes)** — no salting, making identical passwords produce identical hashes, enabling hash reuse attacks
- **SYSTEM privilege** is required to access the SAM; attackers must first escalate privileges before dumping
- The **SYSTEM registry hive** is also needed alongside SAM to decrypt the hashes, since SAM is encrypted with a key stored in SYSTEM
- Defenders detect this via Event ID **4656/4663** (object access on SAM) and monitoring for `reg save` or `vssadmin` command-line activity
