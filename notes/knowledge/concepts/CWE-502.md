# CWE-502

## What it is
Like accepting a sealed envelope from a stranger and letting whatever's inside rearrange your house — deserialization of untrusted data means your application reconstructs objects from external byte streams without verifying their integrity or safety first. Precisely: CWE-502 occurs when an application deserializes data from an untrusted source, allowing attackers to craft malicious serialized objects that execute arbitrary code or manipulate application logic during reconstruction.

## Why it matters
The 2017 Apache Struts vulnerability (CVE-2017-9805) exploited unsafe XML deserialization to achieve remote code execution — the same attack vector used in the Equifax breach that exposed 147 million records. Defenders mitigate this by implementing allowlists of acceptable classes before deserialization, using data-only formats like JSON with strict schema validation instead of native serialization, and signing serialized data with HMAC to detect tampering.

## Key facts
- Deserialization attacks can achieve **Remote Code Execute (RCE)** — the highest-severity outcome — without any memory corruption
- Java's `ObjectInputStream`, Python's `pickle`, and PHP's `unserialize()` are the classic dangerous deserialization sinks
- Listed as **OWASP Top 10 A08:2021** (Software and Data Integrity Failures), previously its own category (A8:2017)
- Gadget chains are pre-existing code paths in libraries (e.g., Apache Commons Collections) that attackers chain together to execute payloads — no malicious code upload required
- Detection technique: monitor for **deserialization of unexpected class types** using Java agents or serialization filters (introduced in Java 9 via `ObjectInputFilter`)
