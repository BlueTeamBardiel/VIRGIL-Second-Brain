# CWE-266

## What it is
Like a hotel janitor who keeps a master key even after being reassigned to the parking garage, this weakness occurs when a process or user retains elevated privileges longer than necessary — or transfers them incorrectly to child processes. CWE-266 (Incorrect Privilege Assignment) happens when software assigns permissions to an entity that exceed what the task actually requires, creating an expanded attack surface.

## Why it matters
In 2021, a misconfigured Linux service daemon inherited root privileges from its parent process despite only needing read access to a config file. An attacker who exploited a buffer overflow in that daemon instantly owned the machine — a direct consequence of incorrect privilege assignment rather than the vulnerability itself. Proper privilege separation would have limited the blast radius to a sandboxed, low-privilege process.

## Key facts
- Distinct from privilege escalation exploits — this is a *configuration/design* flaw, not an active attack; the privilege is incorrectly assigned *by design* or *by mistake*
- Commonly seen in SUID/SGID binaries on Unix systems where executables run with the file owner's privileges rather than the calling user's
- setuid(0) calls that fail silently but allow execution to continue as root are a classic manifestation
- Violates the **Principle of Least Privilege (PoLP)** — a core concept in both Security+ and CySA+ frameworks
- Found frequently in containerized environments where Docker containers are launched with `--privileged` flags unnecessarily, granting full host kernel access
