# CWE-436

## What it is
Like two diplomats who speak different dialects assuming the other means the same thing by "withdraw," two software components can interpret the same input data in conflicting ways. CWE-436 (Interpretation Conflict) occurs when multiple components of a system parse or process the same data differently, creating a gap that attackers can exploit. The disagreement between interpretations—not any single component's flaw—is the vulnerability.

## Why it matters
HTTP Request Smuggling is the canonical exploitation of this weakness: a front-end load balancer and a back-end server disagree on where one HTTP request ends and the next begins (using `Content-Length` vs. `Transfer-Encoding` headers). An attacker crafts a request that the front-end sees as one message but the back-end splits into two, allowing the attacker to poison the request queue, bypass access controls, or hijack other users' sessions.

## Key facts
- **Not a single-component bug**: The vulnerability only exists in the *space between* two systems—each component may behave "correctly" on its own.
- **HTTP Request Smuggling (CVE patterns)** is the most tested real-world example; CL.TE and TE.CL are the two primary attack variants exploiting header interpretation conflicts.
- **Security tools can be blind to it**: WAFs and IDS appliances may parse traffic differently than the target server, allowing malicious payloads to slip through undetected.
- **URL parsing conflicts** (e.g., `%2F` decoded by one layer but not another) are another common manifestation, enabling path traversal or auth bypass.
- **Mitigation** requires enforcing a single canonical parsing layer or ensuring all components use identical, strictly validated parsing logic—not just patching one end.
