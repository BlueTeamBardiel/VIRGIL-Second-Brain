# T1218.015

## What it is
Like hiring a licensed contractor to smuggle contraband past a security checkpoint, this technique uses **Electron applications** — legitimate, signed apps built on web technologies — to execute malicious JavaScript and bypass application controls. Specifically, adversaries abuse Electron's `--inspect` or `--inspect-brk` debug flags to inject arbitrary Node.js code into trusted processes like Slack, VS Code, or Teams.

## Why it matters
In a 2023 campaign, threat actors abused the `--inspect` flag against a legitimate Electron-based app to achieve code execution without triggering application whitelisting controls, since the parent process was a signed, trusted binary. Defenders who only monitor for unsigned executables completely miss this vector because the malicious payload rides inside a recognized application's process.

## Key facts
- Electron apps expose a Node.js runtime, meaning JavaScript executed inside them has full OS-level access — file system, network, child processes — not just browser sandbox privileges
- The `--inspect=PORT` flag opens a Chrome DevTools Protocol (CDP) websocket that accepts arbitrary code execution if reachable
- Attackers can also modify an Electron app's `app.asar` archive (the bundled source code) to persist malicious JavaScript across restarts
- This technique bypasses application whitelisting (e.g., AppLocker, WDAC) because the abused binary is legitimately signed
- Detections focus on: anomalous command-line arguments containing `--inspect`, unexpected network listeners on debug ports (default 9229), and `asar` file modifications

## Why it matters
Electron's combination of web technology flexibility and native OS access makes it a high-value target — defenders should treat Electron apps like interpreted scripting engines, not ordinary executables.
