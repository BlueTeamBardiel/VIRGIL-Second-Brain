# T1003.004

## What it is
Like a thief who knows exactly which drawer the spare keys are kept in, attackers target the LSA Secrets store because it holds some of the most valuable credentials on a Windows system. LSA Secrets is a protected registry location (`HKLM\SECURITY\Policy\Secrets`) where Windows stores service account passwords, cached domain credentials, and VPN/dial-up credentials in an encrypted but recoverable format. With SYSTEM-level privileges, an attacker can decrypt and extract these secrets using tools like `reg.exe` or Mimikatz.

## Why it matters
In the SolarWinds supply chain attack, post-compromise lateral movement relied heavily on credential harvesting techniques including LSA Secret dumping to impersonate service accounts across the enterprise. Because many organizations run services under domain accounts, a single LSA Secret dump can yield credentials that allow an attacker to pivot from a workstation to critical servers without triggering obvious authentication anomalies.

## Key facts
- LSA Secrets are stored encrypted in `HKLM\SECURITY\Policy\Secrets` and require SYSTEM privileges to access — Administrator alone is insufficient without privilege escalation
- Common secrets extracted include: `$MACHINE.ACC` (machine account password), `_SC_<ServiceName>` (service credentials), `DPAPI_SYSTEM`, and cached RAS/VPN credentials
- Tools used: `Mimikatz` (`lsadump::secrets`), `secretsdump.py` (Impacket), and direct registry export via `reg save`
- Detection: Monitor for `reg save` on the SECURITY hive, unusual LSASS access events (Event ID 4656), and privileged process access to `HKLM\SECURITY`
- Mitigation: Group Managed Service Accounts (gMSA) eliminate stored service passwords; limiting service accounts to least-privilege reduces the blast radius
