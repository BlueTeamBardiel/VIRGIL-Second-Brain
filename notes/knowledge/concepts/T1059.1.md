# T1059.1

## What it is
Like handing a master locksmith a blueprint of every lock in the building, PowerShell gives attackers a native, trusted scripting engine with direct access to Windows internals, the file system, network, and registry. Precisely, T1059.1 describes adversary abuse of Windows PowerShell to execute commands, run scripts, and download payloads — all while living inside a signed Microsoft tool the OS trusts by default.

## Why it matters
In the 2020 SolarWinds supply chain attack, threat actors used PowerShell to execute encoded commands that downloaded secondary payloads and exfiltrated data, specifically because PowerShell traffic blended into normal administrative activity and bypassed perimeter controls. Defenders who had enabled PowerShell Script Block Logging (Event ID 4104) were able to reconstruct attacker commands even when obfuscation was used.

## Key facts
- Attackers frequently use `-EncodedCommand` (Base64) and `-ExecutionPolicy Bypass` flags to evade detection and circumvent execution policy restrictions
- PowerShell can download and execute payloads entirely in memory using `IEX (New-Object Net.WebClient).DownloadString()`, leaving minimal artifacts on disk
- Key Windows Event IDs for detection: **4103** (module logging), **4104** (script block logging), **4688** (process creation with command-line auditing enabled)
- Constrained Language Mode (CLM) and AMSI (Antimalware Scan Interface) are primary mitigations — AMSI hooks into PowerShell to expose script content to AV engines before execution
- PowerShell v5+ logging is significantly more robust than v2; attackers sometimes force downgrade to v2 (`-Version 2`) to avoid modern logging capabilities
