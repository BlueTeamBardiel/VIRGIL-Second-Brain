# CWE-184

## What it is
Like a bouncer with a "banned list" who only turns away people whose names he recognizes — anyone not on the list walks right in. CWE-184 is an **Incomplete List of Disallowed Inputs (Denylist/Blacklist)**, where a security control blocks known-bad values but fails to account for all malicious variants, leaving gaps an attacker can exploit. It's a fundamental flaw in blocklist-based input validation logic.

## Why it matters
A web application filtering SQL injection might block the word `SELECT` but miss `SeLeCt`, `%53ELECT` (URL-encoded), or `SEL/**/ECT` (comment-injected). Attackers routinely bypass weak denylists using case variation, encoding tricks, null bytes, or Unicode equivalents — making this vulnerability a staple of real-world web application penetration testing and a common finding in WAF bypass research.

## Key facts
- **Root cause**: Trusting an enumerable blocklist of "bad" patterns is inherently incomplete; attackers only need to find *one* gap.
- **Secure alternative**: Allowlisting (whitelisting) — explicitly permitting only known-good input — is the recommended mitigation because it closes all unlisted variants by default.
- **Common attack vectors**: Case manipulation, URL/HTML encoding, Unicode normalization, comment injection, and double encoding are standard bypass techniques.
- **Affected input types**: File extensions (blocking `.php` but missing `.PHP` or `.php5`), SQL keywords, shell metacharacters, and script tags in XSS filters.
- **OWASP relevance**: Appears under A03:2021 (Injection) and A03's input validation failures; frequently tested in CySA+ scenarios involving WAF bypass and secure coding review.
