# T1055.015

## What it is
Like a parasite that hijacks a healthy organism's body to survive and act undetected, ListPlanting is a process injection technique where an attacker replaces the function pointer of a ListView control's message handler with malicious code, causing it to execute inside a legitimate process. The attacker plants shellcode into a target process's memory, then manipulates a Windows Common Controls ListView widget to call that shellcode as if it were normal UI event handling.

## Why it matters
In a red team engagement, an adversary with local access could inject shellcode into explorer.exe using ListPlanting, making their payload appear as benign GUI activity from a trusted Windows process. Because the execution is triggered through a legitimate UI control callback, behavior-based EDR solutions that monitor for classic injection patterns like `CreateRemoteThread` or `SetWindowsHookEx` may miss it entirely.

## Key facts
- ListPlanting abuses the `LVM_SORTITEMS` message sent to a ListView control, which calls a comparison function — attackers redirect this pointer to shellcode
- The shellcode is first written into target process memory (e.g., via `WriteProcessMemory`), then the function pointer is overwritten using `SendMessage`
- Classified under MITRE ATT&CK T1055 (Process Injection) sub-technique .015, introduced to capture UI control abuse patterns
- Execution occurs within the context of the victim process, inheriting its security token and evading process-level allowlisting
- Detection focus should be on anomalous `SendMessage`/`PostMessage` calls combined with prior cross-process memory writes to processes hosting ListView controls
