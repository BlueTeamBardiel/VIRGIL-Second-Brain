# CWE-522

## What it is
Imagine a bank vault where the combination is written on a sticky note taped to the door — that's essentially what insufficiently protected credentials are. CWE-522 describes the weakness where authentication credentials (passwords, API keys, tokens) are stored or transmitted without adequate protection, making them trivially recoverable by an attacker. This includes plaintext storage in config files, weak encoding like Base64, or transmission over unencrypted channels.

## Why it matters
In 2021, the Twitch breach exposed an internal credential store containing API keys and passwords with minimal protection, allowing attackers to pivot through internal systems. Had credentials been properly hashed (for passwords) or encrypted at rest (for keys/tokens), the blast radius would have been dramatically reduced. This is why secrets management tools like HashiCorp Vault exist — to prevent credentials from sitting naked in source code or flat files.

## Key facts
- Storing passwords with reversible encoding (Base64, ROT13) is **not** encryption and is explicitly covered under CWE-522 — only strong one-way hashing (bcrypt, Argon2, PBKDF2) is acceptable for passwords
- Hardcoded credentials in source code (often caught by tools like TruffleHog or GitGuardian) are a primary manifestation of this weakness
- CWE-522 is distinct from CWE-312 (Cleartext Storage of Sensitive Information) — 522 is specifically scoped to **credentials**
- Transmission of credentials over HTTP (rather than HTTPS) also qualifies as insufficient protection under this CWE
- NIST SP 800-63B provides the authoritative guidance for credential protection standards, including the prohibition of password hints and knowledge-based authentication as sole factors
