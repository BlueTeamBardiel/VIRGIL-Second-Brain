# T1127

## What it is
Like a criminal using a licensed locksmith's van to avoid suspicion, attackers use trusted developer tools already installed on a system to execute malicious code without triggering alerts. T1127 (Trusted Developer Utilities Proxy Execution) describes abusing legitimate build and debugging tools — such as MSBuild.exe, dnSpy, or csc.exe — to compile and run attacker-controlled code while hiding behind the tool's trusted signature.

## Why it matters
In a 2019 campaign, threat actors used Microsoft's MSBuild.exe to execute a Remote Access Trojan by embedding malicious C# code inside a legitimate .proj XML file — bypassing application whitelisting entirely because MSBuild itself was an allowed, signed Microsoft binary. Defenders must therefore whitelist at the *behavior* level, not just the binary level, monitoring for MSBuild spawning network connections or unusual child processes.

## Key facts
- **MSBuild.exe** is the most commonly abused utility; it can compile and execute inline C# tasks directly from XML project files without writing a separate binary to disk
- This technique is specifically designed to **bypass application whitelisting** (e.g., AppLocker, WDAC) since the proxy binary is a trusted, signed Microsoft tool
- Other abused utilities include **msdeploy.exe**, **tracker.exe**, **rcsi.exe**, and **csc.exe** (Roslyn compiler)
- Detection focus: monitor for developer tools running in **non-development environments** or spawning unusual child processes (e.g., cmd.exe, PowerShell, network connections)
- Mitigation: use WDAC **publisher-level rules** combined with path restrictions, and restrict developer utilities to dedicated build systems via Group Policy
