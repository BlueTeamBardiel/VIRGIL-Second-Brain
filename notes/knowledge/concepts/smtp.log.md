# smtp.log

## What it is
Like a postal inspector's manifest recording every letter that passed through a sorting facility, smtp.log is Zeek's (formerly Bro) structured record of every SMTP transaction observed on the network. It captures email communication metadata — sender, recipient, subject, message ID, and server handshake details — without storing the actual email body.

## Why it matters
During a phishing campaign investigation, smtp.log lets a SOC analyst pivot from a single malicious email back to every recipient in the organization who received messages from the same spoofed domain or mail server IP. It can also expose data exfiltration via email — an insider sending large volumes of messages to external addresses will produce anomalous smtp.log entries that SIEM correlation rules can flag automatically.

## Key facts
- Key fields include `ts`, `uid`, `id.orig_h`, `id.resp_h`, `mailfrom`, `rcptto`, `subject`, `msg_id`, `last_reply`, and `tls` (indicating STARTTLS usage)
- The `last_reply` field captures the final SMTP response code (e.g., 250 = success, 550 = mailbox unavailable), useful for detecting rejected spam floods or enumeration attempts
- A missing or mismatched `msg_id` domain (e.g., `Message-ID` domain differs from `MAIL FROM` domain) is a classic indicator of email spoofing
- smtp.log is generated by Zeek's SMTP analyzer and is a primary log source in network security monitoring (NSM) frameworks like Security Onion
- Unlike firewall logs, smtp.log operates at Layer 7, capturing application-layer content that port-based filtering completely misses
