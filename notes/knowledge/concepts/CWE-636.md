# CWE-636

## What it is
Like a bank vault that automatically opens when the alarm system fails instead of locking down harder, this weakness describes software that defaults to an overly permissive state when it encounters an unexpected condition. CWE-636 (Not Failing Securely) occurs when error-handling logic grants access or reduces security controls during failure modes rather than denying access and maintaining a safe state.

## Why it matters
In 2012, a flaw in certain authentication middleware caused the access control check to return a permissive result when it threw an unhandled exception — attackers deliberately triggered the exception to bypass login. A properly hardened system would catch the exception, log it, and **deny** access by default; instead, the broken logic treated "error" as equivalent to "success."

## Key facts
- The secure default principle states: **fail closed, not open** — denials should be the fallback, never grants
- CWE-636 is distinct from CWE-755 (Improper Handling of Exceptional Conditions) — the specific danger here is that the *direction* of failure is toward insecurity
- Common manifestations: authentication functions returning `true` on exception, firewall rules disabling on crash, or null pointer dereferences that skip authorization checks
- Closely tied to **Principle of Least Privilege** — a failing component should lose privileges, not gain them
- OWASP lists "Insecure Design" (A04) as the category; fail-open logic is a classic design-level flaw that code reviews frequently miss
