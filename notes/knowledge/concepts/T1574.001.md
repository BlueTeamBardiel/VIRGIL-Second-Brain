# T1574.001

## What it is
Like a con artist who intercepts a letter meant for the bank and replaces it with a forged one, DLL Search Order Hijacking exploits the fact that Windows searches for DLLs in a predictable sequence of directories. An attacker places a malicious DLL with a legitimate name in a directory that Windows searches *before* the real DLL's location, causing the target application to load the malicious code instead.

## Why it matters
In 2020, attackers targeting the Netwrix Auditor software placed a rogue DLL in a writable directory earlier in the search path, achieving persistent code execution under the context of a trusted application. Defenders counter this by auditing writable directories in the system PATH, monitoring for unexpected DLL loads via Sysmon Event ID 7, and enforcing SafeDllSearchMode in the registry.

## Key facts
- Windows default DLL search order: application directory → System32 → System → Windows → CWD → PATH directories
- **SafeDllSearchMode** (enabled by default) moves the Current Working Directory (CWD) further down the search order, reducing but not eliminating risk
- Attackers need write access to a directory earlier in the search path than the legitimate DLL — often achievable without admin rights
- This technique achieves **persistence and privilege escalation** simultaneously if the hijacked application runs as a higher-privileged account
- Detection relies on file integrity monitoring, EDR telemetry, and Sysmon Event ID 7 (image loaded) with unsigned DLL alerting
