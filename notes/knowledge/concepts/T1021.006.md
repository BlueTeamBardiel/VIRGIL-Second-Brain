# T1021.006

## What it is
Like a master key that lets you remotely operate every power switch in a building, Windows Remote Management (WinRM) gives administrators a network protocol to execute commands on remote machines using standard HTTP/HTTPS ports. Precisely, WinRM is Microsoft's implementation of the WS-Management protocol, enabling PowerShell remoting and remote command execution, typically over TCP port 5985 (HTTP) or 5986 (HTTPS).

## Why it matters
During the SolarWinds breach, attackers leveraged WinRM alongside stolen credentials to move laterally across victim networks, executing commands on remote hosts without deploying additional malware — blending seamlessly with legitimate IT operations. Defenders must monitor for unusual WinRM connections, especially from non-administrative workstations or outside business hours, since the traffic looks identical to routine admin activity.

## Key facts
- WinRM uses **TCP 5985** (HTTP, cleartext) and **TCP 5986** (HTTPS, encrypted); legacy implementations may use port 80/443
- Enabled by default on **Windows Server 2012+**; must be manually enabled on Windows client OS via `winrm quickconfig`
- Attackers typically authenticate with **valid credentials** (Pass-the-Hash or stolen passwords), making detection credential-behavior-focused rather than signature-based
- PowerShell's `Enter-PSSession` and `Invoke-Command` are the primary built-in tools that weaponize WinRM for lateral movement
- Detection focus: **Event ID 4624** (logon type 3 or 10) combined with WinRM service activity in Windows event logs; also monitor `wsmprovhost.exe` spawning child processes
