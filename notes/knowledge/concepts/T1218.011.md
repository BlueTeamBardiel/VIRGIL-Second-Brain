# T1218.011

## What it is
Like borrowing a master key from the building manager to sneak into rooms you're not supposed to enter, attackers borrow `rundll32.exe`—a legitimate Windows utility—to execute malicious code without spawning a suspicious custom process. Specifically, T1218.011 describes abusing `rundll32.exe` to load and run arbitrary DLLs or COM scriptlets, bypassing application whitelisting controls that trust this signed Microsoft binary.

## Why it matters
During the Cobalt Strike-heavy campaigns against financial institutions, red teams and actual threat actors used `rundll32.exe` to load malicious DLLs hosted on attacker-controlled SMB shares, achieving code execution while evading EDR solutions that allowed the signed binary. Defenders who monitor only process names miss this entirely—the key is logging command-line arguments and the DLL paths being passed to `rundll32.exe`.

## Key facts
- `rundll32.exe` lives in `C:\Windows\System32\` and is signed by Microsoft, making it inherently trusted by most whitelisting solutions
- Attackers can call arbitrary exported functions using the syntax: `rundll32.exe malicious.dll,ExportedFunction`
- It can also execute remote SCT (scriptlet) files via `rundll32.exe javascript:` or URL-based COM object loading, enabling fileless execution
- This technique is classified under **Signed Binary Proxy Execution** (T1218), a broader LOLBAS (Living Off the Land Binaries and Scripts) strategy
- Detection focus: alert on `rundll32.exe` spawning child processes, loading DLLs from temp/user-writable directories, or making network connections
