# T1098.003

## What it is
Like slipping a spare key onto a janitor's keyring so you can enter the building long after your contractor badge expires, this technique involves adding credentials to existing cloud accounts to maintain persistent access. Specifically, adversaries add additional SSH keys, service account keys, or access tokens to legitimate cloud or SaaS accounts without creating new accounts — staying hidden in plain sight.

## Why it matters
In the 2020 SolarWinds-linked campaign, attackers added credentials to Azure AD service principals after gaining initial access, allowing them to authenticate to Microsoft 365 environments even after passwords were reset. Defenders who only monitored new account creation missed the persistence entirely because the compromised accounts already looked legitimate.

## Key facts
- Affects cloud platforms including Azure AD, AWS IAM, and Google Cloud — attackers may add API keys, OAuth tokens, or SSH public keys to existing identities
- Differs from T1136 (Create Account) because no new account is created — the hijacked account's normal activity provides camouflage
- Detection relies on auditing credential modification events: `UpdateServicePrincipal` in Azure, `CreateAccessKey` in AWS CloudTrail, or `ssh-keys` metadata changes in GCP
- A classic red flag is a service account suddenly having multiple active access keys, especially if one was created outside change-management windows
- Mitigations include enforcing MFA, applying least-privilege IAM policies, and using AWS Config rules or Azure Policy to alert on unauthorized key additions
