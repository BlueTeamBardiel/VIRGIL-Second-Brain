# T1059.001

## What it is
Like handing an attacker the keys to a fully-equipped machine shop already built into every Windows system, PowerShell gives adversaries a powerful scripting engine with direct access to the OS, .NET framework, and network stack. It is a MITRE ATT&CK sub-technique describing adversary abuse of PowerShell to execute commands, run scripts, and download payloads during an intrusion. Because PowerShell is a legitimate admin tool, malicious use frequently blends with normal operations.

## Why it matters
In the 2020 SolarWinds compromise, attackers used PowerShell to execute encoded commands and download secondary payloads from C2 infrastructure, specifically leveraging `-EncodedCommand` to obfuscate their activity from log-naive defenders. A SOC analyst monitoring for PowerShell process creation with suspicious flags like `-NoProfile -NonInteractive -ExecutionPolicy Bypass` would have had a meaningful detection opportunity.

## Key facts
- **AMSI (Antimalware Scan Interface)** is Microsoft's primary defense layer that intercepts and scans PowerShell scripts at runtime; attackers routinely attempt to patch or bypass AMSI in memory before executing malicious code.
- **Constrained Language Mode (CLM)** restricts PowerShell capabilities and is enforced by Windows Defender Application Control (WDAC); dropping into CLM significantly limits attacker options.
- **Script Block Logging** (Event ID 4104) records the full content of executed script blocks — the single most valuable log source for detecting obfuscated PowerShell.
- **Module Logging** and **Transcription Logging** are complementary policies that capture pipeline execution and session I/O respectively, and should be enabled alongside Script Block Logging.
- PowerShell v2 (`powershell.exe -Version 2`) can bypass many modern logging controls because it lacks AMSI integration — organizations should disable it explicitly.
