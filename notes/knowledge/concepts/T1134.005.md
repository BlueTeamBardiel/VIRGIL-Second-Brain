# T1134.005

## What it is
Like a con artist who doesn't steal a judge's robes but instead borrows them with permission to walk into a courtroom unquestioned, SID-History Injection plants a trusted identity into a user's token without actually changing their account. Specifically, adversaries inject Security Identifier (SID) values from privileged accounts (like Domain Admins) into the SID-History attribute of a compromised account, causing Windows to grant rights associated with those injected SIDs during authentication.

## Why it matters
During the infamous domain migration attacks seen in red team engagements, an attacker who compromises any domain account can inject the SID of the Enterprise Admins group into that account's SID-History — gaining forest-wide administrative access while appearing to be a low-privilege user. Defenders miss this because the account itself hasn't changed groups, making traditional group membership audits blind to the escalation.

## Key facts
- SID-History is a legitimate AD attribute designed to preserve access rights during domain migrations
- Injection requires **SeDebugPrivilege** or Domain Admin rights (often via DCSync or NTDS.dit access) to write to the attribute
- The `mimikatz` command `sid::patch` + `sid::add` is the canonical tool for performing this attack
- Because Windows evaluates *all* SIDs in the token (including SID-History), injected SIDs grant full access to resources ACL'd for those SIDs
- Detection: Monitor Windows Event ID **4765** (SID History added) and **4766** (SID History add attempt failed); these are rarely triggered in non-migration environments
