# RFC 3227

## What it is
Like a crime scene investigator who photographs the body before touching anything, RFC 3227 tells you what to collect and in what order during digital forensics. Formally, it is an IETF informational document titled "Guidelines for Evidence Collection and Archiving" that establishes the **order of volatility** for collecting digital evidence from compromised systems.

## Why it matters
During a live incident response on a breached web server, an analyst who reboots the machine first destroys RAM contents — wiping active network connections, running processes, and encryption keys that existed only in memory. RFC 3227 prevents this mistake by mandating that the most volatile data (RAM, CPU cache) is captured before less volatile data (hard drives, backup tapes), preserving evidence that could prove an attacker's presence and methods.

## Key facts
- **Order of volatility (most → least):** CPU registers/cache → RAM → Network state/connections → Running processes → Disk storage → Remote logging/backups → Paper/printouts
- The document establishes that evidence collection should **minimize contamination** — every action taken on a live system should be documented because it changes the artifact
- RFC 3227 emphasizes maintaining a **chain of custody** — documenting who touched evidence, when, and what tools were used
- Investigators should use **trusted tools from read-only media** (not the compromised system's own binaries) to avoid running attacker-modified utilities
- The document distinguishes between **collection** (gathering raw data) and **archiving** (securely storing and hashing it) — cryptographic hashing (MD5/SHA) is recommended to prove integrity
