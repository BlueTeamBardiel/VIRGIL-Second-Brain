# CWE-670

## What it is
Like a pilot who always manually overrides the autopilot mid-flight regardless of conditions — creating the illusion of safety while actually removing a working control — CWE-670 describes code that always invokes a control flow transfer that renders subsequent logic unreachable or meaningless. Formally, it is "Always-Incorrect Control Flow Implementation," where a branch, return, or exception always executes in a way that defeats the programmer's intended logic, typically through unconditional early returns or misplaced break/continue statements.

## Why it matters
A classic manifestation is a security check that always returns "authorized" before the actual credential validation code can execute — the authentication function exists on paper but is functionally dead code. An attacker probing the application finds no enforcement occurs, effectively bypassing access controls without exploiting any memory corruption or injection vulnerability.

## Key facts
- CWE-670 is a **quality/logic flaw**, not a memory or injection issue — static analysis tools must trace control flow graphs to detect it
- Often introduced by **copy-paste errors** or incomplete refactoring, where a `return true` is accidentally placed before validation logic
- Can manifest in **exception handlers** that always catch and suppress errors, masking security-relevant failures silently
- The vulnerability is particularly dangerous because **unit tests may pass** if the test itself calls the function expecting the always-true result
- Classified under CWE's **"Code Quality"** category; relevant in secure code review for CSSLP and CySA+ scenarios involving logic flaws
