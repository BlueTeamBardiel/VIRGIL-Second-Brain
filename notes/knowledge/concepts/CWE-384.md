# CWE-384

## What it is
Imagine a hotel key card that stays active even after the previous guest checks out — a new guest moves in, but an attacker with the old card can still open the door. Session Fixation is exactly that: an attacker forces a victim's browser to use a known session identifier *before* authentication, then hijacks the session after the victim logs in without the server issuing a fresh token.

## Why it matters
In a classic attack, an adversary sends a victim a crafted link like `https://bank.com/login?sessionid=abc123`. The victim authenticates, but the server keeps the same `abc123` token rather than regenerating it. The attacker, who already knows `abc123`, now has full authenticated access to the victim's account without ever knowing the password.

## Key facts
- The root cause is failure to **invalidate and regenerate session tokens upon privilege escalation** (i.e., after login).
- Differs from session hijacking (CWE-287 adjacent): fixation *plants* the session ID beforehand; hijacking *steals* it afterward.
- Common delivery vectors include URL parameters, hidden form fields, and cookie injection via subdomain attacks.
- The defense is simple and mandatory: call `session_regenerate_id(true)` (PHP) or equivalent immediately after successful authentication.
- OWASP Top 10 A07:2021 (Identification and Authentication Failures) directly encompasses this weakness.
