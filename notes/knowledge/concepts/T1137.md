# T1137

## What it is
Like hiding a spare key inside a fake rock near your front door, Office application startup mechanisms let attackers embed malicious code inside trusted Office features that run automatically. T1137 covers **Office Application Startup persistence**, where adversaries abuse legitimate Microsoft Office extensibility features — add-ins, templates, Office Test registry keys, and macros — to execute malicious code whenever Office applications launch.

## Why it matters
In the 2020 SolarWinds-adjacent campaigns, threat actors planted malicious Word add-ins (`.wll` files) in the Office startup folder, ensuring their payload re-executed every time a target opened Word — surviving reboots and even some AV scans because the loader appeared as a trusted Office component. Defenders hunting this technique should monitor `%APPDATA%\Microsoft\Word\Startup\` and equivalent paths for unexpected `.wll` or `.xlam` files.

## Key facts
- **Sub-techniques include**: Office Template Macros (T1137.001), Office Test registry key (T1137.002), Outlook Home Page (T1137.004), Outlook Rules (T1137.005), and Add-ins (T1137.006)
- The **Office Test** registry key (`HKCU\Software\Microsoft\Office Test\Special\Perf`) is a legacy debugging hook — it executes any DLL path stored there on every Office launch
- Malicious **XLL add-ins** (Excel) have surged since macro blocking by default in Office 365 (2022), making T1137.006 a primary evasion route
- Legitimate add-in directories to monitor: `%APPDATA%\Microsoft\AddIns\`, `%APPDATA%\Microsoft\Excel\XLSTART\`, and the Office COM add-in registry keys
- Detection hinges on **file integrity monitoring** of startup paths and watching for Office spawning unusual child processes (e.g., `cmd.exe`, `powershell.exe`)
