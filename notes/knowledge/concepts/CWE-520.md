# CWE-520

## What it is
Like leaving a master key under the doormat because it's "easier for the staff," .NET misconfiguration that grants excessive trust to assemblies removes a critical security boundary. CWE-520 refers to the use of `FullTrust` permission sets in .NET applications, where code is granted unrestricted access to system resources when it should be operating under a restricted, least-privilege sandbox.

## Why it matters
An attacker who compromises a plugin or third-party library loaded by a .NET application with `FullTrust` gains the same permissions as the host process — potentially allowing file system access, registry manipulation, or process injection. This is particularly dangerous in enterprise applications that dynamically load assemblies from network shares or user-controlled directories, turning a library hijack into full system compromise.

## Key facts
- `FullTrust` bypasses .NET Code Access Security (CAS), meaning no permission demands, stack walks, or sandbox restrictions apply to the trusted assembly
- The vulnerability often appears in configuration files (`<trust level="Full"/>` in `web.config`) or programmatic `PolicyLevel` assignments
- .NET Framework's Code Access Security was deprecated in .NET Core/.NET 5+, but legacy .NET Framework applications remain widely deployed and vulnerable
- CWE-520 is a child of CWE-732 (Incorrect Permission Assignment) and relates to the broader principle of least privilege violation
- Remediation involves applying the principle of least privilege via `PermissionSet` restrictions and validating assembly sources using strong naming or Authenticode signatures
