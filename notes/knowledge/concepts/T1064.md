# T1064

## What it is
Like a burglar who speaks only French to confuse the security guard, scripting lets attackers issue commands in a language the OS understands but defenders may struggle to monitor. T1064 (now subsumed under T1059) refers to adversary use of scripting languages — PowerShell, VBScript, Bash, Python — to automate attack tasks, evade defenses, and execute payloads without dropping traditional executables.

## Why it matters
In the 2016 SANS holiday hack and countless APT campaigns, attackers used malicious VBScript embedded in Office macros to download second-stage payloads entirely in memory — no `.exe` ever touched disk, leaving traditional file-based AV blind. Defenders counter this by enabling Script Block Logging in PowerShell and monitoring Windows Event ID 4104, which captures decoded script content before execution.

## Key facts
- T1064 is a **legacy technique ID** — MITRE ATT&CK v9+ merged it into **T1059 (Command and Scripting Interpreter)** with sub-techniques (e.g., T1059.001 for PowerShell)
- Scripts are favored for **living-off-the-land (LotL)** attacks because they abuse built-in OS tools, reducing the malware footprint
- **PowerShell Constrained Language Mode** and **AMSI (Antimalware Scan Interface)** are primary Windows defenses against script-based attacks
- **Script Block Logging** (Event ID 4104) and **Module Logging** (Event ID 4103) are the two key Windows logging controls for detecting malicious scripting
- Attackers frequently **Base64-encode** or obfuscate scripts to bypass simple string-matching defenses; tools like `Invoke-Obfuscation` automate this
