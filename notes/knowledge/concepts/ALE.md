# ALE

## What it is
Think of ALE like a security guard who logs *everything* that happens at the door—not just who enters, but timestamps, what they carried, and how long they stayed. Application Log Entry (ALE) is a record generated by an application when a security-relevant event occurs, containing details like user actions, authentication attempts, data access, or errors. It's the application's way of creating an audit trail for forensic investigation and compliance.

## Why it matters
When an attacker compromises a web application to steal customer data, security teams rely on ALEs to reconstruct *exactly* what happened: which accounts accessed the database, what queries ran, and when. Without rich application logging, you're flying blind—you can't prove a breach occurred, show regulators what data was exposed, or catch the attack in progress. This is why PCI-DSS and HIPAA mandate detailed application logging.

## Key facts
- ALEs must include timestamp, user ID, action performed, resource accessed, and success/failure status
- Common sources: web servers, databases, APIs, authentication systems, file access logs
- ALEs are useless if not *centralized*—scattered logs across servers become impossible to correlate
- Log rotation and retention policies matter: you need enough history to detect slow, persistent attacks
- Sensitive data (passwords, credit cards) should *never* be logged, even in error messages
