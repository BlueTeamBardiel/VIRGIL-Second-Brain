# CWE-352

## What it is
Imagine a forged letter with your signature that your bank acts on without checking — that's CSRF. Cross-Site Request Forgery tricks an authenticated user's browser into sending unauthorized requests to a trusted site, exploiting the fact that browsers automatically attach session cookies to every request, regardless of origin.

## Why it matters
In 2008, attackers exploited CSRF against uTorrent's web interface by embedding a malicious image tag in a forum post — visiting the post silently changed the victim's download directory to an attacker-controlled path. Any authenticated user who loaded the page was compromised without clicking anything.

## Key facts
- CSRF abuses **trust the server has in the browser**, while XSS abuses trust the browser has in the server — they are complementary, not the same attack
- The primary defense is a **synchronizer token (CSRF token)**: a unique, unpredictable value embedded in forms and validated server-side on each state-changing request
- **SameSite cookie attribute** (set to `Strict` or `Lax`) is a modern, highly effective mitigation that prevents cookies from being sent on cross-origin requests
- CSRF only affects **state-changing requests** (POST, PUT, DELETE) — read-only GET requests don't cause damage but should still avoid triggering side effects
- Double-submit cookie pattern and **custom request headers** (e.g., `X-Requested-With`) are additional defenses leveraging same-origin policy restrictions on header injection
