# T1218.002

## What it is
Like hiring a licensed contractor to smuggle contraband past a security checkpoint, Control Panel Items abuse Windows' own trusted `.cpl` files to execute malicious code under the OS's blessing. Precisely: adversaries craft or trojanize Control Panel Item files (`.cpl` extensions, which are actually DLLs) and execute them via `control.exe` or `rundll32.exe shell32.dll,Control_RunDLL`, bypassing application whitelisting that trusts signed Windows binaries.

## Why it matters
In several commodity malware campaigns, attackers distributed phishing emails with `.cpl` attachments disguised as invoices. When the victim double-clicked the file, Windows automatically invoked `control.exe` to load it, executing embedded shellcode while security tools saw only a routine Control Panel interaction — a signed, legitimate Windows process initiating the payload.

## Key facts
- `.cpl` files are PE DLLs that export a `CPlApplet` function; Windows loads them automatically when double-clicked or called via `control.exe`
- Execution can be triggered multiple ways: direct double-click, `control.exe malware.cpl`, or `rundll32.exe shell32.dll,Control_RunDLL malware.cpl`
- Because `control.exe` is a signed Microsoft binary, execution often evades application whitelisting and basic EDR rules that don't inspect child DLL loads
- Detection focus: monitor for `control.exe` loading `.cpl` files from unusual paths (temp folders, `%AppData%`, downloads) rather than `%SystemRoot%\System32`
- Mitigation includes Windows Defender Attack Surface Reduction (ASR) rules and AppLocker policies blocking unsigned `.cpl` execution
