# CWE-354

## What it is
Like accepting a sealed envelope but never checking whether the wax seal was actually broken, CWE-354 is the failure to verify the integrity of a value after it has been checked. Precisely: the software checks the integrity of data at one point but then uses a different, potentially modified version of that data later — a classic TOCTOU (Time-of-Check to Time-of-Use) variant applied to integrity verification.

## Why it matters
In 2020, researchers demonstrated this pattern against software update mechanisms where an application verified a package's cryptographic hash before downloading, but then executed a *different* file from the same directory path — one that an attacker had swapped in during the window between verification and execution. This is the foundation of many supply chain attacks: the check is real, but it doesn't protect the actual data used.

## Key facts
- The vulnerability is defined by a **gap between verification and use** — the integrity check is performed, but the verified object is not the object consumed.
- Distinct from CWE-353 (missing integrity check) — here the check *exists* but is **decoupled from the actual data used**.
- Common attack vector: race conditions in file systems, where an attacker swaps a file between `stat()` verification and `open()` execution.
- Mitigations include **cryptographically binding the verified copy** (e.g., loading directly from a memory buffer after hashing, never re-reading from disk).
- Appears frequently in **software update clients, package managers, and firmware loaders** — high-value targets for persistent malware installation.
