# CWE-441

## What it is
Like shouting an order through a chain of waiters where each one passes it along without reading it — you trust the message arrived intact, but any one of them could have altered it. CWE-441, **Unintended Proxy or Intermediary ("Confused Deputy")**, occurs when a system component with elevated privileges performs actions on behalf of a lower-privileged requester without validating whether that requester is actually authorized to trigger those actions. The trusted intermediary becomes an unwitting accomplice in privilege abuse.

## Why it matters
A classic exploitation scenario: a web application's server-side component has permission to read sensitive configuration files. An attacker with no direct file access tricks the application into fetching an internal resource (e.g., `http://localhost/admin/config`) by submitting it as a user-controlled URL parameter — a Server-Side Request Forgery (SSRF) attack. The server, acting as the confused deputy, uses its own elevated network trust to retrieve data the attacker could never reach directly.

## Key facts
- The "confused deputy" problem was first formally described by Norm Hardy in 1988 using a compiler example
- The deputy (intermediary) holds a **capability or authority** the attacker lacks, and the vulnerability is exploiting that gap
- SSRF (CWE-918) and CSRF (CWE-352) are both concrete manifestations of the confused deputy pattern
- Mitigation requires the intermediary to validate **both** the identity of the requestor *and* whether that requestor has authorization for the specific action being delegated
- Common in cloud environments where metadata APIs (e.g., `169.254.169.254`) are reachable via confused deputy paths, leading to credential theft
