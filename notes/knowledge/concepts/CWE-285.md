# CWE-285

## What it is
Like a nightclub bouncer who checks your ID at the door but never verifies you're actually on the VIP list once inside — CWE-285 is Improper Authorization, where a system authenticates a user correctly but then fails to verify whether that user actually has *permission* to perform a requested action. Authentication answers "who are you?"; authorization answers "what are you allowed to do?" — CWE-285 collapses that second question.

## Why it matters
In the 2019 Facebook API breach, authenticated users could query the API with valid tokens but access other users' private data because authorization checks on resource ownership were inconsistently enforced. An attacker who obtains any valid session token — even a low-privilege one — can escalate horizontally or vertically by requesting resources the server never properly guards. The fix isn't stronger passwords; it's enforcing access control checks on every sensitive operation server-side.

## Key facts
- CWE-285 is the parent of more specific weaknesses including **CWE-862** (Missing Authorization) and **CWE-863** (Incorrect Authorization)
- It maps directly to **OWASP Top 10 A01:2021 — Broken Access Control**, the #1 web application risk
- **Horizontal privilege escalation** (user A accessing user B's data) and **vertical privilege escalation** (regular user accessing admin functions) are both manifestations of CWE-285
- Authorization must be enforced **server-side** — client-side controls (hidden buttons, disabled fields) are trivially bypassed
- Common in REST APIs where endpoints authenticate via JWT but skip per-resource ownership validation
