# RFC 826

## What it is
Think of it like a neighborhood directory where your computer shouts "Who lives at IP address 192.168.1.1?" and the rightful owner shouts back their MAC address — RFC 826 defines the **Address Resolution Protocol (ARP)**, the mechanism that maps IP addresses to MAC addresses on local area networks. Published in 1982, it operates at the boundary of Layer 2 and Layer 3 with no built-in authentication whatsoever.

## Why it matters
Because ARP has zero authentication, an attacker on the same LAN can send unsolicited (gratuitous) ARP replies poisoning a victim's ARP cache — redirecting traffic meant for a gateway through the attacker's machine instead. This **ARP spoofing** attack enables man-in-the-middle interception, credential theft, and session hijacking on any unprotected switched network. Defenses like **Dynamic ARP Inspection (DAI)** on managed switches validate ARP packets against a trusted DHCP snooping binding table to block this.

## Key facts
- ARP operates at **Layer 2 (Data Link)** and resolves IPv4 addresses only; IPv6 uses Neighbor Discovery Protocol (NDP) instead
- ARP cache entries are stored temporarily (typically **20 minutes** on Windows, 1–4 minutes on Linux) — poisoned entries expire but can be continuously refreshed by an attacker
- **Gratuitous ARP** — an unsolicited ARP reply a host sends to announce its own IP/MAC — is the core mechanism exploited in ARP spoofing attacks
- ARP has **no authentication or verification mechanism** — this is a deliberate design choice from 1982, not an oversight
- Tools like **Arpwatch** detect anomalous MAC-to-IP pairings, while static ARP entries can prevent poisoning on high-value hosts
