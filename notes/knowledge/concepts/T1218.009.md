# T1218.009

## What it is
Like hiring a notary to forge your documents because notaries are inherently trusted, attackers use `Regsvcs.exe` and `Regasm.exe` — Microsoft-signed .NET utilities — to execute malicious code that Windows trusts by default. These two COM component registration tools can load and execute arbitrary .NET assemblies, bypassing application whitelisting controls because they are legitimate, signed Microsoft binaries.

## Why it matters
In a real attack, a threat actor drops a malicious .NET DLL on a compromised host and invokes `Regasm.exe /U malicious.dll` — the `/U` (unregister) flag triggers the `[ComUnregisterFunction]` decorated method in the assembly, executing attacker code without ever running an unsigned executable. This technique was used by APT groups to evade endpoint controls that blocked unknown executables but permitted signed Microsoft tools.

## Key facts
- Both `Regsvcs.exe` (Register Services) and `Regasm.exe` (Register Assembly) are .NET Framework utilities located in `C:\Windows\Microsoft.NET\Framework[64]\`
- Malicious code executes via specially decorated class methods: `[ComRegisterFunction]` (triggered on register) or `[ComUnregisterFunction]` (triggered on unregister with `/U`)
- The attack **does not require admin privileges** in all configurations, lowering the barrier to exploitation
- These binaries are commonly whitelisted by application control policies (AppLocker, WDAC), making them effective LOLBins (Living Off the Land Binaries)
- Detection focus: monitor for `Regsvcs.exe` or `Regasm.exe` loading DLLs from non-standard paths (e.g., `%TEMP%`, user-writable directories)
