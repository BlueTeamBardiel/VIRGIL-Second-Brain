# VIIII. Building an Incident Response Program

## What it is
Like a hospital trauma center — you don't design the response *after* the patient arrives bleeding; you train the team, stock the supplies, and rehearse the procedures *before* the emergency. An Incident Response (IR) Program is the formal, organization-wide framework that defines how security incidents are detected, contained, eradicated, and recovered from, *before* they occur. It encompasses people, processes, and tools organized into a repeatable lifecycle.

## Why it matters
In 2017, Equifax suffered a breach exposing 147 million records partly because their IR program lacked clear ownership — no one acted on the initial vulnerability alert for 78 days. A mature IR program with defined roles, escalation paths, and pre-approved containment authorities would have dramatically reduced dwell time and damage.

## Key facts
- The **NIST SP 800-61** IR lifecycle has four phases: **Preparation → Detection & Analysis → Containment/Eradication/Recovery → Post-Incident Activity**
- **SANS** uses a six-phase model: Preparation, Identification, Containment, Eradication, Recovery, Lessons Learned (**PICERL**)
- The **IR Policy** defines authority and scope; the **IR Plan** defines procedures; **IR Playbooks** define step-by-step responses to *specific* incident types (e.g., ransomware, insider threat)
- A **Computer Security Incident Response Team (CSIRT)** must include representation from legal, HR, communications, and IT — not just security engineers
- **Tabletop exercises** test the IR plan without disrupting production; **full-scale simulations** test technical response capabilities under realistic conditions
