# ModSecurity

## What it is
Think of it as a TSA checkpoint embedded directly inside your web server — every HTTP request and response passes through it and gets screened against a ruleset before anything dangerous reaches the application. ModSecurity is an open-source Web Application Firewall (WAF) that operates as a module for Apache, Nginx, and IIS, inspecting and filtering HTTP traffic in real time using signature-based and anomaly-scoring detection rules.

## Why it matters
In 2017, attackers exploited the Apache Struts vulnerability (CVE-2017-5638) — the same flaw behind the Equifax breach — by injecting malicious payloads into the Content-Type HTTP header. Organizations running ModSecurity with the OWASP Core Rule Set (CRS) had signatures that detected and blocked this exact injection pattern, buying critical time before patches were applied. This is the textbook case for why a WAF sitting in front of vulnerable applications saves breaches.

## Key facts
- ModSecurity operates in three modes: **Detection Only** (logs but doesn't block), **Prevention** (actively blocks), and **Self-Contained** (per-transaction processing)
- The **OWASP Core Rule Set (CRS)** is the standard ruleset used with ModSecurity, providing protection against the OWASP Top 10 including SQLi, XSS, and RFI
- It uses an **anomaly scoring system** — each matched rule adds points; if the total exceeds a threshold, the request is blocked, reducing false positives
- ModSecurity can perform **virtual patching** — deploying a WAF rule to block exploitation of a known vulnerability before the underlying code is patched
- Logs generated by ModSecurity are critical for **SIEM correlation** and qualify as evidence during incident response
