# RFC 5545

## What it is
Like a standardized shipping manifest that every calendar app in the world agrees to read, RFC 5545 defines the iCalendar format — a plain-text specification for encoding calendar events, tasks, and scheduling data using `.ics` files. It establishes the syntax and semantics for calendar object interchange across disparate systems, using a MIME-type structure of `text/calendar`.

## Why it matters
Attackers have weaponized `.ics` files as phishing delivery mechanisms — a malicious iCalendar attachment can embed URLs in event fields (DESCRIPTION, URL, LOCATION) that bypass email security gateways because the payload arrives as a calendar invite, not a suspicious link. Apple and Google Calendar auto-render these invites, creating unsolicited calendar spam or credential-harvesting landing pages that victims encounter even without clicking an email attachment directly.

## Key facts
- iCalendar files use a `VCALENDAR` wrapper containing components like `VEVENT`, `VTODO`, `VJOURNAL`, and `VFREEBUSY`
- Each property follows a `PROPERTY;PARAMETER=value:VALUE` syntax — e.g., `DTSTART;TZID=America/New_York:20240101T090000`
- RFC 5545 replaced the older RFC 2445 (iCalendar v1.0) and is the current baseline standard
- The `URL` property within a `VEVENT` component is a common phishing vector — security tools must inspect `.ics` attachments as rigorously as HTML or PDF files
- iCalendar data can be transported via iTIP (RFC 5546) for meeting request workflows, making calendar-based social engineering attacks particularly convincing since they mimic legitimate business scheduling
