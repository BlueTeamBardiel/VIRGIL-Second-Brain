# CWE-177

## What it is
Like a bouncer who checks IDs at the door but lets anyone in through the back window after they've been verified — this weakness occurs when input is validated *before* it's canonicalized (normalized). CWE-177 (Improper Handling of URL Encoding) describes failures where an application validates a URL or string in its encoded form but then processes a decoded or canonicalized version that bypasses those checks entirely.

## Why it matters
A classic exploit: an attacker submits `%2F..%2F..%2Fetc%2Fpasswd` to a web server. The input validation layer checks the encoded string and sees no obvious `../` path traversal — it passes. The server then decodes it to `/../../../etc/passwd` and serves the file. This double-decode or encode-then-validate anti-pattern was central to the IIS Unicode directory traversal vulnerability (CVE-2000-0884).

## Key facts
- CWE-177 is a child of CWE-116 (Improper Encoding or Escaping of Output) and sits within the broader **Improper Neutralization** category.
- The correct sequence is: **decode/canonicalize FIRST, validate SECOND** — never the reverse.
- Percent-encoding (`%XX`), double encoding (`%252F` → `%2F` → `/`), and Unicode normalization are common attack vectors exploiting this weakness.
- This vulnerability often enables **path traversal**, **filter bypass**, and **access control violations** in web applications.
- OWASP recommends using framework-level canonicalization functions rather than hand-rolled URL parsers, since manual implementations frequently miss encoding edge cases.
