# T1036.009

## What it is
Like a con artist wearing a judge's robe to walk unchallenged through a courthouse, this technique involves an attacker renaming or disguising a malicious binary to look like a trusted system tool. T1036.009 — **Invalid Code Signature** — occurs when malware uses a code signing certificate that is expired, revoked, self-signed, or otherwise invalid to create a false impression of legitimacy without actually passing cryptographic verification.

## Why it matters
In the 2020 SolarWinds supply chain attack, threat actors used legitimately signed code, but lesser sophisticated campaigns frequently deploy binaries with forged or invalid signatures betting that endpoint controls won't strictly enforce signature validation. A defender who configures AppLocker or Windows Defender Application Control (WDAC) to *require* valid trusted signatures — not just the *presence* of any signature — would block this technique cold.

## Key facts
- An invalid signature is worse than no signature: it actively attempts to deceive automated and human reviewers into assuming legitimacy.
- Windows does **not** block execution of invalidly-signed binaries by default — it only warns SmartScreen users; strict policy enforcement must be explicitly configured.
- Attackers often repurpose expired legitimate certificates stolen from software vendors to lend visual credibility in tools like Process Explorer or VirusTotal previews.
- Detection pivot: use Sigcheck (Sysinternals) or EDR telemetry to flag binaries where `SignatureStatus ≠ Valid` but a signature field is present.
- This sub-technique falls under the broader **Masquerading (T1036)** tactic, used to bypass both human analysis and automated allow-listing controls.
