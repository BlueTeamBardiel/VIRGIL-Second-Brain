# T1218.008

## What it is
Like hiring a trusted bank employee to smuggle cash out the back door, attackers use `odbcconf.exe` — a legitimate Windows tool for configuring ODBC drivers — to execute malicious DLLs while hiding behind Microsoft's own signature. Specifically, it abuses the `/A {REGSVR <dll_path>}` argument to load arbitrary DLLs without ever touching `regsvr32.exe` directly. This makes it a signed binary proxy execution technique under the broader living-off-the-land (LoTL) umbrella.

## Why it matters
In documented campaigns, threat actors have dropped malicious DLLs and used `odbcconf.exe` to register and execute them, bypassing application whitelisting solutions that trust Microsoft-signed binaries. Defenders who only block `regsvr32.exe` are blind to this equivalent pathway — security tools must monitor `odbcconf.exe` command-line arguments, not just the binary itself.

## Key facts
- `odbcconf.exe` is a native Windows binary signed by Microsoft, located in `C:\Windows\System32\`
- The `/A {REGSVR <dll>}` flag triggers `DllRegisterServer` export in the target DLL — functionally identical to `regsvr32` behavior
- Response files (`.rsp`) can be used with `odbcconf.exe` to obscure the malicious command in a separate file, evading command-line logging
- Detections should focus on: `odbcconf.exe` spawning unusual child processes, loading DLLs from user-writable directories, or using the `/A` flag with non-standard paths
- MITRE maps this under **Defense Evasion** and **Execution** tactics — dual-purpose like most proxy execution techniques
