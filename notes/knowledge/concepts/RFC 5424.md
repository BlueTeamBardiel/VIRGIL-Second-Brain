# RFC 5424

## What it is
Think of it as a universal postal address format for system events — just as every letter needs a consistent "From/To/Date/Subject" structure to be sorted reliably, RFC 5424 defines the standard message format for the Syslog protocol. It is the 2009 IETF standard that specifies how log messages are structured, including a defined header, structured data fields, and a free-form message body, replacing the older and ambiguous RFC 3164 (BSD Syslog).

## Why it matters
During incident response, a SOC analyst correlating events across a Linux firewall, a Windows IDS, and a Cisco router needs all three to speak the same logging language. RFC 5424's structured data fields (SD-Elements) allow SIEMs like Splunk or QRadar to parse logs programmatically without regex guesswork — meaning an attacker who tampers with log timestamps or injects malformed entries is far more likely to be caught when logs conform to a strict, validated schema.

## Key facts
- **Severity levels 0–7**: Emergency (0), Alert (1), Critical (2), Error (3), Warning (4), Notice (5), Informational (6), Debug (7) — same scale inherited from BSD Syslog but formally standardized
- **Facility codes** (0–23) identify the source process type (e.g., kernel=0, mail=2, security/auth=4) and are combined with severity to produce the **PRI value**: `PRI = (Facility × 8) + Severity`
- **HEADER includes**: VERSION, TIMESTAMP (RFC 3339 format), HOSTNAME, APP-NAME, PROCID, and MSGID — enabling precise log correlation
- **Structured Data (SD)**: Key-value pairs inside brackets (e.g., `[exampleSDID@32473 iut="3"]`) allow machine-readable metadata without parsing free text
- RFC 5424 operates over **UDP port 514** (traditional) or **TCP 6514 with TLS** (RFC 5425) for encrypted, integrity-protected log transport
