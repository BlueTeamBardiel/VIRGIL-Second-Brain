# CWE-420

## What it is
Imagine a bank with a high-security front door but a forgotten side entrance that still has an old, simple lock — attackers just walk in through the back. CWE-420 (Unprotected Alternate Channel) occurs when a system provides an alternative communication path or interface that bypasses the security controls applied to the primary channel. An attacker exploits this unguarded route to gain access, exfiltrate data, or execute commands that the main pathway would have blocked.

## Why it matters
In 2021, researchers discovered industrial control systems (ICS) that enforced strict authentication over their primary SCADA interfaces but exposed legacy Telnet ports on the same device — completely unauthenticated. Attackers targeting these systems skipped the hardened HTTPS interface entirely and connected via Telnet to issue control commands, demonstrating that the weakest channel defines the actual security posture of the whole system.

## Key facts
- The alternate channel doesn't have to be obscure — it can be a debug port, a backup API endpoint, a legacy protocol (Telnet, FTP), or even a physical interface like JTAG/UART on embedded devices
- This weakness directly undermines defense-in-depth: controls on one channel are irrelevant if another channel exists without them
- Common in IoT and embedded systems where manufacturers leave diagnostic channels open in production firmware
- Distinct from CWE-419 (Unprotected Primary Channel) — CWE-420 is specifically about the *alternate* or secondary path being unprotected
- Mitigation requires a full attack surface audit: enumerate *all* interfaces (network, physical, administrative), not just the primary one, and apply consistent access controls across each
