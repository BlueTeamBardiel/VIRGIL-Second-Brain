# CWE-226

## What it is
Like a hotel that forgets to erase the previous guest's whiteboard before the next occupant checks in, CWE-226 occurs when a product fails to clear sensitive information from resources before making them available for reuse. Formally, it is the failure to neutralize sensitive data (passwords, keys, PII) from memory, buffers, files, or registers before those resources are reallocated or released.

## Why it matters
In the 2014 Heartbleed vulnerability (CVE-2014-0160), OpenSSL returned heap memory contents without zeroing them first — attackers could repeatedly query the server and harvest private keys, session tokens, and passwords that lingered in uncleared buffers. A proper defense requires explicit memory zeroing (e.g., `memset_s()` or `SecureZeroMemory()`) before freeing buffers containing cryptographic material.

## Key facts
- Closely related to CWE-459 (Incomplete Cleanup) and CWE-244 (Improper Clearing of Heap Memory Before Release); CWE-226 is the parent concept
- Using standard `memset()` to zero sensitive data is unreliable — compilers may optimize it away as "dead code"; language-specific secure wipe functions must be used instead
- Applies beyond RAM: swap files, temp files, log files, and CPU registers can all retain sensitive residue
- The OWASP Top 10 category "Cryptographic Failures" (A02) encompasses this weakness, making it exam-relevant for CySA+ and Security+
- Mitigation includes OS-level features like `mlock()` to prevent secrets from being paged to disk, combined with explicit zeroing before `free()`
