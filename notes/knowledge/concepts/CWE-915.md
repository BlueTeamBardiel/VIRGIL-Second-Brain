# CWE-915

## What it is
Imagine handing a hotel clerk a form and they blindly update *every field in the database* based on whatever you wrote — including "employee_discount: true" you snuck in. That's mass assignment: when an application automatically maps user-supplied data to internal object properties without filtering which fields are allowed to be modified. This vulnerability arises when frameworks like Rails, Spring, or Django bind request parameters directly to model attributes with no allowlist enforcement.

## Why it matters
In 2012, a researcher exploited mass assignment in GitHub by adding his own SSH public key to the Rails organization repository — he simply included `organization_id` in a POST request, which Rails auto-bound to the user model without restriction. This gave him write access to any repo. Developers must explicitly define "strong parameters" (allowlist) or use DTOs to prevent untrusted fields from reaching the persistence layer.

## Key facts
- Also called **"over-posting"** in .NET/ASP.NET MVC environments and **"auto-binding"** in Java Spring contexts
- The fix is an **allowlist (permit list)**, not a denylist — explicitly declare which fields can be set from user input
- Ruby on Rails introduced **Strong Parameters** (Rails 4+) specifically to mitigate this after the GitHub incident
- Common dangerous targets: `role`, `admin`, `is_verified`, `price`, `balance` — privilege-escalation-relevant fields
- Classified under **OWASP API Security Top 10** as **API6:2023 – Unrestricted Access to Sensitive Business Flows** and overlaps with **A08:2021 – Software and Data Integrity Failures**
