# CWE-787

## What it is
Like a carpenter hammering nails past the end of a board and splitting the wood beneath, Out-of-Bounds Write occurs when a program writes data beyond the allocated memory buffer boundary. Precisely: it's a vulnerability where software writes to a memory location before the start or after the end of an intended buffer, corrupting adjacent memory, crashing the process, or enabling arbitrary code execution.

## Why it matters
In 2021, a CWE-787 flaw in the Windows Print Spooler (PrintNightmare, CVE-2021-34527) allowed attackers to write arbitrary data beyond buffer boundaries, achieving SYSTEM-level code execution remotely. Defenders patched it via emergency out-of-band updates and disabled the Print Spooler service on domain controllers — a classic example of compensating control when a patch can't be immediately applied.

## Key facts
- **#1 most dangerous software weakness** in MITRE's CWE Top 25 for multiple consecutive years (2020–2023), reflecting its prevalence and exploitability
- Commonly caused by missing bounds checks in C/C++ code using functions like `strcpy()`, `memcpy()`, or `sprintf()` without size validation
- Can lead to **heap corruption** (heap overflow) or **stack smashing** (stack overflow), both enabling control-flow hijacking via overwritten return addresses or function pointers
- **Mitigations include**: compiler protections (stack canaries, AddressSanitizer), memory-safe languages (Rust, Go), OS-level ASLR + DEP/NX, and static analysis tools (Coverity, CodeQL)
- Distinct from CWE-125 (Out-of-Bounds Read) — the *write* variant is typically more severe because it enables code execution, not just information disclosure
