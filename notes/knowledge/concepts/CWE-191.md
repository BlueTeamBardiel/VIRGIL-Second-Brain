# CWE-191

## What it is
Like an odometer rolling from 000000 back to 999999 when you try to subtract a mile from zero, integer underflow happens when a calculation produces a value below the minimum a variable can hold. Precisely: CWE-191 occurs when an arithmetic operation on an integer produces a result smaller than the minimum representable value for that data type, causing the value to wrap around to a large positive number.

## Why it matters
In 2002, a classic underflow bug in the OpenSSH challenge-response code allowed an attacker to send a specially crafted packet causing a `nresp` variable to wrap from 0 to an enormous number, leading to a heap overflow and remote code execution. Defenders must validate that subtraction operations on unsigned integers cannot produce values below zero before the operation executes, not after.

## Key facts
- Unsigned integers are especially dangerous: subtracting 1 from a `uint` value of 0 typically yields 4,294,967,295 (2³²−1) on 32-bit systems
- Underflow frequently leads to buffer overflows — the wrapped value is used to allocate or index memory, creating a secondary, more exploitable vulnerability
- C and C++ have **undefined behavior** for signed integer underflow, making compiler optimizations unpredictable and exploits harder to reason about
- Safe mitigation: check `if (a < b) { /* handle error */ }` *before* computing `a - b` when working with unsigned types
- CWE-191 is the conceptual mirror of CWE-190 (Integer Overflow); both fall under the broader CWE-682 (Incorrect Calculation) category
