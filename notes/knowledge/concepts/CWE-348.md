# CWE-348

## What it is
Like a bouncer who checks your ID only after you've already told him your name — and then trusts that name is real — this flaw occurs when software relies on the IP address found in the `X-Forwarded-For` HTTP header to identify a client. Since this header is user-controlled, any attacker can spoof it to impersonate trusted sources.

## Why it matters
An attacker targeting an admin panel that whitelists internal IPs (e.g., `192.168.1.1`) simply adds `X-Forwarded-For: 192.168.1.1` to their request, bypassing IP-based access controls entirely. This was exploited in real-world WAF bypasses and rate-limiting circumventions, where attackers spoofed trusted IPs to avoid throttling or trigger privileged functionality.

## Key facts
- `X-Forwarded-For` is set by *clients or proxies* — it is **not** a reliable source of truth for access control decisions
- The correct approach is to use the **TCP-layer source IP** (socket address) or a verified, server-controlled header from a trusted reverse proxy
- This vulnerability enables **authentication bypass**, **IP allowlist evasion**, and **rate-limit bypass**
- Classified under **CWE-348: Use of Less Trusted Source** — the software uses a less-trusted input channel when a more trustworthy channel is available
- Related to **CWE-290 (Authentication Bypass by Spoofing)** — spoofing the header achieves authentication bypass as a downstream consequence
