# T1134.001

## What it is
Like a contractor who steals a building manager's keycard to access restricted floors, Token Impersonation/Theft is when an attacker duplicates or hijacks an existing Windows access token from a higher-privileged process to assume that identity. Specifically, it involves stealing the security token associated with a running process (such as one owned by SYSTEM or a domain admin) and using it to spawn new processes or perform actions under that stolen identity.

## Why it matters
In the infamous pass-the-token lateral movement scenario, an attacker who has compromised a helpdesk machine runs Mimikatz to harvest tokens from a logged-in domain admin's active session. Without needing the admin's password, the attacker duplicates that token and spawns a command shell running as the domain admin, then pivots across the network to domain controllers.

## Key facts
- Requires `SeImpersonatePrivilege` or `SeAssignPrimaryTokenPrivilege` — commonly held by service accounts (IIS, SQL Server), making them prime targets
- Windows API calls `DuplicateToken()` and `ImpersonateLoggedOnUser()` are the core mechanics attackers abuse
- Tools like Incognito (Meterpreter module) and Mimikatz automate token enumeration and theft
- Differs from `CreateProcess with Token` (T1134.002) — impersonation is thread-level and temporary; primary token replacement affects the entire process
- Detection focus: look for unusual process ancestry, processes spawned by services running under high-privilege accounts, and unexpected use of `NtCreateToken` or `ZwDuplicateToken` syscalls
