# CWE-440

## What it is
Like a contract that promises delivery "within a reasonable time" but never defines what that means — leaving both parties with incompatible assumptions — Expected Behavior Violation occurs when a component behaves differently than its documented or implied specification. Precisely: CWE-440 describes situations where software fails to perform according to its documented design contract, creating a gap between what callers expect and what actually happens.

## Why it matters
In 2014, Apple's infamous `goto fail` SSL bug (CVE-2014-1266) demonstrated this perfectly: the SSL verification function's documented behavior was to validate certificates, but due to a duplicated `goto` statement, it silently returned success without actually checking the certificate chain. Attackers could perform man-in-the-middle attacks against any iOS/OS X application trusting that function, because every caller assumed the contract was honored.

## Key facts
- CWE-440 is a **behavioral weakness**, not a structural one — the code may look syntactically correct while violating its semantic contract
- It differs from CWE-436 (Interpretation Conflict) — CWE-440 is about a *single* component breaking its own promises, not two components disagreeing
- Security controls that sit "upstream" of the violation inherit the broken guarantee, multiplying blast radius
- Common triggers: error-handling shortcuts, optimization refactors that skip edge cases, and undocumented return value changes across library versions
- Detection requires behavioral/specification testing (fuzzing, contract testing, property-based testing), not just code review — the bug often hides in logic that looks intentional
