# CWE-707

## What it is
Like a nightclub bouncer who waves through anyone wearing a "VIP" sticker — even if it's hand-drawn in crayon — CWE-707 describes improper neutralization where a system trusts the *form* of data without validating its *meaning*. Precisely: it is a weakness where software fails to ensure that messages or data structures are syntactically and semantically consistent with what is expected, allowing attackers to smuggle malicious content past surface-level checks.

## Why it matters
HTTP Request Smuggling attacks exploit CWE-707 directly: an attacker crafts an HTTP request with ambiguous `Content-Length` and `Transfer-Encoding` headers, causing a front-end proxy and back-end server to disagree about where one request ends and the next begins. This mismatch lets the attacker prepend malicious content to another user's request — bypassing WAFs, hijacking sessions, or achieving cache poisoning — all because both servers accepted structurally *plausible* but semantically *inconsistent* messages.

## Key facts
- CWE-707 is a **class-level weakness** (a parent/abstraction) in MITRE's CWE hierarchy, covering many specific neutralization failures including injection, encoding bypass, and message inconsistency issues
- It underpins vulnerabilities where **different system components interpret the same data differently** — a fundamental smuggling/confusion pattern
- Proper defense requires **canonical form validation**: normalize data to a single, unambiguous representation *before* any security check is applied
- It is distinct from CWE-20 (Improper Input Validation) — CWE-707 emphasizes structural and semantic *consistency*, not just whether input is present or reasonable
- Commonly manifests in **HTTP, XML, JSON, and SQL parsers** where multiple parsing paths exist
