# T1003.003

## What it is
Like photographing a bank's master key mold instead of stealing the key itself, NTDS.dit dumping extracts the entire Active Directory credential database without touching individual accounts. NTDS.dit is the database file on Windows Domain Controllers that stores all domain user password hashes, Kerberos keys, and account metadata. Attackers who obtain this file can crack or pass every credential in the entire domain offline.

## Why it matters
During the 2020 SolarWinds campaign, threat actors moved laterally to Domain Controllers and extracted NTDS.dit to gain persistent, domain-wide access — no noisy brute-force required. A single successful extraction hands the attacker every user hash, enabling offline cracking or Pass-the-Hash attacks against any domain account, including administrators. Defenders must monitor for Volume Shadow Copy access and unusual `ntdsutil` execution as early indicators.

## Key facts
- **Location:** `%SystemRoot%\NTDS\NTDS.dit` — only accessible on Domain Controllers, locked by the OS during normal operation
- **Extraction methods:** Attackers use `ntdsutil`, Volume Shadow Copy Service (VSS) snapshots, or tools like `Impacket secretsdump.py` to bypass the file lock
- **What's inside:** NTDS.dit stores NTLM hashes, Kerberos AES keys, password history, and group membership for all domain objects
- **SYSTEM hive required:** The NTDS.dit file is encrypted; attackers must also steal the SYSTEM registry hive (containing the Boot Key/SysKey) to decrypt the hashes
- **Detection:** Monitor Event ID 4688 (process creation) for `ntdsutil.exe`, `vssadmin`, and `diskshadow`; alert on shadow copy creation on Domain Controllers
