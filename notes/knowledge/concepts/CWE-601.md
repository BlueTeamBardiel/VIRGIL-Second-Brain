# CWE-601

## What it is
Like a hotel concierge who hands you a taxi ticket to *any address a stranger whispers*, an Open Redirect blindly forwards users to attacker-controlled URLs. It occurs when a web application accepts an unvalidated URL parameter and redirects the browser to that destination without verifying it stays within a trusted domain.

## Why it matters
Attackers exploit Open Redirects to craft convincing phishing URLs — the link reads `https://bank.com/redirect?url=https://evil.com`, so the victim trusts the domain prefix before clicking. This technique was used in real OAuth phishing campaigns where the legitimate redirect endpoint laundered attacker URLs, bypassing email filters that only checked the first domain in the link.

## Key facts
- **Attack vector**: The malicious URL is embedded in the `url=`, `next=`, `redirect=`, or `return=` parameter — these are the first places to audit.
- **Primary impact**: Phishing and credential harvesting; the browser legitimately *leaves* the trusted site, landing on an attacker clone page.
- **Secondary impact**: Can be chained with Server-Side Request Forgery (SSRF) when the redirect target is consumed server-side rather than browser-side.
- **Remediation**: Use an allowlist of permitted redirect destinations; never construct redirect URLs from raw user input. Relative paths (e.g., `/dashboard`) are safe; absolute URLs from parameters are dangerous.
- **OWASP relevance**: Listed under OWASP Top 10 A03 (Injection-adjacent) and historically under "Unvalidated Redirects and Forwards" — a frequent Security+/CySA+ exam topic when discussing input validation controls.
