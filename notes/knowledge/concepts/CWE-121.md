# CWE-121

## What it is
Imagine a parking garage with only 10 spaces, but a valet keeps cramming in the 11th car — smashing into whatever is parked next door. A stack-based buffer overflow occurs when a program writes more data to a stack-allocated buffer than it can hold, overwriting adjacent memory including the return address, which attackers redirect to execute arbitrary code.

## Why it matters
The 1988 Morris Worm exploited a stack buffer overflow in the Unix `fingerd` daemon by sending an oversized string that overwrote the return address, allowing shellcode execution — infecting roughly 6,000 machines (10% of the internet at the time). Modern variants still appear in embedded systems and legacy C code where bounds checking is absent.

## Key facts
- The **return address** on the stack is the primary target — overwriting it lets attackers redirect execution to attacker-controlled shellcode or ROP gadgets
- Mitigation: **Stack Canaries** (GCC's `-fstack-protector`) place a random sentinel value before the return address; if it's overwritten, the program terminates
- **ASLR** (Address Space Layout Randomization) randomizes memory layout to make it harder to predict where shellcode lands, but can be defeated via information leaks
- **NX/DEP** (No-Execute / Data Execution Prevention) marks the stack non-executable, forcing attackers to use Return-Oriented Programming (ROP) instead of direct shellcode
- Root cause is almost always unsafe C functions: `strcpy()`, `gets()`, `sprintf()` — all lack bounds checking and are explicitly listed in CERT C standards as banned functions
