# CWE-1025

## What it is
Like a bouncer checking IDs by only glancing at the first digit and assuming "9" means over 21 — the check runs, but it's comparing the wrong thing. CWE-1025 is an **Incorrect Comparison** weakness where code compares values using an inappropriate operator or against the wrong reference, producing logically flawed access or validation decisions. The check appears to function, but its logic guarantees wrong answers under specific conditions.

## Why it matters
A classic exploitation appears in authentication bypasses where a string comparison uses `==` in PHP against a hash that starts with `0e` — causing PHP's loose comparison to treat the hash as scientific notation (zero), meaning two completely different passwords produce a "match." An attacker who discovers this can authenticate with a known alternate string that also hashes to a `0e...` value, bypassing login entirely without knowing the real password.

## Key facts
- PHP's loose `==` operator is the canonical example: `"0e1234" == "0e5678"` evaluates to `TRUE` because both are cast to `0.0` as floats
- The fix is almost always switching to **strict comparison** (`===` in PHP, `equals()` in Java) that checks both value AND type
- This weakness often masquerades as correct code during code review — the comparison executes without error but is semantically wrong
- Common contexts: authentication tokens, version number comparisons, cryptographic hash validation, and input range checks
- CWE-1025 is a child of **CWE-697 (Incorrect Comparison)** in the CWE hierarchy; both appear on OWASP and SANS Top 25 radar
