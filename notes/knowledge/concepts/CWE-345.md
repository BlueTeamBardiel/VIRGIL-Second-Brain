# CWE-345

## What it is
Like a delivery driver who hands over a package without checking ID or verifying the sender's return address, this weakness occurs when software accepts data without confirming it actually came from the claimed source. CWE-345 (Insufficient Verification of Data Authenticity) describes the failure to validate that data or messages are genuinely from a trusted, legitimate origin before acting on them.

## Why it matters
BGP hijacking attacks exploit this weakness at a protocol level — routers accept route announcements from peers without cryptographic verification of authenticity, allowing an attacker to redirect internet traffic through malicious infrastructure. The 2008 Pakistan Telecom incident, which briefly took YouTube offline globally, is a textbook example of CWE-345 in the wild. RPKI (Resource Public Key Infrastructure) was developed specifically as a defense to add cryptographic authenticity checking to BGP announcements.

## Key facts
- CWE-345 is the parent of several child weaknesses including CWE-346 (Origin Validation Error) and CWE-347 (Improper Verification of Cryptographic Signature)
- It frequently manifests in software update mechanisms that download and execute packages without verifying digital signatures
- DNS cache poisoning attacks succeed partly because original DNS lacks message authentication, making responses unverifiable — DNSSEC was designed to address this
- Attackers leverage this weakness via replay attacks, man-in-the-middle substitution, and spoofed API responses
- CVSS scores for this weakness commonly land in the High range (7.0–8.9) when exploitable remotely without authentication
