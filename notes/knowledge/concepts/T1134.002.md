# T1134.002

## What it is
Like a thief who doesn't pick the lock themselves but instead steals the locksmith's master keychain, this technique involves an attacker creating a new process that inherits the security token of another user — without ever knowing their password. Token Impersonation/Theft (specifically, creating a process with a token) allows adversaries to spawn child processes under a stolen access token, effectively running code as a different, more privileged user.

## Why it matters
In the infamous pass-the-hash lateral movement chains, attackers using tools like Mimikatz or Metasploit's `getsystem` will steal a SYSTEM-level token from a running process and use `CreateProcessWithTokenW` to spawn a command shell with full SYSTEM privileges. This bypasses traditional credential requirements entirely, meaning no password is ever cracked — just stolen context.

## Key facts
- Uses Windows API calls like `CreateProcessWithTokenW` or `CreateProcessAsUserW` to launch processes under a stolen token
- Requires `SeAssignPrimaryTokenPrivilege` or `SeImpersonatePrivilege` — privileges often held by service accounts (IIS, SQL Server)
- Differs from T1134.001 (Impersonation) in that a **new process** is created, not just a thread impersonating within the same process
- Commonly seen after initial compromise via a service account — attackers escalate to SYSTEM by stealing tokens from higher-privileged processes
- Detection focus: monitor for `CreateProcessWithTokenW` calls where the parent process token and spawned process token belong to **different users**
