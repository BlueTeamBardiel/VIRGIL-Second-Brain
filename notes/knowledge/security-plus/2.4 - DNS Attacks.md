# 2.4 - DNS Attacks

DNS attacks represent a critical threat vector in modern networks, as they exploit the fundamental infrastructure that translates domain names into IP addresses. Understanding [[DNS]] poisoning, [[Domain Hijacking]], and [[URL Hijacking]] is essential for Security+ candidates because these attacks can redirect legitimate users to malicious sites, compromise data, and undermine trust in network communications. The exam focuses on the mechanics of how these attacks occur, the prerequisites attackers need, and real-world impact scenarios like the 2016 Brazilian bank breach.

---

## Key Concepts

### DNS Poisoning
- **Definition**: Corrupting [[DNS]] cache or responses to redirect users to attacker-controlled destinations
- **Three primary attack vectors**:
  1. **Server-side poisoning**: Modifying the actual [[DNS]] server records (requires direct access or elevated privileges)
  2. **Client-side poisoning**: Altering the victim's local host file (`hosts` file), which takes precedence over [[DNS]] queries
  3. **On-path (Man-in-the-Middle) poisoning**: Intercepting valid [[DNS]] requests and injecting fake responses in real-time
- **On-path attacks**: Require redirection of either the original request or the resulting response; classified as real-time redirection attacks
- **Key distinction**: The attacker doesn't need to compromise the authoritative [[DNS]] server; they only need to intercept traffic or modify client-side resources

### Domain Hijacking
- **Definition**: Gaining unauthorized access to domain registration credentials, granting complete control over DNS configurations and traffic routing
- **Attack prerequisites**: Multiple attack vectors exist:
  - Brute force attacks against weak credentials
  - Social engineering to extract passwords
  - Compromising the email account linked to domain registration (often the weakest link)
  - Exploiting credential reuse across platforms
- **Impact scope**: Attacker controls where all traffic flows without needing to access actual web servers
- **Real-world example**: October 22, 2016 – 36 Brazilian bank domains hijacked for 6 hours, affecting 5 million customers managing $27 billion in assets

### URL Hijacking (Typosquatting / Brandjacking)
- **Definition**: Registering domains that are intentionally similar to legitimate ones to exploit user errors
- **Common typosquatting vectors**:
  - **Outright misspellings**: `professormessor.com` (extra "s") instead of `professormesser.com`
  - **Missing letters**: `professormeser.com` (missing "s")
  - **Extra characters**: `professormessers.com` (plural)
  - **Alternative TLDs**: `professormesser.org` instead of `.com`
  - **Different phrases**: Slight variations that sound similar
- **Monetization methods**:
  - Ad-based revenue from legitimate traffic redirects
  - Selling the domain to the rightful owner at inflated prices
  - Phishing sites mimicking legitimate services ("looks like the real site, please login")
  - Drive-by downloads infecting visitors with [[Malware]]
  - Competitor redirection (legal liability concerns)

### Distinctions (Common Exam Confusion)
- **DNS Poisoning vs. Domain Hijacking**: Poisoning corrupts responses/caches; hijacking takes control of registration credentials
- **URL Hijacking vs. DNS Poisoning**: URL hijacking exploits user typos through legitimate registration; poisoning corrupts legitimate [[DNS]] lookups
- **On-path attack**: Requires active man-in-the-middle positioning; not passive reconnaissance

## How It Works (Feynman Analogy)

**The Analogy**: Imagine a postal service where users need to look up an address in a phone book before mailing a letter. An attacker has three strategies:

1. **Break into the phone book factory** (DNS server) and change the addresses → [[DNS]] poisoning at server
2. **Intercept the person's personal address book** (host file) and change entries there → Client-side poisoning
3. **Stand in the middle of the mail route** and redirect envelopes to the wrong address → On-path [[DNS]] poisoning
4. **Bribe the postal clerk** and take over the registration office entirely → [[Domain Hijacking]]
5. **Print fake phone books** with names like "Amazom" or "Gogle" and wait for people to misread → [[URL Hijacking]]

**The Technical Reality**: 

- **DNS Poisoning** exploits the stateless, open nature of [[DNS]] queries over [[UDP]] port 53. Without [[DNSSEC]] validation, responses aren't cryptographically verified. An attacker can either modify authoritative server records (requires server compromise), poison a client's `/etc/hosts` file (requires local access), or inject fake responses into network traffic (requires on-path positioning or request/response interception).

- **Domain Hijacking** targets the identity verification process at domain registrars. Most registrars rely on email-based account recovery, making the email account the true "master key." Once compromised, an attacker modifies the Name Server (NS) records, completely redirecting [[DNS]] queries for that domain to attacker-controlled [[DNS]] servers.

- **URL Hijacking** relies on human psychology—typos and domain confusion. No technical attack occurs; the attacker simply registers legitimate-looking domains and waits for typos or similar-looking searches to drive traffic.

## Exam Tips

- **DNS Poisoning ≠ Domain Hijacking**: The exam will test your ability to distinguish. Poisoning corrupts lookups; hijacking takes registration control. Know the 2016 Brazilian bank case as a domain hijacking example.

- **Three-layer attack vectors for poisoning**: Remember server, client, and on-path. The exam may ask "which requires least access?" (Answer: on-path if you're already positioned on the network).

- **Host file precedence**: Candidates often forget that `hosts` files override [[DNS]] queries entirely on Windows (`C:\Windows\System32\drivers\etc\hosts`) and Linux (`/etc/hosts`). If an exam question mentions "client modifications," think host file.

- **URL Hijacking monetization**: The exam tests whether you understand that typosquatting is *not* strictly technical—it's about exploiting user mistakes. Ad revenue, phishing, and drive-by downloads are the primary attack goals. Legal domain ownership is legitimate.

- **On-path attack distinction**: Confirm the question specifies "real-time redirection" or "man-in-the-middle"—this is the key identifying feature of on-path [[DNS]] poisoning and differentiates it from static server or client poisoning.

- **Email compromise as root cause**: In domain hijacking scenarios, the exam often tests whether you recognize that compromising the *registrar account email* is the easiest path, not brute-forcing the registrar password.

---

## Common Mistakes

- **Confusing [[DNS]] poisoning with [[DNS]] amplification attacks**: Amplification is a [[DDoS]] technique using [[DNS]] as a volume source; poisoning is about corrupting responses. They're completely different threat models.

- **Assuming [[HTTPS]] prevents [[DNS]] poisoning**: Many candidates think [[TLS]]/[[HTTPS]] protects against [[DNS]] attacks. It doesn't—the user still gets redirected to the wrong IP address before [[TLS]] handshake. [[DNSSEC]] and [[DNS over HTTPS]] (DoH) are the mitigations.

- **Overestimating the difficulty of domain hijacking**: Candidates assume "hacking the registrar" requires sophisticated techniques. In reality, social engineering or email compromise is the most common vector—not server exploitation.

## Real-World Application

In your homelab, [[DNS]] security directly impacts threat detection and lab isolation. Deploying [[Pi-hole]] as a local [[DNS]] sinkhole enables filtering malicious domains, and integrating it with [[Wazuh]] creates alerting for suspicious [[DNS]] queries (typosquatting, C2 exfiltration patterns). For lab Active Directory environments, monitoring [[DNS]] poisoning attempts via [[Wazuh]] log ingestion helps simulate real sysadmin incident response. Additionally, understanding domain hijacking risks informs security policies for production domain registration—treating registrar email accounts with MFA and separate credentials as critical security assets, much like protecting [[Active Directory]] admins.

---

## Mitigation & Prevention Controls

- **[[DNSSEC]]**: Cryptographically signs [[DNS]] responses, preventing poisoning attacks
- **[[DNS over HTTPS]] (DoH) / [[DNS over TLS]] (DoT)**: Encrypts [[DNS]] queries end-to-end, preventing on-path interception
- **Host file monitoring**: [[IDS]]/[[IPS]] and [[SIEM]] (like [[Wazuh]]) should alert on unexpected `/etc/hosts` modifications
- **Domain registrar security**: Enforce MFA on registrar accounts, use dedicated email addresses not tied to other services
- **[[DNS]] query logging**: [[Wazuh]] and [[SIEM]] platforms should ingest [[DNS]] logs to detect anomalous queries or unusual resolution patterns
- **User education**: Training on typosquatting and phishing; don't click suspicious shortened links
