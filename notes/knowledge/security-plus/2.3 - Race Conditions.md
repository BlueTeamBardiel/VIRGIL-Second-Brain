# 2.3 - Race Conditions

Race conditions are a critical software vulnerability where concurrent processes or threads access shared resources without proper synchronization, leading to unpredictable and potentially dangerous outcomes. This topic is essential for Security+ because race conditions can be exploited to bypass security controls, escalate privileges, and corrupt data—and understanding the [[Time-of-Check to Time-of-Use (TOCTOU)]] attack pattern is a key exam focus. Real-world exploits like the 2023 Tesla Model 3 Pwn2Own attack demonstrate that race conditions remain actively exploited in modern systems.

---

## Key Concepts

- **Race Condition**: A flaw in a system's logic where the output depends on the sequence or timing of uncontrollable events. Two or more processes compete for access to a shared resource without proper locking or synchronization.

- **Time-of-Check to Time-of-Use (TOCTOU)**: A specific race condition attack where an attacker exploits the gap between when a system checks a condition and when it uses the result of that check. The attacker modifies the resource between the check and the use.

- **Concurrent Access**: Multiple processes, threads, or users attempting to access or modify the same resource simultaneously. Without synchronization primitives ([[locks]], [[mutexes]], [[semaphores]]), concurrent access creates vulnerability windows.

- **Synchronization Primitives**: Programming constructs (locks, mutexes, atomic operations, semaphores) that ensure only one thread/process can access a critical section at a time, preventing race conditions.

- **Critical Section**: A code segment that accesses shared resources and must not be executed by more than one thread simultaneously.

- **Atomicity**: The guarantee that an operation completes entirely without interruption. Non-atomic operations (those that can be split across multiple CPU cycles) are vulnerable to race conditions.

- **Privilege Escalation via Race Condition**: An attacker exploits timing to gain elevated permissions by manipulating file systems, temporary files, or authentication checks between the security check and the actual use.

- **File System Race Conditions**: Particularly dangerous in systems where file permissions, symbolic links, or temporary file creation can be manipulated between a check and use operation.

## How It Works (Feynman Analogy)

Imagine two friends sharing a single bank account with $100. Both friends check the balance at nearly the same instant and see $100. Friend 1 immediately withdraws $50, leaving $50 in the account. But Friend 2 checked *before* this withdrawal happened and still believes there's $100 available, so they also withdraw $50—bringing the account to $0 or going negative. The problem? Neither friend locked the account during their check-and-withdraw operation, so the second withdrawal was based on stale information.

**Technical reality**: In software, this occurs when:
1. **Thread A checks** a condition (e.g., file exists, user has permission, resource is available)
2. **Before Thread A uses** that result, **Thread B modifies** the underlying resource
3. **Thread A continues** with a decision based on outdated information, bypassing security controls or causing data corruption

The window between check and use is the vulnerability—and attackers can deliberately widen that window using timing attacks or system load.

## Exam Tips

- **TOCTOU is the primary attack pattern tested**: Recognize that race condition questions often frame themselves as "check, then use" scenarios. Look for questions mentioning file permissions, authentication, or resource availability checks followed by operations.

- **Real-world examples matter**: The exam may reference the Mars Rover Spirit (2004 file system reboot loop) or Pwn2Own exploits (Tesla 2023 TOCTOU on Bluetooth). Know that these aren't theoretical—they're proof race conditions are actively exploited.

- **Distinguish from other timing attacks**: Race conditions are *different* from [[timing attacks]] (which measure cryptographic operation duration) and [[side-channel attacks]]. Race conditions involve *concurrent modification* of shared state, not information leakage through timing side effects.

- **Mitigation is the answer on the exam**: Questions rarely ask "what is a race condition?" Instead, they ask how to prevent them. Know that answers include: **serialization** (one-at-a-time execution), **locking mechanisms**, **atomic operations**, **file permission checks** closer to use, and **temporary file creation in secure directories**.

- **Watch for privilege escalation scenarios**: Many exam questions present race conditions in the context of `/tmp` file manipulation or symbolic link attacks. The attacker wins when they can swap a file between the security check and the actual file operation, gaining unauthorized access.

---

## Common Mistakes

- **Confusing race conditions with deadlocks**: A deadlock is when two processes block each other indefinitely waiting for resources. A race condition is uncontrolled concurrent access. They're separate problems (though poor synchronization can cause both).

- **Thinking race conditions only affect databases**: Candidates often limit their mental model to database transactions. Race conditions affect *any* shared resource: files, network sockets, shared memory, configuration files, authentication tokens, and temporary directories. The Mars Rover and Tesla examples prove this.

- **Underestimating the severity in system administration**: Sysadmins may assume race conditions are "developer problems." But privilege escalation via `/tmp` file manipulation, Active Directory synchronization windows, and Cron job timing issues are all race condition exploits relevant to infrastructure. Recognize race conditions as a sysadmin security risk, not just a code-level issue.

---

## Real-World Application

In Morpheus's [[COCYTUS]] homelab, race conditions could manifest when [[Active Directory]] replication doesn't complete between security policy checks and enforcement, or when [[Wazuh]] agent configuration changes are deployed across the fleet—if an attacker can modify configurations during the window between deployment initiation and deployment completion on a target node, they could escalate privileges or disable logging. Similarly, temporary file creation in deployment scripts (especially in shared `/tmp` directories on shared storage) creates TOCTOU windows that could be exploited to insert malicious code. Properly synchronizing distributed operations and using atomic file operations with secure temporary directories mitigates these risks in production infrastructure.

## Detailed Breakdown

### The Banking Scenario (Exam Prototype)

The bank account example from the PDF is a textbook TOCTOU attack:

| Step | User 1 | User 2 | Account A Balance | Account B Balance |
|------|--------|--------|-------------------|-------------------|
| Start | — | — | $100 | $100 |
| Check | Checks A ($100) | Checks A ($100) | $100 | $100 |
| Transfer | Withdraws $50 from A | — | $50 | $100 |
| Deposit | — | — | $50 | $150 |
| Transfer | — | Withdraws $50 from A (believed to have $100) | $0 or -$50 | $150 |
| Deposit | — | Deposits $50 to B | $0 or -$50 | $200 |

**Expected outcome**: Account A: $0, Account B: $200  
**Actual outcome**: Account A: $50, Account B: $200 (or system becomes insolvent)

The vulnerability: Neither user held a lock during their check-and-transfer sequence.

### Real-World Exploit: Mars Rover Spirit (2004)

The Mars Rover Spirit experienced a race condition in its file system handling:
- A process checked the file system health
- If unhealthy, the rover initiated a reboot
- During reboot, the file system continued to degrade
- The rover re-detected the unhealthy state and rebooted again
- **Result**: Infinite reboot loop, rendering the rover unusable

**Lesson for Security+**: Even mission-critical systems (NASA rovers) aren't immune to race conditions. This example shows that race conditions can cause [[Denial of Service (DoS)]] conditions.

### Real-World Exploit: Tesla Model 3 Pwn2Own 2023

Security researchers achieved a TOCTOU attack against a Tesla Model 3's infotainment system:
- **Vector**: Bluetooth pairing interface
- **Attack**: Manipulated device pairing state between the security check (verify user authorization) and the privilege grant (elevate to root access)
- **Result**: Root-level code execution on the infotainment system
- **Prize**: $100,000 USD and they kept the Tesla

**Lesson for Security+**: Modern, safety-critical systems are still vulnerable. TOCTOU attacks are practical and lucrative.

## Mitigation Strategies

### 1. **Synchronization/Locking**
Use [[mutexes]], [[semaphores]], or other synchronization primitives to ensure only one thread accesses a critical section at a time.

```
Lock(resource)
Check(resource)
Use(resource)
Unlock(resource)
```

### 2. **Atomic Operations**
Perform the check and use in a single, indivisible operation that cannot be interrupted.

### 3. **Secure Temporary Files**
Instead of checking `/tmp/file` and then using `/tmp/file`:
- Create the file in a secure directory with restrictive permissions
- Use `mkstemp()` or equivalent to create with unique, unpredictable names
- Use [[file locking]] mechanisms

### 4. **Move the Check Closer to the Use**
Minimize the window between check and use.

### 5. **Use System Calls Atomically**
Prefer system calls that combine check and use (e.g., `open(O_EXCL | O_CREAT)` for atomic file creation).

### 6. **Input Validation at Use Time**
Don't rely solely on an earlier check. Validate permissions and state again immediately before using a resource.
