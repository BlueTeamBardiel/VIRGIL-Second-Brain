# 3.4 - Backups

## Summary

Backups are a critical component of organizational [[Security Architecture]] and disaster recovery planning, enabling recovery of important and valuable data in the event of data loss, corruption, or catastrophic system failure. This section covers backup frequency, storage strategies (onsite vs. offsite), encryption requirements, and the planning complexities involved in maintaining multiple backup sets. The exam tests your understanding of backup implementation strategies, security considerations, and the trade-offs between different backup approaches.

---

## Key Concepts

- **Backup Definition**: A copy of data stored separately from the primary system, designed to enable data recovery after loss, corruption, or disaster.

- **Backup Frequency**: The interval at which backups are performed (hourly, daily, weekly, monthly). Frequency depends on data criticality and change rates; high-volatility systems require more frequent backups.

- **Onsite Backups**: Backup copies stored at the organization's physical location
  - Advantages: No internet link required; data immediately available for rapid recovery; lower cost than offsite
  - Disadvantages: Vulnerable to physical disasters (fire, flood, theft) affecting the primary site; limited geographic redundancy

- **Offsite Backups**: Backup copies stored at a remote geographic location (different building, city, or cloud provider)
  - Advantages: Protection against site-wide disasters; restoration possible from anywhere; geographic redundancy
  - Disadvantages: Requires internet or WAN link for data transfer; restoration delayed by network latency; higher cost

- **Hybrid Backup Strategy**: Organizations maintain both onsite and offsite backups
  - Onsite: For fast recovery and frequent restores
  - Offsite: For disaster recovery and long-term retention
  - Multiple copies: Increases redundancy and restore options

- **Backup Media**: Physical or cloud storage used to hold backup data (magnetic tape, external hard drives, cloud object storage, etc.)

- **Backup Sets**: Multiple independent backup instances, often on different schedules (daily, weekly, monthly)
  - Example: Daily backups for 7 days, weekly for 4 weeks, monthly for 12 months
  - Creates tiered retention policy matching [[RTO]] and [[RPO]] requirements

- **Encryption of Backups**: Critical security control to protect sensitive data on backup media
  - Renders data unreadable without decryption keys
  - Recovery keys must be securely managed and stored separately from backup data
  - Especially important for offsite and cloud backups where media travels over networks or is stored by third parties

- **Backup and Recovery Software**: Tools and platforms that automate backup scheduling, manage backup sets, handle encryption, and facilitate restore operations

- **Backup Planning Considerations**:
  - Total amount of data to be backed up (impacts media capacity and transfer time)
  - Type of backup (full, incremental, differential)
  - Storage location strategy (onsite, offsite, hybrid)
  - Backup frequency and retention policy
  - Day of the week (some organizations use different schedules for different days)
  - Media management and tracking
  - Encryption key management

## How It Works (Feynman Analogy)

**Think of backups like photocopies of important documents:**

You have an original contract (your production data). Making a photocopy and keeping it in your desk drawer (onsite backup) means you can quickly reference it if the original gets spilled on, but if your building burns down, both copies are lost. To be really safe, you also mail a copy to your lawyer's office across town (offsite backup). Now, even if disaster strikes your location, your data survives.

However, if someone breaks into the lawyer's office, they can read the photocopy—unless it's written in a code they can't understand (encryption). The lawyer has a special decoder key that's kept in a separate safe. Without that decoder key, the backup is just gibberish.

**In technical reality:**
- **Production data** = your live systems and databases
- **Onsite backup** = backup storage at your data center (fast, local, vulnerable to site disasters)
- **Offsite backup** = backup storage at a remote location or cloud provider (slower, geographically protected, network-dependent)
- **Encryption** = cipher protecting backup media; only accessible with the correct decryption key
- **Recovery keys** = the "decoder" stored separately and securely

Organizations use both strategies because they have different purposes: onsite for speed, offsite for survival.

## Exam Tips

- **Distinguish between RTO and RPO**: Backup frequency determines your [[Recovery Point Objective]] (RPO). Backup location affects [[Recovery Time Objective]] (RTO). The exam will ask which strategy meets specific RTO/RPO requirements.

- **Backup frequency is business-driven, not technically driven**: The exam tests whether you understand that frequency depends on data criticality and change rates. A database with millions of transactions per day needs hourly backups; a static file server might need weekly.

- **Offsite backups are mandatory for disaster recovery**: The exam emphasizes that onsite-only backups fail against site-wide disasters. Look for questions testing whether candidates understand the necessity of geographic separation.

- **Encryption is a security control, not optional**: If the question involves sensitive or regulated data (healthcare, PCI, finance), encryption of backups is the correct answer. The exam frequently tests encryption of backup data in transit and at rest.

- **Key management is separate from backup media**: The exam will include questions where a candidate correctly encrypts backups but stores the recovery key with the backup media—this is a common mistake. Keys must be stored separately and securely (e.g., in a hardware security module or key management service).

- **Multiple backup sets = complex media management**: The exam tests understanding that maintaining daily, weekly, and monthly backup sets requires significant planning and tracking. This is a realistic scenario, not just theoretical.

---

## Common Mistakes

- **Confusing backup location with backup frequency**: Candidates often mix up "how often to backup" (frequency) with "where to store backups" (location). The exam tests both independently. A backup taken daily can be stored onsite, offsite, or both.

- **Assuming encryption makes backups immune to theft**: While encryption protects data confidentiality, it does not prevent availability issues. If encrypted backups are stolen and the recovery key is also compromised, the backup is still lost. Encryption protects the *contents*, not the *availability* of backup media.

- **Ignoring offsite backups because they're expensive**: Candidates sometimes argue that onsite backups are "good enough" to save money. The exam expects you to understand that offsite backups are a mandatory part of business continuity planning, not an optional luxury. Organizations use both for a reason.

## Real-World Application

In your homelab, implementing a tiered backup strategy is essential: daily backups of [[Active Directory]] and [[Wazuh]] SIEM data stored onsite (NAS) for quick recovery, combined with encrypted weekly backups pushed to a cloud provider or external offsite location. The [[Wazuh]] logs and [[Tailscale]] VPN configuration should be encrypted at rest using [[AES-256]] or equivalent, with decryption keys managed separately in a secure key store. This hybrid approach protects against both accidental deletion and site-wide disasters (hardware failure, fire, ransomware), while ensuring compliance with security auditing requirements. A real sysadmin must balance backup frequency against backup window constraints—backing up a 10TB data center every hour may overwhelm network capacity, requiring strategic scheduling.

## Additional Study Notes

### Backup Types (Complementary Concept)

While not explicitly detailed in the PDF section, the exam may test backup *types* alongside backup *strategy*:

- **Full Backup**: Complete copy of all data. Largest, slowest, but simplest to restore.
- **Incremental Backup**: Only data changed since last backup (full or incremental). Smaller, faster, but requires full + all incrementals to restore.
- **Differential Backup**: Only data changed since last full backup. Medium size, can restore with full + last differential.

Understanding these helps explain *why* organizations use multiple backup sets (daily incrementals, weekly fulls) rather than backing up everything daily.

### Compliance and Backup Encryption

Regulatory frameworks ([[PCI DSS]], [[HIPAA]], [[GDPR]]) often mandate encrypted backups. The exam may include scenario questions where a company processes payment cards and must encrypt all backup media. This is not a choice—it's a compliance requirement.

### RTO/RPO Decision Matrix

| Scenario | Backup Frequency | Storage Location | Reasoning |
|----------|------------------|------------------|-----------|
| Critical database, 1-hour RPO required | Hourly | Onsite + Offsite | Frequent backups meet RPO; onsite for fast RTO; offsite for disaster |
| Static file server, 24-hour RPO acceptable | Daily | Onsite | Infrequent changes; daily sufficient; cost-effective |
| Regulated financial data, compliance audit | Daily (minimum) | Offsite encrypted | Regulatory requirement; encryption mandatory; offsite for audit trail |
| Development/test system | Weekly | Onsite only | Non-critical; infrequent changes; cost savings acceptable |

**Study Strategy**: Focus on understanding *why* organizations choose specific backup strategies, not just memorizing definitions. The exam rewards reasoning about trade-offs.
