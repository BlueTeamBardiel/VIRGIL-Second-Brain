# 3.3 - Protecting Data

Data protection is a foundational responsibility in information security and a core objective of the [[CIA Triad]]. This section covers the mechanisms—[[encryption]], [[hashing]], [[obfuscation]], and [[masking]]—used to safeguard data at rest and in transit, along with geographic and access-based controls that prevent unauthorized access. For the Security+ exam, you must understand how each protection method works, when to apply it, and the specific security properties (confidentiality, integrity, authentication, non-repudiation) each one provides.

---

## Key Concepts

### Data Protection Fundamentals
- **Primary responsibility**: An organization cannot operate without data; protecting it is a core job function
- **Data everywhere**: Information exists on storage drives, across networks, and within CPU memory—all require protection
- **Multi-layered approach**: Combines [[encryption]], [[security policies]], and access controls ([[data permissions]])

### Geographic Restrictions & Geofencing
- **Network location**: Identify users based on [[IP subnet]]; challenging with mobile and remote devices
- **Geolocation methods**:
  - **GPS**: Most accurate for mobile devices
  - **802.11 wireless**: Moderate accuracy
  - **IP address**: Least accurate; can be spoofed
- **Geofencing**: Automatically allow or deny access when a user is/isn't in a defined physical location (e.g., "don't run this app unless near the office")
- **Use case**: Restricts sensitive operations to trusted locations, reducing remote attack surface

### Encryption
- **Definition**: Encodes plaintext (readable) information into ciphertext (unreadable) using a cryptographic algorithm
- **Bidirectional**: Can convert between plaintext and ciphertext if the proper [[encryption key]] is available
- **Confusion principle**: Encrypted data is drastically different from original plaintext, preventing pattern analysis
- **Protects**: Confidentiality (prevents unauthorized reading)
- **Application**: Securing data at rest (storage) and in transit (network)

### Hashing
- **Definition**: Represents data as a short fixed-length string of text (message digest or fingerprint)
- **One-way function**: Impossible to recover the original message from the hash; not reversible
- **No key required**: Unlike [[encryption]], hashing doesn't use keys—same input always produces same output
- **Primary uses**:
  - **Password storage**: Hash passwords instead of storing plaintext; [[CIA Triad|confidentiality]]
  - **Integrity verification**: Confirm downloaded documents haven't been altered by comparing hashes
  - **Digital signatures**: Combines hashing with asymmetric [[encryption]] for [[authentication]], [[non-repudiation]], and integrity
- **Collision resistance**: Different messages should never produce the same hash (ideally)
- **Common algorithms**: MD5 (deprecated), SHA-1 (weakened), SHA-256, SHA-3

### Obfuscation
- **Definition**: Make normally understandable information very difficult to understand
- **Code obfuscation**: Transform readable source code into logically identical but incomprehensible code
- **Developer perspective**: Developer keeps readable code; users receive obfuscated version
- **Security benefit**: Makes reverse engineering and vulnerability discovery harder
- **Limitation**: Not impossible to break; determined attackers can still analyze obfuscated code
- **Use case**: Protecting proprietary [[algorithms]], [[intellectual property]], and application logic from inspection

### Masking
- **Definition**: A specialized form of [[obfuscation]] that hides portions of original data
- **Primary use**: Protects [[PII]] (Personally Identifiable Information) and sensitive data
- **View vs. storage**: Data may remain intact in storage but masked on screen based on user [[permissions]]
- **Techniques**: Substitution, shuffling, [[encryption]], masking out characters (e.g., showing only last 4 digits of credit card)
- **Access control integration**: Different users see different levels of masked data based on [[RBAC]] or [[ABAC]]

## How It Works (Feynman Analogy)

**Imagine a locked safe in a bank vault:**

Think of [[encryption]] like a safe deposit box. You put your valuables (plaintext) inside, lock it with a key (encryption key), and now it's unreadable to anyone without that key (ciphertext). If you have the key, you can open it and retrieve the original valuables. The key is essential—without it, the safe is useless.

[[Hashing]], by contrast, is like a tamper-evident seal on a package. Once you apply the seal (hash function), it's impossible to get back to the original package contents. But if someone opens the package and tries to reseal it with a different seal, you immediately know something changed. You can't reverse-engineer the original contents from the seal, but you can verify authenticity and integrity.

[[Obfuscation]] is like writing assembly instructions in a language no one speaks. The code does exactly the same thing, but no one can understand *how* or *why* it works unless they spend enormous effort decrypting your invented language.

[[Masking]] is like a redaction pen on a document: you still have the full document in your filing cabinet, but when you photocopy it for someone, you black out sensitive parts. The underlying data isn't destroyed—just hidden from that person's view.

**Technical reality:** In a homelab like your homelab fleet, you'd use [[encryption]] (via [[TLS]] on [[Tailscale]]) to protect data in transit, [[hashing]] to store [[Active Directory]] password hashes securely, [[obfuscation]] to protect proprietary monitoring rules in [[Wazuh]], and [[masking]] in [[SIEM]] dashboards so junior analysts don't see full database credentials.

## Exam Tips

- **Hashing ≠ Encryption**: The exam heavily tests this distinction. Hashing is one-way and irreversible; [[encryption]] is two-way. You cannot decrypt a hash. Hashing verifies integrity; [[encryption]] ensures confidentiality.
  
- **Digital signatures use both**: A digital signature combines a hash (for integrity) with asymmetric [[encryption]] (for [[authentication]] and [[non-repudiation]]). Expect questions pairing these concepts.

- **Masking vs. Obfuscation**: Masking is data-focused (hiding PII on-screen); [[obfuscation]] is code-focused (making code unreadable). The exam may ask which protects PII—the answer is [[masking]].

- **Geofencing practicality**: Recognize that IP-based geolocation is unreliable (VPNs, proxies, mobile networks). GPS is most accurate but requires mobile hardware. Expect scenario questions on which method to use.

- **Confidentiality vs. Integrity vs. Authentication**: Match each protection method to its [[CIA Triad]] property:
  - [[Encryption]] → Confidentiality
  - [[Hashing]] → Integrity (and [[authentication]] if used in a digital signature)
  - [[Masking]] → Confidentiality (PII protection)
  - Geofencing → Access control (prevents unauthorized *access*, not data exposure)

---

## Common Mistakes

- **Thinking hashing is reversible**: Candidates often confuse hashing with [[encryption]] and assume you can "unhash" a password. Hashes are one-way by design. If the exam shows "decrypt this hash," the answer is "you cannot—hashes don't have keys."

- **Misidentifying masking as encryption**: [[Masking]] hides data from view but doesn't encrypt it; the underlying data is still readable in storage. If an attacker gains database access, they see unmasked data. [[Masking]] is a *display-layer* control, not a cryptographic one.

- **Overestimating geolocation accuracy**: Assuming IP-based geolocation can reliably restrict access. In reality, VPNs, proxies, and carrier CGNATs defeat IP-based restrictions. GPS is accurate but only for mobile devices. The exam expects you to recognize these limitations.

---

## Real-World Application

In your homelab fleet, data protection manifests as:
- **[[Tailscale]] VPN with [[TLS]] 1.3** encrypts all inter-node traffic, protecting against network sniffing
- **[[Wazuh]] SIEM** uses [[hashing]] to verify log integrity and store passwords securely; [[masking]] rules hide API keys and credentials from junior analyst dashboards
- **[[Active Directory]] password hashing** (NTLM or Kerberos) ensures passwords are never stored in plaintext
- **Geofencing via [[firewall]] rules** restricts administrative access to the homelab from known subnets, reducing attack surface
