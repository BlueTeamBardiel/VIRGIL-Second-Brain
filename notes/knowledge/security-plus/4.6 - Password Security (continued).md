# 4.6 - Password Security (continued)

## Overview

This section covers **just-in-time (JIT) permissions** and **password vaulting**—two critical privilege management strategies that minimize the attack surface of administrative credentials. Understanding these mechanisms is essential for the Security+ exam because they directly implement the principle of least privilege and represent modern approaches to preventing credential compromise and limiting breach scope. The exam tests both the conceptual framework and practical application of these controls in enterprise environments.

---

## Key Concepts

- **Just-in-Time (JIT) Permissions**
  - Temporary, time-limited elevation of user privileges for administrative tasks
  - Access is granted only when needed, revoked automatically after a defined period
  - Administrator accounts never retain permanent elevated rights
  - Requires formal request and approval through a central clearinghouse or authorization system

- **Principle of Least Privilege (PoLP)**
  - Users and accounts should have only the minimum permissions necessary to perform their job
  - Applied to JIT: admins operate as standard users by default, escalate only when required
  - Reduces the attack surface and blast radius of any single compromised account

- **Password Vaulting**
  - Centralized repository where primary (root/admin) credentials are securely stored
  - Vault controls access to credentials; individual users never see the actual passwords
  - Credentials are checked out (logged) and checked in; all access is audited
  - Primary passwords remain protected and are never directly distributed

- **Ephemeral Credentials**
  - Temporary credentials generated during JIT session activation
  - Single-use or session-specific; automatically expire and are deleted after use
  - Derived from or managed by the password vault; primary password remains hidden
  - Contrasts sharply with standing credentials that persist indefinitely

- **Central Clearinghouse**
  - Authorization system that evaluates JIT access requests
  - Applies predefined security policies to grant or deny elevation
  - Logs all requests, approvals, denials, and credential usage
  - Often integrated with [[SIEM]] and [[Identity and Access Management (IAM)]] platforms

- **Breach Scope Limitation**
  - If a user account is compromised, the attacker gains only temporary elevated access (if at all)
  - Time-limited nature prevents long-term persistence with administrative rights
  - Lack of stored admin credentials in user accounts prevents credential harvesting attacks

## How It Works (Feynman Analogy)

Imagine a bank vault where only the branch manager has the combination, and she keeps it memorized. If someone steals her keycard, they can enter the building but still cannot access the vault. Now imagine instead that the bank implements a policy: no one—not even the manager—keeps the combination written down or memorized. Instead, when the manager needs to open the vault, she must call the security office, prove her identity, explain why she needs access, and wait for the security chief to unlock the vault remotely. Once she closes it, the access code is changed again.

**Technical reality**: JIT + password vaulting work exactly this way. System administrators don't store or memorize root/admin passwords. When they need elevated access, they request it through a clearinghouse (like [[Active Directory]] with Privileged Access Workstation or a dedicated PAM solution). The system grants temporary, logged credentials (ephemeral) that expire after one session. The actual admin password lives only in an encrypted vault, never on an admin's workstation. If an attacker compromises the admin's account, they inherit only standard-user permissions—they'd need to compromise the vault or the clearinghouse to escalate, which are heavily monitored and isolated.

## Exam Tips

- **Distinguish JIT from standing privileges**: The exam will test whether you understand that JIT eliminates *permanent* admin rights. A candidate might incorrectly think "JIT is just like regular admin access but with approval." It's not—the key difference is *time-limited and temporary*.

- **Password vault ≠ password manager**: Don't conflate a password manager (like Bitwarden for personal passwords) with enterprise password vaulting. Vaults enforce *centralized control* and *audit logging*; individuals never see the actual credential. This is critical for privileged access management (PAM).

- **Watch for "principle of least privilege" as the underlying philosophy**: Expect a question that asks why JIT matters. The answer is PoLP—reduce standing privileges, narrow the window of vulnerability, and limit breach impact.

- **Ephemeral credentials are one-time or session-bound**: The exam may ask what happens to credentials after a JIT session ends. Correct answer: they're deleted/revoked, not reused. This is a common wrong-answer trap.

- **Central clearinghouse = authorization + audit**: Know that requests go through a system that evaluates policy AND logs everything. This ties directly to [[Incident Response]] and [[Forensics]]—auditing is essential for post-incident investigation.

---

## Common Mistakes

- **Confusing JIT with MFA**: JIT is about *temporary privilege elevation*, not multi-factor authentication. While they often work together, they're distinct controls. MFA is authentication; JIT is authorization/privilege management.

- **Assuming passwords are ever shared in JIT workflows**: Candidates sometimes think "the vault gives the password to the admin." Wrong. The vault either (a) grants temporary credentials that are never visible, or (b) the system performs the action on behalf of the admin without revealing the password. Primary credentials stay locked in the vault.

- **Forgetting that "no permanent admin rights" is the core security win**: Some answers will sound plausible because they mention "security," but they miss the point. The exam rewards answers that emphasize *elimination of standing privileges*, *time-limited access*, and *narrowed blast radius*. A compromised admin account without permanent elevation is far less dangerous.

## Real-World Application

In Morpheus's homelab, JIT + password vaulting would be implemented via [[Active Directory]] with a PAM solution (e.g., Delinea Secret Server or open-source alternatives like HashiCorp Vault) to manage COCYTUS fleet credentials. When Morpheus needs to perform maintenance on critical systems, instead of logging in with a stored domain admin account, he would request elevated access through the PAM system, receive a temporary token, and perform the task while all actions are logged to [[Wazuh]]. This approach dramatically reduces the risk of credential theft from his workstation and provides forensic evidence for any suspicious access attempts—critical in a homelab used for security research and testing.
