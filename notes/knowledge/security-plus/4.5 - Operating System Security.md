# 4.5 - Operating System Security

Operating system security encompasses the mechanisms and policies that protect systems at the kernel and administrative level, ensuring only authorized users and processes can access resources while maintaining system integrity. This section focuses on centralized identity and access management through [[Active Directory]], policy enforcement via [[Group Policy]], and hardening Linux systems with [[Security-Enhanced Linux (SELinux)]]. Mastery of these controls is critical for Security+ candidates because they form the foundation of enterprise security operations and are frequently tested in scenario-based questions about access control, least privilege, and incident containment.

---

## Key Concepts

### Active Directory (AD)
- **Centralized directory service** — A database that stores and manages all network objects (users, computers, printers, file shares, security groups)
- **Primarily Windows-based** — The standard identity and access management solution for Windows enterprises
- **Authentication management** — Users authenticate once with AD credentials and gain access to authorized resources across the network
- **Centralized access control** — Administrators define which users can access which resources from a single console
- **Help desk operations** — IT support staff use AD to reset passwords, provision/deprovision accounts, and manage group memberships
- **LDAP protocol** — [[Active Directory]] typically communicates using [[LDAP]] (Lightweight Directory Access Protocol) for directory queries

### Group Policy
- **Policy-based management** — Administrators create and apply policies that configure computers and users in bulk
- **Scope levels** — Policies can be applied at Local (individual machine) or Domain (organization-wide) scope
- **Group Policy Management Editor** — MMC snap-in tool (gpEdit.msc) used to create, view, and modify policies
- **Configuration options** — Hundreds of settings control security parameters, login scripts, network configurations (QoS), software deployment, and security restrictions
- **Centralized control** — Eliminates the need to manually configure each machine; changes propagate automatically to grouped systems
- **Enforcement mechanism** — Policies are applied at login and periodically refreshed; users cannot override mandatory settings

### Security-Enhanced Linux (SELinux)
- **Mandatory Access Control (MAC)** — Adds fine-grained permission model on top of Linux's traditional Discretionary Access Control (DAC)
- **Linux DAC limitation** — Traditional Linux DAC allows file owners to change permissions; SELinux restricts this flexibility for security
- **Application access restriction** — SELinux policies limit what applications and processes can do, following the principle of least privilege
- **Breach containment** — If an application is compromised, the attacker's access is scoped to only what that application is permitted to do
- **Open source** — Available on most major Linux distributions (Red Hat, CentOS, Ubuntu) and often enabled by default or as an optional hardening layer
- **Three modes** — Enforcing (blocks violations), Permissive (logs violations, allows actions), and Disabled (no enforcement)

### Key Distinctions
- **AD vs. Local accounts** — AD provides single sign-on and centralized management; local accounts are machine-specific and harder to manage at scale
- **DAC vs. MAC** — DAC (traditional Linux/Unix) lets owners control permissions; MAC (SELinux) enforces system-wide mandatory rules regardless of ownership
- **Group Policy vs. SELinux** — Group Policy is Windows-centric policy enforcement; SELinux is Linux-centric kernel-level access control

## How It Works (Feynman Analogy)

**Think of a university campus with multiple buildings (computers), staff (users), and locked rooms (resources):**

In a small organization (local accounts), each building has its own security desk and key management system—messy, inconsistent, and hard to audit. 

[[Active Directory]] is like consolidating all security operations into a single, central administration building. When a new professor (user) joins, they get one campus ID card (AD credentials). That card works in every building and room they're authorized for. When they leave, a single administrative action disables their card everywhere.

[[Group Policy]] is the rulebook posted in every building—a comprehensive set of policies that dictates "all doors lock at 6 PM," "visitors must check in at the desk," or "certain rooms require two-factor authentication." Instead of manually enforcing these rules in each building, administrators publish the rulebook once, and it applies everywhere automatically.

[[SELinux]] is like a security guard assigned to each building who enforces even stricter rules. Even if someone gains access to a room (a user runs an application), the guard restricts what they can do inside that room based on a predefined access policy—they can't just wander into the dean's office.

**Technical reality:** [[Active Directory]] stores identity objects in an LDAP-compliant directory and uses [[Kerberos]] for authentication; [[Group Policy]] uses Group Policy Objects (GPOs) applied via domain controllers; [[SELinux]] enforces policies defined in security contexts tied to users, roles, and types in the kernel's access control layer.

## Exam Tips

- **Scenario questions often test AD + Group Policy together** — Expect questions like "You need to enforce password complexity across 500 Windows machines; what's the best approach?" The answer involves Group Policy at the domain level, not manual configuration on each machine.

- **Distinguish between authentication and access control** — AD handles *authentication* (proving who you are); Group Policy enforces *authorization* (what you can access). The exam may ask which tool solves a specific access problem.

- **SELinux questions often focus on breach containment and least privilege** — If a question describes a compromised application and asks how to limit the damage, SELinux or MAC-based access control is likely the right answer. Remember: traditional DAC would allow the compromised process to do whatever the user who launched it can do.

- **Group Policy scope and inheritance matter** — The exam may ask about policy precedence (Local > Site > Domain > Organizational Unit). Study the order of application and how child OUs inherit parent policies.

- **SELinux modes are frequently tested** — Know the difference: Enforcing blocks violations, Permissive logs them, and Disabled turns off SELinux. A question might ask what mode to use when troubleshooting without compromising security.

---

## Common Mistakes

- **Assuming Group Policy applies to non-Windows machines** — Group Policy is Windows-only. For Linux/Unix systems, use configuration management tools (Ansible, Puppet) or [[SELinux]]/AppArmor. Candidates sometimes incorrectly answer "use Group Policy" for a mixed-OS environment.

- **Confusing DAC with MAC** — DAC (Linux default) lets the file/resource owner set permissions; MAC ([[SELinux]]) enforces policies regardless of ownership. A question asking "how do you prevent a user from changing file permissions even if they own the file?" is testing MAC knowledge, not DAC.

- **Overlooking [[Active Directory]] as a centralized control point** — Candidates may overthink access control problems and miss that [[Active Directory]] + [[Group Policy]] solves most Windows enterprise scenarios. If the question is about managing hundreds of Windows machines, AD is almost always part of the answer.

## Real-World Application

In your homelab, [[Active Directory]] could be deployed on a Windows Server VM to centralize user authentication and machine management across the fleet—critical for managing multiple hypervisors, VMs, and lab infrastructure with consistent access policies. [[Group Policy]] would enforce security baselines (e.g., enforce TLS 1.2, disable legacy protocols, require Windows Defender definitions) across all domain-joined Windows systems without manual per-machine configuration. For Linux nodes in the lab, [[SELinux]] or AppArmor would restrict what containerized services or compromised applications can access, containing blast radius during security testing or breach simulations. Combined with [[Wazuh]] for monitoring and [[Tailscale]] for secure network access, OS-level security controls form the foundation of defense-in-depth.
