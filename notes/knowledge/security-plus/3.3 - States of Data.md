# 3.3 - States of Data

Data exists in three distinct states—at rest, in transit, and in use—and each state requires different security controls and protection mechanisms. Understanding which encryption, access control, and network security tools apply to each state is critical for designing comprehensive data protection strategies. The exam tests your ability to identify the appropriate security measures for each data state and recognize real-world breach scenarios where inadequate controls failed.

---

## Key Concepts

- **Data at Rest**
  - Data stored on physical or logical storage devices (hard drives, SSDs, flash drives, databases, cloud storage)
  - Primary protection: [[Encryption]] (whole disk, database-level, file/folder-level)
  - Secondary protection: [[Access Control Lists]] (ACLs) and permissions to restrict authorized access
  - Most defensible state—you control the environment and can apply strong encryption

- **Data in Transit** (also called Data in Motion)
  - Data actively transmitted across networks between systems, devices, or locations
  - Travels through multiple network devices (switches, routers, firewalls, proxies) outside your direct control
  - Primary protection: **Transport Encryption**—[[TLS]] (Transport Layer Security) and [[IPsec]] (Internet Protocol Security)
  - Secondary protection: Network-based controls like [[Firewall]] and [[IPS]] (Intrusion Prevention System)
  - Most vulnerable state—exposure points are numerous and distributed

- **Data in Use** (also called Data in Motion in Memory)
  - Data actively processing in system memory (RAM, CPU registers, CPU cache)
  - Almost always exists in **decrypted form**—encryption would prevent processing
  - Highly vulnerable: attackers can extract decrypted data directly from RAM
  - Protection mechanisms are limited (memory protection, secure enclaves, data masking)

- **The Three-State Model**
  - Forms the foundation of comprehensive data protection strategy
  - Each state requires distinct, layered controls
  - No single solution protects all three states equally

---

## How It Works (Feynman Analogy)

**Think of data like cash in a person's life:**

- **At Rest** = Money in a safe or bank vault (highly protected with locks, cameras, guards)
- **In Transit** = Money being transported in an armored truck (protected by armed guards, specific routes, secure containers)
- **In Use** = Money being counted/used at a register (in someone's hands, actively being used, temporary vulnerability)

You wouldn't use the same protection for all three scenarios. You don't need an armored truck parked outside every register. Conversely, a simple envelope isn't secure for long-distance transport.

**In cybersecurity terms:** Whole-disk encryption protects data at rest like a vault protects cash. [[TLS]] encryption protects data in transit like an armored truck. But when data is "in use" in your system's RAM, it's often unencrypted because the CPU needs to process it—like the money must be accessible at the register. This is why the **Target breach (November 2013)** was so catastrophic: attackers accessed point-of-sale (POS) system RAM and extracted 110 million unencrypted credit card numbers *while they were being processed*, bypassing both encryption systems.

---

## Exam Tips

- **Identify the State First**: When a question describes a scenario, always identify whether data is at rest, in transit, or in use. This narrows down the correct security control immediately.
  - *Example:* "Protecting customer data on a file server" = at rest → look for encryption/ACL answers
  - *Example:* "Protecting login credentials sent over the internet" = in transit → look for [[TLS]]/[[IPsec]] answers

- **Common Trap: "Encryption solves everything"**
  - Candidates often default to "encrypt it" for all three states, but data in use *cannot* be fully encrypted during processing
  - If a question asks about protecting data in RAM, encryption isn't the primary answer—look for memory protection, secure enclaves, or access controls instead

- **Real-World Scenarios Matter**
  - The exam loves Target-breach-style questions: "An attacker obtained credit card data from POS RAM despite encryption being enabled. What was NOT in place?"
  - Answer: Data in-use protection (RAM protection, secure processing)

- **Transport Layer Protocols Are Testable**
  - Know the difference: [[TLS]] (application layer, uses port 443 for HTTPS) vs. [[IPsec]] (network layer, transparent to applications, used for VPNs)
  - Know when to use each: TLS for web/API security, IPsec for site-to-site VPNs or end-to-end network encryption

- **Access Control ≠ Encryption**
  - At rest, both [[ACL]]s and encryption are needed—they serve different purposes
  - ACLs control *who can access* the data; encryption makes it unreadable even if accessed
  - The exam may test which is insufficient on its own

---

## Common Mistakes

- **Assuming encryption alone protects data in use**: Candidates often forget that data must be decrypted to be processed. This state requires runtime protections (memory isolation, secure execution environments, data masking), not encryption.

- **Confusing "in transit" with "in use"**: Some students think "in use" means "being transported." In use specifically means actively processing in memory/CPU. In transit means moving across networks. The Target breach involved data *in use*, not in transit.

- **Overlooking network-based controls for in-transit data**: Candidates focus only on encryption ([[TLS]]/[[IPsec]]) but forget that [[Firewall]]s and [[IPS]] are also critical for in-transit data protection. A complete answer requires both encryption AND network monitoring.

---

## Real-World Application

In **COCYTUS homelab operations**, this knowledge directly applies:
- **At Rest**: [[Wazuh]] agent logs stored on the SIEM must be encrypted via [[Active Directory]] GPO whole-disk encryption or database encryption
- **In Transit**: Logs transmitted from fleet systems to the Wazuh server should use [[TLS]] encryption; Tailscale VPN tunnels also encrypt data in transit
- **In Use**: When Wazuh processes security events in RAM, that decrypted data becomes a potential exfiltration target—this is why network segmentation (VLANs, firewall rules) and access controls (LDAP/AD authentication) are critical to prevent unauthorized access to the security operations center

A sysadmin managing a homelab or enterprise environment must implement all three types of protection; relying only on at-rest encryption would be like the Target incident—data could still be stolen from memory.

## Study Questions for Morpheus

1. **Scenario**: A company's database server is protected with AES-256 encryption at rest and [[TLS]] in transit. An attacker gains shell access to the database server and dumps active queries from memory. Why was the company still vulnerable?
   - *Answer*: Data in use was not protected—it must be decrypted for processing, making it vulnerable in RAM.

2. **Scenario**: You're designing security for a Tailscale-connected homelab where sensitive logs flow from edge devices to a central Wazuh server. Which two protections are mandatory?
   - *Answer*: (1) [[TLS]]/encryption in transit (handled by Tailscale), (2) [[Encryption]] at rest on the Wazuh server, (3) [[Access Control]] via [[Active Directory]] or LDAP for in-use data.

3. **Compare & Contrast**: What's the key difference between protecting data at rest with encryption vs. with [[ACL]]s?
   - *Answer*: Encryption makes data unreadable even if accessed; [[ACL]]s prevent unauthorized access in the first place. Both are needed; neither alone is sufficient.
