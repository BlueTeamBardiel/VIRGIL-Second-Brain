# 2.4 - Replay Attacks (continued)

## Summary

This section focuses on **replay attack mechanics through header and cookie manipulation**, emphasizing how attackers intercept, modify, and replay session credentials to hijack user sessions. Understanding these attack vectors and their corresponding mitigations is critical for the Security+ exam because session hijacking represents a real-world threat in both corporate networks and homelab environments. The exam tests your ability to identify vulnerable transmission methods, recommend encryption standards, and distinguish between point-to-point and partial encryption strategies.

---

## Key Concepts

- **Replay Attack**: An attack where an attacker captures legitimate network traffic (session tokens, credentials, or requests) and replays it later to gain unauthorized access or perform unauthorized actions.

- **Session Hijacking**: Capturing or stealing a user's valid session identifier (session ID, cookie, or token) to impersonate the user without needing their credentials.

- **Header Manipulation**: Modifying HTTP headers (User-Agent, Referer, Authorization, etc.) to bypass security controls or inject malicious payloads.

- **Cookie Theft**: Extracting session cookies from the HTTP stream via packet sniffing, then using them to authenticate as the legitimate user.

- **Information Gathering Tools**:
  - **[[Wireshark]]**: Passive packet analyzer that captures and displays live network traffic in real-time.
  - **[[Kismet]]**: Wireless network detector and packet sniffer; effective for capturing unencrypted WiFi traffic.

- **Exploitation Tools**:
  - **[[Cross-site Scripting (XSS)]]**: Injecting malicious JavaScript to steal cookies or session tokens from a victim's browser.
  - **Tamper**: Browser extension to intercept and modify HTTP requests before they reach the server.
  - **Firesheep**: Deprecated but historically important Firefox extension that automated session hijacking on unencrypted WiFi.
  - **[[Scapy]]**: Python packet manipulation library used to craft and replay custom network packets.

- **Cookie Manipulation Tools**:
  - **Cookies Manager+ (Firefox add-on)**: Allows manual inspection, editing, and deletion of browser cookies.

- **End-to-End Encryption (E2EE)**: [[Encryption]] applied from sender to receiver; the server and intermediaries cannot decrypt the payload. Prevents attackers from capturing or replaying session tokens because they remain encrypted in transit.

- **HTTPS (HTTP Secure)**: Uses [[TLS]] encryption to protect HTTP traffic; prevents passive packet sniffing from capturing session IDs in cleartext.

- **HTTPS Everywhere / Force-TLS**: Browser extensions and server policies that enforce encrypted connections, redirecting all HTTP requests to HTTPS.

- **Encrypt End-to-Somewhere**: Partial encryption (e.g., [[VPN]] to corporate gateway, or HTTPS only on the final hop); some segments of the journey remain unencrypted, leaving them vulnerable to capture on local networks.

- **Personal VPN**: User-deployed [[VPN]] to encrypt traffic from their device to a remote VPN server, protecting local wireless capture but not necessarily protecting all intermediate hops.

## How It Works (Feynman Analogy)

**Imagine you're using a post-it note to leave yourself reminders.** You write down your grocery list and stick it on your laptop. An attacker standing nearby photographs your post-it, writes an identical copy, and uses *that copy* to impersonate you at the grocery store. The store clerk trusts the post-it because it looks exactly like yours—he doesn't verify it came directly from you.

In network terms, your **session cookie is the post-it note**. When you log into a website over unencrypted HTTP, your browser sends a session ID in a cookie. An attacker using [[Wireshark]] or [[Kismet]] captures this traffic, extracts the cookie, and replays it to impersonate you. If that cookie is sent in plaintext (not encrypted), the attacker can read it, modify it, or use it indefinitely.

**The technical reality**: When you visit `http://example.com/login` without HTTPS, your browser sends `Cookie: sessionID=abc123xyz` in the HTTP header. An attacker on the same WiFi network sniffs this packet, notes the session ID, and later sends their own request to the server with the same cookie. The server sees a valid, unexpired session and grants access—never knowing it's not you.

**The mitigation**: With [[TLS]] encryption (HTTPS), the entire HTTP request—including headers and cookies—is encrypted. An attacker can see packets flowing, but cannot read the content. Even if they capture a packet, it's gibberish. This forces attackers to target the decryption key instead, which is far harder than simply reading plaintext cookies.

## Exam Tips

- **Wireshark vs. Kismet**: The exam may test the distinction. [[Wireshark]] is for wired and wireless packet capture and analysis; [[Kismet]] is specifically for wireless network discovery and packet sniffing. Both are tools used *before* a replay attack to gather session information.

- **HTTPS Everywhere is not HTTPS**: HTTPS Everywhere is a *browser extension* that enforces HTTPS; the actual encryption is performed by the server and [[TLS]]. Know the difference between the tool and the protocol it enforces.

- **End-to-End vs. End-to-Somewhere**: The exam loves this distinction. **End-to-End [[Encryption]]** (e.g., [[TLS]] from client to server) means attackers cannot intercept in the middle. **End-to-Somewhere** (e.g., unencrypted local WiFi → encrypted [[VPN]] → home lab) leaves some segments exposed. Expect a scenario asking which method is better; end-to-end is always more secure.

- **Session Hijacking prevention**: The exam tests whether you recommend [[Encryption]], [[MFA]], secure cookie flags (HttpOnly, Secure, SameSite), session timeout, and IP pinning. Know that **encryption alone prevents packet sniffing**, but other controls prevent abuse of legitimately obtained tokens.

- **Scapy and packet crafting**: If a question asks what tool allows you to *create and replay custom packets*, that's [[Scapy]] or raw socket programming. Don't confuse it with passive tools like [[Wireshark]].

---

## Common Mistakes

- **Confusing information gathering with exploitation**: Candidates sometimes think that because [[Wireshark]] can capture packets, it has "exploited" the session. In reality, Wireshark is just the first step—gathering the cookie. The *actual attack* is replaying that cookie. These are separate phases.

- **Assuming HTTPS solves all replay attacks**: While [[TLS]] prevents packet sniffing of credentials, it does not prevent replay of legitimately encrypted cookies if the attacker obtains them (e.g., via [[XSS]], malware, or cached data). The exam tests knowing that HTTPS must be *paired* with other controls like session expiration, [[MFA]], and secure cookie attributes.

- **Treating "end-to-end encryption" and "VPN" as equivalent**: A [[VPN]] encrypts traffic from your device to the VPN server, but the VPN server or the recipient can still decrypt it. True end-to-end encryption (like in Signal or WhatsApp) means only the sender and receiver can decrypt. The exam wants you to know that a personal [[VPN]] *adds* security but is not the same as [[E2EE]].

## Real-World Application

In a homelab like **YOUR-LAB**, you might run a web-based dashboard for [[Active Directory]] management or [[Wazuh]] alerting. If that dashboard is exposed over unencrypted HTTP, an attacker on your internal network (or with access to packet captures) could sniff session cookies and hijack your admin account. Enforcing [[TLS]] on all dashboards and using [[Tailscale]] (which provides encrypted tunneling) adds two layers of defense: the attacker cannot see the cookie in transit, and even if they obtain it, [[Tailscale]]'s encryption makes it nearly useless without the VPN keys.

## Study Notes for Morpheus

**Question Pattern**: Expect a scenario like: *"An attacker captures an admin's session cookie over unencrypted WiFi. Which of the following would have prevented this?"*

**Correct Answer Approaches**:
1. Use HTTPS / [[TLS]] — encrypts the cookie in transit
2. Implement [[MFA]] — even if the cookie is stolen, the attacker needs a second factor
3. Use secure cookie flags — HttpOnly prevents JavaScript theft, Secure forces HTTPS-only transmission
4. Session timeout — even if captured, the cookie expires quickly
5. IP pinning — session only valid from the original IP address

**Avoid Answer Trap**: "Use a [[Firewall]]" alone is insufficient; firewalls don't encrypt cookies and won't prevent an authenticated attacker from using a stolen cookie they've already passed through.
