# 2.3 - Cloud-specific Vulnerabilities

Cloud adoption is nearly universal, but organizations are deploying cloud infrastructure without implementing foundational security controls—creating a perfect storm of sensitive data exposure and attack surface expansion. This section covers the unique vulnerabilities inherent to cloud environments, the attack vectors threat actors exploit, and why traditional security practices often fail in cloud deployments. Understanding cloud-specific vulnerabilities is critical for the Security+ exam because cloud infrastructure represents a massive shift in where data lives and who controls the security baseline.

---

## Key Concepts

### Cloud Security Fundamentals
- **Cloud Adoption Gap**: Nearly universal cloud use, but security practices lag significantly behind deployment rates
- **Shared Responsibility Model**: Organizations must understand what the cloud provider secures vs. what they must secure themselves
- **Management Console Access**: Critical control point; 76% of organizations fail to implement [[MFA]] for console users—a basic security best practice
- **Unpatched Code in Production**: 63% of production code contains high or critical vulnerabilities (CVSS ≥ 7.0)—a catastrophic oversight

### Attack Vectors Against Cloud Services

#### Denial of Service (DoS)
- Fundamental attack type that disrupts availability of cloud resources
- Cloud's distributed nature can amplify impact across interconnected services
- Often precursor to other attacks to distract security teams

#### Authentication Bypass
- Exploits weak or faulty authentication mechanisms
- Often targets management consoles, APIs, and service-to-service authentication
- Frequently enabled by missing [[MFA]], default credentials, or weak password policies

#### Directory Traversal
- Faulty configurations allow attackers to navigate file systems beyond intended boundaries
- Cloud storage misconfiguration is a common culprit (e.g., overly permissive IAM roles)
- Results in unauthorized data access

#### Remote Code Execution (RCE)
- Exploitation of unpatched systems and applications
- Allows attacker to execute arbitrary commands with privileges of compromised service
- Critical severity in cloud because single compromised instance can pivot to other resources

#### Web Application Vulnerabilities
- **Log4j / Spring Cloud Function**: Recent high-impact vulnerabilities in popular cloud frameworks
- Easy to exploit with extensive rewards (data access, lateral movement)
- Cloud-native architectures heavily rely on these technologies

#### Cross-Site Scripting (XSS)
- Exploits poor input validation in web applications
- Attacker injects malicious scripts that execute in victim's browser
- Particularly dangerous in multi-tenant cloud environments

#### Out-of-Bounds Write
- Writing data to unauthorized memory areas outside allocated buffer
- Consequences: data corruption, application crashes, or **code execution**
- Often a stepping stone to [[privilege escalation]]

#### SQL Injection
- Direct database manipulation through unsanitized input
- Attacker crafts malicious SQL queries to bypass authentication, exfiltrate data, or modify records
- Cloud databases are high-value targets due to centralized sensitive data storage

## How It Works (Feynman Analogy)

**The Unlocked Apartment Building Analogy:**

Imagine a massive apartment building (cloud infrastructure) where the landlord (cloud provider) is responsible for the building's structural integrity, locks on external doors, and security cameras in hallways. The tenants (organizations) are responsible for locking their individual apartments, not leaving windows open, and not giving spare keys to strangers.

Now imagine 76% of tenants don't even use deadbolts on their apartment doors (no [[MFA]] for management access). Many tenants have never repaired broken windows (63% unpatched vulnerabilities). When a burglar (attacker) shows up, they don't need to pick the lock—they just walk in, steal everything, and laterally move to neighboring apartments because security is so lax.

**Technical Reality:**
In cloud environments, the provider secures the infrastructure, but organizations must secure their own configurations, access controls, and applications. When organizations skip [[MFA]], fail to patch, and misconfigure permissions (like [[IAM]] roles), they hand attackers keys to sensitive data. Unlike on-premises systems where you physically control everything, cloud requires a different security mindset: assume the infrastructure is secure, but validate every access control you configure.

## Exam Tips

- **MFA is non-negotiable**: The exam will test your understanding that multi-factor authentication is a baseline control, especially for privileged access. 76% statistic exists because it's a *common failure*—know that this is a basic control organizations frequently skip.

- **Patching in the cloud isn't optional**: High/critical vulnerabilities (CVSS ≥ 7.0) in production code represent critical risk. The exam will test your ability to prioritize patching and understand the difference between [[vulnerability management]] and patch management.

- **Input validation prevents multiple attacks**: [[XSS]], [[SQL Injection]], and [[directory traversal]] all stem from accepting untrusted input. A single answer option about "validating and sanitizing user input" will solve multiple vulnerability questions.

- **Distinguish RCE from privilege escalation**: RCE = executing code. [[Privilege escalation]] = gaining higher permissions after code execution. The exam often tests whether you understand the *progression* of attacks.

- **Cloud-native vulnerabilities are current**: Log4j and Spring Cloud Function are mentioned specifically because they're recent, high-impact, and widely exploited. Expect questions on "easy-to-exploit, high-reward" vulnerabilities that dominate CVE lists.

---

## Common Mistakes

- **Assuming the cloud provider secures everything**: Organizations blame providers for breaches, but 76% of MFA failures and 63% of unpatched code are *organizational* failures. The exam tests your understanding of the **shared responsibility model**—you are responsible for your own security posture.

- **Conflating vulnerability types**: Candidates mix up [[XSS]] (client-side injection), [[SQL Injection]] (database injection), and [[directory traversal]] (path manipulation). Each has distinct exploitation methods and mitigations. On the exam, the question will hinge on identifying the *correct* vulnerability type, not just any injection attack.

- **Underestimating authentication as an attack surface**: Missing [[MFA]] isn't just "bad practice"—it's a critical vulnerability that enables account takeover. The exam will present scenarios where MFA would have prevented a breach, and you must recognize this immediately.

## Real-World Application

In your homelab, this translates directly to securing [[Tailscale]]-connected cloud instances and on-premises infrastructure: misconfigured [[IAM]] roles on EC2 instances mirror the 76% MFA failure rate, unpatched container images replicate the 63% production vulnerability problem, and misconfigured security groups introduce [[directory traversal]] and RCE risks. When designing the homelab's multi-environment setup (dev/staging/prod), you must implement [[MFA]] universally, scan container images with vulnerability scanners, and use [[Wazuh]] to detect exploitation attempts (e.g., [[XSS]] payloads in web logs, SQL injection patterns in database audit logs). The principle is identical: cloud or on-premises, assume attackers will probe for these foundational misconfigurations and plan defenses accordingly.
