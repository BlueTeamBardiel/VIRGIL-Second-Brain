# 1.2 - Gap Analysis

Gap Analysis is a foundational security assessment methodology that identifies the difference between an organization's current security posture and its desired future state (the target baseline). This process requires comprehensive research across people, processes, and technology, and often takes weeks or months to complete properly. Understanding gap analysis is critical for the Security+ exam because it appears in governance, risk, and compliance (GRC) contexts and represents how organizations systematically approach security improvements and compliance audits.

---

## Key Concepts

- **Gap Definition**: The measurable difference between "where you are" (current state) and "where you want to be" (desired state/target baseline)
- **Current State Assessment**: Detailed evaluation of existing security systems, policies, processes, and personnel capabilities
- **Target Baseline**: The desired security posture, often defined by formal standards like [[NIST]] Special Publication 800-171 or [[ISO/IEC 27001]]
- **Baseline Selection**: Organizations must choose a recognized framework or standard to measure against—either internally defined goals or external compliance requirements
- **Three Pillars of Gap Analysis**:
  - **People**: Employee training, experience levels, knowledge of security policies
  - **Processes**: Existing IT procedures, security policy documentation, operational workflows
  - **Technology**: Current systems, tools, configurations, and their effectiveness
- **Scope Expansion**: Analysis must examine broad security categories, then break them into smaller, manageable segments
- **Actionable Outcomes**: The gap analysis report must include a clear path from current state to desired state, including time, budget, and change control requirements
- **Formal Standards Used**:
  - [[NIST SP 800-171 Rev. 2]] — Protecting Controlled Unclassified Information (CUI) in nonfederal systems
  - [[ISO/IEC 27001]] — Information security management systems framework
- **Research Intensity**: Extensive data gathering, technical research, multiple stakeholders, and significant email/documentation trails
- **Deliverable**: A formal gap analysis report documenting current state, target objectives, and prioritized recommendations

## How It Works (Feynman Analogy)

Imagine you're a homeowner who wants to improve your house's security. Right now, you have some old locks and a basic alarm system (current state). You've decided you want a smart home security setup with cameras, motion sensors, and keyless entry (desired state). 

A gap analysis is the detailed plan you create by:
1. **Documenting what you have**: "I have two front-door locks, one side-door lock, no cameras, no monitoring."
2. **Defining where you want to go**: "I want full video coverage, smart locks on all doors, 24/7 monitoring, automated alerts."
3. **Finding the gap**: "I need 4 cameras, 3 smart locks, a monitoring subscription, and professional installation."
4. **Creating a roadmap**: "Phase 1 costs $2,000 and takes 2 weeks. Phase 2 costs $1,500 and takes 1 week. Total timeline: 3 weeks, $3,500."

In an organization, the principle is identical—but instead of a house, you're assessing security across entire IT systems, compliance requirements, and hundreds of employees. You're comparing your current security posture against a formal standard (like [[NIST]] or [[ISO 27001]]) and documenting exactly what investments, training, and process changes are needed to reach that target.

## Exam Tips

- **"Gap" vs. "Baseline" vs. "Assessment"**: The exam may ask you to distinguish these. A **baseline** is the target standard; an **assessment** is the evaluation process; the **gap** is the difference between current and baseline. Gap analysis *combines* all three.
- **Framework Selection is Crucial**: Expect questions on when to use [[NIST SP 800-171]] (CUI/nonfederal systems) versus [[ISO/IEC 27001]] (general information security management). Know the context clues in the question.
- **Three Components to Remember**: People, Processes, and Technology. If a question asks what gap analysis must evaluate, all three are typically involved. Don't assume it's only technical.
- **Timeline and Resource Expectations**: Gap analysis takes weeks/months and requires extensive stakeholder participation. Answer choices may test whether you understand this is not a quick audit—it's a comprehensive study.
- **Report Deliverables**: The exam may ask what a gap analysis report must include. The answer should include: current state documentation, target objectives, identified gaps, prioritized recommendations, timeline, budget, and change control strategy.

---

## Common Mistakes

- **Confusing Gap Analysis with Vulnerability Scanning**: Gap analysis is strategic and process-oriented; it's not a technical scan. [[Nmap]], [[Nessus]], or [[Qualys]] scans show *vulnerabilities*, not gaps. Gap analysis *uses* vulnerability data as input but encompasses much more (people, processes, compliance).
- **Assuming Gap Analysis is One-Time**: Candidates forget that gap analysis must be repeated periodically. Security posture, regulations, and business goals change. A gap analysis report from 2022 may be outdated by 2024. The exam may test whether you know gap analysis is part of a continuous cycle.
- **Treating Gap Analysis as IT-Only**: Many candidates forget the "people" and "process" components. A gap analysis that only examines technology and ignores employee training, policy documentation, or incident response procedures is incomplete. The exam tests holistic understanding.

## Real-World Application

In Morpheus's homelab environment ([[COCYTUS fleet]]), a gap analysis would assess current monitoring capabilities with [[Wazuh]], network segmentation via [[Tailscale]], identity management with [[Active Directory]], and compare them against [[NIST]] Cybersecurity Framework or [[ISO 27001]] objectives. For example: "Current state: Wazuh collects logs from 5 systems; Desired state: Full endpoint detection and response with threat hunting capability." The gap might be: "Need EDR agents on 20 additional systems, analyst training, and automated response playbooks—estimated 8 weeks, $15K."

In real sysadmin work, gap analysis is the formal document that justifies budget requests to management. It's how you prove you need that new SIEM, extra security staff, or network redesign.
