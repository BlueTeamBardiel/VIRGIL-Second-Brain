# 3.2 - Port Security

Port security is a network access control mechanism that prevents unauthorized devices from connecting to network ports until proper authentication is completed. This section covers [[IEEE 802.1X]] and [[EAP (Extensible Authentication Protocol)]], the foundational standards that enforce port-based authentication on both wired and wireless networks. Understanding these technologies is critical for the Security+ exam because they represent how modern organizations enforce the principle of "never trust, always verify" at the network edge—a cornerstone of [[Zero Trust]] architecture.

---

## Key Concepts

- **Port Security (General)**: A network-level access control strategy that requires authentication before a device gains full network access; prevents unauthorized or rogue devices from connecting
  
- **IEEE 802.1X**: A port-based network access control (NAC) standard that acts as a gatekeeper, denying network access until authentication succeeds; works on both wired (Ethernet) and wireless networks

- **Port-Based Network Access Control (NAC)**: The specific function of 802.1X—it controls access at the port level of network switches and access points, not the device level

- **EAP (Extensible Authentication Protocol)**: An authentication **framework** (not a protocol itself) that provides a flexible method for carrying authentication protocols; defined by RFC standards

- **EAP Flexibility**: Manufacturers can implement proprietary EAP methods tailored to their security requirements; examples include EAP-TLS, EAP-PEAP, EAP-MD5, EAP-TTLS

- **Supplicant**: The client device attempting to connect to the network (e.g., a laptop, smartphone, or IoT device)

- **Authenticator**: The network device (switch port or wireless access point) that enforces port security and blocks traffic until authentication succeeds

- **Authentication Server**: The backend system (e.g., [[RADIUS]], [[LDAP]], [[TACACS+]], [[Kerberos]]) that validates the supplicant's credentials and communicates the result back to the authenticator

- **Authentication Factors**: Port security can enforce single-factor (username/password) or [[MFA]] (multi-factor authentication) depending on the EAP method chosen

- **Network Access Until Success**: A critical distinction—the supplicant has **zero** network access (except for authentication traffic) until the authentication process completes successfully

---

## How It Works (Feynman Analogy)

Think of port security like a nightclub with a bouncer at the door. When you arrive, the bouncer (authenticator) checks your ID (credentials) by contacting a registry of approved guests (authentication server). Only after the registry confirms you're legitimate does the bouncer unlock the velvet rope and let you onto the dance floor (network). Until that moment, you're stuck outside—you can't get past the door, and you certainly can't access the bar or VIP areas.

In technical reality, 802.1X works the same way: when a device connects to a network port, the authenticator (switch or AP) blocks all traffic except for special EAP authentication frames. The supplicant sends credentials encapsulated in EAP packets. The authenticator forwards these to an authentication server (RADIUS/LDAP/etc.), which validates them. Once authentication succeeds, the authenticator opens the port and the device gets full network access. If authentication fails, the port stays locked.

## Exam Tips

- **802.1X vs. EAP**: These are **not** the same thing. 802.1X is the *standard*, EAP is the *protocol framework* that 802.1X uses. Expect questions that ask which one does what.

- **Three-Party Architecture**: Always identify the three roles in port security scenarios: **Supplicant** (client), **Authenticator** (network device), and **Authentication Server** (backend system). Exam questions often test whether you know which device performs which role.

- **Port-Level Enforcement**: Port security operates at the **port level** of a switch or access point, not at the device or user level. This is a subtle but important distinction—two different users on the same device trigger separate authentications.

- **Authentication Methods**: EAP is an *extensible framework*—manufacturers build different EAP methods on top of it. Don't memorize every variant, but know that EAP-TLS (certificate-based), EAP-PEAP (tunneled), and EAP-MD5 exist and have different security profiles.

- **Common Trap**: Candidates confuse port security with **port-level [[VLAN]]** assignment. Port security is about authentication; VLAN assignment is a separate (though often combined) feature that places authenticated devices into isolated network segments.

- **Answer Strategy**: If a question describes a scenario where devices are locked out of the network until credentials are verified, and the standard is mentioned as 802.1X, immediately look for the role descriptions (supplicant, authenticator, server) to confirm your understanding.

---

## Common Mistakes

- **Treating EAP as a standalone protocol**: EAP is a *framework*—it doesn't authenticate by itself. Candidates often forget that a specific EAP method (TLS, PEAP, etc.) must be selected and that a backend authentication server (RADIUS/LDAP) must exist. Test questions sometimes ask "What is missing from this 802.1X deployment?" and the answer is "An authentication server" or "A specific EAP method."

- **Confusing "before authentication" vs. "after authentication" states**: A supplicant connected to a port but not yet authenticated has **no network access** (except for EAP traffic). Candidates sometimes think the device can still ping the gateway or resolve DNS before authentication completes—it cannot. This is a frequent exam gotcha.

- **Forgetting that port security works on wired networks too**: Many candidates associate 802.1X only with wireless networks (because it's commonly deployed in enterprise WiFi). In reality, 802.1X and port security work on wired Ethernet ports as well, and the exam tests this distinction.

## Real-World Application

In Morpheus's [[COCYTUS]] homelab, port security using 802.1X would be implemented on managed switches and [[Tailscale]] exit nodes to ensure only authenticated devices (or users) can join the trusted network mesh. The authentication could be backed by [[Active Directory]] (using [[Kerberos]]) or a [[RADIUS]] server, providing centralized control over who accesses which network segments. On the security monitoring side, [[Wazuh]] would flag any repeated failed authentication attempts on port security, helping detect brute-force attacks or rogue devices trying to breach the network perimeter.
