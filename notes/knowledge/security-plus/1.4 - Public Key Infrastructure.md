# 1.4 - Public Key Infrastructure

## Summary

[[Public Key Infrastructure]] (PKI) is a comprehensive framework of policies, procedures, hardware, software, and people designed to create, distribute, manage, store, and revoke [[digital certificates]]. PKI enables secure communication by binding public keys to individuals and devices through trusted [[Certificate Authority|Certificate Authorities]], making it the foundational system that enables modern cryptographic trust on the internet and in enterprise environments. Understanding PKI is critical for the Security+ exam because it underpins virtually all modern secure communications, from HTTPS to VPNs, email encryption, and code signing.

---

## Key Concepts

### Symmetric vs. Asymmetric Encryption
- **[[Symmetric Encryption]]**: Uses a single, shared secret key to both encrypt and decrypt data
  - Fast and efficient but difficult to distribute securely
  - If the key is compromised, all encrypted data is at risk
  - Common algorithms: [[AES]], [[DES]], [[3DES]]
  - Does not scale well—each pair of communicators needs a unique key

- **[[Asymmetric Encryption]]** (Public Key Cryptography): Uses two mathematically related keys
  - Data encrypted with the public key can only be decrypted with the private key
  - Data encrypted with the private key can be decrypted with the public key (digital signatures)
  - Solves the key distribution problem but computationally more expensive
  - Common algorithms: [[RSA]], [[ECC]] (Elliptic Curve Cryptography)

### The Key Pair
- **[[Public Key]]**: Can be distributed freely to anyone; no need to keep secret
- **[[Private Key]]**: Must be kept secret by the owner; never transmitted or shared
  - Cannot be derived from the public key mathematically
  - The only key capable of decrypting data encrypted with the corresponding public key
  - Loss of the private key means loss of access to encrypted data and ability to create digital signatures

### Key Generation
- Both public and private keys are generated simultaneously during initialization
- Involves extensive randomization and complex mathematics
- Uses large prime numbers (typically 2048-bit or 4096-bit keys in [[RSA]])
- The mathematical relationship between the two keys ensures security while allowing one to be public

### Digital Certificates
- Bind a public key to an identity (person, device, service)
- Issued by trusted [[Certificate Authority|Certificate Authorities]]
- Include the public key, identity information, certificate expiration date, and the CA's digital signature
- Can be revoked via [[Certificate Revocation List]] (CRL) or [[Online Certificate Status Protocol]] (OCSP)
- Standard format: [[X.509]]

### Certificate Authority (CA)
- Trusted third party responsible for verifying identity before issuing certificates
- Creates the binding between the public key and the entity
- Signs certificates with its own private key, allowing verification of certificate authenticity
- Root CAs are at the top of the chain of trust; intermediate CAs operate below them

### Key Escrow
- A third party (often the organization or government agency) holds copies of decryption keys
- Allows authorized personnel to access encrypted data without the owner's private key
- May be required by law enforcement or for compliance purposes
- Highly controversial due to privacy concerns but sometimes legally mandated
- Requires strict access controls, legal procedures, and court orders

### Trust Model
- PKI is fundamentally about establishing and managing trust
- Trust flows from the Certificate Authority down through the chain of certificates
- Clients must trust the root CA to trust all subordinate certificates
- The strength of the entire system depends on the security practices of the CA

## How It Works (Feynman Analogy)

**The Mailbox Analogy:**

Imagine Alice has a public mailbox on her front porch with her name on it. Bob can drop a sealed letter into the mailbox, and anyone can see it's addressed to Alice. However, only Alice has the key to open that mailbox and read the letter inside. She keeps that key hidden in her house.

In this analogy:
- The **public mailbox** = Alice's public key (everyone can use it)
- The **mailbox key** = Alice's private key (only Alice has it)
- The **sealed letter** = encrypted data (readable only with the private key)

**Technical Reality:**

In PKI, Bob uses Alice's publicly available public key (like the mailbox address) to encrypt a message. The encryption algorithm performs complex mathematical operations that scramble the plaintext into ciphertext. Only Alice, who possesses the mathematically paired private key, can reverse these operations to decrypt the message back into plaintext. The relationship between the two keys is so mathematically tight that deriving the private key from the public key is computationally infeasible with current technology.

This solves the fundamental problem with [[Symmetric Encryption]]: how do two parties share a secret key securely over an untrusted network? With PKI, Alice never has to transmit her private key; she only publishes her public key, and anyone can securely send her encrypted messages.

## Exam Tips

- **Understand the asymmetry**: On the exam, you'll see questions asking "which key can decrypt data encrypted with the public key?" The answer is always the private key. Conversely, data encrypted with the private key can be decrypted with the public key (this is the basis for digital signatures).

- **Key distribution is the point**: Asymmetric encryption exists specifically to solve the key distribution problem of symmetric encryption. If a question asks why we use asymmetric encryption despite it being slower, the answer relates to secure key exchange and scalability.

- **Certificate vs. Key confusion**: Don't confuse [[Digital Certificates]] with keys. A certificate is a package that binds a public key to an identity—it's signed by a CA. The certificate itself is public; the private key it references is private.

- **CA = Trust authority**: Recognize that the [[Certificate Authority]] is the single point of trust. If you trust the CA, you trust all certificates it has issued. Questions often test whether you understand why the CA's private key must be protected more carefully than any end-entity private key.

- **Key escrow red flags**: When a question describes a scenario where a third party holds your private keys, identify it as [[Key Escrow]]. Know that while this enables organizational access to encrypted data, it's controversial and requires strict legal/procedural controls.

- **Chain of trust**: Understand hierarchical PKI with root CAs and intermediate CAs. A client validates a certificate by checking the issuing CA's certificate, and that CA's certificate, and so on up to a trusted root CA.

## Common Mistakes

- **Confusing which key does what**: Candidates often mix up which key encrypts vs. decrypts. Remember: public key encrypts → private key decrypts. But for signatures: private key signs → public key verifies.

- **Thinking symmetric and asymmetric are alternatives, not complements**: Many students don't realize that modern systems (like [[TLS]]) use asymmetric encryption to securely exchange symmetric keys, then use the symmetric key for the bulk of data encryption (because it's faster). They're used together, not instead of each other.

- **Underestimating the importance of the CA**: Candidates often overlook the [[Certificate Authority]] as the weak point. If a CA is compromised, all certificates it issued become untrustworthy. The exam tests deep understanding of CA security practices.
