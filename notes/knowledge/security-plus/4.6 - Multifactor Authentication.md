# 4.6 - Multifactor Authentication

Multifactor authentication ([[MFA]]) is a security control that requires users to prove their identity using two or more independent authentication factors from different categories. This topic is critical for the Security+ exam because MFA is the industry standard for protecting against credential compromise, phishing, and unauthorized access—and it directly supports the [[Zero Trust]] model that modern security architectures rely upon. Understanding the four factor types, their strengths, weaknesses, and real-world implementation is essential for both the exam and securing production systems.

---

## Key Concepts

- **Authentication Factors (The Four Types)**
  - **Something You Know**: Information only you should possess (password, PIN, security questions)
  - **Something You Have**: Physical or digital objects in your possession (smart cards, USB security keys, hardware tokens)
  - **Something You Are**: Unique biological characteristics (biometrics—fingerprints, iris scans, voice prints, facial recognition)
  - **Somewhere You Are**: Geographic or network location context (GPS location, IP address geolocation, network-based detection)

- **Multifactor vs. Multi-Step Authentication**
  - [[MFA]] requires factors from *different categories*
  - Asking for a password twice = multi-step, not multifactor
  - The exam tests this distinction heavily

- **"Something You Know" Examples**
  - **Password**: A memorized string of characters; most common but least secure when used alone
  - **PIN (Personal Identification Number)**: Typically 4–8 digits; not usually stored on the card/device itself, validated server-side
  - **Pattern**: A visual unlock pattern (e.g., Android pattern lock); only the user knows the correct sequence

- **"Something You Have" Examples**
  - **Smart Card**: Integrated circuit card (ICC) that stores certificates or keys; often requires a PIN in addition (adding a second factor)
  - **USB Security Key**: Hardware token (e.g., YubiKey) containing a certificate or cryptographic material; plugs into USB port for authentication
  - **Hardware Token**: Generates time-based or event-based pseudo-random codes (TOTP, HOTP)
  - **Mobile Device**: Acts as an authenticator; can receive SMS codes, push notifications, or run authenticator apps

- **"Something You Are" (Biometrics)**
  - **Fingerprint Scanning**: Captures ridge patterns; stored as mathematical representation, not the actual fingerprint image
  - **Iris/Retina Scan**: Scans unique patterns in the eye; highly accurate but less common due to cost and usability
  - **Voice Print**: Analyzes voice characteristics; vulnerable to spoofing and audio recording attacks
  - **Facial Recognition**: Optical capture of facial geometry; increasingly used on smartphones and physical access systems
  - **Critical Limitation**: Biometric data cannot be changed like a password; if compromised, the user cannot "reset" their biometric
  - **Scope**: Used in very specific, high-security situations (data centers, classified facilities, mobile device unlock)
  - **Fallacy**: Biometrics are often presented as "foolproof," but they can be spoofed or fail due to environmental factors

- **"Somewhere You Are" (Geolocation Factor)**
  - **IP Address Geolocation**: Determines location based on the source IP address; not perfect because VPNs, proxies, and IP ranges can be misleading
  - **IPv6 Limitation**: Geolocation works better with IPv4; IPv6 addresses don't correlate to location in the same predictable way
  - **GPS/Mobile Services**: Uses Global Positioning System or cellular network location data for precise geolocation
  - **802.11 Triangulation**: Uses nearby known WiFi networks to infer location without GPS
  - **Use Case**: Transaction may only complete if the user is within an authorized geographic region (e.g., only authenticate banking from the home country)
  - **Caveat**: Not a perfect identifier; GPS can be spoofed, cell data can be inaccurate, and users roam legitimately

## How It Works (Feynman Analogy)

**Simple Analogy:**
Imagine you're trying to access a high-security room at a bank. With single-factor authentication, you just need to show your employee ID card—and if someone steals your card, they're in. With multifactor authentication, you need *both* your card (something you have) *and* a PIN (something you know). Even if someone steals your card, they can't use it without knowing your PIN. If you add a third factor—your fingerprint—they'd need your card, your PIN, *and* your actual finger.

**Technical Reality:**
In practice, when you log into a service with MFA enabled:
1. You enter your username and password (something you know) — **Factor 1**
2. The server prompts you to verify with a second factor: scan your fingerprint, enter a code from an authenticator app, or approve a push notification on your phone (something you are/have) — **Factor 2**
3. The server validates both factors independently and grants access only if *both* succeed

The security gain is exponential: compromising a single factor (e.g., phishing your password) no longer grants access. An attacker would need to compromise *multiple* independent factors simultaneously—a much harder target.

## Exam Tips

- **Test the Distinction**: The exam will ask "which is an example of multifactor authentication?" and include distractors like "password + security question" (both are "something you know," so not multifactor). Always verify factors come from *different categories*.

- **Biometric Limitations Are Testable**: Expect questions like "Why aren't biometrics used as the sole authentication method?" Answer: they cannot be changed if compromised, they're not foolproof, and they're context-specific. The exam loves this nuance.

- **Geolocation is Imperfect**: Questions often present geolocation as a security panacea. Be ready to identify its weaknesses: IP geolocation can be spoofed with [[VPN]]s, GPS can be jammed, and IPv6 doesn't correlate to geography well. Geolocation adds a layer but isn't foolproof.

- **Token Types Matter**: Distinguish between:
  - **TOTP (Time-based One-Time Password)**: Generates codes based on the current time; no internet required (e.g., Google Authenticator, Authy)
  - **HOTP (HMAC-based One-Time Password)**: Generates codes based on a counter; increments with each use
  - **SMS-based codes**: Codes sent via text message; convenient but vulnerable to SIM swapping and interception

- **Common Scenario**: "A user authenticates with a password and then receives an SMS code on their phone. Which factors are in use?" Answer: *Something you know* (password) and *something you have* (phone)—two different categories = multifactor.

---

## Common Mistakes

- **Confusing Multi-Step with Multifactor**: Asking for a password twice, or a password followed by a security question, is multi-step but *not* multifactor (both are "something you know"). The exam loves this trap.

- **Assuming Biometrics are Unhackable**: Candidates often think "biometric = unbreakable." In reality, biometrics can be spoofed (fake fingerprints, deepfake voice/face), fail due to environmental conditions, and cannot be reset if compromised. The exam expects you to know these limitations.

- **Overestimating Geolocation Security**: Treating IP-based geolocation or GPS as a primary security factor. Both can be circumvented or spoofed. Geolocation is a *supplementary* factor, not a primary one. The exam tests whether you understand it's a weak factor used alongside stronger ones.

## Real-World Application

In your homelab, implementing [[MFA]] across [[Active Directory]], [[Wazuh]], and Tailscale would dramatically reduce the attack surface. For example, requiring MFA on all AD administrative accounts (using TOTP or hardware keys) prevents lateral movement even if a sysadmin's password is phished. Similarly, enabling MFA on Tailscale and Wazuh dashboards ensures that compromising a single credential doesn't grant an attacker full visibility into the fleet. In production, this is non-negotiable: the 2023 CISA directives explicitly mandate MFA for remote access and administrative functions.
