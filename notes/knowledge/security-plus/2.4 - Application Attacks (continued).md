# 2.4 - Application Attacks (continued)

This section covers critical application-layer vulnerabilities including [[Privilege Escalation]], [[Cross-Site Request Forgery (CSRF)]], and [[Directory Traversal]] attacks. These vulnerabilities represent some of the most frequently exploited attack vectors on the Security+ exam because they directly compromise application security and user data. Understanding the mechanics, mitigation strategies, and real-world implications of these attacks is essential for identifying and defending against contemporary threats in both traditional and cloud environments.

---

## Key Concepts

### Privilege Escalation
- **Definition**: Gaining higher-level access to a system than originally granted, typically escalating from user-level to administrator or system-level privileges
- **Root Cause**: Exploitation of vulnerabilities—bugs, design flaws, or misconfigurations in the operating system or application
- **Impact**: Once an attacker gains elevated access, they can:
  - Install malware or [[Rootkits]]
  - Access sensitive data
  - Modify system configurations
  - Disable security controls like [[Antivirus]] or [[Firewall]]
- **Priority**: These are **high-priority security patches** because any unprivileged user could potentially become an administrator
- **Horizontal Privilege Escalation**: A variant where User A accesses User B's resources at the same privilege level (lateral movement within the same privilege tier)
- **Vertical Privilege Escalation**: Traditional escalation from low-privilege user to high-privilege (admin/system) account

### Privilege Escalation Mitigations
- **Patch Management**: Apply security updates immediately—privilege escalation vulnerabilities are critical
- **Anti-Virus/Anti-Malware**: Updated signatures detect known exploits and malicious payloads
- **[[Data Execution Prevention (DEP)]]**: CPU-level protection ensuring only executable memory regions can execute code (prevents shellcode injection)
- **[[Address Space Layout Randomization (ASLR)]]**: Randomizes memory addresses so attackers cannot target fixed memory locations for buffer overflow attacks
- **Real-World Example**: **CVE-2023-29336** — Win32k Elevation of Privilege Vulnerability (May 2023)
  - Affects: Windows Server 2008, 2008 R2, 2012, 2012 R2, 2016; Windows 10
  - Win32k kernel driver flaw allowing attacker to gain **SYSTEM privileges** (highest access level)

### Cross-Site Requests (Legitimate)
- Modern web applications load resources from multiple sources (text, images, videos, stylesheets)
- Example: Visiting ProfessorMesser.com loads content from YouTube and Instagram
- These are **normal, expected, and authenticated requests** in standard web architecture
- The browser automatically includes stored credentials ([[Cookies]], [[Session Tokens]]) when making requests to trusted domains

### Cross-Site Request Forgery (CSRF/XSRF, "Sea Surf")
- **Definition**: One-click attack exploiting the trust relationship between a user's browser and a web application
- **Attack Mechanism**:
  1. Attacker creates a malicious request (e.g., "transfer $1,000 from your bank account")
  2. Attacker tricks user into clicking a link via [[Phishing]], fake website, or malicious email
  3. User's browser automatically includes stored credentials when making the request
  4. Web server trusts the request (it came from the user's authenticated browser)
  5. Action executes without user knowledge or consent (attacker posts to victim's Facebook, transfers funds)
- **Root Cause**: Web application design flaw—application trusts all requests from authenticated users without additional validation
- **Mitigation**: Implement **anti-forgery tokens** (cryptographic tokens unique to each session/request) that must be submitted with state-changing requests

### Directory Traversal (Path Traversal)
- **Definition**: Reading or accessing files outside the intended web server directory (e.g., accessing Windows system files from a web request)
- **Attack Example**: Using `../` sequences to traverse parent directories and access `/etc/passwd` or sensitive configuration files
- **Exploitation Levels**:
  - **Web Server Vulnerability**: Web server software doesn't properly restrict file access
  - **Web Application Code Vulnerability**: Poorly written application code fails to validate user input and sanitize file paths
- **Impact**: Attackers can read configuration files, source code, credentials, or other sensitive data not intended for public access

## How It Works (Feynman Analogy)

### Privilege Escalation
Imagine a parking garage with two levels: visitor parking and staff-only parking. A visitor exploits a broken barrier arm (vulnerability) to drive into the staff level, gaining access to reserved spots, shortcuts, and the manager's office. Once there, they can do anything: steal cars, vandalize equipment, or even lock out the real staff. **Patching the barrier** (applying security updates) closes the exploit and prevents anyone without legitimate staff credentials from reaching that level.

**Technical Reality**: An unprivileged user discovers a flaw in the OS kernel or application, crafts a malicious input, and escalates to administrator/SYSTEM privileges, gaining complete control over the system.

---

### CSRF: The Impersonation Attack
You're logged into your bank website. A friend sends you a cute cat picture link. You click it. Unbeknownst to you, that "cat picture" website contains hidden HTML that says "transfer $500 to attacker@evil.com." Your browser, still logged into the bank, automatically sends that request with your stored credentials. The bank thinks it's you (you're logged in!) and executes the transfer.

**Technical Reality**: The attacker exploits the browser's automatic credential submission (cookies/tokens) by crafting a request that appears legitimate to the server but was never authorized by the user. The server cannot distinguish between a genuine user action and a forged request without additional validation tokens.

---

### Directory Traversal: Escaping the Sandbox
A web server is designed to serve only files in `/var/www/html/` (public website files). An attacker submits a request like `GET /files/../../../../../../etc/passwd`, using relative path sequences to "climb out" of the intended directory and access system-level files the web server shouldn't expose.

**Technical Reality**: The application fails to sanitize or validate user-supplied file paths, allowing path traversal sequences to access unauthorized files on the filesystem.

## Exam Tips

- **Privilege Escalation vs. Horizontal Escalation**: The exam distinguishes between vertical (user→admin) and horizontal (user A→user B at same level). Know that horizontal escalation is harder to detect but equally dangerous for lateral movement.

- **CSRF Requires User Action**: CSRF is not a password-stealing attack. The victim must click the malicious link or visit the attacker's site. The browser does the work automatically—expect questions about why the application itself should validate requests, not just rely on browser security.

- **Token-Based MITIGATION**: When the exam asks "how to prevent CSRF," the answer is almost always **anti-forgery tokens** (cryptographic tokens in forms that the server validates). Answers mentioning SOP (Same-Origin Policy) or CORS are related but less direct.

- **DEP and ASLR**: These are memory protection mechanisms that prevent buffer overflow → privilege escalation chains. They don't patch the vulnerability; they make exploitation harder. Understand the distinction.

- **CVE Awareness**: The exam may reference specific CVEs like CVE-2023-29336. You don't need to memorize CVE numbers, but understand what type of system (Win32k kernel) and what impact (SYSTEM privileges) are at risk. This tests your ability to assess severity.

- **Directory Traversal ≠ SQL Injection**: Both are input validation flaws, but directory traversal is file-system access; [[SQL Injection]] is database access. Know the difference.

## Common Mistakes

- **Confusing CSRF with XSS**: [[Cross-Site Scripting (XSS)]] injects malicious script into the page itself; CSRF tricks the user's browser into making unwanted requests. XSS executes in the victim's browser; CSRF uses the victim's existing session. Both are web app attacks but fundamentally different.

- **Thinking CSRF Requires No User Interaction**: Candidates often think CSRF is fully automated. In reality, the attacker must trick the user into clicking a link or visiting a malicious site. However, that "trick" can be as simple as an email with a hidden image request (`<img src="http://bank.com/transfer?amount=500&to=attacker">`), which loads automatically without explicit user action.

- **Underestimating Privilege Escalation Priority**: Candidates sometimes treat privilege escalation patches as routine updates. The exam emphasizes these are **critical, high-priority patches** because they grant unrestricted access. If asked "which vulnerability requires immediate patching," privilege escalation is almost always the answer.

- **Forgetting about Horizontal Privilege Escalation**: Many candidates focus only on user→admin escalation and miss horizontal scenarios (user A accessing user B's files). In a multi-user system or [[Active Directory]] environment, this is a major lateral movement concern.

## Real-World Application

**For Morpheus's COCYTUS Homelab**: Privilege escalation vulnerabilities are critical in lab environments where you run [[Active Directory]], domain controllers, and multiple user accounts. A privilege escalation flaw in a [[Windows Server]] instance could allow a compromised low-privilege account to become Domain Admin, compromising the entire lab. [[Wazuh]] should monitor for suspicious privilege escalation attempts (e.g., unusual process creation with elevated tokens). CSRF is less relevant in internal lab testing but critical when publishing any internal web applications—use anti-forgery tokens in any custom tooling. Directory traversal is a frequent finding in lab web apps; always sanitize file path inputs and use allowlists of permitted directories.
