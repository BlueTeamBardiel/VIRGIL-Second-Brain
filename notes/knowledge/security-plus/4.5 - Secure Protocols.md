# 4.5 - Secure Protocols

## Summary

Secure protocols are the foundation of protecting data in transit across networks by implementing built-in encryption and authentication mechanisms. This section covers the critical difference between encrypted and unencrypted protocols, how to identify them, and why protocol selection is a key operational security decision. For the exam, you must understand which protocols are inherently insecure, recognize common port assignments, and know that port numbers alone do **not** guarantee security.

---

## Key Concepts

- **Unencrypted Protocols**: Network communication sent "in-the-clear" without encryption; includes [[Telnet]], [[FTP]], [[SMTP]], and [[IMAP]]
  - Data is readable by anyone with network access (sniffing/[[Wireshark]])
  - Credentials transmitted as plaintext

- **HTTP vs. HTTPS**
  - **HTTP (Port 80)**: Hypertext Transfer Protocol; unencrypted web traffic
  - **HTTPS (Port 443)**: HTTP Secure; uses [[TLS]]/[[SSL]] for encryption and authentication
  - Port number does **not** guarantee security—verify features are actually enabled

- **Secure Protocol Selection**
  - Choose protocols with built-in [[encryption]]
  - Assess availability—sometimes a secure alternative may not exist (business decision/deal-breaker)
  - Implement alternatives (e.g., SFTP instead of FTP, SMTP over [[TLS]] instead of plaintext SMTP)

- **Port Assignment Strategy**
  - Same application may run on both secure and insecure ports simultaneously
  - Example: HTTP on 80 AND HTTPS on 443; SMTP on 25 AND SMTPS on 587/465
  - Administrators must enforce use of the secure port and monitor for insecure traffic

- **Packet Capture Verification**
  - Use tools like [[Wireshark]] to verify encryption in practice
  - Confirm security features are actually enabled and functioning
  - Critical for compliance audits and incident investigation

- **Common Insecure Protocols**
  - [[Telnet]]: Remote access in cleartext (replaced by [[SSH]])
  - [[FTP]]: File transfer without encryption (replaced by SFTP/[[FTPS]])
  - [[SMTP]]: Mail submission without encryption (use SMTP-TLS or SMTPS)
  - [[IMAP]]: Mail retrieval without encryption (use IMAPS on port 993)

---

## How It Works (Feynman Analogy)

Imagine you're sending postcards through the mail versus sealed letters:

**Unencrypted protocols = Postcards**: Anyone handling the mail (postman, postal workers, neighbors) can read your message. Similarly, with protocols like [[FTP]] or [[Telnet]], anyone with access to the network wire (network tap, [[ARP spoofing]], WiFi eavesdropping) can capture and read your data—including passwords.

**Encrypted protocols = Sealed letters**: Only the intended recipient has the key to open and read the message. [[HTTPS]], [[SFTP]], and [[SSH]] use [[encryption]] to scramble data so that even if someone intercepts the network traffic, it appears as gibberish without the decryption key.

**The critical exam point**: The envelope color (port number) doesn't tell you if the letter is sealed. You must **verify** the actual security mechanism is in place through configuration checks and packet analysis.

---

## Exam Tips

- **Port numbers are a red herring**: The exam may present a scenario with "traffic on port 80"—don't assume it's insecure just by the port. Confirm the protocol. Similarly, port 443 doesn't guarantee HTTPS is configured correctly.

- **Know the insecure protocol list**: Telnet, FTP, SMTP (plaintext), IMAP (plaintext) appear frequently. Be ready to identify why they're problematic and what secure alternatives exist ([[SSH]], [[SFTP]], SMTPS/SMTP-TLS, IMAPS).

- **Distinguish between "available" and "enabled"**: A secure protocol may exist as an option but not be enforced. The exam tests whether you understand that configuration and verification are required.

- **Packet capture as evidence**: When the question asks "how would you verify encryption is in place?" the answer is packet capture analysis. If you see plaintext credentials in [[Wireshark]], the protocol is not providing adequate security.

- **Business vs. technical decisions**: Some scenarios involve legacy systems without secure alternatives. Recognize this as a **risk acceptance** decision requiring compensating controls (network segmentation, [[VPN]], [[IDS]]/[[IPS]]), not a technical fix.

---

## Common Mistakes

- **Assuming port number = security**: Candidates often think "port 443 = HTTPS = secure" without understanding that a misconfigured system could run unencrypted HTTP on 443 or insecure protocols on alternate ports. Always verify the **protocol**, not just the port.

- **Forgetting about SMTP variants**: Many candidates know FTP → SFTP and HTTP → HTTPS, but miss SMTP/IMAP plaintext variants. Remember: SMTPS (port 465), SMTP-TLS (587), and IMAPS (993) are the secure versions. On the exam, watch for questions mixing these up.

- **Overlooking packet verification in scenarios**: When a question describes "we're using HTTPS," candidates assume it's secure without recognizing that the exam may ask you to validate this claim. Know that packet captures and configuration reviews are the way to confirm.

## Real-World Application

In Morpheus's homelab (COCYTUS fleet), secure protocols are enforced across all systems: [[SSH]] for remote access instead of [[Telnet]], HTTPS for web management interfaces, and SMTP-TLS for mail submission to avoid credential exposure in lab traffic analysis with [[Wazuh]]. When integrating legacy equipment (e.g., older network appliances) that only support plaintext protocols, Morpheus isolates them on a management [[VLAN]] and routes traffic through a [[VPN]] gateway or [[IDS]]/[[IPS]] to enforce encryption and detect anomalies. This mirrors real-world [[SOC]] operations where protocol compliance is part of baseline security posture.
