# 4.5 - Firewalls (continued)

Firewalls are the cornerstone of network security, controlling traffic flow through rules and access control lists (ACLs) while modern implementations integrate intrusion prevention capabilities. This section covers firewall rule architecture, the critical concept of implicit deny, and how [[IPS]] (Intrusion Prevention System) detection methods differ in identifying and responding to threats. Understanding both rule construction and IPS detection mechanisms is essential for domain 4.0 and directly applicable to configuring [[NGFW]] devices in production environments.

---

## Key Concepts

### Firewall Rules Architecture
- **Logical Path Evaluation**: Rules are processed sequentially, typically from top to bottom, with the first matching rule determining the action
- **Rule Specificity Hierarchy**: Specific rules are placed at the top of the rule set to take precedence over broader, more general rules below them
- **Implicit Deny**: A critical security principle where an implicit deny rule exists at the bottom of every firewall's rule set by default, blocking all traffic not explicitly allowed
  - This occurs automatically, even if you did not explicitly configure it
  - Represents the "deny by default" security posture

### Access Control Lists (ACLs)
- **Definition**: Groupings of allow/deny conditions organized by traffic categories and attributes
- **Granular Control Parameters**:
  - Source IP address
  - Destination IP address
  - Port number (source and destination)
  - Time of day (temporal rules)
  - Application/protocol layer (L7 filtering)
  - Direction (ingress/egress)
- **ACL Actions**: Binary decisions to allow or disallow traffic matching the specified conditions

### Screened Subnet (DMZ - Demilitarized Zone)
- **Purpose**: An additional security perimeter between internal resources and the Internet
- **Traffic Segregation**:
  - Public access is permitted to publicly-facing resources (web servers, mail servers, DNS)
  - Private corporate data remains behind a secondary firewall, inaccessible from the screened subnet
- **Defense-in-Depth Implementation**: Two firewall boundaries create layered protection

### IPS (Intrusion Prevention System) Rules
- **Integration Model**: Typically embedded within [[NGFW]] (Next-Generation Firewall) platforms
- **Detection Methods**:
  - **Signature-Based Detection**: Matches traffic patterns against a database of known malicious signatures (perfect pattern matching)
  - **Anomaly-Based Detection**: Establishes a baseline of "normal" network behavior, then flags deviations and unusual traffic patterns
- **Scale and Complexity**: Thousands of rules (or more) can be active simultaneously
- **Response Actions**: Operator determines outcome when unwanted traffic is detected:
  - Block the traffic
  - Allow the traffic through
  - Send an alert to the [[SOC]] or SIEM
  - Quarantine/drop packets
  - Log for forensic analysis
- **Rule Customization**: Rules can be applied at group level (departments, subnets) or as individual rules for granular control

### Tuning and Optimization
- **Alert Fatigue Challenge**: Balancing security effectiveness against false positives and alert "noise"
- **Iterative Process**: Finding the optimal rule configuration takes time and ongoing monitoring
- **False Positive Reduction**: Excessive alerts lead to alert fatigue and missed genuine threats

## How It Works (Feynman Analogy)

**Think of a firewall as a security checkpoint at an airport entrance:**

You arrive at the airport, and a security officer checks your credentials against a list of rules. The officer works through the list from top to bottom: "Do you have a ticket to a departing flight? Yes, proceed. Are you carrying prohibited items? No, proceed. Is your ID valid? Yes, go ahead." If you reach the bottom without matching any "allow" rule, the implicit security principle kicks in: the officer denies entry by default.

A screened subnet is like having *two* security checkpoints—the first one checks you into a public waiting area where you can buy tickets, and a second checkpoint guards the entrance to the secure gates where only ticketed passengers are allowed.

An [[IPS]] is like a more sophisticated security system that not only checks your credentials but also watches your *behavior*. The signature-based detection is like having photos of known criminals—if you match one, you're stopped. Anomaly-based detection is like a trained officer who notices when someone is acting suspiciously (pacing nervously, looking at surveillance cameras repeatedly, avoiding eye contact) even if they're not on any "known bad guys" list.

**The technical reality:** Firewalls use ordered packet-matching logic against ACL rules; [[IPS]] engines run packet inspection engines (inline or out-of-band) that compare traffic against threat signatures or behavioral models, making real-time block/alert decisions before packets reach the application layer.

## Exam Tips

- **Implicit Deny is Default**: The exam assumes you know that firewalls deny by default at the bottom of the rule list. Questions often test whether you understand that explicit allow rules must be written—nothing is permitted automatically.
  
- **Signature vs. Anomaly Distinction**: Expect a scenario question asking you to identify whether an [[IPS]] would catch a threat with signature-based or anomaly-based detection. Signature-based catches *known* threats; anomaly-based catches *unusual* behavior (including zero-days).

- **Rule Order Matters**: A common exam trap is presenting a rule set where a general rule appears before a specific rule, causing the specific rule to never be evaluated. Answer questions carefully about what rule is "used" or "applied first."

- **ACL Granularity**: The exam tests whether you understand that ACLs can filter on multiple dimensions (not just IP addresses). Know that modern firewalls can filter by application, time of day, and protocol in addition to network-layer attributes.

- **Screened Subnet Purpose**: Distinguish between a screened subnet and a [[VLAN]]. A screened subnet is a *physical or logical network segment with firewall boundaries*; a [[VLAN]] is just *broadcast domain isolation*. The exam may ask which technology provides better protection for public-facing servers.

- **Alert Tuning Trade-off**: Questions about [[IPS]] configuration often highlight the tension between *detection accuracy* and *false positive rate*. More aggressive rules catch more threats but generate more noise; conservative rules reduce noise but miss threats.

---

## Common Mistakes

- **Confusing Implicit Deny with Default Allow**: Candidates sometimes assume that a rule not explicitly blocking something is allowed. Wrong—the implicit deny rule at the end blocks everything by default. You must *explicitly allow* traffic.

- **Treating Signature and Anomaly Detection as Complementary Only**: While they can work together, the exam wants you to understand their fundamental difference: signatures catch known threats, anomalies catch *unknown/unusual* threats. A novel attack may bypass signatures but trigger anomaly alerts.

- **Assuming All Firewall Rules are Stateless**: While the exam covers stateless ACL rules, modern firewalls (especially [[NGFW]]) are *stateful*—they track connection states and remember bidirectional traffic flows. Don't confuse a basic ACL with stateful inspection capabilities.

## Real-World Application

In your homelab, firewall rules govern traffic between [[Tailscale]] VPN peers and internal resources, with a screened subnet protecting the [[Wazuh]] manager from direct Internet access. The [[Wazuh]] agent on each host forwards logs to the manager, and [[IPS]] rules (if using an [[NGFW]] appliance like pfSense with Suricata) would detect anomalous outbound connections or known malware signatures in network traffic—tuning the rule set to reduce false positives while catching genuine threats is an ongoing operational challenge in any SOC environment.

## Study Notes

### Firewall Rule Processing Order (Critical for Exam)
1. Rules are evaluated top-to-bottom
2. First matching rule is applied; remaining rules are not evaluated
3. Specific rules should always appear before general rules
4. The implicit deny at the bottom catches everything not explicitly allowed

### IPS Tuning Considerations
- **Too Strict**: Many false positives → alert fatigue → operators ignore alerts → real threats missed
- **Too Lenient**: Many threats pass through → network compromise risk
- **Sweet Spot**: Balanced configuration with regular review of rule effectiveness metrics

### Screened Subnet Architecture
```
Internet → Firewall 1 → [Screened Subnet (DMZ)] → Firewall 2 → Internal Network
                             (web, mail, DNS)
```

### Comparison Table: Firewall Technologies

| Feature | Stateless Firewall | Stateful Firewall | NGFW |
|---------|-------------------|-------------------|------|
| Rule Type | ACL-based | Connection-aware | ACL + IPS + App-aware |
| Layer | L3/L4 | L3/L4 + state | L3-L7 |
| Threat Detection | No | No | Yes (IPS) |
| Performance Impact | Minimal | Moderate | Moderate-High |
| Modern Use | Legacy | Common | Enterprise Standard |
