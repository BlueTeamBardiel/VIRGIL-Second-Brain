# Windows Event Log Monitoring

## What it is
Think of Windows Event Logs as the ship's black box recorder — silently capturing every takeoff, turbulence, and crash, whether anyone's watching or not. Windows Event Log Monitoring is the systematic collection, analysis, and alerting on log entries generated by the Windows operating system, capturing authentication events, process execution, service changes, and system errors. These logs are stored in `.evtx` files and organized into channels such as Security, System, and Application.

## Why it matters
During the 2020 SolarWinds attack, adversaries used legitimate credentials to move laterally — but Event ID 4624 (successful logon) and 4672 (special privileges assigned) left forensic breadcrumbs in Security logs. Organizations with centralized SIEM monitoring detected anomalous after-hours privileged logins that manual log review would have missed entirely, demonstrating that log aggregation is the difference between breach detection in hours versus months.

## Key facts
- **Event ID 4625** = Failed logon attempt; a burst of these signals a brute-force attack
- **Event ID 4688** = A new process was created; critical for detecting malicious execution like `powershell.exe` spawning from `winword.exe`
- **Event ID 4698/4702** = Scheduled task created/modified; a common persistence mechanism used by attackers
- **Event ID 1102** = Security audit log was cleared — a high-fidelity indicator of an attacker covering tracks
- Windows logs are stored locally and can be forwarded using **Windows Event Forwarding (WEF)** to a central collector without requiring third-party agents
- **Sysmon** (System Monitor) dramatically enhances default logging by capturing network connections, file hashes, and registry changes that native logs miss
