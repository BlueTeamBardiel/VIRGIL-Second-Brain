# 3.2 - Secure Infrastructures

This section covers the foundational principles of designing and deploying secure network architectures, focusing on device placement, security zones, attack surface reduction, and connectivity security. Understanding secure infrastructure design is critical for the Security+ exam because it forms the basis for implementing [[Zero Trust]], defense-in-depth strategies, and layered security controls across enterprise and homelab environments. The exam tests your ability to recognize appropriate placement of security technologies and understand how zoning strategies reduce risk and simplify policy management.

---

## Key Concepts

- **Device Placement**: Strategic positioning of security appliances and services to maximize protection and control traffic flows between network segments.

- **[[Firewall|Firewalls]]**: Core security devices that separate trusted from untrusted network zones and enforce security policies by inspecting and filtering traffic.

- **Security Zones**: Logical groupings of network resources based on trust levels and functionality, rather than IP address ranges alone—simplifies policy management and enhances flexibility.

- **Zone Types**:
  - **Trusted Zone**: Internal resources controlled and managed by the organization.
  - **Untrusted Zone**: External, potentially hostile networks (e.g., the Internet).
  - **Screened Zone** (DMZ): Intermediate zone containing public-facing services with restricted access to internal networks.
  - **Internal Zone**: Core company/organizational resources.
  - **Server Zone**: Dedicated segment for server resources.
  - **Database Zone**: Restricted segment protecting sensitive data repositories.

- **Attack Surface**: The total number of possible entry points and vulnerabilities an attacker can exploit in a system or network.
  - Includes open ports, exposed applications, authentication mechanisms, and human error vectors.
  - Directly correlates to risk: larger surface = greater exposure.

- **Attack Surface Reduction Strategies**:
  - Code auditing and secure development practices.
  - Port and service hardening (disable unnecessary services).
  - Real-time network traffic monitoring.
  - Principle of least privilege and minimal exposure.

- **Connectivity Security**:
  - **Physical Security**: Securing network cabling and access points to prevent tampering and eavesdropping.
  - **Application-Level Encryption**: [[TLS]], [[SSL]], application-specific protocols—protects data in transit at the application layer.
  - **Network-Level Encryption**: [[IPsec]] tunnels, [[VPN|VPNs]]—encrypts all traffic between endpoints regardless of application.

- **Supplementary Security Technologies**:
  - **[[Honeypots]]**: Decoy systems designed to attract and study attacker behavior.
  - **Jump Servers** (Bastion Hosts): Hardened intermediaries for administrative access to sensitive systems.
  - **Load Balancers**: Distribute traffic to improve availability and can implement security checks.
  - **Network Sensors**: [[IDS|IDS]]/[[IPS|IPS]] devices monitoring for intrusions and anomalies.

- **Policy Simplification via Zones**:
  - Policies are defined between zones (e.g., Trusted ↔ Untrusted) rather than individual IP addresses.
  - Reduces complexity and improves maintainability.
  - Common policy flows: Untrusted → Screened (allowed), Untrusted → Trusted (blocked by default).

## How It Works (Feynman Analogy)

**The Castle Model:**

Imagine a medieval castle protecting a kingdom's treasury. The castle doesn't rely on a single wall; instead, it uses multiple layers:

1. **The Outer Wall (Untrusted Zone)**: Guards at the main gate inspect everyone entering.
2. **The Moat & Guard Towers (Firewalls & Sensors)**: Monitor and control traffic between zones.
3. **The Inner Courtyard (Screened Zone/DMZ)**: Merchants and traders operate here—separated from the treasury.
4. **The Keep (Trusted Internal Zone)**: Only authorized nobility can access the vault (database zone).
5. **Secret Tunnels (VPN/IPsec)**: Secure encrypted paths for trusted messengers between distant locations.

**If one guard fails or one gate is breached**, the other layers still protect the treasury. The castle doesn't rely on a single point of defense.

**In technical terms**: Secure infrastructure applies this multi-layer principle by:
- Using **firewalls** as the outer walls and internal gates.
- Segmenting the network into **zones** based on trust and function (not just IP ranges).
- Reducing the **attack surface** by disabling unnecessary services, similar to sealing unused entrances.
- Encrypting communications (**TLS**, **IPsec**, **VPN**) like encoding secret messages.
- Deploying **sensors and honeypots** like having scouts watching for suspicious behavior.

The key insight: **No single technology secures infrastructure. It's the combination of layered technologies, intelligent placement, and zone-based policies that creates resilience.**

## Exam Tips

- **Zone-Based vs. IP-Based Policies**: The exam often presents scenarios asking you to implement policies. Remember: zone-based security is more **flexible and scalable** than static IP-based rules. When you see questions about simplifying security across multiple subnets, think "zones."

- **Attack Surface Questions**: The exam will ask, "What reduces attack surface?" The correct answers almost always involve:
  - Disabling unnecessary ports/services.
  - Code audits and patch management.
  - Real-time monitoring and segmentation.
  - *Not*: adding more appliances (that increases complexity, not security).

- **Device Placement Scenarios**: You'll see questions like, "Where should you place a honeypot?" or "Why use a jump server?" Remember:
  - **Honeypots** go in the DMZ/screened zone to attract attackers away from real assets.
  - **Jump servers** are hardened intermediaries in the screened zone for administrative access.
  - **Load balancers** often reside at the network edge to distribute and filter traffic.
  - **Sensors (IDS/IPS)** are placed at zone boundaries to monitor inter-zone traffic.

- **Encryption Layers**: Distinguish between:
  - **Application-level encryption** (TLS in HTTPS): Protects specific applications.
  - **Network-level encryption** (VPN, IPsec): Protects all traffic in a tunnel.
  - The exam tests whether you understand *when to use each*: Use IPsec for site-to-site, TLS for web services, VPN for remote access.

- **Common Answer Pattern**: If a question asks "What is the BEST way to..." regarding infrastructure security, the answer usually involves **layering** and **segmentation** rather than a single technology. Look for answers mentioning zones, multiple controls, or defense-in-depth.

## Common Mistakes

- **Confusing Attack Surface with Network Surface**: Candidates often think attack surface = number of hosts or IP addresses. Actually, attack surface = number of ways an attacker can penetrate and compromise the system. A single web server with unpatched code has a larger attack surface than 10 hardened, patched servers.

- **Assuming Zones = Subnets**: While zones *may* map to subnets, they are not the same. Zones are **logical security groupings** that can span multiple subnets or VLANs. A screened zone (DMZ) might include servers from different physical subnets, all managed under the same security policy. The exam tests this distinction in scenario questions.

- **Forgetting Physical Security in Connectivity**: Candidates focus on encryption and overlook physical security ("secure network cabling"). The exam includes questions like, "What protects against wiretapping on a local network?" The answer includes both **physical security** (locking down cable runs, restricting access to patch panels) and **encryption** (TLS, IPsec). Miss physical security, miss the question.

## Real-World Application

In your homelab environment, this translates to:

- **Device Placement**: [[Firewall|Firewalls]] (e.g., pfSense) separate the external network from internal resources, with [[Wazuh|Wazuh agents]] deployed on key systems to monitor zone boundaries.
- **Security Zones**: [[Active Directory|AD domain controllers]], [[DNS|DNS servers]], and databases reside in protected internal zones; external services (Tailscale exit nodes, web frontends) are confined to screened zones.
- **Attack Surface Reduction**: Disabling unused services on [[YOUR-LAB]] nodes, regular vulnerability scans with [[Nmap]], and using [[Zero Trust]] principles via [[Tailscale]] ensure minimal exposure.
- **Connectivity**: [[Tailscale|Tailscale VPN]] provides network-level encryption for remote access; application-level [[TLS|TLS]] secures web services; [[IPsec]] tunnels can secure site-to-site connections to other labs.
