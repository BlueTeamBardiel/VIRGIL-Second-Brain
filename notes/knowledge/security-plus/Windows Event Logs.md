# Windows Event Logs

## What it is
Think of Windows Event Logs as the ship's manifest on a cargo vessel — every passenger, package, and port stop gets recorded, whether the captain intended it or not. Precisely defined, they are structured, timestamped records generated by the Windows OS and applications that document system events, security audits, and application activity. They are stored in `.evtx` format and managed through the Windows Event Log service (`wevtsvc`).

## Why it matters
During incident response to a ransomware attack, analysts pivoting through Event Logs can identify the exact moment an attacker used Event ID 4624 (successful logon) with Logon Type 3 (network logon) to move laterally from a compromised workstation to a domain controller. Without these logs — or if an attacker cleared them using `wevtutil cl Security` — reconstructing the attack chain becomes nearly impossible, highlighting why log forwarding to a SIEM is critical.

## Key facts
- **Security Log (Event ID 4624/4625):** Successful and failed logon attempts — the most forensically valuable log for detecting brute force and pass-the-hash attacks
- **System Log:** Records OS-level events like service startups/crashes; Event ID 7045 flags a newly installed service, a common persistence mechanism
- **Application Log:** Captures software-generated events; not security-controlled and can be written to by any application
- **Event ID 4688:** Logs new process creation — requires audit policy enabled; critical for detecting LOLBin abuse (e.g., `powershell.exe`, `wscript.exe`)
- **Log clearing (Event ID 1102):** When the Security Log is cleared, this event fires — itself an IOC that attackers tried to cover tracks
- Logs are stored at `C:\Windows\System32\winevt\Logs\` and can be forwarded via Windows Event Forwarding (WEF) to a central collector
