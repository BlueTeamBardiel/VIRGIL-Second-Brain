# dhcp.log

## What it is
Like a hotel front desk ledger recording every guest check-in and check-out with timestamps and room numbers, `dhcp.log` records every IP address assignment event on a network — who got which IP, when, and for how long. It is a log file generated by DHCP servers (and captured in tools like Zeek/Bro) that documents DISCOVER, OFFER, REQUEST, and ACK exchanges between clients and the server.

## Why it matters
During a ransomware incident response, analysts use `dhcp.log` to reconstruct which hostname held a suspicious IP address at the exact moment malicious traffic was observed — without it, an attacker's lateral movement can vanish into the noise of a rotating IP pool. It's also the primary artifact for detecting **rogue DHCP servers**: a flood of OFFER messages from an unexpected MAC address signals an attacker attempting to redirect victim traffic through their gateway (DHCP starvation + spoofing attack).

## Key facts
- Contains fields: timestamp, client MAC address, assigned IP, lease duration, hostname, and transaction ID
- Critical for **timeline correlation** — maps IP addresses to specific devices at specific times, bridging firewall/IDS logs to endpoint identity
- **DHCP starvation attacks** (exhausting the IP pool via spoofed MACs) are detectable as abnormal spikes in DISCOVER messages with no corresponding ACKs
- In Zeek (formerly Bro), `dhcp.log` is auto-generated and is a core file in network security monitoring (NSM) workflows
- Lease duration gaps or missing RELEASE messages can indicate a device was forcibly disconnected or spoofed off the network
