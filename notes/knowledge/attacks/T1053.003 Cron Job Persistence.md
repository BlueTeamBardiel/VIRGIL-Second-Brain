# T1053.003 Cron Job Persistence

## What it is
Think of cron like a building's automated lighting system — you program it once, and it reliably switches on every night without anyone touching a switch. Cron is a Unix/Linux time-based job scheduler that attackers abuse by inserting malicious commands into crontab files, ensuring their code executes automatically at defined intervals even after reboots or session terminations.

## Why it matters
During the 2020 Rocke cryptomining campaigns, attackers gained initial access via exposed Redis and Jenkins servers, then planted cron jobs to re-download and execute mining payloads every few minutes — effectively making the malware self-healing even if defenders killed the running process. This is why simply killing a suspicious process is never enough; defenders must audit cron tables simultaneously.

## Key facts
- Crontab files live in `/etc/cron*` directories (system-wide) and `/var/spool/cron/crontabs/` (per-user), plus `/etc/cron.d/` — attackers target all three locations
- The five-field time syntax `* * * * *` (minute, hour, day, month, weekday) lets attackers schedule execution with precision or use `@reboot` for boot persistence specifically
- Malicious entries often use `curl | bash` or `wget -O- | sh` patterns to pull remote payloads, making the crontab file itself appear small and benign
- Privilege level matters: root crontabs in `/etc/crontab` execute as root, while user crontabs run under that user's permissions — attackers prefer root-level placement
- Detection focuses on monitoring crontab file modifications via auditd rules, specifically watching for writes to cron directories by unexpected processes
